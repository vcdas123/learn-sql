## 1. Definition & Introduction

A **Columnstore Index** is a type of database index that stores and organizes data **column by column** rather than row by row. It was introduced specifically to accelerate **analytical queries** (aggregations, filtering, grouping) on large datasets.

> 💡 **Simple Definition:** In a traditional (row store) index, each database page holds a group of complete rows — every column of a row is stored together. In a columnstore index, each page holds the values of just **one column** for a group of rows.

### Why It Exists

Traditional row-based indexes are optimized for **OLTP** (Online Transaction Processing) — fast inserts, lookups of full records by ID, frequent updates. But when analysts write queries like:

```sql
SELECT status, COUNT(*) FROM customers GROUP BY status;
```

...the database must read every row — including every column — even though only the `status` column is needed. On a table with 10 million rows and 50 columns, this wastes enormous I/O.

The columnstore index solves this by storing only the relevant columns' data together, dramatically reducing what the database needs to read from disk.

---

## 2. How Data is Stored — Row Store vs Column Store

### Row Store (Traditional)

Each data page stores complete rows. Every column for a given row is stored together.

**Sample Table: `customers`**

| customer_id | name    | status   |
|---|---|---|
| 1           | Alice   | active   |
| 2           | Bob     | inactive |
| 3           | Clara   | active   |
| 4           | David   | active   |
| 5           | Emma    | inactive |

**Row Store — Page Layout:**

```
Data Page 1:
┌────────────────────────────────────────────────┐
│  Row 1: [ 1 | Alice  | active   ]              │
│  Row 2: [ 2 | Bob    | inactive ]              │
│  Row 3: [ 3 | Clara  | active   ]              │
└────────────────────────────────────────────────┘

Data Page 2:
┌────────────────────────────────────────────────┐
│  Row 4: [ 4 | David  | active   ]              │
│  Row 5: [ 5 | Emma   | inactive ]              │
└────────────────────────────────────────────────┘
```

Every page contains all columns of every row. To read `status` alone, you must read every page and extract `status` from each row.

---

### Column Store

Data is split into columns. Each data page stores values of only **one column** for a batch of rows.

```
Column Page — customer_id:          Column Page — name:
┌──────────────┐                    ┌──────────────┐
│  1           │                    │  Alice        │
│  2           │                    │  Bob          │
│  3           │                    │  Clara        │
│  4           │                    │  David        │
│  5           │                    │  Emma         │
└──────────────┘                    └──────────────┘

Column Page — status (compressed):
┌──────────────┐   Dictionary Page:
│  1           │   ┌──────────────────┐
│  2           │   │  1 → "active"    │
│  1           │   │  2 → "inactive"  │
│  1           │   └──────────────────┘
│  2           │
└──────────────┘
```

To answer `COUNT(*) WHERE status = 'active'`, only the `status` column page is needed. The `customer_id` and `name` pages are **never touched**.

## 3. How SQL Builds a Columnstore Index — Step by Step

When you create a columnstore index, SQL goes through a 4-step process:

---

### Step 1 — Divide Rows into Row Groups

SQL first partitions all rows into chunks called **Row Groups**. In SQL Server, each row group holds approximately **1 million rows**.

```
Table: 2,000,000 rows
         │
         ├── Row Group 1 → rows 1 to 1,000,000
         └── Row Group 2 → rows 1,000,001 to 2,000,000
```

> Why rows first, if it's a column store? Row groups enable **parallel processing** — each row group can be independently processed and queried, enabling multi-threaded execution on large datasets.

---

### Step 2 — Segment the Columns

Within each row group, SQL separates each column into its own independent chunk called a **Column Segment**.

```
Row Group 1 (1,000,000 rows):
│
├── Segment: customer_id   → 1,000,000 integer values
├── Segment: name          → 1,000,000 string values
└── Segment: status        → 1,000,000 string values
```

Each segment contains the values of exactly one column, across the rows in that row group.

---

### Step 3 — Compress the Data (Most Important Step)

This is the step that makes columnstore fast and storage-efficient. SQL compresses each segment using a **Dictionary Encoding** strategy:

**Before compression — `status` segment:**

```
active, inactive, active, active, inactive, active, ...  (1,000,000 strings)
```

**Create a Dictionary:**

```
Dictionary:
  1  →  "active"
  2  →  "inactive"
```

**After compression — `status` data stream:**

```
1, 2, 1, 1, 2, 1, ...  (1,000,000 tiny integers instead of long strings)
```

**Why this is powerful:**
- `"active"` (6 bytes) becomes `1` (1 byte) — **~83% size reduction per value**
- For a column with only 2 distinct values repeated 1,000,000 times, the savings are massive
- Columns with lower cardinality (fewer distinct values) compress even more aggressively

Other compression techniques SQL uses internally: **Run-Length Encoding (RLE)**, **Bit-packing**, and **Delta encoding** for numeric ranges.

---

### Step 4 — Store in LOB Data Pages

After compression, SQL stores each column segment in a special page type called a **LOB (Large Object) Page** — not the standard 8KB data page used in row stores.

The LOB page is designed to handle variable-length, large column segment streams efficiently.

---

## 4. The LOB Data Page — Internal Storage Structure

The LOB data page used by columnstore has a distinct structure compared to a standard row store data page:

```
┌────────────────────────────────────────────────────────┐
│  Page Header                                           │
│  (Standard: page ID, type, metadata)                  │
├────────────────────────────────────────────────────────┤
│  Segment Header                                        │
│  ├─ Segment ID                                        │
│  ├─ Row Group ID                                      │
│  ├─ Column ID                                         │
│  └─ Pointer → Dictionary Page ID                      │
├────────────────────────────────────────────────────────┤
│  Data Stream                                           │
│  (Compressed sequence of dictionary IDs)              │
│  [ 1, 2, 1, 1, 2, 1, 1, 1, 2, ... ]                 │
└────────────────────────────────────────────────────────┘

Dictionary Page (separate page type):
┌────────────────────────────────────────────────────────┐
│  Page Header                                           │
├────────────────────────────────────────────────────────┤
│  Mapping:                                              │
│  1  →  "active"                                       │
│  2  →  "inactive"                                     │
└────────────────────────────────────────────────────────┘
```

**Key difference from row store data pages:**

| Feature | Row Store Data Page | Columnstore LOB Page |
|---|---|---|
| Contains | Full rows (all columns) | One column segment only |
| Size | Fixed 8KB | Variable (LOB) |
| Index structure | B-Tree (root → intermediate → leaves) | No B-Tree — direct segment access |
| Compression | None (by default) | Always compressed |
| Dictionary page | None | Yes — separate mapping page |

## 5. Clustered vs Non-Clustered Columnstore Index

### Clustered Columnstore Index

```sql
CREATE CLUSTERED COLUMNSTORE INDEX idx_customers_cs
ON customers;
```

- **Completely replaces** the row-based table structure.
- The table is **fully re-organized** into the columnstore format — no row store remains.
- All columns are included — you cannot select a subset.
- No B-Tree is built — the columnstore structure IS the table.
- **Maximum 1 per table** (same rule as clustered row store).
- You cannot specify column names in the syntax — including all columns is mandatory and implicit.

```
Before: Heap / Row Store Table
         ↓
CREATE CLUSTERED COLUMNSTORE INDEX
         ↓
After: Full Columnstore Table (row store is gone)
```

---

### Non-Clustered Columnstore Index

```sql
CREATE NONCLUSTERED COLUMNSTORE INDEX idx_customers_status_cs
ON customers (status);
```

- **Co-exists alongside** the original row store table — the original table is untouched.
- You can choose **which columns to include** — useful for indexing only the columns used in analytics.
- The original table remains in row store format for OLTP queries.
- Analytical queries that match the columnstore columns use the columnstore automatically.
- In **SQL Server** — maximum **1 columnstore index per table** (clustered or non-clustered, not both).
- In **Azure SQL / newer databases** — multiple non-clustered columnstore indexes may be allowed.

```
Original Table (row store)
        +
Non-Clustered Columnstore (companion structure for selected columns)
```

---

### Choosing Between Them

| Scenario | Use |
|---|---|
| The table is **purely for analytics** (data warehouse, reporting) | ✅ Clustered Columnstore |
| The table serves **both OLTP and analytics** | ✅ Non-Clustered Columnstore (OLTP uses row store, analytics uses columnstore) |
| You want to include **all columns** in analytics index | ✅ Clustered Columnstore |
| You want to index **only specific columns** | ✅ Non-Clustered Columnstore |

## 6. Why Columnstore is Faster for Analytics — Query Execution Flow

### The Query

```sql
SELECT COUNT(*) FROM customers WHERE status = 'active';
```

---

### Execution with Row Store Index

```
Step 1 — Read all data pages:
  Open Page 1 → Read rows: [1, Alice, active], [2, Bob, inactive], [3, Clara, active]
  Open Page 2 → Read rows: [4, David, active], [5, Emma, inactive]

Step 2 — Filter:
  Keep only rows where status = 'active'
  → Rows 1, 3, 4 pass the filter

Step 3 — Aggregate:
  COUNT(*) = 3
```

**What was unnecessary:**
- `customer_id` was read from every row — never needed
- `name` was read from every row — never needed
- All columns of every row were loaded into memory before filtering

**I/O cost:** All data pages read × all columns per row = maximum possible I/O

---

### Execution with Columnstore Index

```
Step 1 — Identify required columns:
  Query needs only: status
  customer_id → SKIP (column page never opened)
  name        → SKIP (column page never opened)

Step 2 — Read ONLY the status column segment:
  Open status LOB page → Data stream: [1, 2, 1, 1, 2]
  Resolve dictionary:   1 = "active", 2 = "inactive"

Step 3 — Filter on compressed stream:
  Keep only values = 1 (active)
  → [1, 1, 1] — 3 matches found

Step 4 — Aggregate:
  COUNT(*) = 3
```

**What was saved:**
- `customer_id` column page → **never opened**
- `name` column page → **never opened**
- Only 1 of 3 column pages read

**I/O cost:** 1 column page vs 3 column pages in row store — **67% less I/O for this query**. On a 50-column table, savings would be ~98%.

---

### Performance Comparison — The Numbers

For a 2,000,000-row table with 50 columns, running `COUNT(*) WHERE status = 'active'`:

| Step | Row Store | Columnstore |
|---|---|---|
| Pages read | All data pages (all 50 columns) | Only status segment pages |
| Data loaded into memory | ~50 columns × 2M rows | ~1 column × 2M rows (compressed) |
| Filter applied on | Full rows | Compressed integer stream |
| Relative I/O | 🔴 100% | 🟢 ~2% (50x less) |

> 💡 The larger the table and the more columns it has, the **more dramatic the columnstore advantage**.

## 7. Row Store vs Column Store — Full Comparison

| Feature | Row Store Index | Columnstore Index |
|---|---|---|
| Data organization | Row by row | Column by column |
| Page type | Standard data page (8KB) | LOB data page (variable) |
| B-Tree structure | ✅ Yes | ❌ No |
| Data compression | ❌ No (by default) | ✅ Yes (dictionary, RLE, etc.) |
| Storage efficiency | ⚠️ More storage | ✅ Significantly less storage |
| Read performance (OLTP) | ✅ Fast (full record lookup) | ⚠️ Slower for single-row reads |
| Read performance (analytics) | ⚠️ Slow (reads all columns) | ✅ Very fast (reads only needed cols) |
| Write performance (INSERT/UPDATE) | ✅ Faster | ⚠️ Slower (multi-step compression) |
| I/O efficiency | ⚠️ High I/O (unnecessary columns) | ✅ Low I/O (targeted column reads) |
| Best system type | **OLTP** (banking, e-commerce) | **OLAP** (data warehouse, BI, analytics) |
| Max per table (SQL Server) | 1 clustered + many non-clustered | 1 total (clustered or non-clustered) |

### OLTP vs OLAP — System Types

| | OLTP | OLAP |
|---|---|---|
| Full name | Online Transaction Processing | Online Analytical Processing |
| Examples | Banking, e-commerce, ERP | Data warehouse, Power BI, Tableau |
| Query pattern | `SELECT * WHERE id = 5` | `SELECT status, SUM(amount) GROUP BY status` |
| Data volume | Moderate | Massive (millions to billions of rows) |
| Index choice | Row Store | **Columnstore** |

## 8. SQL Syntax — Creating Columnstore Indexes

### Create Clustered Columnstore Index

```sql
-- ✅ Correct: no column list — all columns are included
CREATE CLUSTERED COLUMNSTORE INDEX idx_customers_cs
ON customers;

-- ❌ Wrong: specifying columns is not allowed for clustered columnstore
CREATE CLUSTERED COLUMNSTORE INDEX idx_customers_cs
ON customers (customer_id);
-- Error: Key list is not allowed for clustered columnstore index
```

### Create Non-Clustered Columnstore Index

```sql
-- Non-clustered: you CAN specify which columns to include
CREATE NONCLUSTERED COLUMNSTORE INDEX idx_customers_status_cs
ON customers (status, customer_id);
```

### Drop a Columnstore Index

```sql
-- SQL Server
DROP INDEX idx_customers_cs ON customers;
```

### Replace Row Store with Columnstore

```sql
-- Step 1: Drop existing clustered row store index
DROP INDEX idx_customers_pk ON customers;

-- Step 2: Create clustered columnstore
CREATE CLUSTERED COLUMNSTORE INDEX idx_customers_cs
ON customers;
```

### Check Existing Indexes

```sql
-- SQL Server
SELECT
    i.name        AS index_name,
    i.type_desc   AS index_type,
    c.name        AS column_name
FROM sys.indexes i
LEFT JOIN sys.index_columns ic ON i.object_id = ic.object_id AND i.index_id = ic.index_id
LEFT JOIN sys.columns c ON ic.object_id = c.object_id AND ic.column_id = c.column_id
WHERE i.object_id = OBJECT_ID('customers');
```

## 9. Cross-Database Syntax — MSSQL vs MySQL vs PostgreSQL

### SQL Server (MSSQL)

Native, full-featured columnstore support since SQL Server 2012.

```sql
-- Clustered Columnstore Index (full table conversion)
CREATE CLUSTERED COLUMNSTORE INDEX idx_customers_cs
ON dbo.customers;

-- Non-Clustered Columnstore Index (selected columns only)
CREATE NONCLUSTERED COLUMNSTORE INDEX idx_customers_status_cs
ON dbo.customers (status, customer_id);

-- Drop
DROP INDEX idx_customers_cs ON dbo.customers;

-- Check columnstore indexes
SELECT name, type_desc
FROM sys.indexes
WHERE object_id = OBJECT_ID('dbo.customers')
  AND type_desc LIKE '%COLUMNSTORE%';
```

**SQL Server Notes:**
- Supported since **SQL Server 2012** (non-clustered) and **2014** (clustered, updateable).
- Maximum **1 columnstore index per table** in on-premise SQL Server.
- **Azure SQL Database / Azure Synapse** supports multiple non-clustered columnstore indexes.
- Columnstore indexes are a core feature of **Azure Synapse Analytics** (formerly SQL Data Warehouse).
- Use **`sys.column_store_row_groups`** to monitor row group health and compression state.

```sql
-- Monitor row group state (SQL Server / Azure Synapse)
SELECT
    object_name(object_id)  AS table_name,
    state_desc,             -- OPEN, CLOSED, COMPRESSED, TOMBSTONE
    row_count,
    size_in_bytes
FROM sys.column_store_row_groups
WHERE object_id = OBJECT_ID('customers');
```

---

### MySQL

MySQL does **not** have native columnstore index support in the standard InnoDB or MyISAM engines. However, columnstore capability is available via:

**Option A — MariaDB (MySQL-compatible) with Spider/ColumnStore engine:**

MariaDB (the open-source MySQL fork) includes a dedicated **ColumnStore storage engine** (formerly InfiniDB).

```sql
-- MariaDB ColumnStore: create table with ColumnStore engine
CREATE TABLE customers (
    customer_id INT,
    name        VARCHAR(100),
    status      VARCHAR(20)
) ENGINE=ColumnStore;

-- Standard indexes on ColumnStore tables are not used the same way;
-- the engine itself handles columnar storage internally.
```

**Option B — MySQL HeatWave (Oracle's MySQL cloud service):**

MySQL HeatWave adds an in-memory columnstore accelerator as a separate plugin/cluster.

```sql
-- Enable HeatWave on a table (MySQL HeatWave cloud service)
ALTER TABLE customers SECONDARY_ENGINE=RAPID;
LOAD SECONDARY; -- loads data into HeatWave (columnar, in-memory)

-- Queries automatically route to HeatWave engine for analytics
SELECT status, COUNT(*) FROM customers GROUP BY status;
```

**MySQL Notes:**
- Standard on-premise **MySQL does not support columnstore indexes natively**.
- For analytics on MySQL, consider migrating to MariaDB ColumnStore or using MySQL HeatWave (cloud).
- For pure analytics workloads on MySQL-compatible syntax, **ClickHouse** (column-oriented DB) is a popular open-source alternative.

---

### PostgreSQL

PostgreSQL does **not** have a native columnstore index as of PostgreSQL 16. It does offer column-oriented capabilities through extensions:

**Option A — `cstore_fdw` extension (Citus Data / deprecated in favor of newer approaches):**

```sql
-- Install cstore_fdw extension (must be installed server-side first)
CREATE EXTENSION cstore_fdw;

-- Create a foreign server
CREATE SERVER cstore_server FOREIGN DATA WRAPPER cstore_fdw;

-- Create a columnar foreign table
CREATE FOREIGN TABLE customers_cs (
    customer_id INT,
    name        TEXT,
    status      TEXT
)
SERVER cstore_server
OPTIONS (compression 'pglz');
```

**Option B — `pg_mooncake` or `pg_analytics` (newer 2024 extensions):**

Newer PostgreSQL extensions bring DuckDB-backed columnstore to PostgreSQL:

```sql
-- pg_mooncake: creates columnstore tables
CREATE EXTENSION pg_mooncake;

CREATE TABLE customers_cs (
    customer_id INT,
    name        TEXT,
    status      TEXT
) USING columnstore;
```

**Option C — Use Timescale's Columnar Compression (for time-series):**

```sql
-- TimescaleDB: compress chunks of hypertables into columnar format
SELECT compress_chunk(c) FROM show_chunks('customers') c;
```

**Option D — Migrate analytical workloads to Citus (distributed PostgreSQL):**

Citus (now built into PostgreSQL 14+ as a core extension) supports columnar tables:

```sql
-- PostgreSQL + Citus: create columnar table
CREATE TABLE customers_cs (
    customer_id INT,
    name        TEXT,
    status      TEXT
) USING columnar;

-- Convert existing heap table to columnar
SELECT alter_table_set_access_method('customers', 'columnar');
```

**PostgreSQL Notes:**
- PostgreSQL's native analytics alternative to columnstore is **BRIN indexes** (Block Range INdex) for time-series and sequential data.
- For serious analytics workloads, consider **DuckDB** (embeddable columnstore DB with PostgreSQL-compatible SQL) or **Apache Parquet** format.
- **Amazon Redshift** (PostgreSQL-based, fully columnar) is the enterprise choice for PostgreSQL-compatible columnstore.

---

### Analytics Queries That Benefit from Columnstore

#### Query 1 — Count by Status

```sql
SELECT status, COUNT(*) AS total
FROM customers
GROUP BY status;
```

**Output:**

| status   | total |
|---|---|
| active   | 7     |
| inactive | 3     |

**Columns read by columnstore:** Only `status` segment.

---

#### Query 2 — Average Score by Country

```sql
SELECT country, AVG(score) AS avg_score
FROM customers
GROUP BY country
ORDER BY avg_score DESC;
```

**Output:**

| country | avg_score |
|---|---|
| USA     | 750       |
| Canada  | 870       |
| India   | 633       |
| UK      | 360       |

**Columns read by columnstore:** Only `country` and `score` segments. `name`, `customer_id`, `created_at`, `status` pages → **never touched**.

---

#### Query 3 — Count Active Customers Per Year

```sql
SELECT
    YEAR(created_at) AS year,
    COUNT(*)         AS active_count
FROM customers
WHERE status = 'active'
GROUP BY YEAR(created_at)
ORDER BY year;
```

**Output:**

| year | active_count |
|---|---|
| 2022 | 2            |
| 2023 | 3            |
| 2024 | 2            |

**Columns read by columnstore:** `status` and `created_at` segments only.

## 11. Storage Efficiency — Proof with Numbers

A real benchmark from the transcript using the `FactInternetSales` table (~60,000 rows):

| Structure | Data Space | Index Space | Total Storage |
|---|---|---|---|
| **Heap** | ~9 MB | 0 MB | ~9 MB |
| **Row Store (Clustered)** | ~9 MB | ~X MB (B-Tree) | > 9 MB |
| **Columnstore** | **~1 MB** | 0 MB (no B-Tree) | **~1 MB** |

**Columnstore uses ~89% less storage** than the equivalent row store table — because:
- Dictionary encoding eliminates repeated string values
- No B-Tree index pages are needed
- Numeric sequences are delta/bit-packed efficiently

> At enterprise scale (billions of rows), this storage difference translates to **terabytes of savings** and **dramatically lower cloud storage costs**.

## 12. Real-World Analytics Use Cases

| Use Case | Why Columnstore Fits |
|---|---|
| **Sales dashboards** (total revenue by region/month) | Aggregates 2-3 columns from 100M+ rows — only those columns read |
| **Customer segmentation reports** (count by status, country) | Low-cardinality columns compress massively; fast GROUP BY |
| **Financial reporting** (SUM of amounts by account type) | Only `amount` and `account_type` columns needed; ignores 40+ other columns |
| **Data warehouse fact tables** (star schema) | Fact tables are wide (many columns, few needed per query) — perfect fit |
| **Log analysis** (event counts by type, daily trends) | High-volume, append-mostly, analytics-heavy — columnstore excels |
| **Business Intelligence tools** (Power BI, Tableau) | These tools auto-route to columnstore when available in SQL Server |
| **IoT time-series aggregation** (avg sensor reading per hour) | Massive row counts with repeated low-cardinality category columns |

## 13. Common Mistakes

| Mistake | Problem | Fix |
|---|---|---|
| Specifying column names when creating **clustered** columnstore | `Error: Key list not allowed` | Remove column list — all columns are implicit |
| Creating columnstore on a **write-heavy OLTP table** | Inserts/updates become slow — multi-step compression overhead | Use columnstore only on tables with read-heavy analytics patterns |
| Creating columnstore on a **small table** | No performance benefit; overhead isn't worth it | Columnstore benefits kick in at hundreds of thousands or millions of rows |
| Trying to create **multiple columnstore indexes** (SQL Server on-prem) | `Error: cannot create more than one columnstore index` | Choose either clustered OR non-clustered — not both |
| Forgetting to **drop the row store clustered index** before creating clustered columnstore | Error: cannot have two clustered indexes | Drop the existing clustered index first, then create the columnstore |
| Expecting columnstore to speed up **single-row lookups** (`WHERE id = 5`) | Columnstore is slower than B-Tree row store for point lookups | Use row store (B-Tree) for OLTP point lookups; columnstore for analytics scans |
| Using columnstore on columns with **high cardinality and frequent updates** | Compression is less effective; delta stores fill quickly | Columnstore is best for stable, bulk-loaded, append-only data |

---

## 14. Best Practices

1. **Use clustered columnstore for pure analytics tables** — data warehouses, fact tables in star schemas, reporting tables. These tables are optimized for reads and bulk inserts.

2. **Use non-clustered columnstore for mixed OLTP + analytics tables** — the row store handles fast inserts and point lookups; the columnstore handles analytical queries. SQL's Query Optimizer automatically chooses the right index per query.

3. **Always pair columnstore with batch inserts** — inserting one row at a time causes inefficient "delta store" usage. Use bulk inserts (`INSERT ... SELECT`, `BULK INSERT`, ETL pipelines) for best compression.

4. **Monitor row group health** — in SQL Server, check `sys.column_store_row_groups`. Row groups in `OPEN` state (not yet compressed) indicate small-batch writes. Use `ALTER INDEX REORGANIZE` to compress them.

5. **Column selection for non-clustered columnstore** — include only the columns your analytics queries actually filter or aggregate on. Avoid including high-cardinality, frequently-updated columns.

6. **Let the Query Optimizer decide** — once a columnstore index exists, SQL's optimizer automatically uses it when appropriate. Don't force-hint indexes unless profiling confirms a problem.

7. **Rebuild regularly on data warehouses** — after bulk loads, rebuild or reorganize the columnstore to merge delta stores and optimize compression:

```sql
ALTER INDEX idx_customers_cs ON customers REBUILD;
-- or for online, incremental maintenance:
ALTER INDEX idx_customers_cs ON customers REORGANIZE;
```

8. **Match your architecture to your system type** — don't use columnstore as a silver bullet. OLTP systems with frequent single-row reads should stay on row store B-Tree indexes.

## 15. Final Summary Table

| Concept | Row Store Index | Columnstore Index |
|---|---|---|
| Storage unit | Data page (rows together) | LOB page (one column per page) |
| Data organization | Row by row | Column by column |
| Compression | ❌ None (by default) | ✅ Dictionary, RLE, bit-packing |
| B-Tree | ✅ Always built | ❌ Not used |
| Storage size | Higher | **Up to ~89% smaller** |
| Best for | OLTP (transactions, point lookups) | **OLAP (analytics, aggregations)** |
| Read speed (aggregations) | ⚠️ Slow (reads all columns) | ✅ Very fast (reads only needed columns) |
| Write speed | ✅ Fast | ⚠️ Slower (compression overhead) |
| Max per table (SQL Server) | 1 clustered + many non-clustered | **1 total** (clustered OR non-clustered) |
| Column selection | Row store: all columns always | Non-clustered: choose specific columns |
| Real-world tools | Standard OLTP apps | Power BI, Tableau, Azure Synapse, data warehouses |
| SQL Server support | All versions | SQL Server 2012+ |
| MySQL support | ✅ Native | ❌ Not native (HeatWave / MariaDB) |
