## 1. What Is a SQL Query?

A **SQL query** is a request you write to ask the database a question and retrieve data in response. Your data lives inside tables inside a database. A query is the way you *talk* to that database in its own language — SQL.

```
You (analyst)                    Database
     │                               │
     │  "What is the total sales?"   │
     │  ────────── query ──────────► │
     │                               │  Processes query
     │                               │  Fetches data
     │  ◄──────── results ────────── │
     │                               │
```

### Key Properties of a SELECT Query

- A query **reads** data. It does not modify, insert, update, or delete anything.
- The database processes your query and returns a **result set** — a temporary table of rows and columns shown in your output.
- Queries are made up of **clauses** — named sections like `SELECT`, `FROM`, `WHERE`, each with a specific role.

## 3. SELECT and FROM — The Minimal Query

### 3.1 Definition

- **`SELECT`** — specifies *which columns* you want to see in the result.
- **`FROM`** — specifies *which table* the data comes from.

Together, `SELECT` and `FROM` form the simplest possible SQL query.

### 3.2 Syntax

```sql
-- Retrieve all columns from a table
SELECT *
FROM table_name;

-- Retrieve specific columns
SELECT column1, column2
FROM table_name;
```

`*` (star) is a wildcard that means "all columns."

### 3.3 Example: Retrieve All Customer Data

```sql
SELECT *
FROM customers;
```

**Output:**

| customer_id | first_name | country | score |
|---|---|---|---|
| 1 | Joan   | USA     | 900 |
| 2 | Maria  | Germany | 350 |
| 3 | George | USA     | 750 |
| 4 | Martin | Germany | 500 |
| 5 | Peter  | UK      | 0   |

### 3.4 Example: Retrieve All Orders

```sql
SELECT *
FROM orders;
```

**Output:**

| order_id | customer_id | order_date | amount |
|---|---|---|---|
| 101 | 1 | 2023-01-15 | 250.00 |
| 102 | 3 | 2023-03-22 | 180.00 |
| 103 | 2 | 2023-05-10 | 430.00 |
| 104 | 4 | 2023-06-01 | 120.00 |

### 3.5 Connecting to the Right Database

Before writing queries, make sure you're pointing at the correct database.

```sql
-- SQL Server / T-SQL
USE my_database;

-- PostgreSQL / MySQL — select the database at connection time,
-- or prefix table names:
SELECT * FROM my_database.customers;
```

### 3.6 Comments

Comments are notes inside your code that the database ignores during execution. They document intent and make scripts easier to maintain.

```sql
-- This is a single-line comment

/*
   This is a
   multi-line comment
*/

SELECT *   -- retrieve everything
FROM customers;
```

> ✅ Always comment the purpose of complex queries, especially in shared scripts and stored procedures.

### 3.7 Execution Flow

```
Step 1 → FROM    : Go to the database and load all rows from the table
Step 2 → SELECT  : Keep only the specified columns; remove everything else
```

## 4. Selecting Specific Columns

### 4.1 Definition

Instead of retrieving every column with `*`, you can specify **exactly which columns** to include. This reduces output size, improves readability, and can improve performance on wide tables.

### 4.2 Syntax

```sql
SELECT column1,
       column2,
       column3
FROM table_name;
```

- List column names separated by commas.
- The columns appear in the result **in the order you list them**, regardless of their order in the table.
- **Do not add a comma after the last column** — this is a very common mistake that causes a syntax error.

### 4.3 Example: Retrieve Only Name, Country, and Score

```sql
SELECT first_name,
       country,
       score
FROM customers;
```

**Output:**

| first_name | country | score |
|---|---|---|
| Joan   | USA     | 900 |
| Maria  | Germany | 350 |
| George | USA     | 750 |
| Martin | Germany | 500 |
| Peter  | UK      | 0   |

### 4.4 Column Order Is Yours to Control

```sql
-- Country appears last
SELECT first_name, score, country FROM customers;
```

| first_name | score | country |
|---|---|---|
| Joan | 900 | USA |
| ... | ... | ... |

The database returns columns in the order you specify in `SELECT`.

## 5. Aliases — Renaming Columns in Output

### 5.1 Definition

An **alias** is a temporary display name you assign to a column or expression in your query result. It does not rename anything in the actual database — it only affects how the column header appears in your output.

### 5.2 Syntax

```sql
SELECT column_name AS alias_name
FROM table_name;
```

The `AS` keyword is optional in most databases, but always recommended for clarity.

### 5.3 Example

```sql
SELECT
    country             AS customer_country,
    SUM(score)          AS total_score,
    COUNT(customer_id)  AS total_customers
FROM customers
GROUP BY country;
```

**Output:**

| customer_country | total_score | total_customers |
|---|---|---|
| USA     | 1650 | 2 |
| Germany | 850  | 2 |
| UK      | 0    | 1 |

Without aliases, aggregated columns like `SUM(score)` appear as `(No column name)` in SQL Server or `sum(score)` in PostgreSQL/MySQL — unreadable in reports.

## 6. WHERE — Filtering Rows

### 6.1 Definition

The `WHERE` clause filters the rows returned by a query based on a **condition**. Only rows where the condition evaluates to `TRUE` are included in the result. All other rows are excluded.

### 6.2 Syntax

```sql
SELECT column1, column2
FROM table_name
WHERE condition;
```

### 6.3 Common Comparison Operators

| Operator | Meaning | Example |
|---|---|---|
| `=` | Equal to | `country = 'USA'` |
| `<>` or `!=` | Not equal to | `score <> 0` |
| `>` | Greater than | `score > 500` |
| `<` | Less than | `score < 500` |
| `>=` | Greater than or equal | `score >= 500` |
| `<=` | Less than or equal | `score <= 500` |
| `BETWEEN` | Within a range (inclusive) | `score BETWEEN 300 AND 700` |
| `IN` | Matches any value in a list | `country IN ('USA', 'UK')` |
| `LIKE` | Pattern matching | `first_name LIKE 'M%'` |
| `IS NULL` | Value is missing | `score IS NULL` |
| `IS NOT NULL` | Value is present | `score IS NOT NULL` |

### 6.4 Combining Conditions

```sql
-- Both conditions must be true
WHERE score > 400 AND country = 'USA'

-- At least one condition must be true
WHERE country = 'Germany' OR country = 'UK'

-- Condition must be false
WHERE NOT country = 'USA'
```

### 6.5 String Values Require Single Quotes

```sql
-- ✅ Correct: string values go inside single quotes
WHERE country = 'Germany'

-- ❌ Wrong: double quotes or no quotes cause errors in most dialects
WHERE country = "Germany"
WHERE country = Germany
```

Numbers do **not** need quotes:

```sql
WHERE score > 500       -- ✅ correct
WHERE score > '500'     -- ⚠️ works in some DBs but is bad practice
```

### 6.6 Example: Customers with Score Not Equal to Zero

```sql
SELECT *
FROM customers
WHERE score <> 0;
```

**Output:**

| customer_id | first_name | country | score |
|---|---|---|---|
| 1 | Joan   | USA     | 900 |
| 2 | Maria  | Germany | 350 |
| 3 | George | USA     | 750 |
| 4 | Martin | Germany | 500 |

*(Peter with score = 0 is excluded)*

### 6.7 Example: Customers from Germany

```sql
SELECT first_name, country
FROM customers
WHERE country = 'Germany';
```

**Output:**

| first_name | country |
|---|---|
| Maria  | Germany |
| Martin | Germany |

### 6.8 Execution Flow

```
Step 1 → FROM  : Load all rows from the table
Step 2 → WHERE : Check each row against the condition
                 → TRUE  : keep the row
                 → FALSE : remove the row
Step 3 → SELECT: Keep only the specified columns
```

## 7. ORDER BY — Sorting Results

### 7.1 Definition

`ORDER BY` sorts the result set by one or more columns, either **ascending** (lowest to highest) or **descending** (highest to lowest).

### 7.2 Syntax

```sql
SELECT column1, column2
FROM table_name
ORDER BY column_name ASC | DESC;
```

- `ASC` — ascending order (default if omitted, but always specify explicitly for clarity).
- `DESC` — descending order.

### 7.3 Example: Sort by Highest Score First

```sql
SELECT *
FROM customers
ORDER BY score DESC;
```

**Output:**

| customer_id | first_name | country | score |
|---|---|---|---|
| 1 | Joan   | USA     | 900 |
| 3 | George | USA     | 750 |
| 4 | Martin | Germany | 500 |
| 2 | Maria  | Germany | 350 |
| 5 | Peter  | UK      | 0   |

### 7.4 Example: Sort by Lowest Score First

```sql
SELECT *
FROM customers
ORDER BY score ASC;
```

| customer_id | first_name | country | score |
|---|---|---|---|
| 5 | Peter  | UK      | 0   |
| 2 | Maria  | Germany | 350 |
| 4 | Martin | Germany | 500 |
| 3 | George | USA     | 750 |
| 1 | Joan   | USA     | 900 |

### 7.5 Nested Sorting — Multiple Columns

When sorting by a column that contains **repeated values** (like `country`), rows with the same value may appear in an arbitrary order. Adding a second sort column refines the result within each group.

```sql
-- Sort by country (ascending), then by score (highest first) within each country
SELECT *
FROM customers
ORDER BY country ASC, score DESC;
```

**Output:**

| customer_id | first_name | country | score |
|---|---|---|---|
| 4 | Martin | Germany | 500 |
| 2 | Maria  | Germany | 350 |
| 5 | Peter  | UK      | 0   |
| 1 | Joan   | USA     | 900 |
| 3 | George | USA     | 750 |

The column listed **first** in `ORDER BY` has priority. Secondary columns only act as tiebreakers within identical values of the primary sort column.

> ⚠️ Nested sorting only changes results when there are **duplicate values** in the primary sort column. If every value is unique, a second sort column has no effect.

## 8. GROUP BY — Aggregating Data

### 8.1 Definition

`GROUP BY` **combines rows that share the same value** in a specified column and applies **aggregate functions** to summarize the values within each group. It answers questions like "what is the total/average/count *per* category?"

### 8.2 Common Aggregate Functions

| Function | Description | Example |
|---|---|---|
| `COUNT()` | Count rows | `COUNT(customer_id)` |
| `SUM()` | Sum of values | `SUM(score)` |
| `AVG()` | Average value | `AVG(score)` |
| `MIN()` | Minimum value | `MIN(score)` |
| `MAX()` | Maximum value | `MAX(score)` |

### 8.3 Syntax

```sql
SELECT category_column,
       AGG_FUNCTION(value_column) AS alias
FROM table_name
GROUP BY category_column;
```

### 8.4 Example: Total Score Per Country

```sql
SELECT
    country,
    SUM(score)  AS total_score
FROM customers
GROUP BY country;
```

**How SQL processes this:**

```
Original 5 rows:
  Joan    USA     900  ─┐
  Maria   Germany 350  ─┼─ 2 Germany rows → combined → 350+500 = 850
  George  USA     750  ─┤
  Martin  Germany 500  ─┘
  Peter   UK      0

After GROUP BY country:
  Germany → SUM = 850
  USA     → SUM = 1650
  UK      → SUM = 0
```

**Output:**

| country | total_score |
|---|---|
| Germany | 850  |
| USA     | 1650 |
| UK      | 0    |

### 8.5 Example: Total Score and Total Customers Per Country

```sql
SELECT
    country,
    COUNT(customer_id)  AS total_customers,
    SUM(score)          AS total_score
FROM customers
GROUP BY country;
```

**Output:**

| country | total_customers | total_score |
|---|---|---|
| Germany | 2 | 850  |
| USA     | 2 | 1650 |
| UK      | 1 | 0    |

### 8.6 The Golden Rule of GROUP BY

> Every column in your `SELECT` must be either:
> 1. Listed in the `GROUP BY` clause, **or**
> 2. Wrapped in an aggregate function.

```sql
-- ❌ ERROR: first_name is neither in GROUP BY nor aggregated
SELECT country, first_name, SUM(score)
FROM customers
GROUP BY country;

-- ✅ FIXED: add first_name to GROUP BY
SELECT country, first_name, SUM(score) AS total_score
FROM customers
GROUP BY country, first_name;
```

> Adding more columns to `GROUP BY` creates more groups — with `country, first_name` you get 5 groups (one per unique combination) instead of 3.

## 9. HAVING — Filtering After Aggregation

### 9.1 Definition

`HAVING` filters the result **after** `GROUP BY` has been applied. It works like `WHERE`, but operates on aggregated values rather than raw row data.

| Clause | Filters | Timing |
|---|---|---|
| `WHERE` | Individual rows | **Before** aggregation |
| `HAVING` | Grouped/aggregated rows | **After** aggregation |

### 9.2 Syntax

```sql
SELECT category_column, AGG_FUNCTION(value_column) AS alias
FROM table_name
WHERE row_level_condition          -- applied BEFORE GROUP BY
GROUP BY category_column
HAVING aggregate_level_condition;  -- applied AFTER GROUP BY
```

### 9.3 Example: Countries with Total Score > 800

```sql
SELECT
    country,
    SUM(score) AS total_score
FROM customers
GROUP BY country
HAVING SUM(score) > 800;
```

**Output:**

| country | total_score |
|---|---|
| USA     | 1650 |
| Germany | 850  |

*(UK with total_score = 0 is excluded)*

### 9.4 Combined WHERE and HAVING

**Task:** Find the average score per country, considering only customers with a score not equal to zero, and return only countries with an average score greater than 430.

```sql
SELECT
    country,
    AVG(score) AS average_score
FROM customers
WHERE score <> 0               -- Pre-filter: remove zero-score rows first
GROUP BY country
HAVING AVG(score) > 430;       -- Post-filter: keep only qualifying groups
```

**Execution walkthrough:**

```
1. FROM    → Load all 5 rows
2. WHERE   → Remove Peter (score = 0) → 4 rows remain
3. GROUP BY→ Germany: [350, 500] → avg=425 | USA: [900, 750] → avg=825
4. HAVING  → Germany avg=425 < 430 → excluded | USA avg=825 > 430 → kept
```

**Output:**

| country | average_score |
|---|---|
| USA | 825 |

### 9.5 How to Choose: WHERE or HAVING?

Ask yourself: *"Am I filtering raw rows or aggregated results?"*

- Filter based on `score` (a raw column) → **WHERE**
- Filter based on `AVG(score)` (an aggregated value) → **HAVING**

```sql
-- WHERE: "only customers with score > 400"
WHERE score > 400

-- HAVING: "only countries whose average score > 400"
HAVING AVG(score) > 400
```

## 10. DISTINCT — Removing Duplicates

### 10.1 Definition

`DISTINCT` removes duplicate rows from the result, ensuring each unique value (or combination of values) appears only once.

### 10.2 Syntax

```sql
SELECT DISTINCT column1, column2
FROM table_name;
```

`DISTINCT` is placed **immediately after `SELECT`**, before column names.

### 10.3 Example: Unique List of Countries

```sql
-- Without DISTINCT: duplicates appear
SELECT country FROM customers;
```

| country |
|---|
| USA     |
| Germany |
| USA     |
| Germany |
| UK      |

```sql
-- With DISTINCT: each value appears once
SELECT DISTINCT country FROM customers;
```

| country |
|---|
| USA     |
| Germany |
| UK      |

### 10.4 DISTINCT on Multiple Columns

`DISTINCT` applies to the **combination** of all listed columns:

```sql
SELECT DISTINCT country, score
FROM customers;
```

This returns unique `(country, score)` pairs — not unique countries alone.

### 10.5 Performance Warning

`DISTINCT` is an **expensive operation** — the database must compare every row to identify duplicates. Do not use it reflexively on every query.

```sql
-- ✅ Needed: country has repeated values
SELECT DISTINCT country FROM customers;

-- ❌ Unnecessary: customer_id is already unique (primary key)
SELECT DISTINCT customer_id FROM customers;
-- Wastes resources — produces identical output to SELECT customer_id
```

> Only apply `DISTINCT` when you know duplicates exist and you genuinely need to remove them.

## 11. TOP / LIMIT — Restricting Row Count

### 11.1 Definition

`TOP` (SQL Server) or `LIMIT` (MySQL/PostgreSQL) restricts how many rows are returned in the result, regardless of conditions. It is based purely on **row position** in the result.

### 11.2 Syntax

```sql
-- SQL Server
SELECT TOP n *
FROM table_name;

-- MySQL / PostgreSQL
SELECT *
FROM table_name
LIMIT n;
```

### 11.3 Example: Retrieve Only 3 Customers

```sql
-- SQL Server
SELECT TOP 3 *
FROM customers;
```

```sql
-- MySQL / PostgreSQL
SELECT *
FROM customers
LIMIT 3;
```

**Output:**

| customer_id | first_name | country | score |
|---|---|---|---|
| 1 | Joan   | USA     | 900 |
| 2 | Maria  | Germany | 350 |
| 3 | George | USA     | 750 |

*(First 3 rows by their storage order)*

### 11.4 Combining TOP with ORDER BY — Top/Bottom Analysis

This is where `TOP` becomes truly powerful. By combining it with `ORDER BY`, you can find the top N or bottom N values.

**Task: Top 3 customers with the highest scores**

```sql
-- SQL Server
SELECT TOP 3 *
FROM customers
ORDER BY score DESC;
```

```sql
-- MySQL / PostgreSQL
SELECT *
FROM customers
ORDER BY score DESC
LIMIT 3;
```

**Output:**

| customer_id | first_name | country | score |
|---|---|---|---|
| 1 | Joan   | USA     | 900 |
| 3 | George | USA     | 750 |
| 4 | Martin | Germany | 500 |

**Task: 2 most recent orders**

```sql
-- SQL Server
SELECT TOP 2 *
FROM orders
ORDER BY order_date DESC;
```

**Output:**

| order_id | customer_id | order_date | amount |
|---|---|---|---|
| 104 | 4 | 2023-06-01 | 120.00 |
| 103 | 2 | 2023-05-10 | 430.00 |

**Task: 2 lowest-scoring customers**

```sql
-- SQL Server
SELECT TOP 2 *
FROM customers
ORDER BY score ASC;
```

| customer_id | first_name | country | score |
|---|---|---|---|
| 5 | Peter | UK      | 0   |
| 2 | Maria | Germany | 350 |

## 12. Coding Order vs. Execution Order

This is one of the most important concepts in SQL — and the most commonly misunderstood.

### 12.1 Coding Order (How You Write It)

When writing a query, you must follow this exact order or the database will return a syntax error:

```sql
SELECT DISTINCT TOP n          -- 1st
    column1,
    column2,
    AGG_FUNCTION(column3)
FROM table_name                -- 2nd
WHERE row_condition            -- 3rd
GROUP BY column1               -- 4th
HAVING aggregate_condition     -- 5th
ORDER BY column1 ASC;          -- 6th
```

### 12.2 Execution Order (How the Database Processes It)

The database does **not** execute clauses in the order you write them. It uses this internal execution order:

| Step | Clause | What Happens |
|---|---|---|
| 1 | **FROM** | Load all rows from the specified table |
| 2 | **WHERE** | Filter rows based on row-level conditions (before aggregation) |
| 3 | **GROUP BY** | Combine rows with matching values; compute aggregates |
| 4 | **HAVING** | Filter groups based on aggregate conditions |
| 5 | **SELECT** | Select and project only the specified columns |
| 6 | **DISTINCT** | Remove duplicate rows from the result |
| 7 | **ORDER BY** | Sort the result set |
| 8 | **TOP / LIMIT** | Cut the result to N rows |

### 12.3 Why This Matters

Understanding execution order explains several otherwise confusing behaviors:

**Why can't you use an alias in WHERE?**

```sql
-- ❌ ERROR: 'total_score' alias doesn't exist yet when WHERE runs
SELECT SUM(score) AS total_score
FROM customers
WHERE total_score > 800;

-- ✅ CORRECT: use HAVING (runs after SELECT creates the alias in some DBs,
-- but safest to repeat the expression)
SELECT country, SUM(score) AS total_score
FROM customers
GROUP BY country
HAVING SUM(score) > 800;
```

**Why does WHERE come before GROUP BY in the query?**

Because `WHERE` is executed before `GROUP BY`. If you want to filter raw rows before aggregating — use `WHERE`. If you want to filter after aggregating — use `HAVING`.

**Why is TOP/LIMIT last?**

Because it operates on the **final, fully-processed** result. Sorting happens first, then TOP cuts from the top of the sorted list — this is what makes `TOP 3 ORDER BY score DESC` return the three highest scorers.

### 12.4 Visual Summary

```
CODING ORDER                     EXECUTION ORDER
─────────────────                ─────────────────────────────────
1. SELECT                        1. FROM       ← First: get the data
2. FROM                          2. WHERE      ← Filter rows
3. WHERE                         3. GROUP BY   ← Aggregate
4. GROUP BY                      4. HAVING     ← Filter groups
5. HAVING                        5. SELECT     ← Pick columns
6. ORDER BY                      6. DISTINCT   ← Remove duplicates
                                 7. ORDER BY   ← Sort
                                 8. TOP/LIMIT  ← Cut to N rows
```

## 13. Dialect Differences: MySQL vs MS SQL Server vs PostgreSQL

### 13.1 Limiting Rows

```sql
-- SQL Server: TOP goes inside SELECT, before columns
SELECT TOP 5 * FROM customers;

-- MySQL: LIMIT goes at the end
SELECT * FROM customers LIMIT 5;

-- PostgreSQL: LIMIT goes at the end (same as MySQL)
SELECT * FROM customers LIMIT 5;

-- SQL Server: TOP with PERCENT
SELECT TOP 10 PERCENT * FROM customers;

-- MySQL / PostgreSQL: no native PERCENT; use calculation
SELECT * FROM customers
LIMIT (SELECT CEIL(COUNT(*) * 0.10) FROM customers);
```

### 13.2 OFFSET — Skipping Rows (Pagination)

```sql
-- MySQL / PostgreSQL: LIMIT with OFFSET for pagination
SELECT * FROM customers
ORDER BY customer_id
LIMIT 10 OFFSET 20;   -- skip first 20 rows, return next 10

-- SQL Server 2012+: OFFSET FETCH
SELECT * FROM customers
ORDER BY customer_id
OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY;
```

### 13.3 String Quoting

```sql
-- All three: single quotes for string values ✅
WHERE country = 'USA'

-- MySQL only: also accepts double quotes (non-standard, avoid)
WHERE country = "USA"   -- works in MySQL, fails in SQL Server/PostgreSQL

-- Identifiers (column/table names) with spaces:
-- SQL Server: [column name]
-- MySQL: `column name`
-- PostgreSQL: "column name"
```

### 13.4 NULL Handling in Aggregates

All three databases exclude `NULL` values from aggregate functions (`SUM`, `AVG`, `COUNT(column)`) by default. `COUNT(*)` counts all rows including NULLs.

```sql
-- Safely replace NULLs before aggregating
SELECT AVG(COALESCE(score, 0)) AS avg_score FROM customers;
-- COALESCE returns 0 wherever score is NULL
```

### 13.5 DISTINCT with Aggregates

```sql
-- Count only distinct (unique) values — works in all three
SELECT COUNT(DISTINCT country) AS unique_countries
FROM customers;
```

### 13.6 ORDER BY with NULLs

```sql
-- PostgreSQL: NULLs sort last by default in ASC, first in DESC
-- Control explicitly:
ORDER BY score ASC NULLS LAST;
ORDER BY score DESC NULLS FIRST;

-- SQL Server: NULLs always sort first in ASC, last in DESC (not configurable directly)
-- Workaround:
ORDER BY CASE WHEN score IS NULL THEN 1 ELSE 0 END, score ASC;

-- MySQL: NULLs sort first in ASC
```

### 13.7 GROUP BY Strictness

```sql
-- PostgreSQL / SQL Server: strict — non-aggregated columns MUST be in GROUP BY
SELECT country, first_name, SUM(score) FROM customers GROUP BY country;
-- ❌ ERROR: first_name must be in GROUP BY

-- MySQL with ONLY_FULL_GROUP_BY mode OFF (older default):
-- allows non-aggregated columns NOT in GROUP BY (returns arbitrary values — dangerous!)
-- Always enable ONLY_FULL_GROUP_BY in MySQL for correct behavior:
SET sql_mode = 'ONLY_FULL_GROUP_BY';
```

## 14. Common Mistakes

### ❌ Mistake 1: Trailing Comma After Last Column

```sql
-- ❌ ERROR: trailing comma expects another column
SELECT first_name,
       country,
       score,       -- ← this comma causes a syntax error
FROM customers;

-- ✅ CORRECT
SELECT first_name,
       country,
       score
FROM customers;
```

### ❌ Mistake 2: String Values Without Single Quotes

```sql
-- ❌ ERROR
WHERE country = Germany

-- ✅ CORRECT
WHERE country = 'Germany'
```

### ❌ Mistake 3: Using WHERE Instead of HAVING for Aggregated Conditions

```sql
-- ❌ ERROR: can't use SUM() in WHERE
SELECT country, SUM(score) AS total
FROM customers
WHERE SUM(score) > 800
GROUP BY country;

-- ✅ CORRECT: HAVING filters after GROUP BY
SELECT country, SUM(score) AS total
FROM customers
GROUP BY country
HAVING SUM(score) > 800;
```

### ❌ Mistake 4: Non-Aggregated Column Not in GROUP BY

```sql
-- ❌ ERROR in SQL Server / PostgreSQL
SELECT country, first_name, SUM(score)
FROM customers
GROUP BY country;   -- first_name must be here too

-- ✅ CORRECT
SELECT country, first_name, SUM(score)
FROM customers
GROUP BY country, first_name;
```

### ❌ Mistake 5: Overusing DISTINCT

```sql
-- ❌ Wasteful: customer_id is a primary key — already unique
SELECT DISTINCT customer_id FROM customers;

-- ✅ Only use DISTINCT when duplicates genuinely exist
SELECT DISTINCT country FROM customers;
```

### ❌ Mistake 6: Ignoring Execution Order When Aliasing

```sql
-- ❌ ERROR in SQL Server / PostgreSQL: alias not yet defined when WHERE runs
SELECT SUM(score) AS total_score
FROM customers
WHERE total_score > 500;

-- ✅ CORRECT: reference the original expression in WHERE
-- (or use HAVING for aggregated filters)
SELECT country, SUM(score) AS total_score
FROM customers
GROUP BY country
HAVING SUM(score) > 500;
```

### ❌ Mistake 7: Wrong ORDER of Clauses

```sql
-- ❌ ERROR: GROUP BY must come before ORDER BY
SELECT country, SUM(score)
FROM customers
ORDER BY country
GROUP BY country;

-- ✅ CORRECT: follow the coding order
SELECT country, SUM(score)
FROM customers
GROUP BY country
ORDER BY country;
```

## 15. Real-World Analytics Use Cases

### Use Case 1: Top 5 Customers by Revenue

```sql
-- SQL Server
SELECT TOP 5
    first_name,
    country,
    score
FROM customers
ORDER BY score DESC;
```

### Use Case 2: Sales Report by Country

```sql
SELECT
    c.country,
    COUNT(o.order_id)   AS total_orders,
    SUM(o.amount)       AS total_revenue,
    AVG(o.amount)       AS avg_order_value
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.country
ORDER BY total_revenue DESC;
```

### Use Case 3: Countries with Below-Average Engagement

```sql
SELECT
    country,
    AVG(score) AS avg_score
FROM customers
WHERE score IS NOT NULL
GROUP BY country
HAVING AVG(score) < (SELECT AVG(score) FROM customers)
ORDER BY avg_score ASC;
```

### Use Case 4: Unique Countries for a Dropdown Filter

```sql
SELECT DISTINCT country
FROM customers
WHERE country IS NOT NULL
ORDER BY country ASC;
```

### Use Case 5: Most Recent 10 Orders

```sql
-- SQL Server
SELECT TOP 10 *
FROM orders
ORDER BY order_date DESC;

-- MySQL / PostgreSQL
SELECT *
FROM orders
ORDER BY order_date DESC
LIMIT 10;
```

## 16. Best Practices

1. **Always specify the database** at the top of your script with `USE database_name` (SQL Server) or by prefixing table names, to avoid accidentally querying the wrong database.

2. **Avoid `SELECT *` in production.** Name your columns explicitly. `SELECT *` retrieves unnecessary data, breaks code when table structures change, and makes queries harder to understand.

3. **Always give aliases to aggregated columns.** `SUM(score)` without an alias produces an unreadable column header in reports. Use `SUM(score) AS total_score`.

4. **Comment your queries.** Add inline or block comments explaining what each section does, especially in complex multi-clause queries.

5. **Filter early with WHERE.** Remove unwanted rows before aggregation to reduce the data the database needs to group and sort — this significantly improves performance on large tables.

6. **Use HAVING only for aggregated conditions.** For raw row conditions, always use WHERE. Misusing HAVING for row-level filters forces the database to aggregate data it will then discard.

7. **Always specify ASC or DESC in ORDER BY.** Even though ASC is the default, being explicit makes your intent clear to anyone reading the query.

8. **Use DISTINCT sparingly.** Only apply it when you have confirmed that duplicates exist and need to be removed. It is a potentially expensive operation.

9. **Test queries incrementally.** Start with `SELECT * FROM table`, add `WHERE`, verify the row count, then add `GROUP BY`, verify the groups, then add `HAVING`. Build complexity one clause at a time.

10. **Understand the execution order.** When a query behaves unexpectedly, trace through the execution order (FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → TOP) to find where it diverges from your expectation.

## 17. Final Summary Table

| Clause | Purpose | Position in Query | Execution Step |
|---|---|---|---|
| `SELECT` | Choose which columns to display | 1st | 5th |
| `DISTINCT` | Remove duplicate rows | After SELECT | 6th |
| `TOP` / `LIMIT` | Return only N rows | After SELECT (SS) / End (MySQL/PG) | 8th (last) |
| `FROM` | Specify the source table | 2nd | 1st |
| `WHERE` | Filter rows before aggregation | After FROM | 2nd |
| `GROUP BY` | Group rows and compute aggregates | After WHERE | 3rd |
| `HAVING` | Filter groups after aggregation | After GROUP BY | 4th |
| `ORDER BY` | Sort the result set | Last clause | 7th |

### Five Types of Filtering in a Single Query

| Filter Type | Clause | What It Filters |
|---|---|---|
| Column filter | `SELECT col1, col2` | Removes unwanted columns |
| Duplicate filter | `SELECT DISTINCT` | Removes repeated rows |
| Row filter (pre-aggregation) | `WHERE` | Removes rows based on raw data conditions |
| Row filter (post-aggregation) | `HAVING` | Removes groups based on aggregate conditions |
| Row count limit | `TOP` / `LIMIT` | Keeps only the first N rows |

### Quick Reference: Execution Order

```
FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → TOP/LIMIT
 1        2        3          4        5         6           7           8
```
