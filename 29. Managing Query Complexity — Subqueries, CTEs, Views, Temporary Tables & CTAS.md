## 1. Why Query Complexity Techniques Exist

### 1.1 Definition

**Query complexity techniques** are structured SQL patterns — subqueries, Common Table Expressions (CTEs), views, temporary tables, and CTAS — that allow developers and analysts to:

- Break large, monolithic queries into smaller, named, reusable steps
- Encapsulate repeated logic in a single definition
- Control who can access what data
- Improve performance by caching intermediate results
- Reduce cognitive load when reading and maintaining SQL code

### 1.2 The Core Problem They Solve

In a simple one-person project, a single SELECT statement is usually sufficient. In any real data project, however, the same database is accessed by many people with different roles, different skill levels, and different goals — all simultaneously. Without structured techniques, the result is redundancy, inconsistency, poor performance, and security risk.

### 2.2 The Four Core Challenges

#### Challenge 1 — Redundant Logic

Different analysts, data engineers, and scientists write queries independently without coordination. When you compare their scripts, the same business logic (date filters, customer segmentation, KPI formulas) is duplicated across dozens of files — often implemented differently by each person.

```sql
-- Analyst A's version of "active customers"
WHERE last_order_date >= DATEADD(DAY, -90, GETDATE())

-- Analyst B's version (different threshold, different column)
WHERE status = 'active' AND created_date >= '2023-01-01'

-- Data Engineer's version (different logic entirely)
WHERE customer_tier IN ('Gold', 'Silver') AND churn_flag = 0
```

**Result:** Reports from different teams show different numbers for the same metric. Nobody knows which version is correct.

#### Challenge 2 — Performance Degradation

When many analysts run large, complex, unoptimized queries simultaneously against the same database, the server experiences enormous strain.

| Actor | Query Complexity | Estimated Runtime |
|---|---|---|
| Data Engineer (ETL) | Joins 8 tables, 50M rows | 5 hours |
| Financial Analyst | 4 aggregation levels, 10M rows | 40 minutes |
| Pre-BI data prep query | Window functions, 5M rows | 30 minutes |
| Data Scientist feature extraction | Full table scan, 20M rows | 1 hour |

Each of these running concurrently degrades performance for everyone.

#### Challenge 3 — Data Model Complexity

Enterprise databases are built for applications — not for ad-hoc analysis. The physical data model includes dozens or hundreds of normalized tables with relationships only the original developers understand.

```
"How do I join orders to products?"
"Which table holds the customer's region?"
"Why am I getting duplicate rows after joining these two tables?"
"What does the 'status_cd' column mean?"
```

Analysts flood the database team with questions. The database team repeatedly explains the same schema. Both sides lose productivity.

#### Challenge 4 — Security and Access Control

Giving every analyst direct access to every table is unacceptable in any production environment. A financial analyst should not see salary data. A marketing analyst should not see raw payment records. Without controlled access, sensitive data is exposed.

## 3. The Five Techniques — Overview

Each of the five techniques addresses one or more of the challenges above.

| Technique | Definition | Primary Benefit |
|---|---|---|
| **Subquery** | A query nested inside another query | Breaks complex logic into sequential steps inline |
| **CTE (Common Table Expression)** | A named, temporary result set defined with `WITH` | Improves readability; eliminates redundant subqueries |
| **View** | A saved, named query stored in the database | Reusable encapsulation; access control; hides complexity |
| **Temporary Table** | A physical table stored in tempdb for the session | Materialize intermediate results; improve performance |
| **CTAS (Create Table As Select)** | Creates a permanent new table from a SELECT result | Persist transformed data; stage ETL outputs |

> Each of these techniques is covered in depth in subsequent chapters. This chapter focuses on the foundational architecture that governs how all five behave.

## 4. Database Architecture

### 4.1 Client Side vs Server Side

Every SQL interaction involves two sides:

```
┌───────────────────────────┐        ┌───────────────────────────────────────────┐
│        CLIENT SIDE        │        │               SERVER SIDE                 │
│                           │        │                                           │
│  You write SQL queries    │──────► │  ┌─────────────────────────────────────┐  │
│                           │        │  │         Database Engine              │  │
│  SELECT * FROM orders     │        │  │  (Brain of the database)            │  │
│                           │◄─────  │  │  Handles: store, retrieve, manage   │  │
│  Results returned         │        │  └────────────────┬────────────────────┘  │
└───────────────────────────┘        │                   │                       │
                                     │         ┌─────────┴──────────┐           │
                                     │         ▼                    ▼           │
                                     │  ┌─────────────┐   ┌───────────────────┐ │
                                     │  │    Cache     │   │   Disk Storage    │ │
                                     │  │ (Short-term) │   │   (Long-term)     │ │
                                     │  │  Fast        │   │   Slow but large  │ │
                                     │  │  Temporary   │   │   Permanent       │ │
                                     │  └─────────────┘   └───────────────────┘ │
                                     └───────────────────────────────────────────┘
```

### 4.2 The Database Engine

The **database engine** is the core component that processes every SQL query. Its responsibilities include:

- Parsing and validating your SQL syntax
- Generating and caching execution plans
- Reading data from disk into cache
- Performing joins, filters, sorts, and aggregations
- Writing results back to the client

### 4.3 Cache vs Disk Storage

| Property | Cache (RAM) | Disk Storage |
|---|---|---|
| **Speed** | Very fast | Slow |
| **Capacity** | Limited | Very large |
| **Durability** | Temporary — lost on restart | Permanent — survives restarts |
| **Purpose** | Hold frequently accessed data for fast retrieval | Permanently store all database data |
| **Analogy** | RAM in your computer | Hard drive in your computer |

The engine always checks the cache first. A cache hit means data is served instantly without touching the disk — this is why repeated queries on the same data are faster than the first run.

## 5. Storage Types: User Data, System Catalog, Temporary Storage

Inside a SQL Server database, data is organized into three distinct storage areas. Understanding these areas explains where each of the five techniques stores its results.

```
┌─────────────────────────────────────────────────────────────────────┐
│                         SQL Server Storage                          │
│                                                                     │
│  ┌───────────────────┐  ┌───────────────────┐  ┌────────────────┐  │
│  │   User Data       │  │  System Catalog   │  │  Temporary     │  │
│  │   Storage         │  │  (Metadata)       │  │  Storage       │  │
│  │                   │  │                   │  │  (tempdb)      │  │
│  │ Tables you create │  │ Descriptions of   │  │ Short-lived    │  │
│  │ and query         │  │ your tables,      │  │ tables and     │  │
│  │                   │  │ columns, indexes, │  │ sort buffers   │  │
│  │ customers         │  │ views, schemas    │  │                │  │
│  │ orders            │  │                   │  │ #temp_orders   │  │
│  │ products          │  │ information_schema│  │ ##global_temp  │  │
│  └───────────────────┘  └───────────────────┘  └────────────────┘  │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.1 User Data Storage

**Definition:** The primary content area of the database. This is where all application data lives — the rows and columns that users actually care about.

```sql
-- These tables live in User Data Storage
SELECT * FROM customers;
SELECT * FROM orders;
SELECT * FROM employees;
```

This is the storage area all five techniques ultimately draw from, transform, or write back into.

### 5.2 System Catalog (Metadata Storage)

**Definition:** The internal storage area where the database keeps track of its own structure — tables, columns, data types, indexes, views, constraints, and schemas.

**Metadata** = data about data.

**Example:** When you create the `customers` table:

```sql
CREATE TABLE customers (
    customer_id   INT           NOT NULL PRIMARY KEY,
    first_name    VARCHAR(50)   NOT NULL,
    last_name     VARCHAR(50)   NOT NULL,
    email         VARCHAR(100)  NULL,
    created_date  DATE          NOT NULL
);
```

The database stores two things:

1. **User Data** — the actual customer rows you INSERT
2. **Metadata** (System Catalog) — a structural description of the table:

| Metadata Field | Value |
|---|---|
| Table name | `customers` |
| Schema | `dbo` |
| Column 1 | `customer_id`, INT, NOT NULL, PRIMARY KEY |
| Column 2 | `first_name`, VARCHAR(50), NOT NULL |
| Column 3 | `last_name`, VARCHAR(50), NOT NULL |
| Column 4 | `email`, VARCHAR(100), NULL |
| Column 5 | `created_date`, DATE, NOT NULL |

Every table you create has a "twin" in the system catalog that describes its structure. The database engine uses this metadata to validate your queries, plan execution, and enforce constraints — before touching a single row of user data.

### 5.3 Temporary Storage (tempdb)

**Definition:** A special system database (`tempdb` in SQL Server) that holds short-lived data created during query processing — temporary tables, sort buffers, hash join intermediates, and spool operations.

Temporary data is automatically cleaned up when:
- The session ends
- The stored procedure completes
- An explicit `DROP TABLE` is executed

```
Object Explorer → System Databases → tempdb → Temporary Tables
```

> In production environments, regular users and developers typically cannot access system databases directly — only the DBA has that level of access. In a local development environment, you have full visibility.

## 6. The System Catalog and Information Schema

### 6.1 Accessing Metadata with INFORMATION_SCHEMA

The `INFORMATION_SCHEMA` is a standardized, read-only schema built into SQL Server (and most other databases) that exposes the system catalog through queryable views. You can use it to explore the structure of any database without needing DBA-level access.

```sql
-- List all views available in INFORMATION_SCHEMA
SELECT * FROM INFORMATION_SCHEMA.TABLES;         -- All tables and views
SELECT * FROM INFORMATION_SCHEMA.COLUMNS;        -- All columns in all tables
SELECT * FROM INFORMATION_SCHEMA.VIEWS;          -- All views
SELECT * FROM INFORMATION_SCHEMA.ROUTINES;       -- Stored procedures and functions
```

### 6.2 Practical Metadata Queries

```sql
-- List all tables in the current database
SELECT DISTINCT TABLE_NAME
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_TYPE = 'BASE TABLE'
ORDER BY TABLE_NAME;
```

| TABLE\_NAME |
| ----------- |
| customers   |
| employees   |
| orders      |
| products    |

```sql
-- Inspect all columns in the customers table
SELECT
    COLUMN_NAME,
    ORDINAL_POSITION,
    DATA_TYPE,
    CHARACTER_MAXIMUM_LENGTH,
    IS_NULLABLE
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_NAME = 'customers'
ORDER BY ORDINAL_POSITION;
```

| COLUMN\_NAME | ORDINAL\_POSITION | DATA\_TYPE | CHARACTER\_MAXIMUM\_LENGTH | IS\_NULLABLE |
| ------------ | ----------------- | ---------- | -------------------------- | ------------ |
| customer\_id | 1                 | int        | NULL                       | NO           |
| first\_name  | 2                 | varchar    | 50                         | NO           |
| last\_name   | 3                 | varchar    | 50                         | NO           |
| email        | 4                 | varchar    | 100                        | YES          |
| created\_date | 5                | date       | NULL                       | NO           |

```sql
-- Find tables you may have forgotten about (test tables, staging tables, etc.)
SELECT TABLE_NAME
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_TYPE = 'BASE TABLE'
  AND TABLE_NAME LIKE '%test%';
```

> **Practical Use:** When inheriting an unfamiliar database, querying `INFORMATION_SCHEMA` is the fastest way to understand the data model without any documentation.

## 7. How a Simple Query Executes

Understanding this flow explains *why* each of the five techniques behaves differently in terms of performance and where results are stored.

```
Client submits:
SELECT order_id, total_amount FROM orders WHERE order_date = '2024-01-15';

                            │
                            ▼
                ┌───────────────────────┐
                │   Database Engine     │
                │   receives query      │
                └───────────┬───────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  Check Cache          │◄── Is orders data already in cache?
                │  (Buffer Pool)        │
                └───────────┬───────────┘
                            │
              ┌─────────────┴─────────────┐
              │                           │
         Cache HIT                   Cache MISS
              │                           │
              │                           ▼
              │              ┌────────────────────────┐
              │              │  Read from Disk        │
              │              │  (User Data Storage)   │
              │              │  Load into Cache       │
              │              └────────────┬───────────┘
              │                           │
              └───────────────┬───────────┘
                              │
                              ▼
                ┌─────────────────────────┐
                │  Execute Query          │
                │  Apply WHERE filter     │
                │  Select columns         │
                └─────────────┬───────────┘
                              │
                              ▼
                ┌─────────────────────────┐
                │  Return Results         │
                │  to Client              │
                └─────────────────────────┘
```

### 7.1 Where Each Technique Fits in This Flow

| Technique | Stored Where | Persisted? | Cache Eligible? |
|---|---|---|---|
| **Subquery** | Evaluated inline, in engine memory | ❌ No | Partial (depends on optimizer) |
| **CTE** | Evaluated inline, in engine memory | ❌ No | Partial (materialized or not, optimizer decides) |
| **View** | Definition in System Catalog; data from User Storage at runtime | Definition only | ✅ Yes (same as base table) |
| **Temporary Table** | Written to tempdb (Temporary Storage) on disk | Session-scoped | ✅ Yes (can be indexed) |
| **CTAS** | Written to User Data Storage as permanent table | ✅ Permanent | ✅ Yes (full table) |

## 9. Cross-Database Comparison: MySQL, MSSQL, PostgreSQL

### 9.1 INFORMATION_SCHEMA Support

All three major databases support `INFORMATION_SCHEMA`, but with differences in what views are available and their naming conventions.

| View | MySQL | MSSQL | PostgreSQL |
|---|---|---|---|
| List all tables | `INFORMATION_SCHEMA.TABLES` | `INFORMATION_SCHEMA.TABLES` | `INFORMATION_SCHEMA.TABLES` |
| List all columns | `INFORMATION_SCHEMA.COLUMNS` | `INFORMATION_SCHEMA.COLUMNS` | `INFORMATION_SCHEMA.COLUMNS` |
| List all views | `INFORMATION_SCHEMA.VIEWS` | `INFORMATION_SCHEMA.VIEWS` | `INFORMATION_SCHEMA.VIEWS` |
| Stored routines | `INFORMATION_SCHEMA.ROUTINES` | `INFORMATION_SCHEMA.ROUTINES` | `INFORMATION_SCHEMA.ROUTINES` |
| Database-specific catalog | `information_schema.SCHEMATA` | `sys.databases` | `pg_catalog.pg_database` |

```sql
-- MSSQL: Additional system catalog views
SELECT * FROM sys.tables;        -- All user tables
SELECT * FROM sys.columns;       -- All columns
SELECT * FROM sys.indexes;       -- All indexes
SELECT * FROM sys.views;         -- All views

-- PostgreSQL: pg_catalog
SELECT * FROM pg_catalog.pg_tables WHERE schemaname = 'public';
SELECT * FROM pg_catalog.pg_indexes WHERE tablename = 'orders';

-- MySQL: Equivalent to sys.tables
SELECT * FROM information_schema.TABLES WHERE TABLE_SCHEMA = 'your_db_name';
```

### 9.2 Temporary Storage Location

| Database | Temp Table Location | Scope | Syntax |
|---|---|---|---|
| **MSSQL** | `tempdb` system database | Session (`#`) or global (`##`) | `CREATE TABLE #temp_orders (...)` |
| **MySQL** | In-memory or temp file (engine-dependent) | Session only | `CREATE TEMPORARY TABLE temp_orders (...)` |
| **PostgreSQL** | `pg_temp` schema within the session | Session only | `CREATE TEMP TABLE temp_orders (...)` |

```sql
-- MSSQL: Local temp table (visible only to current session)
CREATE TABLE #temp_active_customers (
    customer_id   INT,
    customer_name VARCHAR(100),
    total_orders  INT
);

-- MSSQL: Global temp table (visible to all sessions)
CREATE TABLE ##global_active_customers (
    customer_id   INT,
    customer_name VARCHAR(100)
);

-- MySQL
CREATE TEMPORARY TABLE temp_active_customers (
    customer_id   INT,
    customer_name VARCHAR(100)
);

-- PostgreSQL
CREATE TEMP TABLE temp_active_customers (
    customer_id   INT,
    customer_name VARCHAR(100)
);
```

### 9.3 Cache Architecture Terminology

| Concept | MSSQL Term | MySQL Term | PostgreSQL Term |
|---|---|---|---|
| In-memory data cache | Buffer Pool | InnoDB Buffer Pool | Shared Buffers |
| Query plan cache | Plan Cache | Query Cache (deprecated 8.0+) | `pg_stat_statements` (extension) |
| Temporary sort buffer | TempDB spill | Sort buffer | `work_mem` |
| Temp storage | tempdb | tmp\_table\_size / tmpdir | `temp_tablespaces` |

### 9.4 CTAS Syntax Comparison

```sql
-- MSSQL: SELECT INTO (most common CTAS pattern)
SELECT
    customer_id,
    SUM(total_amount) AS lifetime_value
INTO customer_lifetime_value          -- creates a new permanent table
FROM orders
GROUP BY customer_id;

-- MySQL: CREATE TABLE ... SELECT
CREATE TABLE customer_lifetime_value AS
SELECT
    customer_id,
    SUM(total_amount) AS lifetime_value
FROM orders
GROUP BY customer_id;

-- PostgreSQL: CREATE TABLE ... AS SELECT
CREATE TABLE customer_lifetime_value AS
SELECT
    customer_id,
    SUM(total_amount) AS lifetime_value
FROM orders
GROUP BY customer_id;
```

### 9.5 View Creation (All Three Databases)

Views use identical syntax across all three:

```sql
-- Works in MSSQL, MySQL, and PostgreSQL
CREATE VIEW vw_active_customers AS
SELECT
    c.customer_id,
    c.first_name,
    c.last_name,
    COUNT(o.order_id) AS total_orders
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE c.status = 'active'
GROUP BY c.customer_id, c.first_name, c.last_name;

-- Query the view just like a table
SELECT * FROM vw_active_customers WHERE total_orders > 5;
```

## 10. Common Mistakes

### 10.1 Solving Everything with Subqueries

**Mistake:** Nesting subquery inside subquery inside subquery, creating a deeply nested, unreadable block of SQL.

```sql
-- ❌ Bad: Triple-nested subquery — nearly impossible to debug
SELECT customer_id, total_revenue
FROM (
    SELECT customer_id, SUM(order_total) AS total_revenue
    FROM (
        SELECT o.customer_id, o.order_id,
               (SELECT SUM(li.unit_price * li.quantity)
                FROM line_items li WHERE li.order_id = o.order_id) AS order_total
        FROM orders o
        WHERE o.status = 'completed'
    ) order_totals
    GROUP BY customer_id
) revenue_summary
WHERE total_revenue > 10000;

-- ✅ Better: Refactor with a CTE for each logical step
WITH line_item_totals AS (
    SELECT order_id, SUM(unit_price * quantity) AS order_total
    FROM line_items
    GROUP BY order_id
),
completed_orders AS (
    SELECT o.customer_id, o.order_id, lit.order_total
    FROM orders o
    JOIN line_item_totals lit ON o.order_id = lit.order_id
    WHERE o.status = 'completed'
),
customer_revenue AS (
    SELECT customer_id, SUM(order_total) AS total_revenue
    FROM completed_orders
    GROUP BY customer_id
)
SELECT customer_id, total_revenue
FROM customer_revenue
WHERE total_revenue > 10000;
```

### 10.2 Using a View When a Temporary Table Is Needed

**Mistake:** Using a view for a large, complex transformation that runs in an ETL job — the view re-executes the full logic on every call.

```sql
-- ❌ Bad: View re-runs an expensive 50M-row join every time it is queried
CREATE VIEW vw_etl_transformed_sales AS
SELECT ... FROM fact_sales JOIN dim_date JOIN dim_product JOIN ...
-- 50M rows, complex logic, called 10 times by downstream queries = 10× the work

-- ✅ Better: Materialize once into a temp table, query multiple times
SELECT ... INTO #transformed_sales FROM fact_sales JOIN ...;
-- 50M rows executed ONCE; downstream queries read the cached result
```

### 10.3 Forgetting to Drop Temporary Tables

**Mistake:** Creating temporary tables inside long-running sessions without dropping them, causing tempdb to grow unexpectedly.

```sql
-- ✅ Always clean up temp tables when done
DROP TABLE IF EXISTS #temp_active_customers;
CREATE TABLE #temp_active_customers AS ...;
-- ... use the table ...
DROP TABLE IF EXISTS #temp_active_customers;  -- explicit cleanup
```

### 10.4 Treating the Estimated Plan as Truth

**Mistake:** Assuming the optimizer's estimated row counts about a view or CTE are accurate without checking the Actual Execution Plan.

Views and CTEs are not always materialized — the optimizer may fold them back into the main query. The actual behavior depends on the database engine and query structure. Always verify with the Actual Execution Plan.

### 10.5 Ignoring the System Catalog

**Mistake:** Asking colleagues "what columns does this table have?" instead of querying `INFORMATION_SCHEMA`.

```sql
-- ✅ Self-serve: explore your database's structure without asking anyone
SELECT COLUMN_NAME, DATA_TYPE, IS_NULLABLE
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_NAME = 'orders'
ORDER BY ORDINAL_POSITION;
```

## 11. Real-World Analytics Use Cases

### 11.1 Data Warehouse ETL Pipeline (Temp Tables + CTAS)

**Scenario:** A data engineer builds a nightly ETL process to load and transform 10M rows of sales data into the warehouse.

```sql
-- Step 1: Extract raw data into a temp table (session-scoped staging area)
SELECT
    s.sale_id,
    s.customer_id,
    s.product_id,
    s.sale_date,
    s.quantity,
    s.unit_price,
    s.quantity * s.unit_price AS gross_amount
INTO #raw_sales
FROM source_system.dbo.sales s
WHERE s.sale_date >= CAST(GETDATE() AS DATE);   -- today's records only

-- Step 2: Transform — enrich and clean
SELECT
    r.sale_id,
    c.region,
    p.category,
    r.sale_date,
    r.gross_amount,
    r.gross_amount * 0.85 AS net_amount          -- apply discount factor
INTO #transformed_sales
FROM #raw_sales r
JOIN dim_customers c ON r.customer_id = c.customer_id
JOIN dim_products  p ON r.product_id  = p.product_id
WHERE r.gross_amount > 0;                        -- remove zero-value rows

-- Step 3: Load into warehouse (CTAS-style permanent table)
INSERT INTO fact_sales_daily
SELECT * FROM #transformed_sales;

-- Cleanup
DROP TABLE IF EXISTS #raw_sales;
DROP TABLE IF EXISTS #transformed_sales;
```

### 11.2 Analyst Self-Service with Views (Hiding Model Complexity)

**Scenario:** A database team creates views so financial analysts can query clean, pre-joined data without understanding the underlying normalized schema.

```sql
-- DBA creates a view that hides 5-table join complexity
CREATE VIEW vw_sales_summary AS
SELECT
    o.order_id,
    o.order_date,
    c.first_name + ' ' + c.last_name   AS customer_name,
    c.region,
    p.english_name                      AS product_name,
    cat.category_name,
    o.quantity,
    o.unit_price,
    o.quantity * o.unit_price           AS revenue
FROM orders o
JOIN customers c   ON o.customer_id  = c.customer_id
JOIN products p    ON o.product_id   = p.product_id
JOIN categories cat ON p.category_id = cat.category_id
JOIN regions r     ON c.region_id    = r.region_id;

-- Analyst writes a simple query — no join knowledge required
SELECT
    region,
    category_name,
    SUM(revenue) AS total_revenue
FROM vw_sales_summary
WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31'
GROUP BY region, category_name
ORDER BY total_revenue DESC;
```

### 11.3 Consistent KPI Definitions Across Teams (Views + CTEs)

**Scenario:** Three teams define "active customer" differently. The BI team creates a canonical view that all teams must use.

```sql
-- Single source of truth: active customer definition
CREATE VIEW vw_active_customers AS
SELECT
    c.customer_id,
    c.first_name,
    c.last_name,
    c.email,
    MAX(o.order_date)   AS last_order_date,
    COUNT(o.order_id)   AS total_orders,
    SUM(o.total_amount) AS lifetime_value
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_date >= DATEADD(DAY, -90, GETDATE())  -- canonical: 90-day window
  AND o.status = 'completed'
GROUP BY c.customer_id, c.first_name, c.last_name, c.email;

-- All teams now query the same definition
-- Financial Analyst:
SELECT region, COUNT(*) FROM vw_active_customers GROUP BY region;

-- Data Scientist:
SELECT customer_id, lifetime_value FROM vw_active_customers WHERE lifetime_value > 1000;

-- Marketing:
SELECT email FROM vw_active_customers WHERE total_orders >= 3;
```

## 12. Best Practices

1. **Use Views to hide data model complexity** — analysts should query clean, pre-joined views rather than raw tables with cryptic column names and complex relationships.

2. **Use Views for access control** — grant SELECT on the view, not on the underlying tables. This protects sensitive columns and rows at the database level.

3. **Use Temporary Tables for multi-step ETL** — materialize expensive intermediate results once, then query them many times rather than re-running a complex view repeatedly.

4. **Use CTEs for readability** — when a query has 3+ logical steps, a CTE per step is clearer than nested subqueries. Name each CTE to document its purpose.

5. **Use CTAS to persist transformed data** — when the same transformation is needed by many downstream consumers over days or weeks, persist it as a real table rather than recomputing it on every query.

6. **Query INFORMATION_SCHEMA before asking colleagues** — it is the fastest way to understand any unfamiliar database schema.

7. **Always clean up temporary tables** — use `DROP TABLE IF EXISTS` before creation (idempotent) and after use to prevent tempdb bloat.

8. **Do not treat views as performance tools** — a view does not cache data. It re-executes its full query on every call. For performance, use Temp Tables, CTAS, or Materialized Views (where supported).

9. **Define shared business metrics in Views** — "active customer," "completed order," "net revenue" should have one canonical definition accessible to all teams.

10. **Understand where each technique stores its data** — inline (Subquery, CTE), System Catalog + runtime (View), tempdb (Temp Table), User Storage (CTAS). Storage location determines performance, persistence, and shareability.
