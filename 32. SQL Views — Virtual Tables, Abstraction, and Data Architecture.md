## 1. Database Structure & DDL Recap

A view is not a query you write ad hoc — it is a **named database object** that lives inside the database alongside tables. To understand views, you need to understand where they sit in the database hierarchy.

### 1.1 Database Hierarchy

```
SQL Server  (top-level container; manages all databases)
└── Database  (e.g., SalesDB — a collection of related data)
    └── Schema  (logical grouping of related objects, e.g., "sales", "hr")
        ├── Tables   (physical storage — where data actually lives)
        └── Views    (virtual tables — no data, just a stored query)
            └── Columns  (name + data type; defined in both tables and views)
```

A **schema** is the namespace that groups related objects. For example, a `sales` schema can contain `sales.customers`, `sales.orders`, and `sales.v_monthly_summary` (a view) — all logically grouped under one roof.

> If you create an object without specifying a schema, most databases assign it to a default schema (`dbo` in MS SQL Server, `public` in PostgreSQL, the current user schema in MySQL).

### 1.2 DDL — Data Definition Language

DDL is the set of SQL commands used to **define and manage database structure** (as opposed to DML, which manipulates data).

| DDL Command | Purpose |
|---|---|
| `CREATE` | Create a new database object (table, view, schema, index…) |
| `ALTER` | Modify the structure of an existing object |
| `DROP` | Permanently remove a database object |

These are the exact commands used to create, modify, and delete views.

## 2. The Three-Level Database Architecture

Before writing views, it helps to understand *why* they exist architecturally. Databases are designed around three levels of abstraction:

```
┌─────────────────────────────────────────────┐
│  View Level  (External Layer)               │  ← End users & applications
│  Friendly views, role-specific subsets      │
├─────────────────────────────────────────────┤
│  Logical Level  (Conceptual Layer)          │  ← Developers & data engineers
│  Tables, relationships, indexes, procedures │
├─────────────────────────────────────────────┤
│  Physical Level  (Internal Layer)           │  ← DBAs
│  Data files, partitions, blocks, caches     │
└─────────────────────────────────────────────┘
```

| Level | Who Accesses It | What They Deal With | Complexity |
|---|---|---|---|
| Physical | Database Administrators | Files, partitions, logs, caches, backups | Highest |
| Logical | Developers / Data Engineers | Tables, relationships, indexes, procedures | Medium |
| View | End Users / BI Tools / Applications | Views — only what they need to see | Lowest |

> **Why this matters for views:** Views are the mechanism that implements the View Level. They decouple end users from the physical and logical complexity of the database, which is precisely what makes them so powerful.

## 3. What Is a View?

**Definition:** A view is a **virtual table in SQL** that is based on the result of a stored query, without actually storing any data in the database.

More precisely: a view is a **persisted SQL query** stored in the database catalog. When queried, it executes that stored query on the fly and returns results — but nothing is physically stored.

```
Physical Table:  stores data permanently on disk
View:            stores a query permanently in the catalog; data is fetched on demand
```

### How a View Works

```
User writes:     SELECT * FROM sales.v_monthly_summary;
                          ↓
Database checks: Is this a table or a view?
                          ↓  (it's a view)
Database fetches: The stored SQL query from the catalog
                          ↓
Database executes: The view's query against the underlying physical tables
                          ↓
Result returned to the user
```

The view acts as an **abstraction layer between the user and the physical data**. The user queries the view; the view secretly queries the table.

---

## 4. Views vs. Tables

| Dimension | Physical Table | View |
|---|---|---|
| Stores data | ✅ Yes — permanently on disk | ❌ No — virtual; data comes from underlying tables |
| Performance | Faster (direct disk read) | Slower (executes the view's query first, then the user's query) |
| Flexibility | Hard to change (schema migrations required) | Easy to change (just update the query) |
| Maintenance | High effort for changes | Low effort — modify the `CREATE VIEW` statement |
| Read / Write | Both read and write | Read-only by default |
| Dropping | Destroys actual data | Destroys only query + metadata; no user data lost |
| Schema changes | Affect all downstream consumers immediately | Shield consumers from schema changes |

> **Key insight:** Dropping a view does **not** delete any data. It only removes the stored query and its metadata from the catalog. The underlying tables are completely unaffected.

---

## 5. Views vs. CTEs

Both views and CTEs produce named intermediate result sets. The key difference is **scope and persistence**.

| Dimension | CTE | View |
|---|---|---|
| Persistence | Temporary — lives only for one query | Permanent — stored in the database catalog |
| Scope | Single query only | Available to all queries, all users, all sessions |
| Reusability | Within one query | Across multiple queries and users |
| Maintenance | Zero — cleaned up automatically | Requires `CREATE`, `ALTER`/`DROP` lifecycle management |
| Best for | Reducing redundancy *within* a complex query | Reducing redundancy *across* many queries and users |
| Portability | Inline, no object created | Named object in the database |

**Rule of thumb:**
- Use a **CTE** when the logic is only relevant to the current query.
- Use a **view** when the logic is important enough to be shared, reused, or persisted for multiple users and queries.

## 6. How the Database Executes a View

### Creating a View (DDL Phase)

```
Developer runs: CREATE VIEW sales.v_top_customers AS SELECT ...
                          ↓
Database engine stores in the system catalog:
  - Metadata about the view (name, schema, column definitions)
  - The SQL query text itself
                          ↓
No data is written anywhere. No user-data tables created.
```

### Querying a View (DML Phase)

```
User runs:    SELECT * FROM sales.v_top_customers WHERE country = 'DE';
                          ↓
Database recognises v_top_customers as a view (not a table)
                          ↓
Step 1: Fetch the stored SQL from the system catalog
Step 2: Execute the view's SQL against the physical tables (e.g., orders, customers)
Step 3: Apply the user's filters (WHERE country = 'DE') on top
Step 4: Return the combined result to the user
```

### Dropping a View (DDL Phase)

```
Developer runs: DROP VIEW sales.v_top_customers;
                          ↓
Database removes from the system catalog:
  - The metadata
  - The stored SQL query
                          ↓
The underlying physical tables are completely untouched.
No user data is lost.
```

### 7.1 CREATE VIEW

```sql
-- Syntax
CREATE VIEW schema_name.view_name AS
(
    SELECT ...
    FROM   ...
    WHERE  ...
);

-- Example: monthly aggregation summary
CREATE VIEW sales.v_monthly_summary AS
(
    SELECT   DATETRUNC('month', order_date) AS order_month,
             SUM(sales)    AS total_sales,
             COUNT(order_id) AS total_orders,
             SUM(quantity) AS total_quantities
    FROM     sales.orders
    GROUP BY DATETRUNC('month', order_date)
);
```

> Always prefix the view name with its schema (`sales.v_monthly_summary`) to avoid objects landing in the default `dbo` / `public` schema unintentionally.

### 7.2 Querying a View

Once created, a view is queried exactly like a table:

```sql
SELECT * FROM sales.v_monthly_summary
ORDER BY order_month;
```

You can also build on top of it with window functions, filters, joins — anything you would do with a table:

```sql
-- Running total on top of the view
SELECT order_month,
       total_sales,
       SUM(total_sales) OVER (ORDER BY order_month) AS running_total
FROM   sales.v_monthly_summary;
```

### 7.3 DROP VIEW

```sql
-- Drop a single view
DROP VIEW sales.v_monthly_summary;

-- Drop only if it exists (avoids error if missing)
DROP VIEW IF EXISTS sales.v_monthly_summary;   -- MySQL, PostgreSQL
```

### 7.4 Updating a View

The approach differs across databases. See the full dialect table in [Section 14](#14-dialect-comparison-mysql-vs-ms-sql-server-vs-postgresql).

```sql
-- PostgreSQL & MySQL: CREATE OR REPLACE (atomic, no drop needed)
CREATE OR REPLACE VIEW sales.v_monthly_summary AS
(
    SELECT   DATETRUNC('month', order_date) AS order_month,
             SUM(sales) AS total_sales
    FROM     sales.orders
    GROUP BY DATETRUNC('month', order_date)
);

-- MS SQL Server approach 1: DROP then CREATE
DROP VIEW sales.v_monthly_summary;
CREATE VIEW sales.v_monthly_summary AS ( ... );

-- MS SQL Server approach 2: T-SQL conditional drop-and-recreate (preferred)
IF OBJECT_ID('sales.v_monthly_summary', 'V') IS NOT NULL
    DROP VIEW sales.v_monthly_summary;
GO
CREATE VIEW sales.v_monthly_summary AS
(
    SELECT   DATETRUNC('month', order_date) AS order_month,
             SUM(sales) AS total_sales
    FROM     sales.orders
    GROUP BY DATETRUNC('month', order_date)
);
GO

-- MS SQL Server approach 3: ALTER VIEW (modifies in place)
ALTER VIEW sales.v_monthly_summary AS
(
    SELECT   DATETRUNC('month', order_date) AS order_month,
             SUM(sales) AS total_sales
    FROM     sales.orders
    GROUP BY DATETRUNC('month', order_date)
);
```

## 8. Use Case 1 — Central Business Logic & Reusability

### The Problem

In any analytics project, multiple users need the same foundational data preparation — joining tables, aggregating by period, cleaning nulls. Without views, each analyst copies the same CTE or subquery into every script they write. This creates redundancy, inconsistency, and wasted time.

```
Financial Analyst:   CTE → join + aggregate → then RANK
Budget Analyst:      CTE → join + aggregate → then MAX/MIN
Risk Analyst:        CTE → join + aggregate → then compare

All three are repeating the same first step.
```

### The Solution: Store the Logic as a View

Define the common logic once in a view. Every analyst skips straight to their specific analysis.

```sql
-- Define the shared foundation ONCE as a view
CREATE VIEW sales.v_monthly_summary AS
(
    SELECT   DATETRUNC('month', order_date) AS order_month,
             SUM(sales)       AS total_sales,
             COUNT(order_id)  AS total_orders,
             SUM(quantity)    AS total_quantities
    FROM     sales.orders
    GROUP BY DATETRUNC('month', order_date)
);
```

```sql
-- Financial Analyst — ranking months by revenue
SELECT order_month,
       total_sales,
       RANK() OVER (ORDER BY total_sales DESC) AS revenue_rank
FROM   sales.v_monthly_summary;

-- Budget Analyst — peaks and troughs
SELECT MAX(total_sales) AS peak_month_sales,
       MIN(total_sales) AS lowest_month_sales
FROM   sales.v_monthly_summary;

-- Risk Analyst — month-over-month change
SELECT order_month,
       total_sales,
       LAG(total_sales) OVER (ORDER BY order_month)              AS prior_month,
       total_sales - LAG(total_sales) OVER (ORDER BY order_month) AS mom_change
FROM   sales.v_monthly_summary;
```

Each analyst writes only what is unique to their task. The shared preparation logic lives once, centrally, in the view.

## 9. Use Case 2 — Hiding Complexity & Improving Abstraction

### The Problem

Real-world databases are built for applications, not analysts. They often have cryptic column names, normalised tables that require multi-table joins, and relationships that are impossible to understand without documentation. Forcing end users to navigate this is a nightmare.

### The Solution: A Friendly Combined View

Create a single, well-named view that joins all relevant tables and exposes only meaningful columns with human-readable names.

```sql
CREATE VIEW sales.v_order_details AS
(
    SELECT
        -- From orders
        o.order_id,
        o.order_date,
        o.sales,
        o.quantity,

        -- From products (replace product_id with actual name)
        p.product,
        p.category,

        -- From customers (combine first + last name, show country)
        CONCAT(COALESCE(c.first_name, ''), ' ', COALESCE(c.last_name, '')) AS customer_name,
        c.country AS customer_country,

        -- From employees (replace salesperson_id with name + department)
        CONCAT(e.first_name, ' ', e.last_name) AS sales_name,
        e.department

    FROM       sales.orders    o
    LEFT JOIN  sales.products  p ON p.product_id    = o.product_id
    LEFT JOIN  sales.customers c ON c.customer_id   = o.customer_id
    LEFT JOIN  sales.employees e ON e.employee_id   = o.salesperson_id
);
```

#### Output Preview

| order_id | order_date | sales | quantity | product | category | customer_name | customer_country | sales_name | department |
|---|---|---|---|---|---|---|---|---|---|
| 101 | 2024-01-05 | 999.00 | 1 | Laptop | Electronics | Alice Brown | Germany | Frank Turner | Sales |
| 102 | 2024-01-10 | 250.00 | 2 | Desk | Furniture | Bob Smith | USA | Frank Turner | Sales |
| 103 | 2024-02-01 | 1998.00 | 2 | Laptop | Electronics | Carol Jones | France | Grace Hall | Sales |

Users see clean, meaningful data. No joins needed. No cryptic IDs. The physical model complexity is fully hidden.

## 10. Use Case 3 — Data Security (Row & Column Level)

### The Problem

Different users should see different subsets of the data. A student should not see USA sales data. An analyst should not see a sensitive salary column. Giving everyone direct table access is not secure.

### Column-Level Security

Exclude sensitive columns by simply not including them in the view's SELECT list.

```sql
-- Full access view (for managers)
CREATE VIEW sales.v_orders_managers AS
(
    SELECT order_id, customer_id, product_id,
           order_date, sales, quantity, salesperson_id
    FROM   sales.orders
);

-- Restricted view (for analysts — sales column excluded)
CREATE VIEW sales.v_orders_analysts AS
(
    SELECT order_id, customer_id, product_id,
           order_date, quantity              -- 'sales' intentionally omitted
    FROM   sales.orders
);
```

### Row-Level Security

Filter rows using a `WHERE` clause to restrict which records a user can see.

```sql
-- EU Sales Team view: excludes all USA orders (row-level security)
CREATE VIEW sales.v_order_details_eu AS
(
    SELECT
        o.order_id,
        o.order_date,
        o.sales,
        o.quantity,
        p.product,
        p.category,
        CONCAT(COALESCE(c.first_name, ''), ' ', COALESCE(c.last_name, '')) AS customer_name,
        c.country AS customer_country,
        CONCAT(e.first_name, ' ', e.last_name) AS sales_name
        -- 'department' column also omitted (column-level security)
    FROM       sales.orders    o
    LEFT JOIN  sales.products  p ON p.product_id  = o.product_id
    LEFT JOIN  sales.customers c ON c.customer_id = o.customer_id
    LEFT JOIN  sales.employees e ON e.employee_id = o.salesperson_id
    WHERE      c.country <> 'USA'            -- row-level filter
);
```

#### Output Preview

| order_id | customer_name | customer_country | product | sales |
|---|---|---|---|---|
| 101 | Alice Brown | Germany | Laptop | 999.00 |
| 103 | Carol Jones | France | Laptop | 1998.00 |
| 105 | Alice Brown | Germany | Desk | 500.00 |

All USA rows are absent. The `department` column is hidden. Users with access to this view cannot see anything outside their permitted scope.

### Security Architecture Pattern

```
Physical Table: sales.orders  (no direct user access granted)
        │
        ├── sales.v_orders_managers    → Full data access
        ├── sales.v_orders_analysts    → Column restriction (no sales $)
        ├── sales.v_order_details_eu   → Row restriction (no USA)
        └── sales.v_orders_students    → Row + column restriction
```

Grant each role access only to their view, never to the underlying table.

## 11. Use Case 4 — Flexibility & Database Refactoring

### The Problem

Without views, any change to a physical table — renaming a column, splitting a table, changing a data type — immediately breaks all user queries. This creates pressure to never change anything, or to coordinate changes with dozens of people.

### The Solution: Views as a Stable Contract

Offer users a view instead of direct table access. You are then free to refactor the physical model as long as the view continues to produce the same output.

```sql
-- Original table has a column called 'sales'
-- You decide to rename it to 'revenue' in the physical table.
-- Update the view to map the new name back to the old interface:

ALTER VIEW sales.v_order_details AS
(
    SELECT o.order_id,
           o.revenue AS sales,    -- renamed in table, aliased back in view
           ...
    FROM   sales.orders o
    ...
);
-- Users querying sales.v_order_details still see a column called 'sales'.
-- They notice nothing.
```

Common refactoring scenarios handled transparently through views:

| Physical Change | View Fix |
|---|---|
| Table split into two | Add a JOIN in the view |
| Table renamed | Update the FROM clause in the view |
| Column renamed | Add an alias in the SELECT list |
| Column removed | Supply a default value or NULL in the SELECT |
| Two tables merged | Change the view to query the merged table |

## 12. Use Case 5 — Multi-Language Data Models

Views can expose the same underlying data with translated object and column names, providing a localised experience for international teams.

```sql
-- German-language view of the orders table
CREATE VIEW sales.bestellung AS   -- "bestellung" = order in German
(
    SELECT
        order_id        AS auftragsnummer,    -- order number
        order_date      AS bestelldatum,      -- order date
        sales           AS umsatz,            -- revenue
        quantity        AS menge              -- quantity
    FROM   sales.orders
);

-- Hindi-language view
CREATE VIEW sales.v_aadesh AS    -- "aadesh" ≈ order in Hindi
(
    SELECT
        order_id    AS kramank,
        order_date  AS tithi,
        sales       AS bikri,
        quantity    AS matra
    FROM   sales.orders
);
```

Each team interacts with the same physical data in their own language, without any data duplication.

## 13. Use Case 6 — Virtual Data Marts in a Data Warehouse

### Architecture Overview

A classical data warehouse with physical data marts looks like this:

```
Source Systems → ETL → Data Warehouse (physical tables) → ETL → Data Marts (physical tables) → BI / Reporting
```

The problem: copying data from the warehouse into marts requires ETL pipelines, adds latency, creates consistency risks (two copies of the truth), and increases maintenance burden.

### Virtual Data Mart Pattern (Views)

Replace physical mart tables with views that sit directly on top of the warehouse:

```
Source Systems → ETL → Data Warehouse (physical tables)
                                  │
                    ┌─────────────┼──────────────┐
                    ▼             ▼               ▼
               View: Sales   View: Finance   View: HR
               Data Mart     Data Mart       Data Mart
                    │             │               │
                    ▼             ▼               ▼
              Power BI      Excel Reports    Custom App
```

### Why Virtual Marts Are Better Than Physical Marts

| Concern | Physical Mart (Tables) | Virtual Mart (Views) |
|---|---|---|
| Data freshness | Delayed — only as fresh as the last ETL run | Real-time — always reflects the warehouse |
| Single point of truth | ❌ Data duplicated across warehouse + mart | ✅ Data lives only in the warehouse |
| ETL complexity | High — requires load pipelines to marts | None — no data movement |
| Consistency risk | High — warehouse and mart can diverge | None — views always query the warehouse |
| Flexibility | Low — schema changes require re-ETL | High — change the view query |
| Maintenance | High | Low |

```sql
-- Example: Sales Data Mart as a view on top of the warehouse
CREATE VIEW datamart.v_sales_mart AS
(
    SELECT
        f.order_date,
        d_cust.customer_name,
        d_cust.country,
        d_prod.product,
        d_prod.category,
        f.sales,
        f.quantity
    FROM       warehouse.fact_orders     f
    LEFT JOIN  warehouse.dim_customers   d_cust ON d_cust.customer_key = f.customer_key
    LEFT JOIN  warehouse.dim_products    d_prod ON d_prod.product_key  = f.product_key
    WHERE      f.is_deleted = 0
);
```

BI tools like Power BI connect directly to `datamart.v_sales_mart` and always see live, consistent data from the warehouse.

## 14. Dialect Comparison: MySQL vs. MS SQL Server vs. PostgreSQL

### 14.1 Basic CREATE VIEW Syntax

All three dialects share the same core syntax:

```sql
CREATE VIEW schema_name.view_name AS
(
    SELECT ...
);
```

### 14.2 CREATE OR REPLACE vs. ALTER VIEW

| Operation | MySQL 8.0+ | MS SQL Server | PostgreSQL |
|---|---|---|---|
| Create new view | `CREATE VIEW` | `CREATE VIEW` | `CREATE VIEW` |
| Replace existing view atomically | `CREATE OR REPLACE VIEW` | ❌ Not supported | `CREATE OR REPLACE VIEW` |
| Modify existing view | `CREATE OR REPLACE VIEW` | `ALTER VIEW` | `CREATE OR REPLACE VIEW` |
| Drop if exists | `DROP VIEW IF EXISTS` | `DROP VIEW IF EXISTS` (2016+) | `DROP VIEW IF EXISTS` |
| Conditional drop + recreate | N/A | T-SQL `IF OBJECT_ID(...) IS NOT NULL` | N/A |

```sql
-- MySQL & PostgreSQL — replace in one statement
CREATE OR REPLACE VIEW sales.v_monthly_summary AS
( SELECT ... );

-- MS SQL Server — ALTER VIEW (modifies in place; view must already exist)
ALTER VIEW sales.v_monthly_summary AS
( SELECT ... );

-- MS SQL Server — T-SQL conditional pattern (works whether view exists or not)
IF OBJECT_ID('sales.v_monthly_summary', 'V') IS NOT NULL
    DROP VIEW sales.v_monthly_summary;
GO
CREATE VIEW sales.v_monthly_summary AS
( SELECT ... );
GO
```

### 14.3 Schema Handling

| Feature | MySQL | MS SQL Server | PostgreSQL |
|---|---|---|---|
| Default schema if none specified | Current database / user | `dbo` | `public` |
| Specify schema in view name | `db_name.view_name` | `schema.view_name` | `schema.view_name` |
| Best practice | Prefix with schema | Prefix with schema | Prefix with schema |

### 14.4 View Materialisation

| Type | MySQL | MS SQL Server | PostgreSQL |
|---|---|---|---|
| Standard (non-materialised) view | ✅ | ✅ | ✅ |
| Materialised view (stores data) | ❌ Not natively supported | ✅ Via indexed views | ✅ `CREATE MATERIALIZED VIEW` |
| Refresh materialised view | N/A | Auto (with `SCHEMABINDING` + unique clustered index) | `REFRESH MATERIALIZED VIEW view_name;` |

```sql
-- PostgreSQL: create a materialised view
CREATE MATERIALIZED VIEW sales.mv_monthly_summary AS
(
    SELECT   DATE_TRUNC('month', order_date) AS order_month,
             SUM(sales)  AS total_sales,
             COUNT(*)    AS total_orders
    FROM     sales.orders
    GROUP BY DATE_TRUNC('month', order_date)
);

-- Refresh it manually
REFRESH MATERIALIZED VIEW sales.mv_monthly_summary;

-- MS SQL Server: indexed view (automatically maintained)
CREATE VIEW sales.v_monthly_summary
WITH SCHEMABINDING AS
(
    SELECT   DATETRUNC(month, order_date) AS order_month,
             SUM(sales)     AS total_sales,
             COUNT_BIG(*)   AS total_orders
    FROM     sales.orders
    GROUP BY DATETRUNC(month, order_date)
);
CREATE UNIQUE CLUSTERED INDEX ix_v_monthly_summary
    ON sales.v_monthly_summary (order_month);
```

### 14.5 Date Truncation in Views

A common operation in views is aggregating by month — the syntax differs:

```sql
-- MySQL
DATE_FORMAT(order_date, '%Y-%m-01')   -- truncate to month start

-- MS SQL Server
DATETRUNC(month, order_date)           -- SQL Server 2022+
DATEFROMPARTS(YEAR(order_date), MONTH(order_date), 1)  -- older versions

-- PostgreSQL
DATE_TRUNC('month', order_date)
```

### 14.6 NULL Handling in Concatenation

When building `customer_name` by concatenating first and last name:

```sql
-- MySQL (CONCAT ignores NULLs in CONCAT_WS, but not in plain CONCAT)
CONCAT_WS(' ', first_name, last_name) AS customer_name

-- MS SQL Server
CONCAT(ISNULL(first_name, ''), ' ', ISNULL(last_name, '')) AS customer_name

-- PostgreSQL
CONCAT(COALESCE(first_name, ''), ' ', COALESCE(last_name, '')) AS customer_name
```

## 15. Common Mistakes

### 15.1 Forgetting the Schema Prefix

```sql
-- ❌ Creates view in the default schema (dbo / public)
CREATE VIEW v_monthly_summary AS ( ... );

-- ✅ Creates view in the intended schema
CREATE VIEW sales.v_monthly_summary AS ( ... );
```

### 15.2 Trying to Write Data Through a View

```sql
-- ❌ Views are read-only
INSERT INTO sales.v_monthly_summary (order_month, total_sales)
VALUES ('2024-04-01', 500);
-- Error: cannot insert into a view
```

### 15.3 Using ORDER BY Inside a View Definition

```sql
-- ❌ ORDER BY is not guaranteed in view definitions (and is often disallowed)
CREATE VIEW sales.v_top_customers AS
(
    SELECT customer_id, SUM(sales) AS total
    FROM   sales.orders
    GROUP  BY customer_id
    ORDER  BY total DESC   -- not allowed or not meaningful
);

-- ✅ Sort in the consuming query
SELECT * FROM sales.v_top_customers ORDER BY total DESC;
```

### 15.4 Forgetting `COALESCE` / `ISNULL` for NULLs in Concatenation

```sql
-- ❌ If last_name is NULL, the whole expression becomes NULL
SELECT first_name + ' ' + last_name AS customer_name ...

-- ✅ Handle NULLs explicitly
SELECT CONCAT(COALESCE(first_name,''), ' ', COALESCE(last_name,'')) AS customer_name ...
```

### 15.5 Dropping a View Thinking It Deletes Data

```sql
-- This is SAFE — no user data is lost
DROP VIEW sales.v_order_details;
-- Only the stored query and metadata are removed.
-- The underlying tables (orders, customers, etc.) are completely untouched.
```

### 15.6 Using `CREATE VIEW` When the View Already Exists (MS SQL Server)

```sql
-- ❌ Error if the view already exists
CREATE VIEW sales.v_monthly_summary AS ( ... );

-- ✅ Use ALTER VIEW to modify an existing view
ALTER VIEW sales.v_monthly_summary AS ( ... );

-- ✅ Or use the conditional T-SQL pattern to handle both cases
IF OBJECT_ID('sales.v_monthly_summary', 'V') IS NOT NULL
    DROP VIEW sales.v_monthly_summary;
GO
CREATE VIEW sales.v_monthly_summary AS ( ... );
GO
```

### 15.7 Joining with INNER JOIN When LEFT JOIN Is Needed

```sql
-- ❌ INNER JOIN drops customers with no orders
CREATE VIEW sales.v_customer_summary AS
(
    SELECT c.customer_id, c.first_name, SUM(o.sales) AS total_sales
    FROM   sales.customers c
    INNER JOIN sales.orders o ON o.customer_id = c.customer_id
    GROUP  BY c.customer_id, c.first_name
);
-- Customers with zero orders disappear from the view entirely.

-- ✅ LEFT JOIN preserves all customers
CREATE VIEW sales.v_customer_summary AS
(
    SELECT c.customer_id, c.first_name, SUM(o.sales) AS total_sales
    FROM   sales.customers c
    LEFT JOIN sales.orders o ON o.customer_id = c.customer_id
    GROUP  BY c.customer_id, c.first_name
);
```

## 16. Best Practices

**Name views consistently and descriptively.** A prefix convention like `v_` or `vw_` (e.g., `v_monthly_summary`, `vw_eu_orders`) makes views immediately distinguishable from physical tables in any schema browser.

**Always specify the schema.** Never rely on default schema assignment. `CREATE VIEW sales.v_name` is always better than `CREATE VIEW v_name`.

**Use `LEFT JOIN` in views by default** unless you specifically intend to exclude non-matching rows. Views often serve as general-purpose data products; dropping rows unexpectedly is a common source of bugs.

**Never store sensitive logic only inside a view.** Document the view's purpose, source tables, and any security restrictions in a comment block at the top of the `CREATE VIEW` statement.

**Limit view complexity.** A view with 10 joins is hard to debug and maintain. Consider splitting it into layered views (a foundation view joined by a business view) rather than building one massive view.

**Do not overuse views.** Not every query deserves a view. Reserve views for logic that is genuinely reused across multiple users or queries, or for security and abstraction requirements.

**Test the view's query before creating the object.** Run the `SELECT` statement in isolation to verify results, then wrap it in the `CREATE VIEW` DDL.

**In a data warehouse, prefer views for data marts.** Use views over physical tables in the mart layer to maintain a single source of truth, avoid ETL duplication, and ensure real-time consistency.

**Grant permissions to views, not tables.** Users should never need direct table access if they are working through views. This keeps your security model clean and easy to audit.

## 17. Real-World Analytics Patterns

### Pattern 1: Running Total View

```sql
CREATE VIEW sales.v_running_totals AS
(
    SELECT
        order_month,
        total_sales,
        SUM(total_sales) OVER (ORDER BY order_month) AS cumulative_sales
    FROM (
        SELECT   DATETRUNC('month', order_date) AS order_month,
                 SUM(sales) AS total_sales
        FROM     sales.orders
        GROUP BY DATETRUNC('month', order_date)
    ) monthly
);
```

### Pattern 2: Country-Level Aggregation for BI

```sql
CREATE VIEW sales.v_sales_by_country AS
(
    SELECT
        c.country,
        COUNT(DISTINCT o.order_id)   AS order_count,
        COUNT(DISTINCT o.customer_id) AS unique_customers,
        SUM(o.sales)                 AS total_revenue,
        AVG(o.sales)                 AS avg_order_value
    FROM       sales.orders    o
    LEFT JOIN  sales.customers c ON c.customer_id = o.customer_id
    GROUP BY   c.country
);
```

### Pattern 3: Product Performance View

```sql
CREATE VIEW sales.v_product_performance AS
(
    SELECT
        p.product,
        p.category,
        COUNT(o.order_id)            AS times_ordered,
        SUM(o.quantity)              AS units_sold,
        SUM(o.sales)                 AS total_revenue,
        ROUND(SUM(o.sales) / NULLIF(SUM(o.quantity), 0), 2) AS avg_selling_price
    FROM       sales.orders   o
    LEFT JOIN  sales.products p ON p.product_id = o.product_id
    GROUP BY   p.product, p.category
);
```

### Pattern 4: Layered Views (Foundation → Business)

```sql
-- Layer 1: Foundation view (raw join, no business logic)
CREATE VIEW sales.v_base_order_facts AS
(
    SELECT o.order_id, o.order_date, o.sales, o.quantity,
           p.product, p.category,
           c.country, c.first_name, c.last_name
    FROM       sales.orders    o
    LEFT JOIN  sales.products  p ON p.product_id  = o.product_id
    LEFT JOIN  sales.customers c ON c.customer_id = o.customer_id
);

-- Layer 2: Business view (built on top of the foundation view)
CREATE VIEW sales.v_eu_high_value_orders AS
(
    SELECT *
    FROM   sales.v_base_order_facts
    WHERE  country <> 'USA'
    AND    sales   > 500
);
```

Layering views keeps each level simple and testable, while the combined result handles complex requirements.

## 18. Summary Table

| Dimension | Detail |
|---|---|
| **Definition** | A virtual table based on a stored SQL query; no data is physically stored |
| **Stores data?** | No — the query is stored; data is fetched from physical tables on demand |
| **Stores query?** | Yes — in the system catalog alongside metadata |
| **Scope** | Permanent — available to all queries, users, and sessions |
| **Performance** | Slower than direct table reads (view query + user query both execute) |
| **Read/Write** | Read-only by default |
| **Dropping** | Safe — no user data lost; only the query and metadata are removed |
| **Use Case 1** | Centralise repeated business logic so all users share one definition |
| **Use Case 2** | Hide physical model complexity; expose friendly, combined objects |
| **Use Case 3** | Column-level and row-level data security for different user roles |
| **Use Case 4** | Buffer users from physical schema changes; enable safe refactoring |
| **Use Case 5** | Provide multi-language versions of the same data model |
| **Use Case 6** | Virtual data marts in a data warehouse — single source of truth |
| **vs. CTE** | CTE = temporary, one query only; View = permanent, many queries/users |
| **vs. Table** | Table = faster, stores data; View = flexible, no data, easier to maintain |
| **MySQL DDL** | `CREATE OR REPLACE VIEW`; `DROP VIEW IF EXISTS` |
| **MS SQL Server DDL** | `CREATE VIEW` / `ALTER VIEW`; T-SQL conditional pattern for safe replace |
| **PostgreSQL DDL** | `CREATE OR REPLACE VIEW`; `CREATE MATERIALIZED VIEW` for stored results |
