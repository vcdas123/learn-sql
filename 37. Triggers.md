## 1. What Is a SQL Trigger?

A **trigger** is a special type of stored procedure that **automatically fires in response to a specific event** on a table — without any manual execution. You don't call a trigger; the database calls it for you whenever a defined event occurs.

### The Core Idea

Think of a trigger as an automated *"if this, then that"* rule attached to a table:

```
A change happens to a table
         │
         ▼
   Trigger detects the event
         │
         ▼
   Trigger automatically executes its SQL logic
```

The key word is **automatically**. Unlike a stored procedure — which you must manually call with `EXECUTE` — a trigger fires on its own the moment its conditions are met.

### What Events Can Fire a Trigger?

| Event | Description |
|---|---|
| `INSERT` | A new row is added to the table |
| `UPDATE` | One or more rows are modified |
| `DELETE` | One or more rows are removed |
| Schema changes | A table or view is created, altered, or dropped (DDL triggers) |
| Login | A user session is established (login triggers) |

---

## 2. Stored Procedure vs. Trigger

| Feature | Stored Procedure | Trigger |
|---|---|---|
| **Execution** | Manually called with `EXECUTE` / `CALL` | Automatically fired by an event |
| **Who calls it** | A user, application, or scheduled job | The database engine itself |
| **Attached to** | Nothing — called by name | A specific table and event |
| **Parameters** | Can accept input parameters | Cannot accept parameters |
| **Use case** | Multi-step business logic, reports, ETL | Automation, auditing, enforcement |

A stored procedure is like a vending machine — you press a button and it runs. A trigger is like a smoke detector — it fires automatically when it detects the event it was programmed to watch for.

---

## 3. Types of Triggers

### 3.1 By Event Type

**DML Triggers** — respond to data changes:
- `INSERT`
- `UPDATE`
- `DELETE`

**DDL Triggers** — respond to schema changes:
- `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`
- `CREATE VIEW`, `ALTER VIEW`

**Login Triggers** — respond to user login events (SQL Server only).

> This chapter focuses entirely on **DML triggers**, which are the most commonly used in analytics and data engineering.

### 3.2 By Timing (DML Triggers)

| Type | When It Fires | Keyword |
|---|---|---|
| **AFTER trigger** | After the event completes successfully | `AFTER` |
| **INSTEAD OF trigger** | In place of the event — replaces the actual operation | `INSTEAD OF` |

- Use **AFTER** when you want to react to a completed change (logging, cascading updates).
- Use **INSTEAD OF** when you want to intercept and control the operation before it happens (validation, conditional blocking).

## 5. Trigger Syntax

### 5.1 General Structure (SQL Server / T-SQL)

```sql
CREATE TRIGGER trigger_name
ON table_name
AFTER INSERT | UPDATE | DELETE        -- or: INSTEAD OF
AS
BEGIN
    -- SQL statements that execute when the trigger fires
    -- Can reference INSERTED and DELETED virtual tables
END;
```

### 5.2 Key Components

| Component | Description |
|---|---|
| `trigger_name` | Unique name for the trigger (convention: `trg_` prefix) |
| `ON table_name` | The table this trigger is attached to |
| `AFTER` / `INSTEAD OF` | When the trigger fires relative to the event |
| `INSERT` / `UPDATE` / `DELETE` | The event that fires the trigger |
| `BEGIN...END` | The body — SQL statements to execute |
| `INSERTED` | Virtual table holding the new rows (available in INSERT and UPDATE) |
| `DELETED` | Virtual table holding the old/removed rows (available in DELETE and UPDATE) |

### 5.3 Modifying and Dropping Triggers

```sql
-- Modify an existing trigger
ALTER TRIGGER trigger_name
ON table_name
AFTER INSERT
AS
BEGIN
    -- updated logic
END;

-- Remove a trigger permanently
DROP TRIGGER trigger_name;

-- Disable a trigger temporarily (SQL Server)
DISABLE TRIGGER trigger_name ON table_name;

-- Re-enable it
ENABLE TRIGGER trigger_name ON table_name;
```

## 6. Use Case: Audit Logging on Employee Inserts

### 6.1 The Problem

The `sales_employees` table holds sensitive HR data — new hires, salary changes, terminations. You need a **complete audit trail** that records:
- What changed
- When it changed
- Which employee record was affected

This is essential for compliance, security reviews, and incident investigations. Doing this manually (asking every developer to remember to insert a log row after every insert) is fragile and error-prone.

**Solution:** Attach a trigger to `sales_employees` that automatically writes to `sales_employee_logs` every time a new employee is inserted.

### 6.2 Step 1 — Create the Log Table

```sql
CREATE TABLE sales_employee_logs (
    log_id        INT           PRIMARY KEY IDENTITY(1,1),
    employee_id   INT,
    log_message   VARCHAR(255),
    log_date      DATETIME
);
```

### 6.3 Step 2 — Create the AFTER INSERT Trigger

```sql
CREATE TRIGGER trg_after_insert_employee
ON sales_employees
AFTER INSERT
AS
BEGIN
    INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
    SELECT
        employee_id,
        'New employee added. Employee ID: ' + CAST(employee_id AS VARCHAR),
        GETDATE()
    FROM INSERTED;   -- INSERTED is the virtual table holding the new rows
END;
```

> 📌 `INSERTED` is a special virtual table automatically created by SQL Server during trigger execution. It contains all rows that were just inserted into `sales_employees`. It only exists for the lifetime of the trigger.

### 6.4 Step 3 — Test It: Insert a New Employee

```sql
-- Check logs before insert (should be empty)
SELECT * FROM sales_employee_logs;
```

| log_id | employee_id | log_message | log_date |
|---|---|---|---|
| *(empty)* | | | |

```sql
-- Insert a new employee (this fires the trigger automatically)
INSERT INTO sales_employees (first_name, last_name, position, birth_date, salary, hierarchy)
VALUES ('Maria', 'Garcia', 'HR Manager', '1988-09-14', 62000, 2);
```

```sql
-- Check logs after insert (trigger should have written a record)
SELECT * FROM sales_employee_logs;
```

| log_id | employee_id | log_message | log_date |
|---|---|---|---|
| 1 | 6 | New employee added. Employee ID: 6 | 2023-06-01 09:14:32 |

The trigger fired automatically — no separate `INSERT INTO logs` was needed. Insert another employee and the log grows:

```sql
INSERT INTO sales_employees (first_name, last_name, position, birth_date, salary, hierarchy)
VALUES ('James', 'Lee', 'Sales Associate', '1997-03-05', 41000, 4);

SELECT * FROM sales_employee_logs;
```

| log_id | employee_id | log_message | log_date |
|---|---|---|---|
| 1 | 6 | New employee added. Employee ID: 6 | 2023-06-01 09:14:32 |
| 2 | 7 | New employee added. Employee ID: 7 | 2023-06-01 09:17:55 |

Every new insert is automatically tracked — zero manual effort.

## 7. The `INSERTED` and `DELETED` Virtual Tables

These are **temporary in-memory tables** created by the database engine during DML trigger execution. They hold the row data involved in the event.

| Virtual Table | Available In | Contains |
|---|---|---|
| `INSERTED` | `INSERT` trigger | The newly inserted rows |
| `INSERTED` | `UPDATE` trigger | The new (post-update) values of the changed rows |
| `DELETED` | `DELETE` trigger | The rows that were just deleted |
| `DELETED` | `UPDATE` trigger | The old (pre-update) values of the changed rows |

> ⚠️ These tables only exist **during the execution of the trigger**. You cannot query them outside the trigger body.

### Example: Capturing Both Old and New Values in an UPDATE Trigger

```sql
CREATE TRIGGER trg_after_update_salary
ON sales_employees
AFTER UPDATE
AS
BEGIN
    INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
    SELECT
        i.employee_id,
        'Salary changed from ' + CAST(d.salary AS VARCHAR)
            + ' to ' + CAST(i.salary AS VARCHAR),
        GETDATE()
    FROM INSERTED i
    JOIN DELETED  d ON i.employee_id = d.employee_id
    WHERE i.salary <> d.salary;  -- only log if salary actually changed
END;
```

**Test:**

```sql
UPDATE sales_employees SET salary = 80000 WHERE employee_id = 1;

SELECT * FROM sales_employee_logs;
```

| log_id | employee_id | log_message | log_date |
|---|---|---|---|
| 3 | 1 | Salary changed from 75000 to 80000 | 2023-06-01 10:05:11 |

## 8. Extending the Trigger: UPDATE and DELETE Events

### 8.1 AFTER DELETE Trigger

```sql
CREATE TRIGGER trg_after_delete_employee
ON sales_employees
AFTER DELETE
AS
BEGIN
    INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
    SELECT
        employee_id,
        'Employee record deleted. Employee ID: ' + CAST(employee_id AS VARCHAR),
        GETDATE()
    FROM DELETED;   -- DELETED holds the rows that were removed
END;
```

**Test:**

```sql
DELETE FROM sales_employees WHERE employee_id = 7;

SELECT * FROM sales_employee_logs;
```

| log_id | employee_id | log_message | log_date |
|---|---|---|---|
| 1 | 6 | New employee added. Employee ID: 6 | 2023-06-01 09:14:32 |
| 2 | 7 | New employee added. Employee ID: 7 | 2023-06-01 09:17:55 |
| 3 | 1 | Salary changed from 75000 to 80000 | 2023-06-01 10:05:11 |
| 4 | 7 | Employee record deleted. Employee ID: 7 | 2023-06-01 10:22:04 |

### 8.2 One Trigger for Multiple Events

A single trigger can respond to more than one event type:

```sql
CREATE TRIGGER trg_employee_all_changes
ON sales_employees
AFTER INSERT, UPDATE, DELETE
AS
BEGIN
    -- INSERT events
    IF EXISTS (SELECT 1 FROM INSERTED) AND NOT EXISTS (SELECT 1 FROM DELETED)
    BEGIN
        INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
        SELECT employee_id, 'INSERT event on employee ID: ' + CAST(employee_id AS VARCHAR), GETDATE()
        FROM INSERTED;
    END;

    -- DELETE events
    IF EXISTS (SELECT 1 FROM DELETED) AND NOT EXISTS (SELECT 1 FROM INSERTED)
    BEGIN
        INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
        SELECT employee_id, 'DELETE event on employee ID: ' + CAST(employee_id AS VARCHAR), GETDATE()
        FROM DELETED;
    END;

    -- UPDATE events (both INSERTED and DELETED exist)
    IF EXISTS (SELECT 1 FROM INSERTED) AND EXISTS (SELECT 1 FROM DELETED)
    BEGIN
        INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
        SELECT i.employee_id, 'UPDATE event on employee ID: ' + CAST(i.employee_id AS VARCHAR), GETDATE()
        FROM INSERTED i;
    END;
END;
```

> 💡 During an UPDATE, **both** `INSERTED` (new values) and `DELETED` (old values) are populated. This is how you distinguish between INSERT, UPDATE, and DELETE events within a combined trigger.

## 9. AFTER vs. INSTEAD OF Triggers

### 9.1 AFTER Trigger

Fires **after** the data modification has already been committed to the table. The change has happened; the trigger reacts to it.

```
User runs INSERT
      │
      ▼
Row is inserted into the table  ← The actual insert happens first
      │
      ▼
AFTER trigger fires             ← Then the trigger logic runs
```

**Use when:** You want to log, cascade, or notify after a confirmed change.

### 9.2 INSTEAD OF Trigger

Fires **instead of** the original operation. The triggering statement (INSERT/UPDATE/DELETE) is **replaced** by the trigger's logic. The original operation does not happen unless you explicitly re-issue it inside the trigger.

```
User runs INSERT
      │
      ▼
INSTEAD OF trigger fires        ← Trigger intercepts immediately
      │
      ▼
Original insert is NOT executed ← Unless the trigger does it manually
```

**Use when:** You need to validate, transform, or conditionally block an operation before it reaches the table.

### 9.3 Example: INSTEAD OF INSERT — Block Duplicate Employees

```sql
CREATE TRIGGER trg_instead_of_insert_employee
ON sales_employees
INSTEAD OF INSERT
AS
BEGIN
    -- Only allow the insert if no employee with the same name exists
    IF NOT EXISTS (
        SELECT 1 FROM sales_employees e
        JOIN INSERTED i ON e.first_name = i.first_name AND e.last_name = i.last_name
    )
    BEGIN
        -- Re-issue the original insert manually
        INSERT INTO sales_employees (first_name, last_name, position, birth_date, salary, hierarchy)
        SELECT first_name, last_name, position, birth_date, salary, hierarchy
        FROM INSERTED;

        PRINT 'Employee inserted successfully.';
    END
    ELSE
    BEGIN
        PRINT 'Duplicate employee name detected. Insert blocked.';
    END;
END;
```

## 10. Dialect Differences: MySQL vs MS SQL Server vs PostgreSQL

### 10.1 Creating a Trigger

```sql
-- MS SQL Server (T-SQL)
CREATE TRIGGER trg_after_insert_employee
ON sales_employees
AFTER INSERT
AS
BEGIN
    INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
    SELECT employee_id,
           'New employee added. ID: ' + CAST(employee_id AS VARCHAR),
           GETDATE()
    FROM INSERTED;
END;


-- MySQL
DELIMITER $$
CREATE TRIGGER trg_after_insert_employee
AFTER INSERT ON sales_employees
FOR EACH ROW
BEGIN
    INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
    VALUES (
        NEW.employee_id,
        CONCAT('New employee added. ID: ', NEW.employee_id),
        NOW()
    );
END $$
DELIMITER ;


-- PostgreSQL (requires a trigger function)
-- Step 1: Create the function
CREATE OR REPLACE FUNCTION fn_log_employee_insert()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
    VALUES (
        NEW.employee_id,
        'New employee added. ID: ' || NEW.employee_id::TEXT,
        NOW()
    );
    RETURN NEW;
END;
$$;

-- Step 2: Attach the function to the table as a trigger
CREATE TRIGGER trg_after_insert_employee
AFTER INSERT ON sales_employees
FOR EACH ROW
EXECUTE FUNCTION fn_log_employee_insert();
```

### 10.2 `INSERTED` / `DELETED` vs. `NEW` / `OLD`

| Concept | SQL Server | MySQL | PostgreSQL |
|---|---|---|---|
| New row values (INSERT/UPDATE) | `INSERTED` virtual table | `NEW.column_name` | `NEW.column_name` |
| Old row values (DELETE/UPDATE) | `DELETED` virtual table | `OLD.column_name` | `OLD.column_name` |
| Row-level access | Set-based (`FROM INSERTED`) | Row-by-row (`NEW.*`) | Row-by-row (`NEW.*`) |

> ⚠️ **Important:** SQL Server triggers are **set-based** — `INSERTED` can contain multiple rows if the triggering statement affected multiple rows (e.g., `INSERT INTO ... SELECT ...`). MySQL and PostgreSQL triggers are **row-level** by default (`FOR EACH ROW`), so `NEW` and `OLD` refer to a single row.

```sql
-- MySQL: access new values with NEW.column
CREATE TRIGGER trg_after_update_salary
AFTER UPDATE ON sales_employees
FOR EACH ROW
BEGIN
    IF NEW.salary <> OLD.salary THEN
        INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
        VALUES (
            NEW.employee_id,
            CONCAT('Salary changed from ', OLD.salary, ' to ', NEW.salary),
            NOW()
        );
    END IF;
END;

-- PostgreSQL
CREATE OR REPLACE FUNCTION fn_log_salary_change()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    IF NEW.salary <> OLD.salary THEN
        INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
        VALUES (
            NEW.employee_id,
            'Salary changed from ' || OLD.salary::TEXT || ' to ' || NEW.salary::TEXT,
            NOW()
        );
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_after_update_salary
AFTER UPDATE ON sales_employees
FOR EACH ROW
EXECUTE FUNCTION fn_log_salary_change();
```

### 10.3 INSTEAD OF Triggers

| Feature | SQL Server | MySQL | PostgreSQL |
|---|---|---|---|
| `INSTEAD OF` on tables | ✅ Supported | ❌ Not supported | ✅ Supported (`BEFORE` trigger + `RETURN NULL`) |
| `INSTEAD OF` on views | ✅ Fully supported | ❌ Not supported | ✅ Supported |

```sql
-- PostgreSQL: use BEFORE trigger + RETURN NULL to block an operation
CREATE OR REPLACE FUNCTION fn_block_delete()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    RAISE NOTICE 'Delete blocked for employee ID: %', OLD.employee_id;
    RETURN NULL;   -- returning NULL cancels the DELETE
END;
$$;

CREATE TRIGGER trg_before_delete_employee
BEFORE DELETE ON sales_employees
FOR EACH ROW
EXECUTE FUNCTION fn_block_delete();
```

### 10.4 Disabling / Dropping Triggers

```sql
-- SQL Server: disable and re-enable
DISABLE TRIGGER trg_after_insert_employee ON sales_employees;
ENABLE  TRIGGER trg_after_insert_employee ON sales_employees;

-- SQL Server: drop
DROP TRIGGER trg_after_insert_employee;

-- MySQL: drop (MySQL has no DISABLE — drop and recreate)
DROP TRIGGER trg_after_insert_employee;

-- PostgreSQL: disable and re-enable
ALTER TABLE sales_employees DISABLE TRIGGER trg_after_insert_employee;
ALTER TABLE sales_employees ENABLE  TRIGGER trg_after_insert_employee;

-- PostgreSQL: drop
DROP TRIGGER trg_after_insert_employee ON sales_employees;
```

### 10.5 Statement-Level vs. Row-Level Triggers

| Feature | SQL Server | MySQL | PostgreSQL |
|---|---|---|---|
| **Row-level triggers** | Default (set-based via INSERTED/DELETED) | `FOR EACH ROW` only | `FOR EACH ROW` |
| **Statement-level triggers** | Not explicitly declared (INSERTED handles sets) | Not supported | `FOR EACH STATEMENT` |

```sql
-- PostgreSQL: statement-level trigger (fires once per statement, not once per row)
CREATE TRIGGER trg_statement_level
AFTER INSERT ON sales_employees
FOR EACH STATEMENT
EXECUTE FUNCTION fn_log_employee_insert();
```

## 11. Execution Flow

### AFTER INSERT Trigger Flow

```
User/Application:  INSERT INTO sales_employees VALUES (...)
                            │
                            ▼
         Database validates the INSERT (constraints, data types)
                            │
                            ▼
              Row is written to sales_employees
                            │
                            ▼
         INSERTED virtual table populated with new row(s)
                            │
                            ▼
         Trigger trg_after_insert_employee fires
                            │
                            ▼
         INSERT INTO sales_employee_logs ...
         FROM INSERTED (reads new row data)
                            │
                            ▼
         Log row written to sales_employee_logs
                            │
                            ▼
                      Transaction complete
```

### INSTEAD OF INSERT Trigger Flow

```
User/Application:  INSERT INTO sales_employees VALUES (...)
                            │
                            ▼
         INSERTED virtual table populated with pending row(s)
                            │
                            ▼
         INSTEAD OF trigger fires — original INSERT is SUSPENDED
                            │
             ┌──────────────┴──────────────────┐
             │                                 │
        Condition TRUE                    Condition FALSE
             │                                 │
     Re-issue INSERT manually           Block the INSERT
     inside trigger body                Print warning
             │                                 │
             ▼                                 ▼
    Row written to table              No row written
```

## 12. Common Mistakes

### ❌ Mistake 1: Forgetting That `INSERTED` Can Contain Multiple Rows (SQL Server)

```sql
-- BAD: assumes only one row was inserted — breaks on bulk inserts
CREATE TRIGGER trg_bad
ON sales_employees AFTER INSERT AS
BEGIN
    DECLARE @emp_id INT;
    SELECT @emp_id = employee_id FROM INSERTED;  -- only captures one row!
    INSERT INTO logs VALUES (@emp_id, 'inserted', GETDATE());
END;

-- GOOD: use INSERT...SELECT to handle all rows
CREATE TRIGGER trg_good
ON sales_employees AFTER INSERT AS
BEGIN
    INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
    SELECT employee_id, 'inserted', GETDATE()
    FROM INSERTED;   -- handles 1 or 1000 rows equally
END;
```

### ❌ Mistake 2: Recursive Triggers

If a trigger on Table A modifies Table A, it can fire itself again — creating an infinite loop.

```sql
-- DANGEROUS: this trigger updates sales_employees, which fires itself again
CREATE TRIGGER trg_recursive
ON sales_employees AFTER UPDATE AS
BEGIN
    UPDATE sales_employees SET hierarchy = hierarchy + 1
    WHERE employee_id IN (SELECT employee_id FROM INSERTED);
    -- ↑ This fires the trigger again → infinite loop
END;
```

> ✅ In SQL Server, you can control recursion with the `RECURSIVE_TRIGGERS` database option. In PostgreSQL, use `pg_trigger_depth()` to guard against recursion. In MySQL, direct recursive triggers are not allowed by default.

### ❌ Mistake 3: Using Triggers for Complex Business Logic

Triggers are invisible to most application developers. Hiding critical logic inside triggers makes systems extremely hard to debug and maintain.

> ✅ Use triggers for **cross-cutting concerns** (auditing, logging) only. Put business rules in application code or stored procedures where they are visible and testable.

### ❌ Mistake 4: Not Casting Data Types in Log Messages

```sql
-- BAD: type error — cannot concatenate INT with VARCHAR
'Employee ID: ' + employee_id

-- GOOD
'Employee ID: ' + CAST(employee_id AS VARCHAR)   -- SQL Server
CONCAT('Employee ID: ', employee_id)              -- MySQL (auto-converts)
'Employee ID: ' || employee_id::TEXT              -- PostgreSQL
```

### ❌ Mistake 5: Querying `INSERTED` / `DELETED` Outside the Trigger

```sql
-- This will FAIL — INSERTED only exists during trigger execution
SELECT * FROM INSERTED;   -- Error: Invalid object name 'INSERTED'
```

### ❌ Mistake 6: No Error Handling Inside Triggers

Uncaught errors inside a trigger will roll back the entire originating transaction. Always wrap trigger logic in TRY / CATCH:

```sql
CREATE TRIGGER trg_safe_log
ON sales_employees AFTER INSERT AS
BEGIN
    BEGIN TRY
        INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
        SELECT employee_id, 'New employee added.', GETDATE()
        FROM INSERTED;
    END TRY
    BEGIN CATCH
        PRINT 'Trigger error: ' + ERROR_MESSAGE();
        -- Do NOT re-raise the error if logging failure should not block the INSERT
    END CATCH;
END;
```

## 13. Real-World Analytics Use Cases

### Use Case 1: HR Audit Trail (Covered in This Chapter)

Automatically log every INSERT, UPDATE, and DELETE on the employees table for compliance and forensic analysis.

```sql
-- Query the audit log to see all changes today
SELECT *
FROM sales_employee_logs
WHERE CAST(log_date AS DATE) = CAST(GETDATE() AS DATE)
ORDER BY log_date DESC;
```

### Use Case 2: Enforcing Data Integrity — Blocking Invalid Salaries

```sql
CREATE TRIGGER trg_validate_salary
ON sales_employees
INSTEAD OF INSERT
AS
BEGIN
    IF EXISTS (SELECT 1 FROM INSERTED WHERE salary < 0)
    BEGIN
        RAISERROR('Salary cannot be negative.', 16, 1);
        ROLLBACK;
    END
    ELSE
    BEGIN
        INSERT INTO sales_employees (first_name, last_name, position, birth_date, salary, hierarchy)
        SELECT first_name, last_name, position, birth_date, salary, hierarchy
        FROM INSERTED;
    END;
END;
```

### Use Case 3: Maintaining a Summary Table Automatically

Keep a `department_headcount` summary table in sync whenever employees are added or removed:

```sql
CREATE TRIGGER trg_sync_headcount_insert
ON sales_employees AFTER INSERT AS
BEGIN
    -- Recalculate headcount after every insert
    UPDATE dept_headcount
    SET employee_count = (
        SELECT COUNT(*) FROM sales_employees WHERE position = dept_headcount.position
    );
END;
```

### Use Case 4: Soft Delete Pattern

Instead of physically deleting records, mark them as deleted:

```sql
CREATE TRIGGER trg_soft_delete
ON sales_employees
INSTEAD OF DELETE
AS
BEGIN
    UPDATE sales_employees
    SET is_deleted = 1, deleted_at = GETDATE()
    WHERE employee_id IN (SELECT employee_id FROM DELETED);
END;
```

## 14. Best Practices

1. **Name triggers clearly** with a prefix and the event they respond to:
   `trg_after_insert_employee`, `trg_before_delete_order`, `trg_instead_of_update_salary`

2. **Keep triggers lean.** A trigger should do one focused thing — log a change, sync a field, enforce a constraint. Heavy logic belongs in stored procedures or application code.

3. **Always handle sets, not single rows** (especially in SQL Server). Write trigger logic to handle bulk operations correctly using `INSERT...SELECT FROM INSERTED`.

4. **Add TRY / CATCH** inside every trigger that writes data, so a logging failure doesn't roll back the user's original transaction unintentionally.

5. **Document triggers prominently.** Because they fire invisibly, developers unfamiliar with the schema won't know they exist. Add a comment block at the top of every trigger explaining what it does, when it fires, and why.

6. **Do not use triggers for business logic.** Rules like discount calculations, approval workflows, or pricing should live in the application layer — not in hidden database triggers.

7. **Test with bulk inserts.** Verify that your trigger behaves correctly when 1 row, 100 rows, and 10,000 rows are inserted in a single statement.

8. **Be careful with INSTEAD OF on views.** While powerful, they can be confusing and are a frequent source of bugs. Prefer materialized logic in stored procedures unless the view pattern is well understood by the whole team.

9. **Audit the auditors.** Periodically query your log tables to verify triggers are firing and the data looks correct. Silent trigger failures can go unnoticed for months.

## 15. Final Summary Table

| Feature | AFTER Trigger | INSTEAD OF Trigger |
|---|---|---|
| **When it fires** | After the DML completes | In place of the DML (intercepts it) |
| **Original operation** | Already happened | Does NOT happen unless you re-issue it |
| **Access to data** | `INSERTED` / `DELETED` | `INSERTED` / `DELETED` (pending rows) |
| **Use cases** | Audit logging, cascades, notifications | Validation, conditional blocks, view updates |

### Trigger Types at a Glance

| Trigger Type | Event | Typical Use |
|---|---|---|
| `AFTER INSERT` | Row(s) added | Log new records, sync derived tables |
| `AFTER UPDATE` | Row(s) modified | Log changes, track old/new values |
| `AFTER DELETE` | Row(s) removed | Log deletions, implement soft delete |
| `INSTEAD OF INSERT` | Insert intercepted | Validate input, route to correct table |
| `INSTEAD OF UPDATE` | Update intercepted | Enforce immutability, transform values |
| `INSTEAD OF DELETE` | Delete intercepted | Implement soft delete, block deletions |

### Virtual Tables Quick Reference

| Table | Available In | Contains |
|---|---|---|
| `INSERTED` (`NEW` in MySQL/PG) | INSERT, UPDATE triggers | New / updated row values |
| `DELETED` (`OLD` in MySQL/PG) | DELETE, UPDATE triggers | Deleted / pre-update row values |

### Complete Audit Trigger Template (SQL Server)

```sql
CREATE TRIGGER trg_after_insert_employee
ON sales_employees
AFTER INSERT
AS
BEGIN
    BEGIN TRY
        -- ================================================
        -- Write to audit log for every inserted row
        -- ================================================
        INSERT INTO sales_employee_logs (employee_id, log_message, log_date)
        SELECT
            employee_id,
            'New employee added. Employee ID: ' + CAST(employee_id AS VARCHAR),
            GETDATE()
        FROM INSERTED;
    END TRY
    BEGIN CATCH
        PRINT 'Trigger error: ' + ERROR_MESSAGE();
    END CATCH;
END;
```
