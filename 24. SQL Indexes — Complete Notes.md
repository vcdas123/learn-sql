## 1. What is an Index?

An **index** is a data structure that provides **fast access to rows** in a table, improving query speed — especially on large tables.

### Analogies

**The Book Index:** Instead of flipping through every page of a book to find a topic, you use the index at the back to jump straight to the right page. That's what a database index does for your data.

**The Hotel Map:** Without a room numbering system, you'd walk floor by floor checking every room. With a map from reception, you go straight to the right building, floor, and room. An index gives your database that map.

Without an index → database scans **every row** → slow.
With an index → database **navigates directly** to the right data → fast.

## 2. How Data is Stored Without an Index — Heap Structure

When you create a table and insert data **without any index**, the database stores data in what is called a **Heap Structure**.

### What is a Heap?

- Data is stored in **data files on disk**, divided into fixed-size units called **pages**.
- Rows are inserted **in the order they arrive** — no sorting, no organization.
- The heap has **no concept of order**.

```
Page 1:  ID 12 | ID 5  | ID 15 | ID 6  | ID 7
Page 2:  ID 3  | ID 18 | ID 1  | ID 9  | ID 11
Page 3:  ID 14 | ID 20 | ID 4  | ID 17 | ID 8
Page 4:  ID 2  | ID 16 | ID 13 | ID 10 | ID 19
```

### Reads on a Heap — Full Table Scan

If you query `WHERE customer_id = 14`, the database has **no idea which page it's on**. It must:

1. Read Page 1 — scan all 5 rows — not found
2. Read Page 2 — scan all 5 rows — not found
3. Read Page 3 — scan first row — found!

Even if found on page 3, **4 pages were read, 19 rows scanned** just to find one record.

This is called a **Full Table Scan** — the database reads the entire table page by page, row by row.

> 🔖 **Analogy:** Like throwing all your papers in a drawer with no organization. You can add papers instantly (fast writes) but finding a specific one takes forever (slow reads).

### Heap Pros & Cons

| Aspect | Heap |
|---|---|
| Write speed | ✅ Fastest — just append |
| Read speed | ❌ Slowest — full table scan |
| Storage | ✅ Efficient — no extra structures |

## 3. The Page — Unit of Storage

A **page** is the fundamental unit of data storage in a SQL database.

- Fixed size: **8 KB** per page
- Everything is stored in pages: row data, indexes, metadata
- SQL reads and writes **entire pages** — it never reads a single row in isolation

### Page Structure

```
┌─────────────────────────────────┐
│  Page Header (96 bytes)         │  ← Metadata: page ID (e.g. 1:150), type, etc.
├─────────────────────────────────┤
│  Row Data (variable size)       │  ← Actual table rows stored here
│  Row 1                          │
│  Row 2                          │
│  ...                            │
├─────────────────────────────────┤
│  Offset Array                   │  ← Index of where each row starts within this page
└─────────────────────────────────┘
```

**Page ID format:** `FileID:PageNumber` → e.g., `1:150` means file 1, page 150.

**Offset Array:** A small lookup at the end of each page that records the starting position of each row, so SQL can jump directly to a row without scanning the whole page.

### Two Important Page Types

| Page Type | Contains |
|---|---|
| **Data Page** | Actual table rows |
| **Index Page** | Keys + pointers to other pages (no actual row data) |

## 4. Index Types Overview

Indexes can be categorized in three ways:

### By Structure (How data is organized/referenced)
- **Clustered Index** — physically sorts the data
- **Non-Clustered Index** — separate lookup structure, data untouched

### By Storage (How data is physically stored)
- **Row Store Index** — data stored row by row (traditional)
- **Column Store Index** — data stored column by column (great for analytics)

### By Function (Special behaviors)
- **Unique Index** — enforces uniqueness on a column
- **Filtered Index** — index built on a subset of rows (with a WHERE clause)

> This notes file focuses on the **By Structure** category — Clustered and Non-Clustered indexes.

---

## 5. Clustered Index

A clustered index **physically sorts and stores** the table rows based on the index key column.

### What Happens When You Create a Clustered Index

**Step 1 — Physical Sort:**
All data pages are rearranged so rows are sorted by the index column (e.g., `customer_id` ascending).

```
Page 1:  ID 1  | ID 2  | ID 3  | ID 4  | ID 5
Page 2:  ID 6  | ID 7  | ID 8  | ID 9  | ID 10
Page 3:  ID 11 | ID 12 | ID 13 | ID 14 | ID 15
Page 4:  ID 16 | ID 17 | ID 18 | ID 19 | ID 20
```

**Step 2 — B-Tree Construction:**
A **Balanced Tree (B-Tree)** is built — a hierarchical structure used to navigate to data quickly.

```
                    ┌──────────────────────┐
                    │   ROOT NODE          │
                    │  (Index Page)        │
                    │  1-10 → Page 1:200   │
                    │  11-20 → Page 1:201  │
                    └──────────┬───────────┘
                               │
             ┌─────────────────┴─────────────────┐
             ▼                                   ▼
   ┌──────────────────┐               ┌──────────────────┐
   │ INTERMEDIATE NODE│               │ INTERMEDIATE NODE│
   │  (Index Page)    │               │  (Index Page)    │
   │  1-5  → 1:100    │               │  11-15 → 1:102   │
   │  6-10 → 1:101    │               │  16-20 → 1:103   │
   └────────┬─────────┘               └────────┬─────────┘
            │                                  │
   ┌────────┴────────┐               ┌─────────┴────────┐
   ▼                 ▼               ▼                  ▼
[DATA PAGE]     [DATA PAGE]     [DATA PAGE]       [DATA PAGE]
 ID 1-5          ID 6-10         ID 11-15           ID 16-20
(LEAF LEVEL — actual data lives here)
```

### Key Insight: Leaves = Actual Data

In a clustered index B-Tree, the **leaf level contains the actual data pages** — not pointers to data, but the data itself.

### Query Trace — Finding `customer_id = 14`

```
Start at Root:       14 is in 11-20 → go to Intermediate Node 1:201
Intermediate Node:   14 is in 11-15 → go to Data Page 1:102
Data Page:           Scan page, find ID 14 ✅

Pages read: 3 (2 index pages + 1 data page)
Rows scanned on data page: ~5
```

Compare to Heap: **4 pages read, 19 rows scanned**.
With clustered index: **3 pages read, ~5 rows scanned** — and most reads are cheap index pages.

### Rules for Clustered Index

- **Maximum 1 per table** — because data can only be physically sorted in one way.
- **Best column choice:** A column that is unique and rarely/never updated (e.g., primary key).
- **Avoid:** Columns with frequent updates — every update may require re-sorting data pages.

## 6. Non-Clustered Index

A non-clustered index **does not touch or rearrange the actual data**. Instead, it builds a **separate B-Tree structure** alongside the table that contains sorted key values + pointers (Row Identifiers) pointing back to the actual row locations.

### What Happens When You Create a Non-Clustered Index

The physical data pages remain exactly as they are (heap or clustered order). A completely separate B-Tree is built.

**The Row Identifier (RID):** Instead of just pointing to a data page, the non-clustered index stores the **exact location** of each row:

```
RID format: FileID : PageNumber : Offset
Example:    1 : 100 : 96
            └──┬──┘ └──┬──┘ └─┬─┘
           File 1   Page 100  Offset 96 (exact row position within the page)
```

### B-Tree Structure

```
                    ┌──────────────────────┐
                    │   ROOT NODE          │
                    │  (Index Page)        │
                    │  1-10 → Page 1:300   │
                    │  11-20 → Page 1:301  │
                    └──────────┬───────────┘
                               │
             ┌─────────────────┴─────────────────┐
             ▼                                   ▼
   ┌──────────────────┐               ┌──────────────────┐
   │ INTERMEDIATE NODE│               │ INTERMEDIATE NODE│
   │  (Index Page)    │               │  (Index Page)    │
   └────────┬─────────┘               └────────┬─────────┘
            │                                  │
   ┌────────┴────────┐               ┌─────────┴────────┐
   ▼                 ▼               ▼                  ▼
[INDEX PAGE]    [INDEX PAGE]    [INDEX PAGE]      [INDEX PAGE]
(Row Locator    (Row Locator    (Row Locator       (Row Locator
  Pages)          Pages)          Pages)             Pages)
LEAF LEVEL — index pages with RIDs, pointing to data pages ↓
                       │
                       ▼
              [ACTUAL DATA PAGES]
              (Heap or Clustered — outside the B-Tree)
```

### Key Insight: Leaves = Index Pages with Pointers

Unlike the clustered index, the **leaf level of a non-clustered index contains index pages** (not data pages). These index pages hold the RID for each row, pointing to the data pages which are **outside the B-Tree**.

### Query Trace — Finding `customer_id = 14`

```
Start at Root:         14 is in 11-20 → go to Intermediate Node
Intermediate Node:     14 is in 11-15 → go to Leaf Index Page
Leaf Index Page:       Find ID 14 → RID = 1:103:offset → go to data page
Data Page:             Jump directly to the row offset ✅

Pages read: 3 index pages + 1 data page = 4 total
```

One extra layer vs clustered index — but still vastly better than a full table scan.

### Rules for Non-Clustered Index

- **Multiple allowed per table** — no limit, because the B-Tree only stores pointers (no data duplication).
- **Best for:** Columns used in WHERE, JOIN conditions, or ORDER BY that aren't the primary key.
- **Writes are faster** than clustered — the physical data doesn't need re-sorting.

## 7. Clustered vs Non-Clustered — Side-by-Side

| Feature | Clustered Index | Non-Clustered Index |
|---|---|---|
| Physical data sorting | ✅ Yes — data is physically sorted | ❌ No — data untouched |
| Leaf level contains | Actual data pages | Index pages with RIDs (pointers) |
| Max per table | **1** only | **Unlimited** |
| Read performance | ✅ Faster | ⚠️ Slightly slower (extra hop) |
| Write performance | ⚠️ Slower (re-sort on insert/update) | ✅ Faster |
| Storage overhead | ✅ Lower (no extra data copy) | ⚠️ Higher (extra index pages) |
| B-Tree structure | Root → Intermediate → **Data Pages** | Root → Intermediate → **Index Pages** → Data Pages |
| Best column | Primary key (unique, never updated) | Columns in WHERE / JOIN / ORDER BY |
| Use for | Range queries, primary key lookups | Exact match lookups, multi-column filters |

> 🔖 **Book Analogy:**
> - **Clustered Index** = Table of Contents at the front — chapters are physically arranged in the same order as the contents.
> - **Non-Clustered Index** = Index at the back of the book — a separate alphabetical list pointing to where each topic appears in the book, regardless of chapter order.

## 8. Composite Index

A **composite index** (also called a multi-column index) is an index built on **two or more columns** together.

### When to Use

When your WHERE clause filters on multiple columns together:

```sql
SELECT * FROM customers
WHERE country = 'USA' AND score > 500;
```

Rather than separate indexes on `country` and `score`, a composite index on `(country, score)` is more efficient.

### Creating a Composite Index

```sql
CREATE INDEX idx_tb_customers_country_score
ON sales.tb_customers (country ASC, score DESC);
```

- Column order in the index definition must **match the column order in your most common queries**.
- You can specify **sort direction** (ASC/DESC) per column.

### How It Works Internally

The B-Tree sorts data by the **first column first**, then by the **second column within each first-column group**:

```
Sorted by country first, then score within each country:
  (Canada, 950) → RID
  (Canada, 820) → RID
  (India,  710) → RID
  (USA,   1200) → RID
  (USA,    890) → RID
  (USA,    540) → RID
```

---

## 9. The Leftmost Prefix Rule

The most important rule for composite indexes: **SQL can only use the index if your query starts with the leftmost column(s) of the index, in order, without gaps**.

### Index defined as: `(A, B, C, D)`

| Query uses | Uses index? | Why |
|---|---|---|
| `A` | ✅ Yes | Leftmost column — index used |
| `A, B` | ✅ Yes | Left two columns in order |
| `A, B, C` | ✅ Yes | Left three columns in order |
| `A, B, C, D` | ✅ Yes | All columns |
| `B` | ❌ No | Skips `A` (leftmost) |
| `C` | ❌ No | Skips `A` and `B` |
| `A, C` | ❌ No | Gap between A and C (skips B) |
| `A, B, D` | ❌ No | Gap (skips C) |

### Practical Example — `(country, score)` index

```sql
-- ✅ Uses index (country is leftmost)
SELECT * FROM customers WHERE country = 'USA' AND score > 500;

-- ✅ Uses index (country alone, leftmost column)
SELECT * FROM customers WHERE country = 'USA';

-- ❌ Does NOT use index (skips leftmost column "country")
SELECT * FROM customers WHERE score > 500;
```

> ⚠️ **Critical:** If you query skips the leftmost column of a composite index, the optimizer ignores the index entirely. Either reorder your query columns or recreate the index with the correct column order.

## 10. Index Syntax — SQL Server

### Create Clustered Index

```sql
CREATE CLUSTERED INDEX idx_tb_customers_customer_id
ON sales.tb_customers (customer_id ASC);
```

### Create Non-Clustered Index

```sql
-- Explicit
CREATE NONCLUSTERED INDEX idx_tb_customers_last_name
ON sales.tb_customers (last_name ASC);

-- Shorthand (default is NONCLUSTERED)
CREATE INDEX idx_tb_customers_last_name
ON sales.tb_customers (last_name);
```

### Create Composite Index

```sql
CREATE INDEX idx_tb_customers_country_score
ON sales.tb_customers (country ASC, score DESC);
```

### Create Unique Index

```sql
CREATE UNIQUE INDEX idx_customers_email
ON customers (email);
```

### Drop an Index

```sql
DROP INDEX idx_tb_customers_last_name ON sales.tb_customers;
```

### View Indexes on a Table

```sql
-- SQL Server
SELECT * FROM sys.indexes WHERE object_id = OBJECT_ID('sales.tb_customers');

-- Or via Object Explorer: Tables → tb_customers → Indexes
```

### Naming Convention (Recommended)

```
idx_{table_name}_{column_name(s)}

Examples:
  idx_tb_customers_customer_id
  idx_tb_customers_last_name
  idx_tb_customers_country_score
```

Following this convention makes it instantly clear **which table** and **which columns** the index covers when monitoring or debugging.

## 11. Index Syntax Across SQL Databases

The core syntax is similar across SQL databases — `CREATE [CLUSTERED|NONCLUSTERED] INDEX` — but each database has its own terminology, defaults, and features.

---

### MySQL / MariaDB

MySQL uses **InnoDB** engine where the primary key is always a clustered index. Secondary indexes are non-clustered.

```sql
-- Primary key → automatically creates clustered index
CREATE TABLE customers (
  customer_id INT PRIMARY KEY,  -- clustered by default in InnoDB
  last_name VARCHAR(100),
  email VARCHAR(255)
);

-- Secondary (non-clustered) index
CREATE INDEX idx_customers_last_name
ON customers (last_name);

-- Unique index
CREATE UNIQUE INDEX idx_customers_email
ON customers (email);

-- Composite index
CREATE INDEX idx_customers_country_score
ON customers (country, score);

-- Full-text index (for text search)
CREATE FULLTEXT INDEX idx_customers_bio
ON customers (bio);

-- Drop index
DROP INDEX idx_customers_last_name ON customers;

-- View indexes on table
SHOW INDEX FROM customers;
```

**MySQL Notes:**
- No `CLUSTERED` / `NONCLUSTERED` keywords — InnoDB always clusters on primary key.
- All secondary indexes include the primary key value implicitly as the row locator.
- Use `EXPLAIN SELECT ...` to see if a query uses an index.

---

### PostgreSQL

PostgreSQL supports multiple index types beyond B-Tree (Hash, GiST, GIN, BRIN).

```sql
-- Default B-Tree index (non-clustered by default)
CREATE INDEX idx_customers_last_name
ON customers (last_name);

-- Unique index
CREATE UNIQUE INDEX idx_customers_email
ON customers (email);

-- Composite index
CREATE INDEX idx_customers_country_score
ON customers (country ASC, score DESC);

-- Partial index (filtered index — only index rows meeting a condition)
CREATE INDEX idx_active_customers
ON customers (last_name)
WHERE status = 'active';

-- CONCURRENTLY — build index without locking the table (for production)
CREATE INDEX CONCURRENTLY idx_customers_last_name
ON customers (last_name);

-- Hash index (only for equality checks, not range queries)
CREATE INDEX idx_customers_email_hash
ON customers USING hash (email);

-- GIN index (for arrays, JSONB, full-text search)
CREATE INDEX idx_customers_tags
ON customers USING gin (tags);

-- Cluster table data by an index (one-time physical sort, not maintained)
CLUSTER customers USING idx_customers_last_name;

-- Drop index
DROP INDEX idx_customers_last_name;

-- View indexes
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'customers';
```

**PostgreSQL Notes:**
- No built-in clustered index concept — `CLUSTER` is a one-time physical sort, not maintained on insert.
- Primary key creates a unique B-Tree index automatically.
- `CONCURRENTLY` is critical in production to avoid table locks during index creation.
- `GIN` and `GiST` indexes are unique to PostgreSQL — great for full-text search and spatial data.

---

### SQLite

SQLite is minimal — no clustered/non-clustered distinction. The `rowid` (internal integer) acts as the clustered key.

```sql
-- Basic index
CREATE INDEX idx_customers_last_name
ON customers (last_name);

-- Unique index
CREATE UNIQUE INDEX idx_customers_email
ON customers (email);

-- Composite index
CREATE INDEX idx_customers_country_score
ON customers (country, score);

-- Partial index
CREATE INDEX idx_active_customers
ON customers (last_name)
WHERE status = 'active';

-- Drop index
DROP INDEX idx_customers_last_name;

-- View indexes
PRAGMA index_list('customers');
PRAGMA index_info('idx_customers_last_name');
```

**SQLite Notes:**
- Extremely lightweight — B-Tree indexes only.
- `WITHOUT ROWID` tables use a clustered B-Tree on primary key.
- No `CONCURRENTLY`, no GIN/GiST — simplicity is the design goal.

---

### Oracle Database

Oracle uses **B-Tree** (default) and **Bitmap** indexes. No `CLUSTERED` / `NONCLUSTERED` keywords.

```sql
-- Standard (non-unique) index
CREATE INDEX idx_customers_last_name
ON customers (last_name);

-- Unique index
CREATE UNIQUE INDEX idx_customers_email
ON customers (email);

-- Composite index
CREATE INDEX idx_customers_country_score
ON customers (country ASC, score DESC);

-- Bitmap index (great for low-cardinality columns like status, gender)
CREATE BITMAP INDEX idx_customers_status
ON customers (status);

-- Function-based index (index on an expression)
CREATE INDEX idx_customers_upper_last_name
ON customers (UPPER(last_name));

-- Reverse key index (distributes inserts for sequential keys)
CREATE INDEX idx_customers_id_reverse
ON customers (customer_id) REVERSE;

-- Invisible index (exists but optimizer ignores it — for testing)
CREATE INDEX idx_customers_test
ON customers (last_name) INVISIBLE;

-- Drop index
DROP INDEX idx_customers_last_name;

-- View indexes
SELECT index_name, index_type, uniqueness FROM user_indexes WHERE table_name = 'CUSTOMERS';
```

**Oracle Notes:**
- **Bitmap indexes** are powerful for columns with low cardinality (few distinct values) like `status = 'active'/'inactive'`.
- Function-based indexes (e.g., `UPPER(last_name)`) allow indexing computed values.
- `INVISIBLE` indexes are a testing feature unique to Oracle — great for validating an index before enabling it.
- Primary key automatically creates a unique index in Oracle.

## 12. When to Use Which Index

### Clustered Index — Use When:

- Column values are **unique** (e.g., primary key, UUID, auto-increment ID)
- Column values **rarely/never change** — updates trigger re-sorting
- You frequently run **range queries**: `WHERE id BETWEEN 100 AND 200`, `ORDER BY id`
- You want the **fastest possible read** on the most common lookup

### Non-Clustered Index — Use When:

- Column is used in **WHERE, JOIN, or ORDER BY** but is not the primary key
- You need **multiple indexes** on the same table (you can have unlimited)
- Column values **change frequently** (writes are cheaper with non-clustered)
- You're optimizing **exact match** queries: `WHERE email = 'alice@example.com'`

### Composite Index — Use When:

- Your WHERE clause **consistently uses multiple columns together**
- You want one index to serve multiple related queries (using leftmost prefix)
- Joining on multiple columns

### When NOT to Index:

- **Small tables** — full table scan is fast enough; index overhead isn't worth it
- **Columns with very low cardinality** — e.g., a `gender` column with 2 values (use bitmap index in Oracle instead)
- **Columns that change extremely frequently** — constant re-sorting / pointer updates defeat the purpose
- **Write-heavy tables** where insert speed matters more than read speed

---

## 13. Index Tradeoffs Summary

```
Stronger Index (clustered + multiple non-clustered)
        │
        ▼
  Faster reads
  Direct navigation to data
  ⬇ Slower writes (more structures to maintain)
  ⬇ More storage (index pages)

No Index (Heap)
        │
        ▼
  Fastest writes (just append)
  ⬇ Slowest reads (full table scan)
  ⬇ No storage overhead
```

| Index Type | Read Speed | Write Speed | Storage | Max Per Table |
|---|---|---|---|---|
| No index (Heap) | ❌ Slowest | ✅ Fastest | ✅ Minimal | N/A |
| Clustered | ✅ Fastest | ⚠️ Slower | ✅ Efficient | 1 |
| Non-Clustered | ✅ Fast | ✅ Faster than clustered | ⚠️ Extra pages | Unlimited |
| Composite | ✅ Fast (multi-column) | ⚠️ Moderate | ⚠️ Extra pages | Unlimited |

## 14. Quick Reference Cheat Sheet

```sql
-- SQL SERVER --------------------------------------------------
CREATE CLUSTERED INDEX idx_tbl_col ON schema.table (col ASC);
CREATE NONCLUSTERED INDEX idx_tbl_col ON schema.table (col);
CREATE INDEX idx_tbl_col1_col2 ON schema.table (col1 ASC, col2 DESC); -- composite
DROP INDEX idx_tbl_col ON schema.table;

-- MYSQL -------------------------------------------------------
CREATE INDEX idx_tbl_col ON table (col);
CREATE UNIQUE INDEX idx_tbl_col ON table (col);
CREATE INDEX idx_tbl_col1_col2 ON table (col1, col2);  -- composite
CREATE FULLTEXT INDEX idx_tbl_col ON table (col);
DROP INDEX idx_tbl_col ON table;
SHOW INDEX FROM table;

-- POSTGRESQL --------------------------------------------------
CREATE INDEX idx_tbl_col ON table (col);
CREATE UNIQUE INDEX idx_tbl_col ON table (col);
CREATE INDEX idx_tbl_col1_col2 ON table (col1 ASC, col2 DESC);  -- composite
CREATE INDEX idx_partial ON table (col) WHERE condition;         -- partial
CREATE INDEX CONCURRENTLY idx_tbl_col ON table (col);           -- no lock
CREATE INDEX idx_gin ON table USING gin (col);                   -- GIN
DROP INDEX idx_tbl_col;

-- SQLITE ------------------------------------------------------
CREATE INDEX idx_tbl_col ON table (col);
CREATE UNIQUE INDEX idx_tbl_col ON table (col);
CREATE INDEX idx_partial ON table (col) WHERE condition;  -- partial
DROP INDEX idx_tbl_col;
PRAGMA index_list('table');

-- ORACLE ------------------------------------------------------
CREATE INDEX idx_tbl_col ON table (col);
CREATE UNIQUE INDEX idx_tbl_col ON table (col);
CREATE BITMAP INDEX idx_tbl_col ON table (col);          -- low-cardinality cols
CREATE INDEX idx_func ON table (UPPER(col));              -- function-based
DROP INDEX idx_tbl_col;
SELECT index_name FROM user_indexes WHERE table_name = 'TABLE';
```
