# SQL DDL (Data Definition Language) – Detailed Notes

## 1. What is DDL?

**DDL (Data Definition Language)** is a subset of SQL used to **define, change, and remove the structure of database objects**. Unlike `SELECT` queries, DDL commands do **not return data**. Instead, they modify the database schema itself.

DDL is typically used when:

- Creating tables
- Modifying table structures
- Deleting tables or other database objects

Common DDL commands:

- `CREATE`
- `ALTER`
- `DROP`
- (Sometimes) `TRUNCATE`

---

## 2. CREATE TABLE

### Purpose

`CREATE TABLE` is used to define a **new table** in the database. When created, the table is empty and contains **only structure, no data**.

### General Syntax

```sql
CREATE TABLE table_name (
    column_name data_type [constraints],
    column_name data_type [constraints],
    ...
);
```

---

## 3. Example: Creating the `persons` Table

### Task

Create a table named `persons` with the following columns:

- `id`
- `person_name`
- `birth_date`
- `phone`

### SQL Statement

```sql
CREATE TABLE persons (
    id INT NOT NULL,
    person_name VARCHAR(50) NOT NULL,
    birth_date DATE NULL,
    phone VARCHAR(15) NOT NULL,
    CONSTRAINT pk_persons PRIMARY KEY (id)
);
```

### Explanation

- `INT` → numeric data type for IDs
- `VARCHAR(50)` → variable-length text (up to 50 characters)
- `DATE` → stores date values only
- `NOT NULL` → value is mandatory
- `PRIMARY KEY` → ensures uniqueness and table integrity

### Why Primary Key Is Important

- Ensures each row is unique
- Improves query performance
- Required for relationships with other tables

---

## 3A. Different Ways to Define Constraints in SQL

Constraints can be defined in **two main ways**:

1. **Column-level constraints**
2. **Table-level constraints**

Both are valid and widely used.

---

### 1. Column-Level Constraints

Constraints are defined **inline**, directly after the column definition.

#### Example

```sql
CREATE TABLE persons (
    id INT PRIMARY KEY,
    person_name VARCHAR(50) NOT NULL,
    email VARCHAR(50) UNIQUE,
    birth_date DATE
);
```

#### Characteristics

- Written next to the column
- Simple and readable
- Best for single-column constraints

---

### 2. Table-Level Constraints

Constraints are defined **separately**, after all columns are declared.

#### Example

```sql
CREATE TABLE persons (
    id INT,
    person_name VARCHAR(50) NOT NULL,
    email VARCHAR(50),
    CONSTRAINT pk_persons PRIMARY KEY (id),
    CONSTRAINT uq_persons_email UNIQUE (email)
);
```

#### Characteristics

- Constraints are named explicitly
- Required for multi-column constraints
- Easier to manage and modify later

---

### 3. Common Constraint Types

| Constraint  | Purpose                              |
| ----------- | ------------------------------------ |
| PRIMARY KEY | Uniquely identifies a row            |
| FOREIGN KEY | Enforces relationship between tables |
| UNIQUE      | Prevents duplicate values            |
| NOT NULL    | Disallows NULL values                |
| CHECK       | Enforces a condition                 |
| DEFAULT     | Provides a default value             |

---

### 4. CHECK Constraint Example

```sql
CREATE TABLE customers (
    id INT PRIMARY KEY,
    score INT CHECK (score >= 0)
);
```

Ensures `score` is never negative.

---

### 5. DEFAULT Constraint Example

```sql
CREATE TABLE orders (
    id INT PRIMARY KEY,
    status VARCHAR(20) DEFAULT 'NEW'
);
```

If no value is provided, `status` becomes `NEW`.

---

### 6. FOREIGN KEY Constraint Example

```sql
CREATE TABLE orders (
    id INT PRIMARY KEY,
    person_id INT,
    CONSTRAINT fk_orders_persons
        FOREIGN KEY (person_id)
        REFERENCES persons(id)
);
```

Ensures `person_id` exists in the `persons` table.

---

### 7. Adding Constraints Later Using ALTER TABLE

Constraints can also be added **after table creation**.

```sql
ALTER TABLE persons
ADD CONSTRAINT uq_persons_email UNIQUE (email);
```

---

### Best Practice: Which Style to Use?

- Use **column-level** for simple rules (`NOT NULL`, simple `PRIMARY KEY`)
- Use **table-level** for:
  - Multi-column constraints
  - Named constraints
  - Better maintainability

---

## 4. Executing DDL Commands

When executing a DDL command:

- No rows or columns are returned
- You receive a **success or failure message**
- The database structure is immediately changed

Example check:

```sql
SELECT * FROM persons;
```

This shows columns but no rows (empty table).

---

## 5. Retrieving CREATE Script from Database

If you lose your original DDL script, you can retrieve it from the database metadata.

### Common Approach (GUI Tools)

- Right-click table → **Script Table As** → **CREATE**

This generates:

- Column definitions
- Constraints
- Schema information

⚠️ Best Practice: Always store DDL scripts in **Git repositories**.

---

## 6. ALTER TABLE

### Purpose

`ALTER TABLE` modifies the **structure of an existing table**.

You can:

- Add columns
- Remove columns
- Modify data types
- Add or remove constraints

---

## 7. Adding a Column

### Task

Add a new column `email` to the `persons` table.

### SQL Statement

```sql
ALTER TABLE persons
ADD email VARCHAR(50) NOT NULL;
```

### Key Notes

- New columns are added **at the end of the table**
- Existing rows must satisfy `NOT NULL` (or a default value is required)

---

## 7A. Renaming a Column

### Purpose

Renames an existing column **without losing its data**. This is useful when:

- Column names were poorly chosen initially
- Business terminology changes
- Improving schema readability

### Syntax (SQL Standard / PostgreSQL / MySQL 8+)

```sql
ALTER TABLE table_name
RENAME COLUMN old_column_name TO new_column_name;
```

### Example

```sql
ALTER TABLE persons
RENAME COLUMN person_name TO full_name;
```

### Important Notes

- Data inside the column is preserved
- Constraints remain attached to the column
- Indexes referencing the column are updated automatically

### Database-Specific Variations

#### MySQL (older versions)

```sql
ALTER TABLE persons
CHANGE person_name full_name VARCHAR(50) NOT NULL;
```

⚠️ In MySQL `CHANGE`, you must **redefine the data type**.

#### SQL Server

```sql
EXEC sp_rename 'persons.person_name', 'full_name', 'COLUMN';
```

### Best Practices

- Always check dependent objects (views, procedures)
- Avoid renaming columns in production without testing
- Prefer clear, business-meaningful names

---

## 8. Removing a Column

### Task

Remove the `phone` column from the `persons` table.

### SQL Statement

```sql
ALTER TABLE persons
DROP COLUMN phone;
```

### Warning

- Dropping a column permanently removes **all its data**
- This action cannot be undone without backups

---

## 9. Column Order Limitation

- SQL does **not guarantee column order**
- You cannot insert a column in the middle using `ALTER TABLE`
- To reorder columns, you must:
  1. Drop the table
  2. Recreate it with correct order
  3. Reinsert data (if any)

---

## 10. DROP TABLE

### Purpose

`DROP TABLE` removes a table **completely** from the database.

### SQL Statement

```sql
DROP TABLE persons;
```

### Effects

- Deletes table structure
- Deletes all data inside the table
- Cannot be rolled back in most databases

### Best Practices

- Double-check table name
- Ensure backups exist
- Avoid running in production without confirmation

---

## 10A. DROP DATABASE

### Purpose

`DROP DATABASE` removes an **entire database**, including:

- All tables
- Views
- Indexes
- Stored procedures
- All data

This is the **most destructive DDL command**.

### SQL Statement

```sql
DROP DATABASE database_name;
```

### Example

```sql
DROP DATABASE sales_db;
```

### Effects

- Permanently deletes the whole database
- All objects inside the database are lost
- Usually **cannot be rolled back**

### Safety Checks (Recommended)

```sql
DROP DATABASE IF EXISTS sales_db;
```

Prevents errors if the database does not exist.

### When Should You Use DROP DATABASE?

- Removing test or temporary databases
- Cleaning up unused environments
- Never directly in production without approval

### ⚠️ Critical Warning

- This command is **irreversible** in most systems
- Always:
  - Verify database name
  - Confirm environment (dev / test / prod)
  - Ensure full backups exist

Think of it as **formatting a hard drive**, not deleting a file.

---

## 11. CREATE vs ALTER vs DROP

| Command | Purpose                    | Risk Level |
| ------- | -------------------------- | ---------- |
| CREATE  | Build new structure        | Low        |
| ALTER   | Modify existing structure  | Medium     |
| DROP    | Destroy structure and data | High       |

---

## 12. DDL vs DML

| DDL                 | DML                    |
| ------------------- | ---------------------- |
| Defines structure   | Manipulates data       |
| CREATE, ALTER, DROP | INSERT, UPDATE, DELETE |
| No data returned    | Data affected          |

---

## 13. Summary

DDL commands define the **blueprint of the database**. They:

- Create tables
- Change schemas
- Remove objects

Because DDL directly impacts structure and data integrity, these commands must be used **carefully and deliberately**.
