# SQL DDL Constraints – Complete & Detailed Guide

## 1. What Are Constraints in SQL?

**Constraints** are rules enforced by the database to maintain **data integrity, accuracy, and consistency**. They restrict what values can be inserted, updated, or deleted in a table.

Constraints are evaluated by the database engine itself, making them **more reliable than application-level validations**.

---

## 2. Why Constraints Are Important

Constraints ensure:

- No duplicate or invalid data
- Correct relationships between tables
- Protection against accidental data corruption
- Business rules are enforced at the database level

---

## 3. Ways to Define Constraints

Constraints can be defined in **three ways**:

1. **Column-level constraints** (inline)
2. **Table-level constraints** (named & separate)
3. **Using ALTER TABLE (after creation)**

---

## 4. Column-Level Constraints

Defined directly next to the column definition.

### Example

```sql
CREATE TABLE users (
    id INT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL,
    age INT CHECK (age >= 18),
    created_at DATE DEFAULT CURRENT_DATE
);
```

### When to Use

- Single-column rules
- Simple validations

---

## 5. Table-Level Constraints

Defined after all columns and usually **named explicitly**.

### Example

```sql
CREATE TABLE users (
    id INT,
    username VARCHAR(50),
    email VARCHAR(100),
    age INT,
    CONSTRAINT pk_users PRIMARY KEY (id),
    CONSTRAINT uq_users_username UNIQUE (username),
    CONSTRAINT chk_users_age CHECK (age >= 18)
);
```

### Advantages

- Required for multi-column constraints
- Easier to maintain
- Preferred in production systems

---

## 6. PRIMARY KEY Constraint

### Purpose

- Uniquely identifies each row
- Cannot contain NULL
- Only one primary key per table

### Single-Column Primary Key

```sql
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(50)
);
```

### Composite Primary Key

```sql
CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    CONSTRAINT pk_order_items PRIMARY KEY (order_id, product_id)
);
```

---

## 7. FOREIGN KEY Constraint

### Purpose

- Enforces relationships between tables
- Maintains referential integrity

### Basic Foreign Key

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    CONSTRAINT fk_orders_customers
        FOREIGN KEY (customer_id)
        REFERENCES customers(customer_id)
);
```

### FOREIGN KEY with Actions

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    CONSTRAINT fk_orders_customers
        FOREIGN KEY (customer_id)
        REFERENCES customers(customer_id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
```

### Common Actions

| Action               | Behavior           |
| -------------------- | ------------------ |
| CASCADE              | Propagates change  |
| SET NULL             | Sets FK to NULL    |
| SET DEFAULT          | Sets default value |
| RESTRICT / NO ACTION | Prevents operation |

---

## 8. UNIQUE Constraint

### Purpose

- Prevents duplicate values
- Allows NULL (DB dependent)

### Example

```sql
CREATE TABLE users (
    id INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE
);
```

### Composite UNIQUE Constraint

```sql
CREATE TABLE enrollments (
    student_id INT,
    course_id INT,
    CONSTRAINT uq_enrollments UNIQUE (student_id, course_id)
);
```

---

## 9. NOT NULL Constraint

### Purpose

- Ensures column always has a value

### Example

```sql
CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL
);
```

---

## 10. CHECK Constraint

### Purpose

- Enforces custom business rules

### Example

```sql
CREATE TABLE products (
    id INT PRIMARY KEY,
    price DECIMAL(10,2) CHECK (price > 0),
    discount DECIMAL(5,2) CHECK (discount BETWEEN 0 AND 50)
);
```

### Multi-Column CHECK

```sql
CHECK (end_date >= start_date)
```

---

## 11. DEFAULT Constraint

### Purpose

- Provides default values when none are supplied

### Example

```sql
CREATE TABLE orders (
    id INT PRIMARY KEY,
    status VARCHAR(20) DEFAULT 'NEW',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

---

## 12. Adding Constraints Using ALTER TABLE

### Add PRIMARY KEY

```sql
ALTER TABLE users
ADD CONSTRAINT pk_users PRIMARY KEY (id);
```

### Add UNIQUE

```sql
ALTER TABLE users
ADD CONSTRAINT uq_users_email UNIQUE (email);
```

### Add FOREIGN KEY

```sql
ALTER TABLE orders
ADD CONSTRAINT fk_orders_users
FOREIGN KEY (user_id)
REFERENCES users(id);
```

### Add CHECK

```sql
ALTER TABLE users
ADD CONSTRAINT chk_users_age CHECK (age >= 18);
```

---

## 13. Dropping Constraints

```sql
ALTER TABLE users
DROP CONSTRAINT uq_users_email;
```

⚠️ Constraint name is required.

---

## 14. Best Practices

- Always name constraints
- Prefer table-level constraints in production
- Use CHECK for business rules
- Enforce relationships using FOREIGN KEY
- Keep validation at DB level, not only in code

---

## 15. Interview Tips

- Difference between PRIMARY KEY and UNIQUE
- WHERE vs CHECK
- FOREIGN KEY vs JOIN
- Column-level vs Table-level constraints

---

## 16. Summary

Constraints are the **foundation of reliable databases**. They protect data integrity, enforce relationships, and encode business rules directly into the database schema.

Mastering constraints is essential for:

- Database design
- Backend development
- System architecture
- SQL interviews
