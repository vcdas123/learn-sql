# Window Value Functions (Analytical Functions) in SQL

---

# 1. Introduction

Window value functions (also called analytical functions) allow you to **access a value from another row within the same window**.

They are extremely important in data analytics because they:

- Enable comparisons across rows
- Support time-series analysis
- Avoid complex self-joins
- Allow advanced calculations using simple SQL

The four core value functions are:

1. LEAD()
2. LAG()
3. FIRST_VALUE()
4. LAST_VALUE()

# 3. Why Are They Called Value Functions?

They allow SQL to access values from:

- Previous rows → LAG()
- Next rows → LEAD()
- First row in window → FIRST_VALUE()
- Last row in window → LAST_VALUE()

SQL must always **sort the data first using ORDER BY** to determine row position.

---

# 4. Syntax & Rules (Transcript Accurate)

## 4.1 LEAD / LAG Syntax

```sql
LEAD(expression, offset, default)
OVER (
    PARTITION BY column  -- optional
    ORDER BY column      -- required
)
```

```sql
LAG(expression, offset, default)
OVER (
    PARTITION BY column
    ORDER BY column
)
```

Rules:

- Expression is required
- Offset is optional (default = 1)
- Default value is optional (default = NULL)
- ORDER BY is mandatory
- Frame clause NOT allowed

---

## 4.2 FIRST_VALUE / LAST_VALUE Syntax

```sql
FIRST_VALUE(expression)
OVER (
    PARTITION BY column
    ORDER BY column
)
```

```sql
LAST_VALUE(expression)
OVER (
    PARTITION BY column
    ORDER BY column
    ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING
)
```

Rules:

- Expression required
- ORDER BY mandatory
- Frame optional for FIRST_VALUE
- Frame required for LAST_VALUE to work correctly

# 5. LEAD() and LAG()

## 5.1 Basic Example

```sql
SELECT
    OrderDate,
    Sales,
    LAG(Sales) OVER (ORDER BY OrderDate) AS PreviousSales,
    LEAD(Sales) OVER (ORDER BY OrderDate) AS NextSales
FROM SalesOrders;
```

### Output

| OrderDate  | Sales | PreviousSales | NextSales |
| ---------- | ----- | ------------- | --------- |
| 2024-01-01 | 90    | NULL          | 20        |
| 2024-01-05 | 25    | 90            | 20        |
| 2024-01-10 | 20    | 25            | 60        |
| 2024-01-20 | 60    | 20            | 30        |
| 2024-02-01 | 80    | 60            | 35        |
| 2024-02-05 | 35    | 80            | 30        |
| 2024-02-15 | 30    | 35            | 50        |
| 2024-03-01 | 50    | 30            | NULL      |

Explanation (one line): LAG accesses previous row value, LEAD accesses next row value after sorting.

Execution Flow:

1. Sort by OrderDate
2. Move pointer row by row
3. Fetch value using offset

---

## 5.2 Offset and Default Value Example

```sql
SELECT
    OrderDate,
    Sales,
    LEAD(Sales,2,0) OVER (ORDER BY OrderDate) AS TwoMonthsAhead
FROM SalesOrders;
```

Explanation (one line): LEAD jumps two rows forward and returns 0 if no row exists.

# 6. Month-over-Month Analysis (Time Series)

## 6.1 Step 1: Aggregate Monthly Sales

```sql
SELECT
    MONTH(OrderDate) AS OrderMonth,
    SUM(Sales) AS CurrentMonthSales
FROM SalesOrders
GROUP BY MONTH(OrderDate);
```

---

## 6.2 Step 2: Access Previous Month Using LAG

```sql
SELECT
    OrderMonth,
    CurrentMonthSales,
    LAG(CurrentMonthSales)
        OVER (ORDER BY OrderMonth) AS PreviousMonthSales
FROM (
    SELECT MONTH(OrderDate) AS OrderMonth,
           SUM(Sales) AS CurrentMonthSales
    FROM SalesOrders
    GROUP BY MONTH(OrderDate)
) t;
```

---

## 6.3 Step 3: Calculate Percentage Change

```sql
SELECT *,
ROUND(
   (CurrentMonthSales - PreviousMonthSales)
   / CAST(PreviousMonthSales AS FLOAT) * 100, 1
) AS MonthOverMonthPercentage
FROM (
    SELECT
        OrderMonth,
        CurrentMonthSales,
        LAG(CurrentMonthSales)
        OVER (ORDER BY OrderMonth) AS PreviousMonthSales
    FROM (
        SELECT MONTH(OrderDate) AS OrderMonth,
               SUM(Sales) AS CurrentMonthSales
        FROM SalesOrders
        GROUP BY MONTH(OrderDate)
    ) t
) x;
```

Explanation (one line): Calculates growth or decline compared to previous month.

# 7. Customer Retention Analysis (Time Gap Analysis)

## Step 1: Get Next Order Per Customer

```sql
SELECT
    CustomerID,
    OrderDate,
    LEAD(OrderDate)
      OVER (PARTITION BY CustomerID ORDER BY OrderDate) AS NextOrderDate
FROM SalesOrders;
```

## Step 2: Calculate Days Between Orders

```sql
DATEDIFF(DAY, OrderDate, NextOrderDate)
```

## Step 3: Average Days Per Customer

```sql
SELECT
    CustomerID,
    AVG(DaysBetweenOrders) AS AverageDays
FROM (
    SELECT
        CustomerID,
        DATEDIFF(DAY,
                 OrderDate,
                 LEAD(OrderDate)
                 OVER (PARTITION BY CustomerID ORDER BY OrderDate)
        ) AS DaysBetweenOrders
    FROM SalesOrders
) t
GROUP BY CustomerID;
```

Explanation (one line): Measures customer loyalty based on time between purchases.

# 8. FIRST_VALUE() and LAST_VALUE()

## 8.1 Lowest Sales Per Product

```sql
SELECT
    ProductID,
    Sales,
    FIRST_VALUE(Sales)
    OVER (PARTITION BY ProductID ORDER BY Sales ASC)
    AS LowestSales
FROM SalesOrders;
```

---

## 8.2 Highest Sales Per Product (Correct Way)

```sql
SELECT
    ProductID,
    Sales,
    LAST_VALUE(Sales)
    OVER (
        PARTITION BY ProductID
        ORDER BY Sales ASC
        ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING
    ) AS HighestSales
FROM SalesOrders;
```

Explanation (one line): LAST_VALUE requires custom frame; otherwise it returns current row value.

---

## Alternative Using FIRST_VALUE

```sql
FIRST_VALUE(Sales)
OVER (PARTITION BY ProductID ORDER BY Sales DESC)
```

---

## Alternative Using MAX()

```sql
MAX(Sales)
OVER (PARTITION BY ProductID)
```

---

# 9. Performance Notes

- ORDER BY causes sorting (expensive on large datasets).
- Indexing ORDER BY and PARTITION BY columns improves speed.
- Avoid unnecessary nested window queries.

---

# 10. Common Mistakes

1. Forgetting ORDER BY (mandatory).
2. Not defining frame for LAST_VALUE.
3. Forgetting NULL handling in LEAD/LAG.
4. Mixing aggregation and window incorrectly.
5. Not casting integers in percentage calculations.

---

# 11. Best Practices

- Use LEAD/LAG for time-series comparison.
- Always define frame for LAST_VALUE.
- Prefer FIRST_VALUE with reversed ORDER BY instead of LAST_VALUE when possible.
- Use window functions instead of self-joins.
- Handle NULL using default parameter.

---

# 12. Final Summary Table

| Function    | Purpose               | Frame Needed | Typical Use Case  |
| ----------- | --------------------- | ------------ | ----------------- |
| LAG         | Previous row value    | No           | Month-over-month  |
| LEAD        | Next row value        | No           | Time gap analysis |
| FIRST_VALUE | First value in window | No           | Lowest value      |
| LAST_VALUE  | Last value in window  | Yes          | Highest value     |

Mastering LEAD, LAG, FIRST_VALUE, and LAST_VALUE unlocks advanced SQL analytical capabilities.
