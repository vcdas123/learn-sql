## 1. What Is a Stored Procedure?

A **Stored Procedure** is a named, pre-compiled collection of one or more SQL statements that is saved **inside the database server** and can be called (executed) at any time with a single command.

### The Coffee Shop Analogy

> Every time you go to a coffee shop, you say: *"I'd like a large coffee with coconut milk, no sugar, and extra whipped cream."*
>
> You repeat this word-for-word every single visit.
>
> With a stored procedure, you simply say: **"Give me my usual."**
> The barista already knows exactly what that means.

This is precisely what stored procedures do — they encapsulate a complex sequence of SQL logic under a single, simple name. Instead of re-writing or re-sharing your SQL scripts every time, you just call the procedure.

### Key Concept: Client Side vs. Server Side

Without stored procedures, SQL statements live on the **client side** — in your local files, scripts, or editor. You manually send them to the database each time.

With stored procedures, your SQL statements are stored **inside the database server**:

```
┌─────────────────────┐          ┌──────────────────────────────┐
│    CLIENT SIDE      │          │       SERVER SIDE (DB)       │
│                     │          │                              │
│  EXECUTE            │ ──────►  │  Stored Procedure:           │
│  get_customer_      │          │    1. UPDATE sales.customers  │
│  summary            │          │    2. SELECT COUNT(*)         │
│                     │ ◄──────  │    3. SELECT SUM(sales)       │
│  ← results          │          │                              │
└─────────────────────┘          └──────────────────────────────┘
```

The database executes all statements in the exact defined order, from top to bottom, and returns the results.

## 2. Stored Procedure vs. Plain SQL Query

| Feature | Plain SQL Query | Stored Procedure |
|---|---|---|
| **Number of statements** | One request at a time | Multiple statements in one call |
| **Transactions** | Single interaction | Multiple interactions in one execution |
| **Logic** | None (just a query) | Loops, conditionals, error handling |
| **Parameters** | None | Dynamic input parameters |
| **Variables** | Not supported | Fully supported |
| **Storage** | Client-side (your files) | Server-side (inside the database) |
| **Reusability** | Copy-paste required | Call by name from anywhere |
| **Compilation** | Re-parsed each run | Pre-compiled; execution plan cached |

A plain SQL query is a **single request**. A stored procedure is closer to **a program** — it has control flow, branching logic, variables, error handling, and can perform dozens of operations in sequence.

## 3. Stored Procedure vs. Python

Both stored procedures and Python scripts can execute a series of SQL statements. Here's how they compare:

| Aspect | Stored Procedure | Python Script |
|---|---|---|
| **Location** | Inside the database | External server |
| **Network overhead** | None (runs in-DB) | Requires DB connection (slight latency) |
| **Pre-compilation** | ✅ Yes — syntax-checked and execution-planned in advance | ❌ No — DB sees SQL only at runtime |
| **Complex logic** | Hard to implement (verbose, limited) | ✅ Easy — full Python ecosystem available |
| **Version control** | Difficult (stored in DB catalog) | ✅ Native (Git, CI/CD pipelines) |
| **Debugging & testing** | Very difficult | ✅ Easy (unit tests, logging, IDEs) |
| **Maintainability** | Poor for large projects | ✅ Much better |
| **Platform dependency** | Tied to the database | Platform-independent |

### Instructor's Honest Take

> *"If you have the possibility to put your code in Python — in platforms like Databricks or Snowflake — do it. I've seen a lot of projects using stored procedures, and most of them end in chaos. It's really hard to debug, hard to test. But if you only have a database server and no other option, stored procedures are your tool. For small projects with a few tables, they're fine. Never build a big project on stored procedures alone."*

**Summary rule of thumb:**
- Small project, DB-only environment → Stored procedures are acceptable.
- Large project, modern data platform → Use Python.

## 5. Basic Syntax & Creating Your First Procedure

### 5.1 The Two Steps

Building a stored procedure always involves exactly two steps:

1. **Define (CREATE)** — write the procedure body with its SQL statements.
2. **Execute (EXEC/CALL)** — run the procedure by name.

### 5.2 Syntax

```sql
-- Step 1: Define the procedure
CREATE PROCEDURE procedure_name
AS
BEGIN
    -- SQL statements go here
    SELECT ...;
    INSERT ...;
    UPDATE ...;
END;

-- Step 2: Execute the procedure
EXECUTE procedure_name;
```

### 5.3 Example: Weekly Customer Summary Report

**The task:** For USA customers, find the total number of customers and their average score.

**Step 1 — Write and test the query first:**

```sql
SELECT
    COUNT(*)        AS total_customers,
    AVG(score)      AS average_score
FROM sales_customers
WHERE country = 'USA';
```

**Output:**

| total_customers | average_score |
|---|---|
| 3 | 812 |

*(NULL in Bob's score is excluded from AVG by default)*

**Step 2 — Wrap it in a stored procedure:**

```sql
CREATE PROCEDURE get_customer_summary
AS
BEGIN
    SELECT
        COUNT(*)    AS total_customers,
        AVG(score)  AS average_score
    FROM sales_customers
    WHERE country = 'USA';
END;
```

**Step 3 — Execute it:**

```sql
EXECUTE get_customer_summary;
```

**Output:**

| total_customers | average_score |
|---|---|
| 3 | 812 |

The procedure is now stored permanently in the database. You never need to re-type this query — just call `EXECUTE get_customer_summary` whenever you need the report.

> 📁 In SQL Server's Object Explorer, you can find stored procedures under:
> `Database → Programmability → Stored Procedures`

## 6. Parameters

### 6.1 Definition

A **parameter** is a named placeholder declared in the procedure's signature that accepts a value from the **caller at runtime**. Parameters make stored procedures **flexible, dynamic, and reusable** — instead of one hardcoded procedure per country, you have one procedure that works for any country.

### 6.2 The Problem Parameters Solve

Without parameters, you'd need a separate procedure for each country:

```sql
-- BAD: code duplication
CREATE PROCEDURE get_customer_summary_USA     AS BEGIN ... WHERE country = 'USA'     END;
CREATE PROCEDURE get_customer_summary_Germany AS BEGIN ... WHERE country = 'Germany' END;
```

This violates the **DRY principle** (Don't Repeat Yourself). If the logic needs to change, you'd have to update every copy.

### 6.3 Syntax

```sql
CREATE PROCEDURE procedure_name
    @parameter_name DATATYPE       -- parameter declaration
AS
BEGIN
    SELECT ...
    WHERE column = @parameter_name; -- parameter used inside
END;
```

- Parameter names start with `@` in SQL Server / T-SQL.
- You must declare a data type for each parameter (same rules as column data types).

### 6.4 Example: Dynamic Country Filter

```sql
-- Drop the old version first (or use ALTER — see below)
CREATE PROCEDURE get_customer_summary
    @country VARCHAR(50)          -- ← parameter declared here
AS
BEGIN
    SELECT
        COUNT(*)    AS total_customers,
        AVG(score)  AS average_score
    FROM sales_customers
    WHERE country = @country;     -- ← parameter used here
END;
```

**Execute for USA:**

```sql
EXECUTE get_customer_summary @country = 'USA';
```

| total_customers | average_score |
|---|---|
| 3 | 812 |

**Execute for Germany:**

```sql
EXECUTE get_customer_summary @country = 'Germany';
```

| total_customers | average_score |
|---|---|
| 2 | 425 |

One procedure. Infinite reusability.

### 6.5 Modifying an Existing Procedure

Use `ALTER PROCEDURE` to update a procedure's definition without dropping and recreating it:

```sql
ALTER PROCEDURE get_customer_summary
    @country VARCHAR(50)
AS
BEGIN
    -- updated logic here
END;
```

## 7. Default Parameter Values

If you call a procedure with a certain parameter value **very frequently**, you can define it as the **default** — so the caller can omit it entirely.

### 7.1 Syntax

```sql
CREATE PROCEDURE procedure_name
    @parameter_name DATATYPE = 'default_value'   -- ← default defined here
AS
BEGIN
    ...
END;
```

### 7.2 Example

```sql
ALTER PROCEDURE get_customer_summary
    @country VARCHAR(50) = 'USA'    -- USA is the default
AS
BEGIN
    SELECT
        COUNT(*)    AS total_customers,
        AVG(score)  AS average_score
    FROM sales_customers
    WHERE country = @country;
END;
```

**Call without a parameter (uses default 'USA'):**

```sql
EXECUTE get_customer_summary;
```

| total_customers | average_score |
|---|---|
| 3 | 812 |

**Call with an explicit value (overrides the default):**

```sql
EXECUTE get_customer_summary @country = 'Germany';
```

| total_customers | average_score |
|---|---|
| 2 | 425 |

> ✅ The explicitly passed value always takes priority over the default.

## 8. Multiple SQL Statements in One Procedure

One of the core strengths of stored procedures is the ability to run **multiple SQL statements in a single call** — selects, inserts, updates, and deletes can all coexist in one procedure.

### 8.1 Example: Two Reports in One Procedure

**Task:** For a given country, generate two reports:
1. Total customers and average score.
2. Total number of orders and total sales.

```sql
ALTER PROCEDURE get_customer_summary
    @country VARCHAR(50) = 'USA'
AS
BEGIN

    -- Report 1: Customer summary
    SELECT
        COUNT(*)    AS total_customers,
        AVG(score)  AS average_score
    FROM sales_customers
    WHERE country = @country;

    -- Report 2: Order summary
    SELECT
        COUNT(o.order_id)   AS total_orders,
        SUM(o.sales)        AS total_sales
    FROM sales_orders o
    JOIN sales_customers c ON o.customer_id = c.customer_id
    WHERE c.country = @country;

END;
```

**Execute:**

```sql
EXECUTE get_customer_summary @country = 'USA';
```

**Result Set 1 — Customer Summary:**

| total_customers | average_score |
|---|---|
| 3 | 812 |

**Result Set 2 — Order Summary:**

| total_orders | total_sales |
|---|---|
| 4 | 1008.00 |

> ✅ **Best practice:** Add a semicolon (`;`) at the end of each statement inside the procedure. While the database doesn't always require it, it makes multi-query procedures significantly easier to read and debug.

## 9. Variables

### 9.1 Definition

A **variable** is a named placeholder that lives **inside the stored procedure** and holds a value in memory that can be read and reused throughout the procedure.

**Variables vs. Parameters:**

| | Parameter | Variable |
|---|---|---|
| **Source** | Input from the caller (external) | Created inside the procedure (internal) |
| **Purpose** | Pass values in from outside | Store and reuse intermediate results |
| **Defined with** | In the procedure signature | `DECLARE` statement |

### 9.2 The Three Steps to Using Variables

1. **DECLARE** — tell SQL to create a placeholder in memory.
2. **SET / SELECT INTO** — assign a value to the variable.
3. **USE** — reference the variable anywhere in the procedure.

### 9.3 Syntax

```sql
-- Step 1: Declare
DECLARE @variable_name DATATYPE;

-- Step 2: Assign (option A — using SELECT)
SELECT @variable_name = column_name
FROM table_name
WHERE condition;

-- Step 2: Assign (option B — using SET for a fixed value)
SET @variable_name = 42;

-- Step 3: Use anywhere
PRINT 'The value is: ' + CAST(@variable_name AS VARCHAR);
SELECT @variable_name AS result;
```

### 9.4 Example: Printing a Formatted Text Report

Instead of returning a table, we want to generate a human-readable text message like:

```
Total customers from Germany = 2
Average score from Germany = 425
```

```sql
ALTER PROCEDURE get_customer_summary
    @country VARCHAR(50) = 'USA'
AS
BEGIN

    -- Step 1: Declare variables
    DECLARE @total_customers INT;
    DECLARE @average_score   FLOAT;

    -- Step 2: Assign values from the query
    SELECT
        @total_customers = COUNT(*),
        @average_score   = AVG(score)
    FROM sales_customers
    WHERE country = @country;

    -- Step 3: Use variables in PRINT messages
    PRINT 'Total customers from ' + @country + ': ' + CAST(@total_customers AS VARCHAR);
    PRINT 'Average score from '   + @country + ': ' + CAST(@average_score   AS VARCHAR);

END;
```

**Execute:**

```sql
EXECUTE get_customer_summary @country = 'Germany';
```

**Messages output:**

```
Total customers from Germany: 2
Average score from Germany: 425
```

> ⚠️ `PRINT` only accepts string (VARCHAR) values. Use `CAST(value AS VARCHAR)` to convert numbers before printing.

> ⚠️ When you assign values to variables using `SELECT`, the query **does not return a result set** — its only job is to populate the variables.

## 10. Control Flow — IF / ELSE

### 10.1 Definition

**IF / ELSE** is a control flow construct that lets the procedure follow **different execution paths** based on a condition. This is what separates a stored procedure from a simple query — it can make decisions.

### 10.2 Syntax

```sql
IF <condition>
BEGIN
    -- Executed if condition is TRUE
    ...
END
ELSE
BEGIN
    -- Executed if condition is FALSE
    ...
END;
```

The `EXISTS` keyword is especially useful in conditions — it returns `TRUE` if a subquery returns any rows:

```sql
IF EXISTS (SELECT 1 FROM table WHERE condition)
BEGIN
    -- rows exist → do something
END
ELSE
BEGIN
    -- no rows → do something else
END;
```

### 10.3 Use Case: Cleaning NULL Scores Before Reporting

**Problem:** The `score` column sometimes contains `NULL` values. NULLs distort aggregations. We want to replace NULLs with `0` — but **only if NULLs actually exist**. Running a blanket `UPDATE` every time wastes resources on large tables.

**Solution:** Check first. Update only if needed.

```sql
ALTER PROCEDURE get_customer_summary
    @country VARCHAR(50) = 'USA'
AS
BEGIN

    -- ============================================
    -- STEP 1: Prepare & Clean Data
    -- ============================================

    IF EXISTS (
        SELECT 1
        FROM sales_customers
        WHERE score IS NULL
          AND country = @country
    )
    BEGIN
        PRINT 'Updating NULL scores to 0...';
        UPDATE sales_customers
        SET    score = 0
        WHERE  score IS NULL
          AND  country = @country;
    END
    ELSE
    BEGIN
        PRINT 'No NULL scores found. No update needed.';
    END;

    -- ============================================
    -- STEP 2: Generate Reports
    -- ============================================

    -- Report 1: Customer summary
    SELECT
        COUNT(*)    AS total_customers,
        AVG(score)  AS average_score
    FROM sales_customers
    WHERE country = @country;

    -- Report 2: Order summary
    SELECT
        COUNT(o.order_id)   AS total_orders,
        SUM(o.sales)        AS total_sales
    FROM sales_orders o
    JOIN sales_customers c ON o.customer_id = c.customer_id
    WHERE c.country = @country;

END;
```

**Execute for USA (Bob has a NULL score):**

```sql
EXECUTE get_customer_summary @country = 'USA';
```

**Messages:**
```
Updating NULL scores to 0...
```

**Result Set 1:**

| total_customers | average_score |
|---|---|
| 3 | 541 |

*(Now includes Bob's score of 0, giving a more complete average)*

**Execute for Germany (no NULLs):**

```sql
EXECUTE get_customer_summary @country = 'Germany';
```

**Messages:**
```
No NULL scores found. No update needed.
```

### 10.4 Execution Flow

```
EXECUTE get_customer_summary
          │
          ▼
   IF EXISTS (NULLs?) ──── TRUE ──►  UPDATE score = 0 for NULLs
          │                                     │
        FALSE                                   │
          │                                     │
          ▼                                     ▼
   PRINT "No NULLs"               PRINT "Updating NULLs"
          │                                     │
          └──────────────┬──────────────────────┘
                         ▼
               Run Report 1 (customer summary)
                         │
                         ▼
               Run Report 2 (order summary)
                         │
                         ▼
                      DONE
```

## 11. Error Handling — TRY / CATCH

### 11.1 Definition

**TRY / CATCH** is SQL's error handling mechanism. It lets you define:
- **TRY block:** the code you *attempt* to execute.
- **CATCH block:** what to do *if something goes wrong*.

If no error occurs, the CATCH block is skipped entirely. If any error occurs inside TRY, SQL immediately jumps to CATCH.

### 11.2 Syntax

```sql
BEGIN TRY
    -- Your SQL statements
    -- If anything fails here → jump to CATCH
END TRY
BEGIN CATCH
    -- Error handling logic
    -- Access built-in error functions here
END CATCH;
```

**Built-in error information functions:**

| Function | Returns |
|---|---|
| `ERROR_MESSAGE()` | Human-readable error description |
| `ERROR_NUMBER()` | Numeric error code |
| `ERROR_LINE()` | Line number where the error occurred |
| `ERROR_PROCEDURE()` | Name of the stored procedure that failed |
| `ERROR_SEVERITY()` | Severity level of the error |
| `ERROR_STATE()` | Error state number |

### 11.3 Example: Wrapping the Full Procedure in TRY / CATCH

```sql
ALTER PROCEDURE get_customer_summary
    @country VARCHAR(50) = 'USA'
AS
BEGIN

    BEGIN TRY

        -- ============================================
        -- STEP 1: Prepare & Clean Data
        -- ============================================

        IF EXISTS (
            SELECT 1 FROM sales_customers
            WHERE score IS NULL AND country = @country
        )
        BEGIN
            PRINT 'Updating NULL scores to 0...';
            UPDATE sales_customers
            SET    score = 0
            WHERE  score IS NULL AND country = @country;
        END
        ELSE
        BEGIN
            PRINT 'No NULL scores found.';
        END;

        -- ============================================
        -- STEP 2: Generate Reports
        -- ============================================

        -- Report 1: Calculate total customers and average score
        SELECT
            COUNT(*)    AS total_customers,
            AVG(score)  AS average_score
        FROM sales_customers
        WHERE country = @country;

        -- Report 2: Calculate total orders and total sales
        SELECT
            COUNT(o.order_id)   AS total_orders,
            SUM(o.sales)        AS total_sales
        FROM sales_orders o
        JOIN sales_customers c ON o.customer_id = c.customer_id
        WHERE c.country = @country;

    END TRY
    BEGIN CATCH

        -- ============================================
        -- ERROR HANDLING
        -- ============================================

        PRINT 'An error occurred!';
        PRINT 'Error Message  : ' + ERROR_MESSAGE();
        PRINT 'Error Number   : ' + CAST(ERROR_NUMBER()    AS VARCHAR);
        PRINT 'Error Line     : ' + CAST(ERROR_LINE()      AS VARCHAR);
        PRINT 'Error Procedure: ' + ERROR_PROCEDURE();

    END CATCH;

END;
```

**If a divide-by-zero error were introduced, the CATCH block would output:**

```
An error occurred!
Error Message  : Divide by zero error encountered.
Error Number   : 8134
Error Line     : 35
Error Procedure: get_customer_summary
```

Instead of a raw database error crashing the caller's application, you get a controlled, informative message.

### 11.4 TRY / CATCH Workflow

```
BEGIN TRY
    Execute statement 1 ─── error? ──►  Jump to CATCH immediately
    Execute statement 2 ─── error? ──►  Jump to CATCH immediately
    Execute statement 3 ─── OK
    ...
END TRY
    │ (no errors)
    ▼
  DONE — CATCH block is skipped entirely

BEGIN CATCH (only reached if error occurred)
    Print structured error message
END CATCH
```

## 12. Styling & Organizing a Stored Procedure

As stored procedures grow, readability becomes critical. Use these conventions consistently.

### 12.1 Indentation

Use **tabs** to indent everything inside `BEGIN...END` blocks:

```sql
CREATE PROCEDURE my_procedure
AS
BEGIN                           -- outer BEGIN

    BEGIN TRY                   -- indented 1 level

        IF EXISTS (...)
        BEGIN                   -- indented 2 levels
            UPDATE ...;
        END;

        SELECT ...;             -- indented 2 levels

    END TRY
    BEGIN CATCH

        PRINT ERROR_MESSAGE();  -- indented 2 levels

    END CATCH;

END;                            -- outer END
```

### 12.2 Section Comments

Break the procedure into named sections using comment banners:

```sql
-- ============================================================
-- STEP 1: Prepare & Clean Data
-- ============================================================

-- ============================================================
-- STEP 2: Generate Summary Reports
-- Calculate total customers and average score by country
-- ============================================================

-- ============================================================
-- ERROR HANDLING
-- ============================================================
```

### 12.3 Semicolons

Add a semicolon after every SQL statement inside the procedure body. It's not always syntactically required, but it dramatically improves readability in long, multi-statement procedures.

### 12.4 Naming Conventions

| Object | Convention | Example |
|---|---|---|
| Stored Procedure | `verb_noun` or `get_/usp_` prefix | `get_customer_summary`, `usp_refresh_orders` |
| Parameters | `@descriptive_name` | `@country`, `@start_date` |
| Variables | `@descriptive_name` | `@total_customers`, `@avg_score` |

## 13. Dialect Differences: MySQL vs MS SQL Server vs PostgreSQL

### 13.1 Creating a Stored Procedure

```sql
-- MS SQL Server (T-SQL)
CREATE PROCEDURE get_customer_summary
    @country VARCHAR(50) = 'USA'
AS
BEGIN
    SELECT COUNT(*) AS total_customers
    FROM sales_customers
    WHERE country = @country;
END;

-- MySQL
DELIMITER $$
CREATE PROCEDURE get_customer_summary(
    IN country_param VARCHAR(50)
)
BEGIN
    SELECT COUNT(*) AS total_customers
    FROM sales_customers
    WHERE country = country_param;
END $$
DELIMITER ;

-- PostgreSQL (uses PL/pgSQL)
CREATE OR REPLACE PROCEDURE get_customer_summary(
    country_param VARCHAR DEFAULT 'USA'
)
LANGUAGE plpgsql
AS $$
BEGIN
    -- PostgreSQL procedures cannot return result sets directly
    -- Use RAISE NOTICE for messages, or use a FUNCTION instead
    RAISE NOTICE 'Filtering by country: %', country_param;
END;
$$;
```

> ⚠️ **PostgreSQL caveat:** Standard `PROCEDURE` objects in PostgreSQL (added in v11) **cannot return result sets**. For query-returning logic, use a `FUNCTION` with `RETURNS TABLE` or `RETURNS SETOF` instead (see below).

### 13.2 Executing a Procedure

```sql
-- MS SQL Server
EXECUTE get_customer_summary @country = 'Germany';
-- or shorthand:
EXEC get_customer_summary 'Germany';

-- MySQL
CALL get_customer_summary('Germany');

-- PostgreSQL
CALL get_customer_summary('Germany');
```

### 13.3 Variables

```sql
-- MS SQL Server
DECLARE @total INT;
SELECT @total = COUNT(*) FROM sales_customers WHERE country = 'USA';
PRINT CAST(@total AS VARCHAR);

-- MySQL
DECLARE total_var INT DEFAULT 0;
SELECT COUNT(*) INTO total_var FROM sales_customers WHERE country = 'USA';
SELECT total_var;

-- PostgreSQL (inside PL/pgSQL function)
DECLARE total_var INT;
BEGIN
    SELECT COUNT(*) INTO total_var FROM sales_customers WHERE country = 'USA';
    RAISE NOTICE 'Total: %', total_var;
END;
```

### 13.4 IF / ELSE

```sql
-- MS SQL Server
IF EXISTS (SELECT 1 FROM sales_customers WHERE score IS NULL AND country = @country)
BEGIN
    UPDATE sales_customers SET score = 0 WHERE score IS NULL AND country = @country;
END
ELSE
BEGIN
    PRINT 'No NULLs found.';
END;

-- MySQL
IF EXISTS (SELECT 1 FROM sales_customers WHERE score IS NULL AND country = country_param) THEN
    UPDATE sales_customers SET score = 0 WHERE score IS NULL AND country = country_param;
ELSE
    SELECT 'No NULLs found.' AS message;
END IF;

-- PostgreSQL (PL/pgSQL)
IF EXISTS (SELECT 1 FROM sales_customers WHERE score IS NULL AND country = country_param) THEN
    UPDATE sales_customers SET score = 0 WHERE score IS NULL AND country = country_param;
ELSE
    RAISE NOTICE 'No NULLs found.';
END IF;
```

### 13.5 Error Handling

```sql
-- MS SQL Server — TRY/CATCH
BEGIN TRY
    UPDATE sales_customers SET score = score / 0;  -- intentional error
END TRY
BEGIN CATCH
    PRINT 'Error: ' + ERROR_MESSAGE();
    PRINT 'Line: '  + CAST(ERROR_LINE() AS VARCHAR);
END CATCH;

-- MySQL — DECLARE HANDLER
DECLARE EXIT HANDLER FOR SQLEXCEPTION
BEGIN
    GET DIAGNOSTICS CONDITION 1
        @msg = MESSAGE_TEXT;
    SELECT CONCAT('Error: ', @msg) AS error_message;
END;

-- PostgreSQL — EXCEPTION block (PL/pgSQL)
BEGIN
    UPDATE sales_customers SET score = score / 0;
EXCEPTION
    WHEN division_by_zero THEN
        RAISE NOTICE 'Caught error: division by zero';
    WHEN OTHERS THEN
        RAISE NOTICE 'Caught error: %', SQLERRM;
END;
```

### 13.6 Default Parameter Values

```sql
-- MS SQL Server
@country VARCHAR(50) = 'USA'

-- MySQL (uses DEFAULT keyword in CALL, or set via IF)
-- MySQL doesn't natively support default parameter values in procedures.
-- Workaround:
IF country_param IS NULL THEN SET country_param = 'USA'; END IF;

-- PostgreSQL
country_param VARCHAR DEFAULT 'USA'
```

### 13.7 Returning Results from Procedures

| Database | Method |
|---|---|
| **MS SQL Server** | `SELECT` inside procedure body — returns result sets directly |
| **MySQL** | `SELECT` inside procedure body — returns result sets directly |
| **PostgreSQL** | Use a `FUNCTION` with `RETURNS TABLE` or `RETURNS SETOF record`; standard `PROCEDURE` cannot return data |

```sql
-- PostgreSQL: use a FUNCTION to return data
CREATE OR REPLACE FUNCTION get_customer_summary(country_param VARCHAR DEFAULT 'USA')
RETURNS TABLE(total_customers BIGINT, average_score NUMERIC)
LANGUAGE plpgsql
AS $$
BEGIN
    RETURN QUERY
        SELECT COUNT(*), AVG(score::NUMERIC)
        FROM sales_customers
        WHERE country = country_param;
END;
$$;

-- Call it like a table:
SELECT * FROM get_customer_summary('USA');
```

## 14. Common Mistakes

### ❌ Mistake 1: Using `CREATE` Instead of `ALTER` When Updating

```sql
-- BAD: will fail if the procedure already exists
CREATE PROCEDURE get_customer_summary ...

-- GOOD: modifies an existing procedure
ALTER PROCEDURE get_customer_summary ...
```

> In PostgreSQL and MySQL, use `CREATE OR REPLACE PROCEDURE`.

### ❌ Mistake 2: Hardcoding Values Instead of Using Parameters

```sql
-- BAD: only ever works for USA
WHERE country = 'USA'

-- GOOD: works for any country
WHERE country = @country
```

### ❌ Mistake 3: Running Updates Without a WHERE Clause

```sql
-- CATASTROPHIC: updates every row in the table
UPDATE sales_customers SET score = 0 WHERE score IS NULL;
-- (forgot AND country = @country)
```

> Always double-check that UPDATE and DELETE statements inside procedures include the correct WHERE clause.

### ❌ Mistake 4: Printing Numbers Without CAST

```sql
-- BAD: type mismatch error in SQL Server
PRINT 'Total: ' + @total_customers;

-- GOOD
PRINT 'Total: ' + CAST(@total_customers AS VARCHAR);
```

### ❌ Mistake 5: Skipping TRY / CATCH in Production Procedures

Without error handling, a failure in the middle of a multi-step procedure can leave your data in a partially updated, inconsistent state. Always wrap production procedures in TRY / CATCH.

### ❌ Mistake 6: Bloating a Single Procedure with Too Much Logic

If a procedure exceeds ~100 lines or handles more than 2–3 distinct responsibilities, split it into smaller, focused procedures. Smaller procedures are easier to test, debug, and reuse.

## 15. Real-World Analytics Use Cases

### Use Case 1: Automated Weekly KPI Report

A weekly job calls one procedure that cleans data, calculates KPIs, and prints a structured summary — replacing a manual, error-prone script-sharing workflow.

```sql
EXECUTE get_customer_summary @country = 'USA';
```

### Use Case 2: ETL Data Pipeline Step

A stored procedure handles one stage of an ETL pipeline — extracting, cleaning, and loading data into a staging table:

```sql
CREATE PROCEDURE usp_load_staging_orders
    @batch_date DATE
AS
BEGIN
    BEGIN TRY
        -- Extract
        INSERT INTO staging_orders
        SELECT * FROM raw_orders WHERE order_date = @batch_date;

        -- Log success
        INSERT INTO etl_log (step, status, run_date)
        VALUES ('load_staging_orders', 'SUCCESS', GETDATE());
    END TRY
    BEGIN CATCH
        INSERT INTO etl_log (step, status, error_msg, run_date)
        VALUES ('load_staging_orders', 'FAILED', ERROR_MESSAGE(), GETDATE());
    END CATCH;
END;
```

### Use Case 3: Dynamic Country Report Generator

An analyst runs the same procedure for multiple countries by changing a single parameter, instead of maintaining separate queries per region.

```sql
EXECUTE get_customer_summary @country = 'USA';
EXECUTE get_customer_summary @country = 'Germany';
EXECUTE get_customer_summary @country = 'India';
```

### Use Case 4: Data Quality Check Procedure

```sql
CREATE PROCEDURE usp_data_quality_check
    @table_name VARCHAR(100)
AS
BEGIN
    DECLARE @null_count INT;

    SELECT @null_count = COUNT(*)
    FROM sales_customers
    WHERE score IS NULL;

    IF @null_count > 0
        PRINT 'WARNING: ' + CAST(@null_count AS VARCHAR) + ' NULL scores found.';
    ELSE
        PRINT 'Data quality check passed.';
END;
```

## 16. Best Practices

1. **Always write and test the SQL query first**, then wrap it in a stored procedure. Never write a procedure from scratch.

2. **Use parameters** for any value that might change between executions. Never hardcode filter values.

3. **Set default parameter values** for frequently used inputs to reduce repetitive typing.

4. **Add semicolons** after every statement inside the procedure body for readability.

5. **Declare all variables at the top** of the procedure body, immediately after `BEGIN`. This mirrors the convention from languages like Python and makes the procedure easier to scan.

6. **Always use TRY / CATCH in production** procedures. At minimum, log the error message, number, and line.

7. **Use IF EXISTS before UPDATE/DELETE** when you only need to modify data under certain conditions — avoid blind bulk updates that waste resources.

8. **Organize with section comments.** Label each logical block (`STEP 1: Prepare Data`, `STEP 2: Generate Reports`, `ERROR HANDLING`) so the procedure reads like a structured document.

9. **Indent consistently.** Each `BEGIN...END` block should be indented one level deeper than its parent.

10. **Keep procedures focused.** One procedure = one responsibility. Combine procedures at the application layer, not inside a single monolithic procedure.

11. **For large or complex projects, prefer Python** over stored procedures. Python offers better version control, testing, debugging, and maintainability.

## 17. Final Summary Table

| Feature | Description | Key Syntax |
|---|---|---|
| **Create** | Define a new stored procedure | `CREATE PROCEDURE name AS BEGIN ... END` |
| **Modify** | Update an existing procedure's definition | `ALTER PROCEDURE name AS BEGIN ... END` |
| **Execute** | Run the procedure | `EXECUTE name` / `EXEC name` / `CALL name` |
| **Drop** | Permanently delete the procedure | `DROP PROCEDURE name` |
| **Parameter** | Dynamic input value from the caller | `@param DATATYPE` in signature |
| **Default Value** | Fallback if no parameter is provided | `@param DATATYPE = 'default'` |
| **Variable** | Internal value holder | `DECLARE @var DATATYPE; SELECT @var = ...` |
| **IF / ELSE** | Conditional branching logic | `IF <condition> BEGIN ... END ELSE BEGIN ... END` |
| **TRY / CATCH** | Error handling | `BEGIN TRY ... END TRY BEGIN CATCH ... END CATCH` |
| **Error Functions** | Built-in error metadata | `ERROR_MESSAGE()`, `ERROR_NUMBER()`, `ERROR_LINE()` |

### Stored Procedure Anatomy — Full Template

```sql
CREATE PROCEDURE usp_procedure_name
    @param1 VARCHAR(50) = 'default_value',
    @param2 INT         = 0
AS
BEGIN

    -- Variable declarations (always at the top)
    DECLARE @var1 INT;
    DECLARE @var2 FLOAT;

    BEGIN TRY

        -- ================================================
        -- STEP 1: Prepare & Clean Data
        -- ================================================

        IF EXISTS (SELECT 1 FROM table WHERE condition AND col = @param1)
        BEGIN
            UPDATE table SET col = 0 WHERE col IS NULL AND filter_col = @param1;
        END
        ELSE
        BEGIN
            PRINT 'No cleanup needed.';
        END;

        -- ================================================
        -- STEP 2: Assign Variables
        -- ================================================

        SELECT @var1 = COUNT(*), @var2 = AVG(score)
        FROM table
        WHERE col = @param1;

        PRINT 'Result for ' + @param1 + ': ' + CAST(@var1 AS VARCHAR);

        -- ================================================
        -- STEP 3: Generate Reports
        -- ================================================

        SELECT COUNT(*) AS total, AVG(score) AS avg_score
        FROM table WHERE col = @param1;

        SELECT COUNT(order_id) AS orders, SUM(sales) AS revenue
        FROM orders o
        JOIN table t ON o.id = t.id
        WHERE t.col = @param1;

    END TRY
    BEGIN CATCH

        -- ================================================
        -- ERROR HANDLING
        -- ================================================

        PRINT 'An error occurred!';
        PRINT 'Message  : ' + ERROR_MESSAGE();
        PRINT 'Number   : ' + CAST(ERROR_NUMBER() AS VARCHAR);
        PRINT 'Line     : ' + CAST(ERROR_LINE()   AS VARCHAR);
        PRINT 'Procedure: ' + ERROR_PROCEDURE();

    END CATCH;

END;
```
