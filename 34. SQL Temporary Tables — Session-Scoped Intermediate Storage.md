## 1. What Are Temporary Tables?

**Definition:** A temporary table (often called a *temp table*) is a table that stores intermediate results in a temporary storage area of the database **for the duration of a session**. The database automatically drops it when the session ends — no manual cleanup required.

The mechanics are nearly identical to a permanent CTAS table:

```
Permanent Table (CTAS):
  Query → executes → result stored in permanent user table → lives until DROP TABLE

Temporary Table:
  Query → executes → result stored in temporary system storage → lives until session ends
```

The only difference is **lifetime** and **storage location**. The creation syntax, the ability to query and manipulate data, and the performance characteristics are the same.

### What Is a Session?

A **session** is the time between a client connecting to the database and disconnecting from it.

```
Connect to DB   →   [Active Session]   →   Disconnect / Close client
      ↓                    ↓                         ↓
  Session starts     Temp tables usable         Temp tables auto-dropped
```

Examples of session boundaries:
- Opening and closing SQL Server Management Studio (SSMS)
- Opening and closing a DBeaver or DataGrip connection
- A script execution block in an application
- A stored procedure call (in some contexts)

> **Key implication:** Temporary tables created in one session are not visible in another session (with the exception of global temp tables in MS SQL Server, covered in [Section 8](#8-dialect-comparison-mysql-vs-ms-sql-server-vs-postgresql)).

## 2. Where Temporary Tables Live

Temporary tables do **not** live inside your user database (e.g., `SalesDB`). They are stored in a dedicated system database.

| Database | System Storage Location |
|---|---|
| MS SQL Server | `tempdb` system database |
| MySQL | Temporary table storage (in-memory or on-disk, configured by `tmp_table_size`) |
| PostgreSQL | `pg_temp` schema (session-scoped temporary schema) |

In MS SQL Server's object explorer:

```
SQL Server
├── SalesDB                    ← your permanent tables live here
│   └── Tables
│       ├── sales.customers
│       ├── sales.orders
│       └── ...
└── System Databases
    └── tempdb                 ← temporary tables live here
        └── Temporary Tables
            └── #orders        ← auto-dropped when session ends
```

> Because they live in a system database rather than your user database, temporary tables don't pollute your schema and require zero cleanup — the database handles it automatically.

## 3. Syntax and Creation

### Sample Schema

```sql
-- Permanent table used as the source
CREATE TABLE sales.orders (
    order_id     INT,
    customer_id  INT,
    order_date   DATE,
    sales        DECIMAL(10,2),
    quantity     INT,
    order_status VARCHAR(20)    -- values: 'Shipped', 'Delivered', 'Pending'
);

INSERT INTO sales.orders VALUES
  (1, 1, '2024-01-05', 999.00, 1, 'Delivered'),
  (2, 2, '2024-01-10', 250.00, 2, 'Shipped'),
  (3, 3, '2024-02-01', 200.00, 2, 'Delivered'),
  (4, 1, '2024-02-14', 10.00,  2, 'Shipped'),
  (5, 4, '2024-03-01', 500.00, 2, 'Delivered'),
  (6, 3, '2024-03-15', 15.00,  3, 'Shipped'),
  (7, 2, '2024-03-20', 85.00,  1, 'Pending'),
  (8, 1, '2024-03-25', 400.00, 2, 'Delivered'),
  (9, 4, '2024-04-01', 120.00, 1, 'Shipped'),
  (10,2, '2024-04-10', 60.00,  1, 'Delivered');
```

### MS SQL Server: `SELECT INTO #table`

The `#` prefix is what tells SQL Server to create a temporary table instead of a permanent one.

```sql
-- Create a temporary copy of the orders table
SELECT *
INTO   #orders             -- # prefix = local temporary table (session-scoped)
FROM   sales.orders;

-- Verify it was created
SELECT * FROM #orders;
```

```sql
-- Global temporary table (visible across all sessions)
SELECT *
INTO   ##orders_global     -- ## prefix = global temporary table
FROM   sales.orders;
```

### MySQL: `CREATE TEMPORARY TABLE`

```sql
-- MySQL
CREATE TEMPORARY TABLE orders_temp AS
(
    SELECT * FROM sales.orders
);

-- Or with explicit structure:
CREATE TEMPORARY TABLE orders_staging (
    order_id     INT,
    customer_id  INT,
    order_date   DATE,
    sales        DECIMAL(10,2),
    order_status VARCHAR(20)
);

INSERT INTO orders_staging
SELECT order_id, customer_id, order_date, sales, order_status
FROM   sales.orders;
```

### PostgreSQL: `CREATE TEMP TABLE`

```sql
-- PostgreSQL
CREATE TEMP TABLE orders_temp AS
(
    SELECT * FROM sales.orders
);

-- With explicit structure
CREATE TEMP TABLE orders_staging (
    order_id     INT,
    customer_id  INT,
    order_date   DATE,
    sales        DECIMAL(10,2),
    order_status VARCHAR(20)
);

INSERT INTO orders_staging
SELECT order_id, customer_id, order_date, sales, order_status
FROM   sales.orders;
```

## 4. How the Database Executes Temporary Tables

```
Step 1: Client executes a query with SELECT INTO #temp / CREATE TEMP TABLE
                    ↓
Step 2: Database engine parses the query
                    ↓
Step 3: Execute the SELECT query → fetch data from source table (e.g., sales.orders)
                    ↓
Step 4: Store metadata in the system catalog (column names, types, session ID)
                    ↓
Step 5: Write the result rows to temporary disk storage (tempdb / pg_temp)
                    ↓
Step 6: Temp table is now available for the session
                    ↓
[Session Active — multiple queries can read/write the temp table]
                    ↓
Step 7: Client disconnects / session ends
                    ↓
Step 8: Database automatically drops the temp table and frees storage
        (no user action required)
```

### What Gets Stored Where

| Component | Location |
|---|---|
| Row data | Temporary disk storage (`tempdb` / `pg_temp`) |
| Column metadata | System catalog (session-scoped entry) |
| Indexes (if added) | Temporary disk storage |
| Source table | Unmodified — temp table is a copy |

## 5. Working with Temporary Tables: Full Workflow

This is the most common pattern: copy → transform → analyse → optionally persist.

### Step 1: Create the Temporary Copy

```sql
-- MS SQL Server: copy all orders to a temp table
SELECT *
INTO   #orders
FROM   sales.orders;
```

### Step 2: Manipulate the Copy Freely

Because it is a copy, any changes do not affect the original table.

```sql
-- Delete rows you do not need for this analysis
DELETE FROM #orders
WHERE  order_status = 'Delivered';

-- Check the remaining rows
SELECT * FROM #orders;
```

#### Output

| order_id | customer_id | order_date | sales | quantity | order_status |
|---|---|---|---|---|---|
| 2 | 2 | 2024-01-10 | 250.00 | 2 | Shipped |
| 4 | 1 | 2024-02-14 | 10.00 | 2 | Shipped |
| 6 | 3 | 2024-03-15 | 15.00 | 3 | Shipped |
| 7 | 2 | 2024-03-20 | 85.00 | 1 | Pending |
| 9 | 4 | 2024-04-01 | 120.00 | 1 | Shipped |

### Step 3: Analyse the Temporary Table

```sql
-- Aggregate the modified copy
SELECT   order_status,
         COUNT(*)    AS order_count,
         SUM(sales)  AS total_sales
FROM     #orders
GROUP BY order_status;
```

### Step 4 (Optional): Persist the Result to a Permanent Table

If the result of your temporary work is valuable beyond the session, copy it back to a permanent table before disconnecting.

```sql
-- Save the modified temp table as a permanent table for future use
SELECT *
INTO   sales.orders_non_delivered      -- permanent table, no # prefix
FROM   #orders;
```

After this step:
- `#orders` — still in `tempdb`; auto-dropped when session ends
- `sales.orders_non_delivered` — permanent; survives session end
- `sales.orders` — original table, completely untouched

### Step 5: Session Ends → Auto-Cleanup

```
Client disconnects
        ↓
Database drops #orders from tempdb automatically
        ↓
sales.orders_non_delivered remains in SalesDB (permanent)
sales.orders remains in SalesDB (unmodified)
```

## 6. Use Case — Intermediate Results in ETL Pipelines

### The Pattern

In data engineering and ETL (Extract, Transform, Load) workflows, data from a source system must be cleaned, deduplicated, filtered, and aggregated before it can be loaded into a data warehouse. None of these transformations should touch the original source table.

Temporary tables provide a safe, isolated workspace for multi-step transformations within a single session.

```
Source DB                    Temp Storage              Data Warehouse
─────────────────────────────────────────────────────────────────────
sales.orders (permanent)
       │
       │  Step 1: Extract
       ▼
   #orders_raw (temp)
       │
       │  Step 2: Remove duplicates
       ▼
   #orders_deduped (temp)
       │
       │  Step 3: Handle NULLs / clean
       ▼
   #orders_clean (temp)
       │
       │  Step 4: Aggregate / transform
       ▼
   #orders_final (temp)
       │
       │  Step 5: Load
       ▼
warehouse.fact_orders (permanent)   ← lives forever

[Session ends → all #temp tables auto-dropped]
```

### Full ETL Example

```sql
-- Step 1: Extract raw copy
SELECT *
INTO   #orders_raw
FROM   sales.orders;

-- Step 2: Remove duplicates (keep the row with the highest order_id per customer+date)
SELECT   order_id, customer_id, order_date, sales, quantity, order_status
INTO     #orders_deduped
FROM (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY customer_id, order_date
               ORDER BY order_id DESC
           ) AS rn
    FROM   #orders_raw
) deduped
WHERE rn = 1;

-- Step 3: Handle NULLs and filter out invalid rows
SELECT   order_id,
         customer_id,
         order_date,
         ISNULL(sales, 0)        AS sales,         -- replace NULL sales with 0
         ISNULL(quantity, 1)     AS quantity,       -- replace NULL quantity with 1
         order_status
INTO     #orders_clean
FROM     #orders_deduped
WHERE    order_date IS NOT NULL                     -- drop rows with no date
AND      order_status <> 'Cancelled';               -- exclude cancelled orders

-- Step 4: Aggregate by customer and month
SELECT   customer_id,
         DATETRUNC(month, order_date)   AS order_month,
         COUNT(order_id)                AS total_orders,
         SUM(sales)                     AS total_revenue,
         SUM(quantity)                  AS total_units
INTO     #orders_final
FROM     #orders_clean
GROUP BY customer_id,
         DATETRUNC(month, order_date);

-- Step 5: Load into the data warehouse (permanent table)
INSERT INTO warehouse.fact_customer_monthly
    (customer_id, order_month, total_orders, total_revenue, total_units)
SELECT customer_id, order_month, total_orders, total_revenue, total_units
FROM   #orders_final;

-- Session ends → #orders_raw, #orders_deduped, #orders_clean, #orders_final
-- are all automatically dropped. No manual cleanup needed.
```

### When to Use Temp Tables vs. Permanent Tables for ETL Intermediates

> "If there is something wrong in the data warehouse, you would like to always check the copy where the transformations are done in order to debug and find issues."

This is the key trade-off:

| Scenario | Recommendation |
|---|---|
| Intermediate steps that have no value after the load | Temporary table — auto-cleanup saves effort |
| Debugging a complex ETL pipeline | Permanent staging table — survives session end; lets you inspect step-by-step |
| Quick personal analysis or one-off scripts | Temporary table — no cleanup burden |
| Shared ETL that others may need to inspect or rerun | Permanent staging table |
| Simple multi-step transformation within a script | CTE — no table created at all |

## 7. Temporary Tables vs. CTEs vs. CTAS vs. Views

| Dimension | CTE | Temp Table | CTAS (Permanent) | View |
|---|---|---|---|---|
| **Lifespan** | One query | One session | Until `DROP TABLE` | Until `DROP VIEW` |
| **Stores data on disk** | No (in memory) | Yes (temp storage) | Yes (user database) | No |
| **Visible to other sessions** | No | No (local) | Yes | Yes |
| **Auto-cleanup** | Yes (query ends) | Yes (session ends) | No | No |
| **Indexable** | No | Yes | Yes | No |
| **Reusable within session** | No (one query only) | Yes | Yes | Yes |
| **Data freshness** | Always live | Snapshot at creation | Snapshot at creation | Always live |
| **Maintenance effort** | None | None (auto-drop) | Drop + recreate to refresh | Update the query |
| **Best for** | Readability in a single complex query | Multi-step session work, ETL intermediates | Pre-computed results for performance | Dynamic, shared data products |

### Decision Guide

```
Do you need the result across multiple queries?
  ├── No, just within one query         → Use a CTE
  └── Yes
      ├── Only within this session      → Use a Temp Table
      └── Beyond this session
          ├── For many users/queries    → Use a View
          └── For performance          → Use CTAS (permanent table)
```

## 8. Dialect Comparison: MySQL vs. MS SQL Server vs. PostgreSQL

### 8.1 Creation Syntax

| Feature | MySQL | MS SQL Server | PostgreSQL |
|---|---|---|---|
| Local temp table | `CREATE TEMPORARY TABLE t AS (SELECT ...)` | `SELECT ... INTO #t FROM ...` | `CREATE TEMP TABLE t AS (SELECT ...)` |
| Explicit structure | `CREATE TEMPORARY TABLE t (col type, ...)` | `CREATE TABLE #t (col type, ...)` | `CREATE TEMP TABLE t (col type, ...)` |
| Global temp table | ❌ Not supported | `SELECT ... INTO ##t FROM ...` | ❌ Not supported natively |
| Keyword | `TEMPORARY` | `#` prefix (local), `##` prefix (global) | `TEMP` or `TEMPORARY` |
| Auto-drop | On session end | On session end (local) / all connections disconnect (global) | On session end |

```sql
-- MS SQL Server: local temp table
SELECT order_id, sales
INTO   #orders_temp
FROM   sales.orders
WHERE  order_status = 'Shipped';

-- MS SQL Server: global temp table (visible to all sessions)
SELECT order_id, sales
INTO   ##orders_global
FROM   sales.orders
WHERE  order_status = 'Shipped';

-- MySQL
CREATE TEMPORARY TABLE orders_temp AS
(
    SELECT order_id, sales
    FROM   sales.orders
    WHERE  order_status = 'Shipped'
);

-- PostgreSQL
CREATE TEMP TABLE orders_temp AS
(
    SELECT order_id, sales
    FROM   sales.orders
    WHERE  order_status = 'Shipped'
);
```

### 8.2 Dropping Temporary Tables

```sql
-- MS SQL Server: explicit drop within session (before session ends)
DROP TABLE IF EXISTS #orders_temp;

-- MS SQL Server: older syntax
IF OBJECT_ID('tempdb..#orders_temp') IS NOT NULL
    DROP TABLE #orders_temp;

-- MySQL
DROP TEMPORARY TABLE IF EXISTS orders_temp;

-- PostgreSQL
DROP TABLE IF EXISTS orders_temp;
```

> In all databases, if you do not explicitly drop a temporary table, it is dropped automatically when the session ends. Explicit drops within a session are used when you want to reuse the same name, free memory sooner, or signal intent clearly.

### 8.3 Adding Indexes to Temporary Tables

Just like permanent tables, temporary tables support indexes for performance:

```sql
-- MS SQL Server
CREATE INDEX idx_status ON #orders_temp (order_status);

-- MySQL
CREATE INDEX idx_status ON orders_temp (order_status);

-- PostgreSQL
CREATE INDEX idx_status ON orders_temp (order_status);
```

### 8.4 Scope Differences

| Scope | MS SQL Server | MySQL | PostgreSQL |
|---|---|---|---|
| Local (current session only) | `#table` | `TEMPORARY TABLE` | `TEMP TABLE` |
| Global (all sessions) | `##table` | ❌ | ❌ (use unlogged tables or a shared schema) |
| Stored procedure scope | Temp tables from calling session are visible inside SP | Session-scoped | Session-scoped |

### 8.5 Storage and Performance Notes

| Database | Where Stored | Memory vs. Disk |
|---|---|---|
| MS SQL Server | `tempdb` database | Disk-based by default; memory-optimised temp tables available (In-Memory OLTP) |
| MySQL | In-memory if result fits `tmp_table_size`; otherwise disk (MyISAM/InnoDB temp) | Configurable |
| PostgreSQL | `pg_temp` schema | Disk-based; unlogged tables (faster, no WAL) available as an alternative |

```sql
-- MySQL: check temp table memory limit
SHOW VARIABLES LIKE 'tmp_table_size';
SHOW VARIABLES LIKE 'max_heap_table_size';

-- PostgreSQL alternative: UNLOGGED TABLE (faster than temp for large intermediates;
-- NOT auto-dropped, but survives crashes as empty)
CREATE UNLOGGED TABLE orders_staging AS
(
    SELECT * FROM sales.orders
);
```

## 9. Common Mistakes

### 9.1 Forgetting the `#` Prefix (MS SQL Server)

```sql
-- ❌ Creates a PERMANENT table, not a temp table
SELECT * INTO orders_temp FROM sales.orders;

-- ✅ Creates a temporary table
SELECT * INTO #orders_temp FROM sales.orders;
```

### 9.2 Trying to Access a Temp Table From Another Session

```sql
-- Session A creates:
SELECT * INTO #orders_temp FROM sales.orders;

-- Session B tries to query it:
SELECT * FROM #orders_temp;
-- Error: Invalid object name '#orders_temp'
-- Local temp tables are session-scoped and invisible to other sessions.
```

### 9.3 Recreating a Temp Table That Already Exists in the Same Session

```sql
-- ❌ Error if #orders_temp was already created earlier in this session
SELECT * INTO #orders_temp FROM sales.orders;
-- "There is already an object named '#orders_temp' in the database."

-- ✅ Drop first, then recreate
DROP TABLE IF EXISTS #orders_temp;
SELECT * INTO #orders_temp FROM sales.orders;
```

### 9.4 Assuming Temp Table Data Survives Session End

```sql
-- ❌ Common misconception: planning to reuse temp table tomorrow
-- You close your client and come back the next morning.
SELECT * FROM #orders_temp;
-- Error: the temp table was dropped when the session ended.

-- ✅ If the result must survive beyond the session, save it to a permanent table
SELECT * INTO sales.orders_staging FROM #orders_temp;  -- permanent copy
```

### 9.5 Using a Temp Table When a CTE Is Simpler

```sql
-- ❌ Overkill for a single-use intermediate result
SELECT *
INTO   #orders_high_value
FROM   sales.orders
WHERE  sales > 500;

SELECT customer_id, SUM(sales) AS total
FROM   #orders_high_value
GROUP BY customer_id;

-- ✅ A CTE is cleaner and requires no storage or cleanup
WITH orders_high_value AS (
    SELECT * FROM sales.orders WHERE sales > 500
)
SELECT customer_id, SUM(sales) AS total
FROM   orders_high_value
GROUP BY customer_id;
```

### 9.6 Not Adding Indexes on Large Temp Tables

```sql
-- ❌ Joining or filtering a large temp table without an index → full scan
SELECT *
INTO   #large_staging
FROM   sales.orders;   -- millions of rows

SELECT * FROM #large_staging WHERE order_status = 'Shipped';  -- slow

-- ✅ Add an index after creation
SELECT * INTO #large_staging FROM sales.orders;
CREATE INDEX idx_status ON #large_staging (order_status);
SELECT * FROM #large_staging WHERE order_status = 'Shipped';  -- fast
```

## 10. Best Practices

**Use temp tables for multi-step, session-scoped work.** If a transformation requires more than two or three dependent steps and cannot be expressed cleanly as a CTE chain, a temp table is the right tool.

**Prefer CTEs for single-query intermediate results.** A CTE requires no storage, no cleanup, and keeps all logic in one readable block. Reach for temp tables only when you need to reference the intermediate result across multiple separate queries within the session.

**Always drop before recreating within the same session.** Use `DROP TABLE IF EXISTS #t` (or `DROP TEMPORARY TABLE IF EXISTS t` in MySQL) at the top of any script that creates temp tables, so the script is safely re-runnable.

**Name temp tables descriptively.** `#orders_deduped`, `#customers_eu`, `#staging_clean` are far better than `#temp1`, `#t`, `#data`.

**Add indexes on large temp tables.** If you will filter, join, or group by a column, add an index after creation — temp tables support indexes just like permanent tables.

**Save intermediate results to permanent tables when debugging.** If an ETL job is misbehaving, replace the temporary tables with permanent staging tables temporarily so you can inspect each step's output after the session ends.

**Do not use global temp tables (`##t`) for shared state between sessions.** They are fragile — the table is dropped as soon as all referencing sessions disconnect. Use a permanent staging table with a clear naming convention instead.

**Understand when to use temp table vs. permanent table for ETL intermediates.** Temporary tables are ideal when the intermediate data has no value after the load. Permanent staging tables are better when you need to debug or audit the pipeline.

## 11. Summary Table

| Dimension | Detail |
|---|---|
| **Definition** | A table stored in temporary system storage that is automatically dropped when the session ends |
| **Storage location** | `tempdb` (MS SQL Server) / `pg_temp` (PostgreSQL) / temp storage (MySQL) |
| **Lifespan** | Duration of the session that created it |
| **Visible to other sessions?** | No (local); Yes (global `##t` in MS SQL Server only) |
| **Auto-cleanup** | Yes — database drops it when session ends |
| **Stores data?** | Yes — physical rows written to temporary disk storage |
| **Supports indexes?** | Yes |
| **Data freshness** | Snapshot at creation time; does not reflect later changes to source tables |
| **MS SQL Server syntax** | `SELECT ... INTO #table FROM ...` (local) / `##table` (global) |
| **MySQL syntax** | `CREATE TEMPORARY TABLE t AS (SELECT ...)` |
| **PostgreSQL syntax** | `CREATE TEMP TABLE t AS (SELECT ...)` |
| **Primary use case** | Multi-step ETL transformations within a session |
| **vs. CTE** | CTE = one query only, no storage; Temp table = entire session, stored on disk, indexable |
| **vs. CTAS** | CTAS = permanent, survives session; Temp table = session-scoped, auto-dropped |
| **vs. View** | View = dynamic, always fresh, no data; Temp table = snapshot, session-scoped, stored |
| **Instructor opinion** | Rarely needed in practice; CTEs handle most single-query cases; permanent staging tables are better for complex ETL debugging |
