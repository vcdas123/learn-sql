## 1. Why These Techniques Matter

In any production database, many users and teams need to run **complex analytical queries** — dashboards, reports, ML feature pipelines, business KPIs, and more. Without structured approaches, you'll encounter:

| Problem | Description |
|---|---|
| **Complexity** | Long, unreadable queries that are hard to debug |
| **Redundancy** | The same logic rewritten by multiple analysts independently |
| **Performance** | Repeated expensive operations with no caching or materialization |
| **Security** | Direct exposure of raw tables to all users |

The five techniques in this chapter directly solve each of these problems.

## 3. Technique 1 — Subqueries

### 3.1 Definition

A **subquery** (also called an *inner query* or *nested query*) is a `SELECT` statement written **inside another SQL query**. The database evaluates the inner query first, stores the intermediate result temporarily in **memory (cache)**, and then uses it in the outer (main) query.

> 💡 Think of a subquery as a query that prepares data so the outer query can work with it.

### 3.2 Syntax

```sql
-- General structure
SELECT column1, column2
FROM table_name
WHERE column1 = (
    SELECT column1          -- ← This is the subquery
    FROM another_table
    WHERE condition
);
```

Subqueries can appear in:
- `WHERE` clause (most common)
- `FROM` clause (inline view)
- `SELECT` clause (scalar subquery)

### 3.3 Example: Find Customers Who Spent Above Average

**Goal:** List all customers whose total spending is above the average order amount.

```sql
SELECT
    c.customer_name,
    o.total_amount
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.total_amount > (
    SELECT AVG(total_amount)   -- Subquery: calculates the average
    FROM orders
);
```

**Query Output:**

| customer_name | total_amount |
|---|---|
| Alice Johnson | 1280.00 |
| Alice Johnson | 400.00 |
| David Smith   | 1200.00 |

*(Average was ~387.50, so orders above that are returned)*

### 3.4 Execution Flow

```
Step 1: DB evaluates the subquery → AVG(total_amount) = 387.50
Step 2: Result stored temporarily in memory/cache
Step 3: Outer query runs: WHERE total_amount > 387.50
Step 4: Cache cleared once the full query finishes
```

### 3.5 Performance Notes

- ⚠️ **Correlated subqueries** (where the inner query references the outer query row-by-row) can be **very slow** on large datasets because they execute once per row.
- Non-correlated subqueries (like the example above) execute only once and are generally efficient.
- Use `EXPLAIN` / `EXPLAIN ANALYZE` to inspect how the database is executing your query.

```sql
-- Example of a correlated subquery (SLOW on large tables)
SELECT customer_name
FROM customers c
WHERE (
    SELECT COUNT(*) FROM orders o
    WHERE o.customer_id = c.customer_id   -- references outer query
) > 2;
```

## 4. Technique 2 — Common Table Expressions (CTEs)

### 4.1 Definition

A **Common Table Expression (CTE)** is a named temporary result set defined using the `WITH` keyword **at the top of a query**. Like subqueries, CTEs store their intermediate results in **memory/cache** — but unlike subqueries, a CTE can be **referenced multiple times** within the same query.

> 💡 Use a CTE when you have logic that you need to reuse in more than one place inside the same query, or when a subquery makes the code too hard to read.

### 4.2 Syntax

```sql
WITH cte_name AS (
    SELECT column1, column2
    FROM table_name
    WHERE condition
)
SELECT *
FROM cte_name;          -- Reference the CTE here
```

Multiple CTEs can be chained:

```sql
WITH
  cte_one AS (
      SELECT ... FROM ...
  ),
  cte_two AS (
      SELECT ... FROM cte_one   -- Can reference a previous CTE
  )
SELECT * FROM cte_two;
```

### 4.3 Example: Total Revenue Per Customer, Then Filter High-Value Customers

```sql
WITH customer_revenue AS (
    -- Step 1: Aggregate total spending per customer
    SELECT
        c.customer_id,
        c.customer_name,
        SUM(o.total_amount) AS total_spent
    FROM customers c
    JOIN orders o ON c.customer_id = o.customer_id
    GROUP BY c.customer_id, c.customer_name
),
avg_spending AS (
    -- Step 2: Calculate the average (reuses customer_revenue)
    SELECT AVG(total_spent) AS avg_total
    FROM customer_revenue
)
-- Step 3: Main query — filter customers above average
SELECT
    cr.customer_name,
    cr.total_spent,
    a.avg_total
FROM customer_revenue cr
CROSS JOIN avg_spending a
WHERE cr.total_spent > a.avg_total;
```

**Query Output:**

| customer_name | total_spent | avg_total |
|---|---|---|
| Alice Johnson | 1680.00 | 554.17 |
| David Smith   | 1200.00 | 554.17 |

### 4.4 Execution Flow

```
Step 1: DB evaluates customer_revenue CTE → stored in memory
Step 2: DB evaluates avg_spending CTE → stored in memory
Step 3: Main query runs using both CTE results
Step 4: All CTE memory is cleared once the query finishes
```

### 4.5 Performance Notes

- CTEs are **not necessarily materialized** — in many databases (especially PostgreSQL ≤ 11), the CTE is re-evaluated each time it's referenced unless `MATERIALIZED` is specified.
- PostgreSQL 12+ introduced the `MATERIALIZED` and `NOT MATERIALIZED` hints for explicit control.
- Avoid chaining more than **5 CTEs** in a single query — it becomes difficult to read and debug.

```sql
-- PostgreSQL 12+: force materialization for performance
WITH customer_revenue AS MATERIALIZED (
    SELECT customer_id, SUM(total_amount) AS total_spent
    FROM orders
    GROUP BY customer_id
)
SELECT * FROM customer_revenue WHERE total_spent > 500;
```

## 5. Technique 3 — Views

### 5.1 Definition

A **View** is a **named, saved SQL query stored permanently in the database**. It does not store any data itself — it is essentially a virtual table. Every time you query a view, the database runs the underlying SQL and fetches **fresh, live data** from the original tables.

> 💡 Views are best for: sharing complex logic across users, enforcing security (hide sensitive columns), and always getting up-to-date data.

### 5.2 Syntax

```sql
-- Create a view
CREATE VIEW view_name AS
SELECT column1, column2
FROM table_name
WHERE condition;

-- Query the view (just like a table)
SELECT * FROM view_name;

-- Drop the view
DROP VIEW view_name;
```

### 5.3 Example: Create a View for Customer Order Summary

```sql
CREATE VIEW vw_customer_order_summary AS
SELECT
    c.customer_id,
    c.customer_name,
    c.country,
    COUNT(o.order_id)       AS total_orders,
    SUM(o.total_amount)     AS total_spent,
    MAX(o.order_date)       AS last_order_date
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.customer_name, c.country;
```

**Now any analyst can simply run:**

```sql
SELECT * FROM vw_customer_order_summary
WHERE country = 'USA';
```

**Query Output:**

| customer_id | customer_name | country | total_orders | total_spent | last_order_date |
|---|---|---|---|---|---|
| 1 | Alice Johnson | USA | 2 | 1680.00 | 2023-04-10 |
| 4 | David Smith   | USA | 1 | 1200.00 | 2023-05-20 |

### 5.4 Execution Flow

```
Analyst runs: SELECT * FROM vw_customer_order_summary
              ↓
DB looks up the view definition in the catalog
              ↓
DB executes the underlying SQL against live tables
              ↓
Fresh results returned — no pre-stored data used
```

### 5.5 Performance Notes

- Since views re-execute every time, **complex views on large tables can be slow**.
- When performance becomes a bottleneck (e.g., the view takes 30+ minutes), consider switching to a **CTAS** (see Section 6) to create a pre-computed physical table.
- Some databases support **Materialized Views** (PostgreSQL, Oracle, Snowflake) — these store data like a table but can be refreshed on schedule.

```sql
-- PostgreSQL: Materialized View (stores data physically)
CREATE MATERIALIZED VIEW mv_customer_summary AS
SELECT customer_id, SUM(total_amount) AS total_spent
FROM orders
GROUP BY customer_id;

-- Refresh when needed
REFRESH MATERIALIZED VIEW mv_customer_summary;
```

## 6. Technique 4 — CTAS (Create Table As Select)

### 6.1 Definition

**CTAS** stands for *Create Table As Select*. It executes a `SELECT` query and **saves the result as a new permanent physical table** stored on disk. The data is captured once at creation time — it does not update automatically if the underlying data changes.

> 💡 Use CTAS when a view is too slow and you need a pre-computed, permanent table that multiple users can query efficiently.

### 6.2 Syntax

```sql
-- Standard CTAS
CREATE TABLE new_table_name AS
SELECT column1, column2
FROM source_table
WHERE condition;
```

### 6.3 Example: Materialize the Customer Revenue Summary

```sql
CREATE TABLE tbl_customer_revenue_snapshot AS
SELECT
    c.customer_id,
    c.customer_name,
    c.country,
    SUM(o.total_amount)  AS total_spent,
    COUNT(o.order_id)    AS total_orders
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.customer_name, c.country;
```

**Querying the result:**

```sql
SELECT * FROM tbl_customer_revenue_snapshot
ORDER BY total_spent DESC;
```

**Query Output:**

| customer_id | customer_name | country | total_spent | total_orders |
|---|---|---|---|---|
| 1 | Alice Johnson | USA   | 1680.00 | 2 |
| 4 | David Smith   | USA   | 1200.00 | 1 |
| 2 | Bob Patel     | India |  350.00 | 1 |
| 3 | Clara Wu      | China |   85.00 | 1 |
| 5 | Eva Rossi     | Italy |   10.00 | 1 |

### 6.4 Performance Notes

- ✅ **Very fast reads** since data is pre-computed and stored on disk.
- ⚠️ **Data is stale** — you must `DROP` and recreate (or `TRUNCATE` + re-insert) to refresh.
- Best used in **ETL pipelines** or scheduled batch jobs where you control the refresh cadence.

```sql
-- Refresh a CTAS table
DROP TABLE tbl_customer_revenue_snapshot;

CREATE TABLE tbl_customer_revenue_snapshot AS
SELECT ... ;  -- Re-run the full query
```

## 7. Technique 5 — Temporary Tables

### 7.1 Definition

A **Temporary Table** is a table created at runtime that is stored on disk (or sometimes in memory, depending on the database engine) and exists **only for the duration of your session**. Once your session ends, the database automatically drops the table.

> 💡 Use temporary tables when you need an intermediate result that you'll query multiple times within a session, but don't want to permanently store it.

### 7.2 Syntax

```sql
-- Create a temporary table
CREATE TEMPORARY TABLE temp_table_name AS
SELECT column1, column2
FROM source_table
WHERE condition;

-- Query it like any other table (within the same session)
SELECT * FROM temp_table_name;

-- Explicitly drop it (optional — auto-dropped at session end)
DROP TEMPORARY TABLE temp_table_name;
```

### 7.3 Example: Stage High-Value Orders for Multi-Step Analysis

```sql
-- Step 1: Create a temporary staging table
CREATE TEMPORARY TABLE temp_high_value_orders AS
SELECT
    o.order_id,
    o.customer_id,
    o.total_amount,
    o.order_date
FROM orders o
WHERE o.total_amount >= 400;

-- Step 2: Query it for customer details (same session)
SELECT
    c.customer_name,
    t.total_amount,
    t.order_date
FROM temp_high_value_orders t
JOIN customers c ON t.customer_id = c.customer_id;
```

**Query Output:**

| customer_name | total_amount | order_date |
|---|---|---|
| Alice Johnson | 1280.00 | 2023-03-01 |
| Alice Johnson |  400.00 | 2023-04-10 |
| David Smith   | 1200.00 | 2023-05-20 |

### 7.4 Performance Notes

- Useful when you need to join a complex intermediate result more than once in a session.
- Avoids re-running expensive computations multiple times.
- ⚠️ Be careful: the data is **frozen at creation time**, just like CTAS — changes to the source table won't be reflected.

## 8. Side-by-Side Comparison

This table gives you the complete picture — the same comparison covered in the original lesson:

| Criteria | Subquery | CTE | View | CTAS | Temp Table |
|---|---|---|---|---|---|
| **Storage Type** | Memory/Cache | Memory/Cache | No storage (virtual) | Disk (physical table) | Disk (or memory) |
| **Lifetime** | Temporary | Temporary | Permanent | Permanent | Temporary |
| **Auto-Deleted** | End of query | End of query | Never (manual DROP) | Never (manual DROP) | End of session |
| **Query Scope** | Single query | Single query | Multiple queries / users | Multiple queries / users | Multiple queries (same session) |
| **Reusability** | ❌ Very low (one place, one query) | 🟡 Low (multiple places, one query) | ✅ Highest (any user, any time) | ✅ High (any user, session-independent) | 🟡 Medium (same session only) |
| **Data Freshness** | ✅ Always fresh | ✅ Always fresh | ✅ Always fresh | ⚠️ Stale (snapshot) | ⚠️ Stale (snapshot) |
| **Performance** | Can be slow (correlated) | Good for moderate complexity | Can be slow on large data | ✅ Fast reads | ✅ Faster repeated access |
| **Best Use Case** | One-off multi-step logic | Reusing logic within one query | Sharing logic across users/teams | Pre-computed tables for performance | Multi-step session-scoped analysis |

## 9. Dialect Differences: MySQL vs MS SQL Server vs PostgreSQL

### 9.1 CTEs

All three support CTEs, but with minor syntax variations:

```sql
-- ✅ MySQL 8.0+, MS SQL Server, PostgreSQL — identical syntax
WITH top_customers AS (
    SELECT customer_id, SUM(total_amount) AS revenue
    FROM orders
    GROUP BY customer_id
    HAVING SUM(total_amount) > 500
)
SELECT * FROM top_customers;
```

> ⚠️ MySQL versions **below 8.0** do **not** support CTEs. Use subqueries instead.

### 9.2 Temporary Tables

| Feature | MySQL | MS SQL Server | PostgreSQL |
|---|---|---|---|
| **Keyword** | `CREATE TEMPORARY TABLE` | `CREATE TABLE #temp_name` | `CREATE TEMPORARY TABLE` or `CREATE TEMP TABLE` |
| **Scope** | Session | Session (local `#`) or Global (`##`) | Session |
| **Auto-dropped** | On session end | On session end (local) | On session end |

```sql
-- MySQL
CREATE TEMPORARY TABLE temp_orders AS
SELECT * FROM orders WHERE total_amount > 500;

-- MS SQL Server
SELECT * INTO #temp_orders
FROM orders WHERE total_amount > 500;

-- PostgreSQL
CREATE TEMP TABLE temp_orders AS
SELECT * FROM orders WHERE total_amount > 500;
```

### 9.3 Views

All three databases support standard views. Syntax is identical across dialects.

```sql
-- Works in MySQL, MS SQL Server, and PostgreSQL
CREATE VIEW vw_high_value_customers AS
SELECT customer_id, SUM(total_amount) AS total_spent
FROM orders
GROUP BY customer_id
HAVING SUM(total_amount) > 500;
```

> ⚠️ `CREATE OR REPLACE VIEW` is supported in **MySQL** and **PostgreSQL**, but **not in MS SQL Server**. In SQL Server, use `ALTER VIEW` instead.

```sql
-- MySQL / PostgreSQL
CREATE OR REPLACE VIEW vw_high_value_customers AS ...;

-- MS SQL Server equivalent
ALTER VIEW vw_high_value_customers AS ...;
```

### 9.4 CTAS

```sql
-- MySQL / PostgreSQL
CREATE TABLE new_table AS
SELECT * FROM orders WHERE total_amount > 500;

-- MS SQL Server — uses SELECT INTO instead
SELECT * INTO new_table
FROM orders WHERE total_amount > 500;
```

### 9.5 Materialized Views

| Feature | MySQL | MS SQL Server | PostgreSQL |
|---|---|---|---|
| **Native Materialized View** | ❌ Not supported | ❌ Not natively (use Indexed Views) | ✅ Full support |
| **Workaround** | Use CTAS + scheduled job | Use Indexed Views | `CREATE MATERIALIZED VIEW` |

```sql
-- PostgreSQL materialized view
CREATE MATERIALIZED VIEW mv_revenue_summary AS
SELECT customer_id, SUM(total_amount) AS revenue
FROM orders
GROUP BY customer_id;

-- Refresh data
REFRESH MATERIALIZED VIEW mv_revenue_summary;
```

## 10. Execution Flow in a Real Project

The following walkthrough illustrates how all five techniques work together in a real analytics project — as described in the lesson.

```
┌─────────────────────────────────────────────────────────────────────┐
│  DATA ENGINEER / DBA                                                │
│                                                                     │
│  1. CREATE TABLE orders (...);          ← DDL: create structure     │
│  2. INSERT INTO orders VALUES (...);    ← DML: populate data        │
└──────────────────────────┬──────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────────────┐
│  DATA ANALYST / DATA SCIENTIST                                      │
│                                                                     │
│  3. Writes a complex two-step query:                                │
│     → Uses SUBQUERY to prepare intermediate result (in memory)      │
│     → Main query reads from that intermediate result                │
│                                                                     │
│  4. Notices repeated logic → extracts into CTE                      │
│     → CTE reused in multiple JOINs within the same query            │
│                                                                     │
│  5. Wants to share logic with teammates                             │
│     → Saves it as a VIEW (permanent, always fresh)                  │
│                                                                     │
│  6. View is too slow (30+ minutes on large data)                    │
│     → Creates a CTAS table (physical snapshot, fast reads)          │
│                                                                     │
│  7. Needs a staging table during session analysis                   │
│     → Creates a TEMP TABLE (auto-cleaned after session)             │
└─────────────────────────────────────────────────────────────────────┘
```

## 11. Common Mistakes

### ❌ Mistake 1: Using Correlated Subqueries on Large Tables

```sql
-- SLOW: executes the subquery once for every row in customers
SELECT customer_name
FROM customers c
WHERE (SELECT SUM(total_amount) FROM orders o
       WHERE o.customer_id = c.customer_id) > 500;

-- BETTER: use a CTE or JOIN instead
WITH spending AS (
    SELECT customer_id, SUM(total_amount) AS total
    FROM orders GROUP BY customer_id
)
SELECT c.customer_name
FROM customers c
JOIN spending s ON c.customer_id = s.customer_id
WHERE s.total > 500;
```

### ❌ Mistake 2: Chaining Too Many CTEs

```sql
-- BAD: extremely hard to follow
WITH a AS (...), b AS (...), c AS (...),
     d AS (...), e AS (...), f AS (...), g AS (...)
SELECT * FROM g;
```

> ✅ **Rule of thumb:** Keep CTEs to **5 or fewer** per query. For anything beyond that, consider splitting into views or CTAS tables.

### ❌ Mistake 3: Querying a Stale CTAS Table as if It Were Live

```sql
-- This might return outdated data if new orders were added
SELECT * FROM tbl_customer_revenue_snapshot;
```

> ✅ Always document the last refresh time of CTAS / temp tables, or schedule automated refreshes via an ETL pipeline.

### ❌ Mistake 4: Dropping a View Instead of Recreating It

```sql
-- SQL Server: this will fail
CREATE OR REPLACE VIEW vw_summary AS ...;   -- ❌ Not valid in SQL Server

-- Correct SQL Server approach
ALTER VIEW vw_summary AS ...;               -- ✅
```

### ❌ Mistake 5: Using Temp Tables Across Sessions

```sql
-- Session A creates:
CREATE TEMPORARY TABLE temp_results AS SELECT ...;

-- Session B tries to use it — FAILS, temp table doesn't exist there
SELECT * FROM temp_results;   -- ❌ Table not found
```

> ✅ Temp tables are **session-scoped**. Use a View or CTAS table when multiple users/sessions need access.

## 12. Real-World Analytics Use Cases

### Use Case 1: Sales Dashboard (View)

A BI team wants a live sales dashboard. They create a view so the dashboard always reflects current data.

```sql
CREATE VIEW vw_daily_sales_summary AS
SELECT
    order_date,
    COUNT(order_id)     AS orders_count,
    SUM(total_amount)   AS daily_revenue
FROM orders
GROUP BY order_date;
```

### Use Case 2: End-of-Month Reporting (CTAS)

Finance runs heavy month-end reports. A nightly job refreshes a CTAS table so analysts get sub-second query times.

```sql
-- Nightly ETL job:
DROP TABLE IF EXISTS tbl_monthly_revenue;

CREATE TABLE tbl_monthly_revenue AS
SELECT
    DATE_FORMAT(order_date, '%Y-%m')  AS month,   -- MySQL
    SUM(total_amount)                 AS revenue,
    COUNT(DISTINCT customer_id)       AS unique_customers
FROM orders
GROUP BY DATE_FORMAT(order_date, '%Y-%m');
```

### Use Case 3: Multi-Step Customer Segmentation (CTE)

A data scientist segments customers into tiers using chained CTEs.

```sql
WITH customer_totals AS (
    SELECT customer_id, SUM(total_amount) AS total_spent
    FROM orders
    GROUP BY customer_id
),
customer_tiers AS (
    SELECT
        customer_id,
        total_spent,
        CASE
            WHEN total_spent >= 1000 THEN 'Gold'
            WHEN total_spent >= 500  THEN 'Silver'
            ELSE                          'Bronze'
        END AS tier
    FROM customer_totals
)
SELECT c.customer_name, ct.total_spent, ct.tier
FROM customers c
JOIN customer_tiers ct ON c.customer_id = ct.customer_id
ORDER BY ct.total_spent DESC;
```

### Use Case 4: Session-Scoped A/B Test Analysis (Temp Table)

An analyst is exploring a new discount campaign. They stage a filtered dataset for the session.

```sql
CREATE TEMPORARY TABLE temp_campaign_orders AS
SELECT *
FROM orders
WHERE order_date BETWEEN '2023-04-01' AND '2023-05-31';

-- Now run multiple exploratory queries against the temp table
SELECT country, SUM(total_amount) AS revenue
FROM temp_campaign_orders t
JOIN customers c ON t.customer_id = c.customer_id
GROUP BY country;
```

## 13. Best Practices

1. **Start with a CTE or subquery** for one-off analysis. Only elevate to a View or CTAS if you need to share or persist.

2. **Prefer Views over CTAS** when data freshness matters. Switch to CTAS only when query performance becomes unacceptable.

3. **Name objects clearly:**
   - Views → prefix with `vw_` (e.g., `vw_customer_summary`)
   - CTAS tables → prefix with `tbl_` or `rpt_` (e.g., `tbl_monthly_revenue`)
   - Temp tables → prefix with `temp_` (e.g., `temp_campaign_data`)
   - CTEs → use descriptive names (e.g., `customer_revenue`, not just `cte1`)

4. **Limit CTE chains to 5 or fewer** per query to maintain readability.

5. **Document stale data sources.** Always add a comment or a `created_at` column to CTAS and temp tables indicating when the data was last refreshed.

6. **Use Views for security.** Grant analysts access only to specific views instead of raw tables — this hides sensitive columns (e.g., PII, pricing costs) without complex row-level security.

7. **Profile before optimizing.** Use `EXPLAIN` (MySQL/PostgreSQL) or `SET STATISTICS` (SQL Server) before deciding whether a view needs to be replaced by a CTAS.

8. **In SQL Server, prefer `SELECT INTO` over `CREATE TABLE AS SELECT`** since CTAS syntax differs.

## 14. Final Summary Table

| Technique | Storage | Lifespan | Data Freshness | Reusability | Best For |
|---|---|---|---|---|---|
| **Subquery** | Memory | Single query execution | ✅ Always live | ❌ One place, one query | Quick two-step logic within one query |
| **CTE** | Memory | Single query execution | ✅ Always live | 🟡 Multiple places, one query | Readable multi-step logic; reuse within one query |
| **View** | None (virtual) | Permanent | ✅ Always live | ✅ Any user, any time | Shared logic, security, always-fresh dashboards |
| **CTAS** | Disk (physical) | Permanent | ⚠️ Snapshot | ✅ Any user, any time | Performance-critical pre-computed tables |
| **Temp Table** | Disk/Memory | Session only | ⚠️ Snapshot | 🟡 Same session only | Multi-step session-scoped intermediate results |

### Instructor's Recommended Ranking

> *"My favorite is Views first, then CTEs, then Subqueries. I use CTAS when Views are too slow. Temporary tables I use the least."* — Course Instructor

| Rank | Technique | Reason |
|---|---|---|
| 🥇 1st | **View** | Always fresh, reusable, great for security and sharing |
| 🥈 2nd | **CTE** | Clean, readable, excellent for complex single queries |
| 🥉 3rd | **Subquery** | Simple and effective for one-off multi-step logic |
