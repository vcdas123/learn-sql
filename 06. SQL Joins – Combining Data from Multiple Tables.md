# SQL Joins – Combining Data from Multiple Tables

## 0. Table Schemas Used in This Chapter

Before learning JOINs, it is very important to clearly understand the **structure (schema)** of the tables we are working with. All examples and outputs in this chapter are based on the following two tables.

### customers table

```sql
CREATE TABLE customers (
    id INT PRIMARY KEY,
    first_name VARCHAR(50) NOT NULL,
    country VARCHAR(50),
    score INT
);
```

**Sample data:**

| id  | first_name | country | score |
| --- | ---------- | ------- | ----- |
| 1   | Maria      | Germany | 350   |
| 2   | John       | USA     | 900   |
| 3   | George     | UK      | 750   |
| 4   | Martin     | Germany | 500   |
| 5   | Peter      | USA     | 0     |

---

### orders table

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    sales INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);
```

**Sample data:**

| order_id | customer_id | sales |
| -------- | ----------- | ----- |
| 101      | 1           | 1200  |
| 102      | 2           | 800   |
| 103      | 3           | 600   |
| 104      | 6           | 300   |

---

### Important Notes About the Schema

- `customers.id` is the **PRIMARY KEY**
- `orders.customer_id` is a **FOREIGN KEY** referencing `customers.id`
- JOIN conditions in this chapter use:

```sql
customers.id = orders.customer_id
```

- Order `104` has `customer_id = 6`, which **does not exist** in the customers table
  - This row is intentionally included to demonstrate **LEFT / RIGHT / FULL JOIN behavior**

## 1. How to Combine Data in SQL

When working with databases, data is often stored in **multiple tables**. Before writing a query, you must first decide **how you want to combine data**:

### Two Ways to Combine Tables

1. **Combine Columns → SQL JOINs**
   - Tables are joined **side by side**
   - Result becomes **wider** (more columns)

2. **Combine Rows → Set Operators**
   - Rows are stacked **one below another**
   - Result becomes **longer** (more rows)

> In this chapter, we focus on **SQL JOINs**.

---

## 2. What Are SQL JOINs?

**SQL JOINs** allow you to retrieve data from **multiple tables in a single query** by linking them using a **common column (key)**.

### Key Concept

- A **JOIN requires a common column** in both tables
- Usually, this column is an **ID / Primary Key / Foreign Key**

Example:

- `customers.id`
- `orders.customer_id`

When the values match, SQL can **combine those rows**.

---

## 3. Why Do We Use JOINs?

### 1. Recombine Data (Big Picture)

Customer data is often split into multiple tables:

- customers
- addresses
- orders
- reviews

JOINs allow you to see **everything in one result**.

---

### 2. Data Enrichment

- One table is the **main (master) table**
- Another table provides **extra information** (lookup/reference table)

Example:

- customers + country zip codes

JOINs enrich your main data with additional context.

---

### 3. Existence / Non‑Existence Checks

JOINs can be used to:

- Find customers **who placed orders**
- Find customers **who did NOT place orders**

Here, the joined table is used **only for filtering**, not for selecting data.

---

## 4. Types of SQL JOINs (Overview)

### Basic JOIN Types (Covered in This Chapter)

| Join Type  | Description               |
| ---------- | ------------------------- |
| INNER JOIN | Only matching rows        |
| LEFT JOIN  | All left + matching right |
| RIGHT JOIN | All right + matching left |
| FULL JOIN  | All rows from both tables |

---

### Advanced JOIN Types (Mentioned Only – Covered Later)

> The following JOIN types are **mentioned by the instructor** but are **not explained in this chapter**. They will be covered in later sections.

| Join Type       | High‑level Idea                                             |
| --------------- | ----------------------------------------------------------- |
| LEFT ANTI JOIN  | Rows from left table with **no match** in right table       |
| RIGHT ANTI JOIN | Rows from right table with **no match** in left table       |
| FULL ANTI JOIN  | Rows that **do not match** in either table                  |
| CROSS JOIN      | Every row from table A combined with every row from table B |

⚠️ **Important**: These joins are intentionally listed **without examples or queries** here, as the instructor introduces them conceptually but teaches them in later chapters.

## 6. INNER JOIN

### Definition

Returns **only rows that exist in both tables**.

### Syntax

```sql
SELECT columns
FROM tableA
INNER JOIN tableB
ON tableA.key = tableB.key;
```

### Characteristics

- Matching data only
- Order of tables **does not matter**

---

### Example

```sql
SELECT c.id, c.first_name, o.order_id, o.sales
FROM customers c
INNER JOIN orders o
ON c.id = o.customer_id;
```

### Output (INNER JOIN)

| id  | first_name | order_id | sales |
| --- | ---------- | -------- | ----- |
| 1   | Maria      | 101      | 1200  |
| 2   | John       | 102      | 800   |
| 3   | George     | 103      | 600   |

### Use Cases

- Recombine related data
- Check existence of records

---

## 7. LEFT JOIN

### Definition

Returns:

- **ALL rows from the left table**
- Only **matching rows from the right table**

If no match exists → **NULL values appear**.

### Syntax

```sql
SELECT columns
FROM tableA
LEFT JOIN tableB
ON tableA.key = tableB.key;
```

### Important Rule

- **Order matters**
- Left table = main table

---

### Example

```sql
SELECT c.id, c.first_name, o.order_id, o.sales
FROM customers c
LEFT JOIN orders o
ON c.id = o.customer_id;
```

### Output (LEFT JOIN)

| id  | first_name | order_id | sales |
| --- | ---------- | -------- | ----- |
| 1   | Maria      | 101      | 1200  |
| 2   | John       | 102      | 800   |
| 3   | George     | 103      | 600   |
| 4   | Martin     | NULL     | NULL  |
| 5   | Peter      | NULL     | NULL  |

### Use Cases

- Data enrichment
- Finding missing related data

---

## 8. RIGHT JOIN

### Definition

Returns:

- **ALL rows from the right table**
- Only **matching rows from the left table**

### Syntax

```sql
SELECT columns
FROM tableA
RIGHT JOIN tableB
ON tableA.key = tableB.key;
```

### Example Output (RIGHT JOIN)

| id   | first_name | order_id | sales |
| ---- | ---------- | -------- | ----- |
| 1    | Maria      | 101      | 1200  |
| 2    | John       | 102      | 800   |
| 3    | George     | 103      | 600   |
| NULL | NULL       | 104      | 300   |

### Notes

- Order matters
- Less commonly used

---

## 9. Alternative to RIGHT JOIN (Recommended)

Instead of using RIGHT JOIN:

```sql
SELECT ...
FROM orders o
LEFT JOIN customers c
ON o.customer_id = c.id;
```

👉 **LEFT JOIN is preferred in real projects**.

---

## 10. FULL JOIN

### Definition

Returns:

- All rows from **both tables**
- Matching rows combined
- Non‑matching rows filled with **NULLs**

### Syntax

```sql
SELECT columns
FROM tableA
FULL JOIN tableB
ON tableA.key = tableB.key;
```

### Output (FULL JOIN)

| id   | first_name | order_id | sales |
| ---- | ---------- | -------- | ----- |
| 1    | Maria      | 101      | 1200  |
| 2    | John       | 102      | 800   |
| 3    | George     | 103      | 600   |
| 4    | Martin     | NULL     | NULL  |
| 5    | Peter      | NULL     | NULL  |
| NULL | NULL       | 104      | 300   |

### Characteristics

- Order does NOT matter
- Includes matching + non‑matching rows

---

## 11. Table Aliases (Best Practice)

### Why Use Aliases?

- Improves readability
- Avoids column name ambiguity
- Shortens long table names

### Example

```sql
SELECT c.id, c.first_name, o.order_id, o.sales
FROM customers AS c
LEFT JOIN orders AS o
ON c.id = o.customer_id;
```

---

## 12. How SQL Executes JOINs (Conceptual)

1. Start from the **left table**
2. Search for matching rows in the right table
3. Apply join rules
4. Fill unmatched values with NULL (if allowed)

---

## 13. Summary

- JOINs combine **columns**, not rows
- Choose join type based on your goal
- LEFT JOIN is the most commonly used
- Always:
  - Use aliases
  - Specify join type
  - Join on keys
