## 1. What Is a Subquery?

### 1.1 Definition

A **subquery** (also called an **inner query** or **nested query**) is a complete SQL SELECT statement embedded inside another SQL query. The enclosing query is called the **main query** or **outer query**.

```
┌────────────────────────────────────────────────────────┐
│                     MAIN QUERY (Outer)                 │
│                                                        │
│   SELECT column_a, column_b                            │
│   FROM   some_table                                    │
│   WHERE  column_c > (                                  │
│                                                        │
│       ┌────────────────────────────────────────┐       │
│       │          SUBQUERY (Inner)              │       │
│       │                                        │       │
│       │   SELECT AVG(price)                    │       │
│       │   FROM   products                      │       │
│       └────────────────────────────────────────┘       │
│                     )                                  │
└────────────────────────────────────────────────────────┘
```

### 1.2 Key Properties of Subqueries

| Property | Detail |
|---|---|
| **Placement** | Inside parentheses `( )` anywhere a value, row, or table is expected |
| **Execution order** | Subquery executes first (for non-correlated); result is passed to main query |
| **Result visibility** | Intermediate result is never shown to the user — it exists only in memory |
| **Scope** | A subquery's intermediate results are local to its parent main query; no external query can access them |
| **Cleanup** | Once the full query finishes, the intermediate result is destroyed and memory is freed |

### 1.3 Terminology Cheat Sheet

| Term | Meaning |
|---|---|
| **Subquery / Inner query** | The nested SELECT inside parentheses |
| **Main query / Outer query** | The enclosing SELECT that uses the subquery's result |
| **Intermediate result** | The temporary output of a subquery, held in cache until the main query consumes it |
| **Non-correlated subquery** | Subquery that runs independently of the main query — executes once |
| **Correlated subquery** | Subquery that references columns from the main query — executes once per row |

## 2. Why Use Subqueries?

### 2.1 Breaking Down Multi-Step Logic

Most real analytical problems cannot be solved in a single flat SELECT. They have natural steps that must be computed in order:

```
Complex Task
     │
     ├── Step 1: Join and prepare the raw data
     │         └── Subquery 1
     │
     ├── Step 2: Filter outliers or apply conditions
     │         └── Subquery 2
     │
     ├── Step 3: Apply transformations, handle NULLs, create new columns
     │         └── Subquery 3
     │
     └── Step 4: Aggregate and present the final result
               └── MAIN QUERY (uses all subquery results as input)
```

Each step is a subquery that prepares data for the next. The main query performs the final aggregation or output.

### 2.2 Enabling Dynamic Filters

Without subqueries, WHERE clauses can only use static values:

```sql
-- Static (brittle) — breaks when average changes
WHERE price > 38.50

-- Dynamic (robust) — always uses the current average
WHERE price > (SELECT AVG(price) FROM products)
```

### 2.3 Supporting the Main Query

The subquery is always a **supporter** — it exists to provide data or context that the main query cannot compute in a single step. The main query owns the final result.

## 3. Subquery Result Types

Subqueries are classified by how many rows and columns they return. This determines where and how they can be used.

### 3.1 Scalar Subquery

**Returns:** Exactly one row, one column — a single value.

```sql
-- Returns a single value: the average price
SELECT AVG(price) FROM products;
-- Result: 38.50  (1 row × 1 column)
```

**Allowed in:** SELECT clause, WHERE clause with comparison operators, JOIN conditions.

### 3.2 Row Subquery

**Returns:** Multiple rows, one column — a list of values.

```sql
-- Returns a list of customer IDs
SELECT customer_id FROM orders WHERE status = 'completed';
-- Result: 1, 3, 4, 7  (4 rows × 1 column)
```

**Allowed in:** WHERE clause with IN, NOT IN, ANY, ALL, EXISTS operators.

### 3.3 Table Subquery

**Returns:** Multiple rows and multiple columns — a full result set that acts like a temporary table.

```sql
-- Returns a result set with two columns
SELECT customer_id, SUM(total_amount) AS total_sales
FROM orders
GROUP BY customer_id;
-- Result: 4 rows × 2 columns
```

**Allowed in:** FROM clause (must be aliased), JOIN clause.

### 3.4 Quick Reference

| Type | Rows | Columns | Use In |
|---|---|---|---|
| **Scalar** | 1 | 1 | SELECT, WHERE (comparison), JOIN |
| **Row** | Many | 1 | WHERE (IN, ANY, ALL, EXISTS) |
| **Table** | Many | Many | FROM clause, JOIN clause |

## 4. Categories of Subqueries

Subqueries can be grouped three ways simultaneously:

```
Subquery Categories
├── By Dependency
│   ├── Non-Correlated (independent, runs once)
│   └── Correlated (depends on outer query, runs per row)
│
├── By Result Type
│   ├── Scalar   (1 row × 1 col)
│   ├── Row      (N rows × 1 col)
│   └── Table    (N rows × M cols)
│
└── By Location in the Main Query
    ├── FROM clause     (table subquery)
    ├── SELECT clause   (scalar subquery only)
    ├── JOIN clause     (table subquery)
    └── WHERE clause
        ├── Comparison operators (=, <, >, <=, >=, <>)  → scalar only
        ├── IN / NOT IN                                  → row subquery
        ├── ANY / ALL                                    → row subquery
        └── EXISTS / NOT EXISTS                         → correlated
```

## 6. FROM Clause Subquery

### 6.1 Definition

A **FROM clause subquery** replaces a table name in the FROM clause with a query that returns a temporary result set. The outer query then treats this result set exactly like a regular table.

> In MSSQL, the subquery in the FROM clause **must** have an alias. In MySQL and PostgreSQL, an alias is strongly recommended and often required.

### 6.2 Syntax

```sql
SELECT column_a, column_b
FROM (
    SELECT column_x, column_y, computed_column
    FROM   base_table
    WHERE  some_condition
) AS subquery_alias         -- alias is mandatory in MSSQL
WHERE  column_a > some_filter;
```

### 6.3 Task: Find Products Priced Above Average

**Step 1 — Subquery:** Calculate the price alongside the overall average using a window function.

```sql
SELECT
    product_id,
    product_name,
    price,
    AVG(price) OVER () AS avg_price     -- window function: no partition = global average
FROM products;
```

| product\_id | product\_name  | price | avg\_price |
| ----------- | -------------- | ----- | ---------- |
| 1           | Mountain Bike  | 89.99 | 43.99      |
| 2           | Yoga Mat       | 24.99 | 43.99      |
| 3           | Coffee Maker   | 49.99 | 43.99      |
| 4           | Desk Lamp      | 19.99 | 43.99      |
| 5           | Wireless Mouse | 34.99 | 43.99      |

**Step 2 — Full Query:** Wrap the subquery in a FROM clause and filter in the main query.

```sql
SELECT product_id, product_name, price, avg_price
FROM (
    SELECT
        product_id,
        product_name,
        price,
        AVG(price) OVER () AS avg_price
    FROM products
) AS t                          -- 't' for table; alias required in MSSQL
WHERE price > avg_price;
```

**Output:**

| product\_id | product\_name | price | avg\_price |
| ----------- | ------------- | ----- | ---------- |
| 1           | Mountain Bike | 89.99 | 43.99      |
| 3           | Coffee Maker  | 49.99 | 43.99      |

> **Debugging Tip:** To inspect subquery results without running the full query, highlight only the inner SELECT (without the parentheses) and execute. SSMS, MySQL Workbench, and pgAdmin all support partial execution this way.

### 6.4 Task: Rank Customers by Total Sales

```sql
SELECT
    customer_id,
    total_sales,
    RANK() OVER (ORDER BY total_sales DESC) AS sales_rank
FROM (
    SELECT
        customer_id,
        SUM(total_amount) AS total_sales
    FROM orders
    GROUP BY customer_id
) AS t;
```

**Output:**

| customer\_id | total\_sales | sales\_rank |
| ------------ | ------------ | ----------- |
| 2            | 700.00       | 1           |
| 1            | 840.00       | 1           |
| 4            | 465.00       | 3           |
| 3            | 405.00       | 4           |

> **Execution Flow:** SQL executes the subquery first, produces the aggregated `total_sales` per customer as an intermediate result, stores it in cache, then executes the RANK window function in the main query against that intermediate result.

## 7. SELECT Clause Subquery

### 7.1 Definition

A **SELECT clause subquery** embeds a query directly inside the SELECT column list of the main query. The subquery runs alongside the other columns and its result is appended as an additional column to every row.

> **Critical Rule:** A SELECT clause subquery must be a **scalar subquery** — it must return exactly one row and one column. Returning multiple rows causes an error.

### 7.2 Syntax

```sql
SELECT
    column_a,
    column_b,
    (SELECT aggregate_function(col) FROM other_table WHERE condition) AS alias
FROM main_table;
```

### 7.3 Task: Show Products with Total Order Count

```sql
SELECT
    product_id,
    product_name,
    price,
    (SELECT COUNT(*) FROM orders) AS total_orders     -- scalar subquery: always returns 1 value
FROM products;
```

**Output:**

| product\_id | product\_name  | price | total\_orders |
| ----------- | -------------- | ----- | ------------- |
| 1           | Mountain Bike  | 89.99 | 10            |
| 2           | Yoga Mat       | 24.99 | 10            |
| 3           | Coffee Maker   | 49.99 | 10            |
| 4           | Desk Lamp      | 19.99 | 10            |
| 5           | Wireless Mouse | 34.99 | 10            |

> The total order count (10) is the same for every row because it's a global aggregate with no filter. It gets appended as a constant column.

### 7.4 What Happens if the Subquery Returns Multiple Rows?

```sql
-- ❌ Error: subquery returns more than one value
SELECT
    product_id,
    (SELECT order_id FROM orders)   -- returns 10 rows — not allowed in SELECT clause
FROM products;

-- ✅ Fix: use an aggregate to reduce to a single value
SELECT
    product_id,
    (SELECT COUNT(*) FROM orders) AS total_orders
FROM products;
```

### 7.5 Execution Flow

```
Step 1: SQL identifies the scalar subquery
        → SELECT COUNT(*) FROM orders
        → Result: 10  (stored in cache)

Step 2: SQL executes the main query
        → SELECT product_id, product_name, price, [10] FROM products
        → The value 10 is substituted in place of the subquery

Step 3: Final result returned — 10 appears as a constant in every row
```

## 8. JOIN Clause Subquery

### 8.1 Definition

A **JOIN clause subquery** uses a subquery as one side of a JOIN operation, creating a dynamic result set to join against instead of a static table.

### 8.2 Syntax

```sql
SELECT c.*, o_summary.total_orders
FROM customers c
LEFT JOIN (
    SELECT customer_id, COUNT(*) AS total_orders
    FROM   orders
    GROUP BY customer_id
) AS o_summary ON c.customer_id = o_summary.customer_id;
```

### 8.3 Task: Show All Customers with Their Total Orders

```sql
SELECT
    c.customer_id,
    c.first_name,
    c.country,
    o_summary.total_orders
FROM customers c
LEFT JOIN (
    SELECT
        customer_id,
        COUNT(*)            AS total_orders,
        SUM(total_amount)   AS total_revenue
    FROM orders
    GROUP BY customer_id
) AS o_summary ON c.customer_id = o_summary.customer_id;
```

**Output:**

| customer\_id | first\_name | country | total\_orders | total\_revenue |
| ------------ | ----------- | ------- | ------------- | -------------- |
| 1            | John        | Germany | 3             | 840.00         |
| 2            | Sarah       | USA     | 3             | 700.00         |
| 3            | Lena        | Germany | 2             | 405.00         |
| 4            | Tom         | UK      | 2             | 465.00         |
| 5            | Anna        | Germany | NULL          | NULL           |

> **Why LEFT JOIN?** Anna (customer 5) has no orders. An INNER JOIN would exclude her entirely. LEFT JOIN preserves all customers and returns NULL for the aggregated columns when no match exists.

> **Performance Note:** Pre-aggregating in the subquery (GROUP BY before the JOIN) is generally more efficient than joining first and then grouping, because the JOIN operates on a smaller, already-summarized result set.

## 9. WHERE Clause Subquery — Comparison Operators

### 9.1 Definition

A comparison operator subquery places a scalar subquery on the right-hand side of a `=`, `<>`, `<`, `>`, `<=`, or `>=` operator in a WHERE clause. The subquery produces a single value used as the comparison target.

> **Rule:** The subquery must return exactly one value (scalar subquery). Multiple rows will cause an error.

### 9.2 Syntax

```sql
SELECT column_a, column_b
FROM   table_one
WHERE  column_c  >  (SELECT aggregate_value FROM table_two WHERE condition);
--     ^^^^^^^^  ^  ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
--     column    op  scalar subquery — must return exactly 1 row, 1 column
```

### 9.3 Available Comparison Operators

| Operator | Meaning |
|---|---|
| `=` | Equal to the subquery result |
| `<>` or `!=` | Not equal to the subquery result |
| `>` | Greater than the subquery result |
| `<` | Less than the subquery result |
| `>=` | Greater than or equal to the subquery result |
| `<=` | Less than or equal to the subquery result |

### 9.4 Task: Find Products Priced Above Average

```sql
SELECT
    product_id,
    product_name,
    price,
    (SELECT AVG(price) FROM products) AS avg_price   -- also shown for reference
FROM products
WHERE price > (SELECT AVG(price) FROM products);
```

**Output:**

| product\_id | product\_name | price | avg\_price |
| ----------- | ------------- | ----- | ---------- |
| 1           | Mountain Bike | 89.99 | 43.99      |
| 3           | Coffee Maker  | 49.99 | 43.99      |

### 9.5 Execution Flow

```
Step 1: SQL identifies and executes the subquery
        → SELECT AVG(price) FROM products
        → Result: 43.99  (stored in cache as intermediate result)

Step 2: SQL injects the value into the main query
        → WHERE price > 43.99

Step 3: SQL executes the main query against the products table
        → Filters: keep only rows where price > 43.99
        → Returns: Mountain Bike (89.99), Coffee Maker (49.99)
```

## 10. WHERE Clause Subquery — IN and NOT IN Operators

### 10.1 Definition

The **IN operator** checks whether a value matches **any** value in a list. When combined with a subquery, the list is generated dynamically rather than typed statically. This allows the filter to adapt automatically as the underlying data changes.

Unlike comparison operators, IN accepts **row subqueries** — subqueries that return multiple rows (one column).

### 10.2 Syntax

```sql
-- Static (brittle)
WHERE customer_id IN (1, 4)

-- Dynamic with subquery (robust)
WHERE customer_id IN (
    SELECT customer_id
    FROM   customers
    WHERE  country = 'Germany'
)
```

### 10.3 Task: Orders from German Customers

```sql
SELECT
    order_id,
    customer_id,
    order_date,
    total_amount
FROM orders
WHERE customer_id IN (
    SELECT customer_id
    FROM   customers
    WHERE  country = 'Germany'
);
```

**Subquery intermediate result:**

| customer\_id |
| ------------ |
| 1            |
| 3            |
| 5            |

**Final output:**

| order\_id | customer\_id | order\_date | total\_amount |
| --------- | ------------ | ----------- | ------------- |
| 101       | 1            | 2024-01-10  | 250.00        |
| 103       | 1            | 2024-02-01  | 380.00        |
| 104       | 3            | 2024-02-10  | 95.00         |
| 105       | 1            | 2024-03-05  | 210.00        |
| 107       | 3            | 2024-03-20  | 310.00        |

### 10.4 NOT IN: Inverting the Filter

```sql
-- Show orders from non-German customers
SELECT order_id, customer_id, order_date, total_amount
FROM orders
WHERE customer_id NOT IN (
    SELECT customer_id
    FROM   customers
    WHERE  country = 'Germany'
);
```

**Output:**

| order\_id | customer\_id | order\_date | total\_amount |
| --------- | ------------ | ----------- | ------------- |
| 102       | 2            | 2024-01-15  | 120.00        |
| 106       | 2            | 2024-03-12  | 450.00        |
| 108       | 4            | 2024-04-02  | 175.00        |
| 109       | 4            | 2024-04-15  | 290.00        |
| 110       | 2            | 2024-05-01  | 130.00        |

### 10.5 Execution Flow

```
Step 1: SQL identifies and executes the subquery
        → SELECT customer_id FROM customers WHERE country = 'Germany'
        → Intermediate result: [1, 3, 5]

Step 2: SQL injects the list into the main query
        → WHERE customer_id IN (1, 3, 5)

Step 3: SQL scans orders row by row:
        order_id 101: customer_id=1  → IN list? ✅ Include
        order_id 102: customer_id=2  → IN list? ❌ Exclude
        order_id 103: customer_id=1  → IN list? ✅ Include
        ... and so on

Step 4: Return all matching rows as final result
```

> **Warning — NOT IN and NULLs:** If the subquery result contains any NULL value, `NOT IN` will return no rows at all. This is because `x NOT IN (1, NULL)` is evaluated as `x != 1 AND x != NULL` — and any comparison with NULL is UNKNOWN, not TRUE. Use `NOT EXISTS` instead when NULLs may be present.

## 11. WHERE Clause Subquery — ANY and ALL Operators

### 11.1 The ANY Operator

**Definition:** `ANY` returns TRUE if the comparison is true for **at least one** value in the subquery result. It is equivalent to asking: "Is my value greater than (or less than, or equal to) at least one value in this list?"

```sql
-- Syntax: compare_value  operator  ANY (subquery)
WHERE salary > ANY (SELECT salary FROM employees WHERE gender = 'Male')
-- True if salary is greater than at least one male salary
```

### 11.2 The ALL Operator

**Definition:** `ALL` returns TRUE if the comparison is true for **every** value in the subquery result. It asks: "Is my value greater than (or less than, or equal to) every value in this list?"

```sql
-- Syntax: compare_value  operator  ALL (subquery)
WHERE salary > ALL (SELECT salary FROM employees WHERE gender = 'Male')
-- True only if salary is greater than every male salary
```

### 11.3 Task: Female Employees Earning More Than Any Male Employee

```sql
SELECT employee_id, first_name, salary
FROM employees
WHERE gender = 'Female'
  AND salary > ANY (
      SELECT salary
      FROM   employees
      WHERE  gender = 'Male'
  );
```

**Male salaries (subquery intermediate result):**

| salary |
| ------ |
| 75,000 |
| 52,000 |
| 48,000 |

**Evaluation per female employee:**

| first\_name | salary | > 75,000? | > 52,000? | > 48,000? | ANY true? |
|---|---|---|---|---|---|
| Mary | 61,000 | ❌ | ✅ | ✅ | ✅ Include |
| Carol | 44,000 | ❌ | ❌ | ❌ | ❌ Exclude |

**Output:**

| employee\_id | first\_name | salary |
| ------------ | ----------- | ------ |
| 4            | Mary        | 61,000 |

### 11.4 Task: Female Employees Earning More Than All Male Employees

```sql
SELECT employee_id, first_name, salary
FROM employees
WHERE gender = 'Female'
  AND salary > ALL (
      SELECT salary
      FROM   employees
      WHERE  gender = 'Male'
  );
```

**Output: (empty — no female earns more than all three males)**

Carol (44,000) and Mary (61,000) are both less than Michael (75,000), so neither passes the ALL condition.

### 11.5 ANY vs ALL vs IN — Comparison

| Operator | Passes if… | Example |
|---|---|---|
| `IN` | Value exactly matches any in the list | `WHERE id IN (1, 2, 3)` |
| `= ANY` | Equivalent to IN | `WHERE id = ANY (subquery)` |
| `> ANY` | Greater than at least one value | `WHERE salary > ANY (...)` |
| `> ALL` | Greater than every value | `WHERE salary > ALL (...)` |
| `< ANY` | Less than at least one value (i.e. not the maximum) | `WHERE price < ANY (...)` |
| `< ALL` | Less than every value (i.e. less than the minimum) | `WHERE price < ALL (...)` |

## 12. Correlated vs Non-Correlated Subqueries

### 12.1 Non-Correlated Subquery

**Definition:** A subquery that is **completely independent** of the main query. It can be extracted and run on its own as a standalone query. It executes **once**, and its result is reused for every row in the main query.

```
Execution model:
  1. Execute subquery once → intermediate result
  2. Execute main query using intermediate result
  3. Return final output
```

```sql
-- Non-correlated: the subquery has no reference to the outer query
SELECT product_id, price
FROM   products
WHERE  price > (SELECT AVG(price) FROM products);
--              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
--              This runs once. No reference to outer 'products' alias.
```

### 12.2 Correlated Subquery

**Definition:** A subquery that **references columns from the main query**. It cannot run independently. It executes **once per row** of the main query, with the current row's column values substituted into the subquery.

```
Execution model (for N rows in main query):
  For each row in main query:
    1. Pass current row's values to the subquery
    2. Execute subquery with those values
    3. Include or exclude the row based on subquery result
  After all N rows processed → return final output
```

```sql
-- Correlated: subquery references c.customer_id from the outer query
SELECT
    c.customer_id,
    c.first_name,
    (SELECT COUNT(*)
     FROM   orders o
     WHERE  o.customer_id = c.customer_id   -- ← c.customer_id comes from the outer query
    ) AS total_orders
FROM customers c;
```

### 12.3 Iteration Visualization (Correlated)

```
Main query: scanning customers table row by row

Row 1: customer_id = 1 (John)
  → Subquery executes: SELECT COUNT(*) FROM orders WHERE customer_id = 1
  → Result: 3  → total_orders = 3  → Row included ✅

Row 2: customer_id = 2 (Sarah)
  → Subquery executes: SELECT COUNT(*) FROM orders WHERE customer_id = 2
  → Result: 3  → total_orders = 3  → Row included ✅

Row 3: customer_id = 3 (Lena)
  → Subquery executes: SELECT COUNT(*) FROM orders WHERE customer_id = 3
  → Result: 2  → total_orders = 2  → Row included ✅

Row 4: customer_id = 4 (Tom)
  → Subquery executes: SELECT COUNT(*) FROM orders WHERE customer_id = 4
  → Result: 2  → total_orders = 2  → Row included ✅

Row 5: customer_id = 5 (Anna)
  → Subquery executes: SELECT COUNT(*) FROM orders WHERE customer_id = 5
  → Result: 0  → total_orders = 0  → Row included ✅ (0, not NULL)
```

### 12.4 Side-by-Side Comparison

| Property | Non-Correlated | Correlated |
|---|---|---|
| **Depends on main query?** | ❌ Independent | ✅ Depends on each row |
| **Executes how many times?** | Once | Once per row |
| **Can run standalone?** | ✅ Yes | ❌ No — references outer columns |
| **Easier to read?** | ✅ Simpler | ⚠️ More complex |
| **Performance** | ✅ Better (runs once) | ⚠️ Worse (runs N times) |
| **Used for** | Static comparison / lookup | Row-by-row comparison / existence check |
| **Debugging** | Highlight subquery, execute | Must test row-by-row with hardcoded value |

### 12.5 Debugging a Correlated Subquery

Since a correlated subquery cannot run standalone, test it by substituting a specific row's value manually:

```sql
-- Original correlated subquery (cannot run alone)
-- WHERE  o.customer_id = c.customer_id   ← c.customer_id is unknown here

-- Debug: hardcode one customer's ID to test
SELECT COUNT(*)
FROM   orders o
WHERE  o.customer_id = 1;     -- manually inject the value from one row
-- Result: 3  → confirms the subquery logic is correct for customer 1
```

## 13. EXISTS and NOT EXISTS Operators

### 13.1 Definition

The **EXISTS** operator tests whether a correlated subquery returns **any rows at all**. It does not care about the value returned — only whether a result exists (TRUE) or is empty (FALSE).

- `EXISTS` → include the row if the subquery returns ≥ 1 row
- `NOT EXISTS` → include the row if the subquery returns 0 rows

> **Best Practice:** Write `SELECT 1` inside an EXISTS subquery. You are not using the data — just testing existence. `SELECT 1` is faster than `SELECT *` because the engine doesn't need to retrieve any column data.

### 13.2 Syntax

```sql
SELECT column_a, column_b
FROM   table_two  t2                    -- main query
WHERE  EXISTS (
    SELECT 1
    FROM   table_one  t1               -- subquery
    WHERE  t1.id = t2.id              -- ← correlated link: references t2 from outer query
);
```

### 13.3 Task: Orders from German Customers (EXISTS version)

```sql
SELECT o.order_id, o.customer_id, o.order_date, o.total_amount
FROM   orders o
WHERE  EXISTS (
    SELECT 1
    FROM   customers c
    WHERE  c.customer_id = o.customer_id   -- correlated: o.customer_id from main query
      AND  c.country = 'Germany'
);
```

**Output:**

| order\_id | customer\_id | order\_date | total\_amount |
| --------- | ------------ | ----------- | ------------- |
| 101       | 1            | 2024-01-10  | 250.00        |
| 103       | 1            | 2024-02-01  | 380.00        |
| 104       | 3            | 2024-02-10  | 95.00         |
| 105       | 1            | 2024-03-05  | 210.00        |
| 107       | 3            | 2024-03-20  | 310.00        |

### 13.4 NOT EXISTS: Customers with No Orders

```sql
-- Find customers who have placed zero orders
SELECT c.customer_id, c.first_name, c.country
FROM   customers c
WHERE  NOT EXISTS (
    SELECT 1
    FROM   orders o
    WHERE  o.customer_id = c.customer_id
);
```

**Output:**

| customer\_id | first\_name | country |
| ------------ | ----------- | ------- |
| 5            | Anna        | Germany |

### 13.5 EXISTS Execution Flow (Row-by-Row)

```
Main query: scanning orders row by row

Order 101: customer_id = 1
  → Subquery: SELECT 1 FROM customers WHERE customer_id=1 AND country='Germany'
  → Customer 1 (John) IS from Germany → returns 1 row
  → EXISTS = TRUE → order 101 ✅ included

Order 102: customer_id = 2
  → Subquery: SELECT 1 FROM customers WHERE customer_id=2 AND country='Germany'
  → Customer 2 (Sarah) is from USA → returns 0 rows
  → EXISTS = FALSE → order 102 ❌ excluded

Order 103: customer_id = 1
  → Subquery: EXISTS = TRUE → order 103 ✅ included
  ... continues for all 10 orders
```

### 13.6 EXISTS vs IN — When to Use Which

| Scenario | Preferred Operator | Reason |
|---|---|---|
| Subquery may return NULLs | `EXISTS` | IN with NULLs can return 0 rows silently |
| Filtering by existence only | `EXISTS` | More semantic; engine can short-circuit on first match |
| Filtering by exact value match | `IN` | Cleaner syntax for simple value lists |
| Checking rows that DON'T match | `NOT EXISTS` | Safer than `NOT IN` when NULLs are possible |
| Performance with large datasets | `EXISTS` | Can stop searching after first match; IN fetches all |

## 14. How the Database Executes Subqueries

### 14.1 Non-Correlated Execution (Single Pass)

```
Client submits query with non-correlated subquery
                    │
                    ▼
        Database Engine receives query
                    │
                    ▼
        ┌─────────────────────────┐
        │  1. Identify subquery   │
        └────────────┬────────────┘
                     │
                     ▼
        ┌─────────────────────────┐
        │  2. Execute subquery    │ ← reads from disk → User Data
        │     (runs ONCE)         │
        └────────────┬────────────┘
                     │
                     ▼
        ┌─────────────────────────┐
        │  3. Store result in     │ ← intermediate result lives in Cache
        │     Cache (temporary)   │
        └────────────┬────────────┘
                     │
                     ▼
        ┌─────────────────────────┐
        │  4. Execute main query  │ ← reads intermediate result from Cache
        │     using cached result │   (no disk access needed)
        └────────────┬────────────┘
                     │
                     ▼
        ┌─────────────────────────┐
        │  5. Return final result │ → sent to client
        └────────────┬────────────┘
                     │
                     ▼
        ┌─────────────────────────┐
        │  6. Clean up Cache      │ ← intermediate result destroyed
        └─────────────────────────┘
```

### 14.2 Correlated Execution (Per-Row Loop)

```
Client submits query with correlated subquery
                    │
                    ▼
        1. Execute main query — get first row
                    │
                    ▼
        2. Pass current row's values to subquery
                    │
                    ▼
        3. Execute subquery with those values
                    │
              ┌─────┴──────┐
         Has result?   No result?
              │               │
              ▼               ▼
        Include row      Exclude row
              │               │
              └──────┬─────────┘
                     │
                     ▼
        4. Move to next row → repeat from step 2
                     │
                     ▼ (after all rows processed)
        5. Return accumulated final results to client
```

## 15. Cross-Database Comparison: MySQL, MSSQL, PostgreSQL

### 15.1 Subquery Alias Requirement (FROM clause)

| Database | Alias Required? | Example |
|---|---|---|
| **MSSQL** | ✅ Yes — mandatory | `FROM (SELECT ...) AS t` |
| **MySQL** | ✅ Yes — mandatory | `FROM (SELECT ...) AS t` |
| **PostgreSQL** | ✅ Yes — mandatory | `FROM (SELECT ...) AS t` or `FROM (SELECT ...)  t` |

All three databases require an alias for FROM clause subqueries.

### 15.2 Scalar Subquery in SELECT — All Three

```sql
-- Works identically in MSSQL, MySQL, PostgreSQL
SELECT
    product_id,
    product_name,
    price,
    (SELECT AVG(price) FROM products) AS avg_price
FROM products;
```

### 15.3 EXIST / NOT EXISTS — All Three

```sql
-- Works identically in MSSQL, MySQL, PostgreSQL
SELECT customer_id, first_name
FROM   customers c
WHERE  EXISTS (
    SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);
```

### 15.4 ANY / ALL Syntax

All three databases support `ANY` and `ALL` with comparison operators. MySQL additionally supports `SOME` as an alias for `ANY`.

```sql
-- MSSQL, MySQL, PostgreSQL: identical syntax
WHERE salary > ANY (SELECT salary FROM employees WHERE gender = 'Male')
WHERE salary > ALL (SELECT salary FROM employees WHERE gender = 'Male')

-- MySQL only: SOME is an alias for ANY
WHERE salary > SOME (SELECT salary FROM employees WHERE gender = 'Male')
```

### 15.5 NOT IN and NULL Behavior

All three databases share the same NULL-related pitfall with `NOT IN`:

```sql
-- ⚠️ All three: if subquery returns any NULL, NOT IN returns 0 rows
WHERE customer_id NOT IN (SELECT customer_id FROM some_table)
-- If some_table has a NULL customer_id → result is always empty!

-- ✅ Safe alternative in all three: use NOT EXISTS
WHERE NOT EXISTS (
    SELECT 1 FROM some_table st
    WHERE st.customer_id = c.customer_id
)
```

### 15.6 Lateral / Cross Apply Subqueries

MSSQL and PostgreSQL support a more powerful form of correlated subquery that allows the subquery to reference the outer query in the FROM clause (not just WHERE):

```sql
-- MSSQL: CROSS APPLY / OUTER APPLY
SELECT c.customer_id, c.first_name, latest_order.order_date
FROM   customers c
OUTER APPLY (
    SELECT TOP 1 order_date
    FROM   orders o
    WHERE  o.customer_id = c.customer_id
    ORDER BY order_date DESC
) AS latest_order;

-- PostgreSQL: LATERAL JOIN
SELECT c.customer_id, c.first_name, latest_order.order_date
FROM   customers c
LEFT JOIN LATERAL (
    SELECT order_date
    FROM   orders o
    WHERE  o.customer_id = c.customer_id
    ORDER BY order_date DESC
    LIMIT 1
) AS latest_order ON true;

-- MySQL 8.0+: LATERAL JOIN (added in 8.0)
-- Same syntax as PostgreSQL LATERAL
```

### 15.7 Performance Tools for Subquery Analysis

| Tool | MSSQL | MySQL | PostgreSQL |
|---|---|---|---|
| View execution plan | SSMS (graphical) or `SET SHOWPLAN_ALL ON` | `EXPLAIN` / `EXPLAIN ANALYZE` | `EXPLAIN ANALYZE` |
| Subquery cost visible? | ✅ Yes — each node shows cost % | ⚠️ Partial — `type` column shows join method | ✅ Yes — nested loop cost shown |
| Detect correlated subquery in plan | Key Lookup / Nested Loop | `DEPENDENT SUBQUERY` in type column | `SubPlan` node in plan output |

```sql
-- MySQL: detect correlated subquery in EXPLAIN output
EXPLAIN
SELECT customer_id, first_name,
       (SELECT COUNT(*) FROM orders o WHERE o.customer_id = c.customer_id) AS total_orders
FROM customers c;
-- Look for: select_type = DEPENDENT SUBQUERY → this is correlated
```

## 16. Common Mistakes

### 16.1 Using a Non-Scalar Subquery in SELECT or Comparison WHERE

```sql
-- ❌ Error: subquery returns multiple rows in SELECT clause
SELECT
    product_id,
    (SELECT order_id FROM orders)   -- returns 10 rows — not scalar!
FROM products;

-- ✅ Fix: aggregate to produce one value
SELECT
    product_id,
    (SELECT COUNT(*) FROM orders) AS total_orders
FROM products;
```

### 16.2 NOT IN with NULLs Silently Returns Empty Results

```sql
-- ❌ Dangerous: if any customer_id in the table is NULL, returns 0 rows
SELECT * FROM orders
WHERE customer_id NOT IN (SELECT customer_id FROM some_table);

-- ✅ Safe: NOT EXISTS handles NULLs correctly
SELECT * FROM orders o
WHERE NOT EXISTS (
    SELECT 1 FROM some_table st WHERE st.customer_id = o.customer_id
);
```

### 16.3 Forgetting the Alias on FROM Clause Subqueries

```sql
-- ❌ Error in MSSQL (and MySQL): Every derived table must have its own alias
SELECT * FROM (SELECT product_id, price FROM products);

-- ✅ Fixed:
SELECT * FROM (SELECT product_id, price FROM products) AS t;
```

### 16.4 Using Correlated Subqueries on Large Tables

```sql
-- ❌ Slow: subquery executes once per row — millions of executions for large tables
SELECT c.customer_id,
       (SELECT SUM(total_amount) FROM orders o WHERE o.customer_id = c.customer_id)
FROM customers c;   -- if customers has 1M rows → 1M subquery executions

-- ✅ Better: pre-aggregate with GROUP BY, then JOIN
SELECT c.customer_id, o_agg.total_revenue
FROM customers c
LEFT JOIN (
    SELECT customer_id, SUM(total_amount) AS total_revenue
    FROM orders GROUP BY customer_id
) AS o_agg ON c.customer_id = o_agg.customer_id;
```

### 16.5 Nesting Subqueries Too Deeply

```sql
-- ❌ Unreadable: triple-nested subqueries
SELECT * FROM (SELECT * FROM (SELECT * FROM orders WHERE ...) t1 WHERE ...) t2 WHERE ...;

-- ✅ Better: refactor into a CTE (covered in Chapter 12)
WITH step1 AS (SELECT * FROM orders WHERE ...),
     step2 AS (SELECT * FROM step1 WHERE ...)
SELECT * FROM step2 WHERE ...;
```

### 16.6 Confusing Correlated and Non-Correlated Subquery Behavior

**Mistake:** Expecting a correlated subquery to run once, not realizing it runs per row.

```sql
-- This looks simple but runs SELECT AVG() once PER ROW in a correlated context
-- Make sure the subquery is truly non-correlated when you want it to run once
SELECT p.product_id,
       (SELECT AVG(price) FROM products)    -- non-correlated → runs ONCE ✅
FROM products p;

SELECT p.product_id,
       (SELECT AVG(price) FROM products p2
        WHERE p2.category = p.category)     -- correlated → runs once PER ROW ⚠️
FROM products p;
```

## 17. Real-World Analytics Use Cases

### 17.1 Financial Reporting: Above-Threshold Customers

**Scenario:** A finance team needs all customers whose lifetime spend is above the average customer spend.

```sql
SELECT
    c.customer_id,
    c.first_name,
    c.country,
    o_agg.total_spent
FROM customers c
JOIN (
    SELECT customer_id, SUM(total_amount) AS total_spent
    FROM   orders
    WHERE  status = 'completed'
    GROUP BY customer_id
) AS o_agg ON c.customer_id = o_agg.customer_id
WHERE o_agg.total_spent > (
    SELECT AVG(customer_total)
    FROM (
        SELECT SUM(total_amount) AS customer_total
        FROM   orders
        WHERE  status = 'completed'
        GROUP BY customer_id
    ) AS avg_calc
);
```

### 17.2 ETL Data Validation: Detect Missing Dimension Keys

**Scenario:** A data engineer checks whether all product keys in the fact table exist in the dimension table before loading.

```sql
-- Find fact rows with no matching product in the dimension
SELECT fact.order_id, fact.product_id
FROM   fact_orders fact
WHERE  NOT EXISTS (
    SELECT 1
    FROM   dim_products dp
    WHERE  dp.product_id = fact.product_id
);
-- Any rows returned = data quality issue to fix before loading
```

### 17.3 HR Analysis: Rank Employees Within Their Department

**Scenario:** An HR analyst wants to rank each employee's salary within their own department, then surface only those ranked #1.

```sql
SELECT employee_id, first_name, department, salary, dept_rank
FROM (
    SELECT
        employee_id,
        first_name,
        department,
        salary,
        RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank
    FROM employees
) AS ranked
WHERE dept_rank = 1;
```

### 17.4 Supply Chain: Products Never Ordered

**Scenario:** A procurement analyst wants to find all products that have never appeared in any order.

```sql
SELECT product_id, product_name, price
FROM products p
WHERE NOT EXISTS (
    SELECT 1
    FROM   order_line_items oli
    WHERE  oli.product_id = p.product_id
);
```

## 18. Best Practices

1. **Choose the right clause for the right result type** — use scalar subqueries only in SELECT and comparison WHERE; use row subqueries with IN/ANY/ALL; use table subqueries in FROM and JOIN.

2. **Prefer non-correlated subqueries over correlated** when possible — non-correlated runs once; correlated runs per row, which is costly on large datasets.

3. **Always alias FROM clause subqueries** — all three major databases require it; omitting it causes an error.

4. **Use `SELECT 1` inside EXISTS** — you are testing existence, not retrieving data. Using `SELECT 1` signals intent and avoids unnecessary column reads.

5. **Prefer NOT EXISTS over NOT IN** when NULL values may exist in the subquery result — NOT IN with NULLs silently returns zero rows.

6. **Debug correlated subqueries by hardcoding one row's value** — since they can't run standalone, substitute a specific value from the main query temporarily to validate the logic.

7. **Replace deep nesting with CTEs** (Chapter 12) — three or more nested levels of subqueries become unreadable; CTEs name each step explicitly.

8. **Pre-aggregate before joining** — computing `GROUP BY` inside a subquery and then joining is generally faster than joining first and grouping after, because the join operates on fewer rows.

9. **Validate intermediate results by highlighting** — in SSMS, MySQL Workbench, and pgAdmin, selecting the inner query text and running only that selection lets you inspect the intermediate result without running the full query.

10. **Know what the optimizer does with your subquery** — use the Actual Execution Plan (Chapter 9) to confirm whether a subquery is being materialized, inlined, or treated as a correlated loop.

## 19. Summary Table

| Concept | Definition | Key Rule | Where Used |
|---|---|---|---|
| **Subquery** | A SELECT statement nested inside another query | Must be enclosed in parentheses `()` | FROM, SELECT, JOIN, WHERE |
| **Main / Outer query** | The enclosing query that uses the subquery's result | Sees the subquery result as an intermediate table or value | — |
| **Intermediate result** | Temporary output of a subquery held in cache | Local to the main query; destroyed after execution | Cache |
| **Scalar subquery** | Returns exactly 1 row × 1 column | Required in SELECT clause and comparison WHERE | SELECT, WHERE (=, <, >, etc.) |
| **Row subquery** | Returns N rows × 1 column | Used with IN, NOT IN, ANY, ALL, EXISTS | WHERE clause |
| **Table subquery** | Returns N rows × M columns | Must be aliased when used in FROM or JOIN | FROM, JOIN |
| **FROM clause subquery** | Subquery acting as a temporary table | Must have an alias in all three databases | FROM |
| **SELECT clause subquery** | Scalar subquery adding a computed column | Must return exactly one value | SELECT |
| **JOIN clause subquery** | Table subquery used as one join side | More efficient to pre-aggregate before joining | JOIN |
| **Comparison operator subquery** | Scalar subquery with =, <, >, etc. | Subquery must return one value | WHERE |
| **IN / NOT IN** | Row subquery for list membership test | NOT IN is unsafe when NULLs exist | WHERE |
| **ANY** | TRUE if condition holds for at least one subquery value | Paired with a comparison operator | WHERE |
| **ALL** | TRUE if condition holds for every subquery value | Paired with a comparison operator | WHERE |
| **EXISTS** | TRUE if the correlated subquery returns ≥ 1 row | Always correlated; use `SELECT 1` | WHERE |
| **NOT EXISTS** | TRUE if the correlated subquery returns 0 rows | Safer than NOT IN for NULL handling | WHERE |
| **Non-correlated subquery** | Independent of the main query; runs once | Can be highlighted and executed alone | Any |
| **Correlated subquery** | References outer query columns; runs per row | Cannot run standalone; slower on large sets | WHERE (EXISTS), SELECT |
| **`SELECT 1` in EXISTS** | Convention for existence-only testing | Does not retrieve data; faster than `SELECT *` | EXISTS subqueries |
| **NOT IN + NULL risk** | NOT IN returns empty if subquery has any NULL | Use NOT EXISTS instead | WHERE |
| **Debugging non-correlated** | Highlight inner SELECT, execute separately | Works in SSMS, MySQL Workbench, pgAdmin | Development |
| **Debugging correlated** | Hardcode one row's value to test logic | Substitute the outer column value manually | Development |
| **LATERAL / CROSS APPLY** | Correlated subquery in FROM clause | MSSQL: CROSS/OUTER APPLY; PG/MySQL 8+: LATERAL | FROM |
