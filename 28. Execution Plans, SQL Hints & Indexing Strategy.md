## 1. What Is an Execution Plan?

### 1.1 Definition

An **execution plan** (also called a query plan or query execution plan) is a detailed, step-by-step description of the exact operations the database engine performs to satisfy a SQL query. It shows:

- How data is **retrieved** from disk (scan type used)
- How tables are **joined** (join algorithm chosen)
- How data is **sorted or aggregated**
- Which **indexes** are used (or ignored)
- The **estimated cost** (CPU + I/O) of each operation

Think of it as a GPS route: before driving anywhere, the navigation app calculates the fastest path. The database does the same before touching a single row of data.

### 1.2 Why It Matters

Without the execution plan, performance tuning is guesswork. The execution plan answers:

- *Why is this query slow?*
- *Is my new index actually being used?*
- *Which step is consuming most of the resources?*
- *Should I add an index on this table or that column?*

## 2. How the Database Engine Processes a Query

When you submit a SQL query, the engine does **not** immediately fetch data. It goes through these stages:

```
SQL Query Submitted
       │
       ▼
┌─────────────────────────────┐
│  1. Parse & Validate SQL    │  — Syntax check, table/column existence
└─────────────────────────────┘
       │
       ▼
┌─────────────────────────────┐
│  2. Check Plan Cache        │  — Has a plan been cached for this query?
└─────────────────────────────┘
       │               │
  Cache HIT       Cache MISS
       │               │
       │               ▼
       │   ┌─────────────────────────────┐
       │   │  3. Generate Execution Plan │  — Optimizer builds a plan using
       │   │     (Query Optimizer)       │    table statistics, indexes, etc.
       │   └─────────────────────────────┘
       │               │
       └───────────────┘
                       │
                       ▼
┌─────────────────────────────┐
│  4. Execute the Plan        │  — Read tables, join, sort, aggregate
└─────────────────────────────┘
       │
       ▼
┌─────────────────────────────┐
│  5. Cache the Plan          │  — Store for reuse on identical queries
└─────────────────────────────┘
       │
       ▼
  Results returned to user
```

> **Key Insight:** A cached plan is reused on re-execution of the same query, skipping optimization entirely. This is faster but can occasionally cause issues if underlying data distribution has changed significantly.

## 3. Types of Execution Plans

SQL Server (MSSQL) provides three kinds of execution plan views. Understanding the difference is essential.

| Plan Type | When Generated | Executes Query? | Best Used For |
|---|---|---|---|
| **Estimated** | Before execution | ❌ No | Quick preview; optimizer's guess |
| **Actual** | After execution | ✅ Yes | Real performance diagnosis |
| **Live Query Statistics** | During execution | ✅ Yes (real-time) | Long-running query monitoring |

### 3.1 Estimated vs Actual

The estimated plan is the optimizer's prediction. The actual plan records what really happened — including real row counts versus estimated row counts.

**Why compare them?**

```
Estimated Rows: 100
Actual Rows:    60,000
```

A large discrepancy between estimated and actual row counts is a red flag indicating **stale statistics** or a poor cardinality estimate. This often leads to a suboptimal execution plan.

## 5. Reading an Execution Plan — Step by Step

### 5.1 Direction of Reading

**Execution plans are read from right to left, top to bottom.** The rightmost node is the first operation performed; the leftmost node is the final output returned to the client.

```
[Table Scan] ──► [Sort] ──► [SELECT / Output]
      ↑ starts here           ↑ ends here
```

Each arrow between operators represents data flowing from one step to the next. The **thickness** of an arrow (in SSMS) indicates the relative number of rows flowing through — a thick arrow means many rows, a thin arrow means few.

### 5.2 Cost Percentage

Each node in the plan displays a **percentage of the total query cost**. This immediately shows you where the bottleneck is. In a healthy query, no single node should dominate with 80–90% of total cost unless it's intentional (e.g., a full scan of a large fact table for a full aggregation).

### 5.3 Node Properties to Check

Right-click any node → **Properties** to see:

| Property | What to Look For |
|---|---|
| **Actual Number of Rows Read** | Should be close to rows returned |
| **Estimated Number of Rows** | Large gap with actual = stale stats |
| **CPU Cost** | High = compute-heavy step |
| **I/O Cost** | High = too much disk reading |
| **Logical Operator** | Table Scan, Index Seek, Hash Match, etc. |
| **Object (Index Used)** | Name of the index the engine chose |
| **Storage** | Row Store vs Column Store |

## 6. Scan Types: Table Scan vs Index Scan vs Index Seek

These are the three most important operators in any execution plan relating to data retrieval.

### 6.1 Definitions

| Operator | Description | When It Appears |
|---|---|---|
| **Table Scan** | Reads every row in the table (full scan of a Heap) | No usable index exists |
| **Index Scan** | Reads all or part of an index's B-Tree pages | Index exists but can't seek; or full range needed |
| **Index Seek** | Navigates directly to matching rows via the B-Tree | Index exists and a selective WHERE/JOIN condition is present |

### 6.2 Performance Tier

```
Worst                                                    Best
  │                                                        │
  ▼                                                        ▼
Table Scan ──────────── Index Scan ──────────── Index Seek
(reads all rows)     (reads index pages)    (finds exact rows)
```

### 6.3 Example — Causing Each Scan Type

```sql
-- TABLE SCAN: no index, no WHERE filter
SELECT * FROM fact_reseller_sales_heap;
-- Execution plan: Table Scan (reads all 60,000 rows)

-- INDEX SCAN: clustered index exists, but no selective filter
SELECT * FROM fact_reseller_sales;
-- Execution plan: Clustered Index Scan (reads all 60,000 rows via index)

-- INDEX SEEK: selective filter, nonclustered index on carrier_tracking_number
SELECT * FROM fact_reseller_sales
WHERE carrier_tracking_number = 'YG8-2654';
-- Execution plan: Index Seek (reads only 12 matching rows!)
```

### 6.4 Output Comparison

| Query | Rows Read | Rows Returned | Operator |
|---|---|---|---|
| Full heap scan | 60,000 | 60,000 | Table Scan |
| Full clustered scan | 60,000 | 60,000 | Clustered Index Scan |
| Seek on tracking number | **12** | 12 | Index Seek ✅ |

## 7. Heap vs Clustered Index — Execution Plan Comparison

### 7.1 What Is a Heap?

A **Heap** is a table with no clustered index. Data is stored in no particular order — rows are inserted wherever space is available. SQL Server identifies it by the absence of a clustered index entry.

### 7.2 What Is a Clustered Index?

A **Clustered Index** physically sorts and stores the data rows in the table according to the indexed key (usually the primary key). The leaf level of the B-Tree **is** the actual data.

### 7.3 Creating Both Structures

```sql
-- Clustered index is created automatically with PRIMARY KEY
CREATE TABLE fact_reseller_sales (
    sales_order_number VARCHAR(20) NOT NULL,
    ...
    CONSTRAINT pk_reseller_sales PRIMARY KEY (sales_order_number)
);

-- Heap: no primary key, no index
CREATE TABLE fact_reseller_sales_heap (
    sales_order_number VARCHAR(20),
    ...
    -- No constraint = Heap
);
```

### 7.4 ORDER BY Comparison

```sql
-- Query on Heap
SELECT * FROM fact_reseller_sales_heap
ORDER BY sales_order_number;
```

**Execution Plan (Heap):**
```
[Table Scan] ──► [Sort] ──► [SELECT]
    Cost: 0.07       Cost: 0.15
```

```sql
-- Query on Clustered Index
SELECT * FROM fact_reseller_sales
ORDER BY sales_order_number;
```

**Execution Plan (Clustered Index):**
```
[Clustered Index Scan] ──► [SELECT]
    Cost: 0.07
```

> **Performance Note:** The clustered index eliminates the Sort operator entirely because the data is already physically ordered by the primary key. This removes both the CPU cost and I/O cost of sorting 60,000 rows in memory.

| Metric | Heap | Clustered Index |
|---|---|---|
| Operators | Table Scan + Sort | Clustered Index Scan |
| CPU Cost | ~0.07 + 0.15 | ~0.07 |
| Sort required? | ✅ Yes | ❌ No |
| Relative speed | Slower | Faster ✅ |

## 8. Nonclustered Indexes in the Execution Plan

### 8.1 Definition

A **nonclustered index** is a separate B-Tree structure that stores a copy of one or more columns alongside a pointer (the row locator) back to the full row in the base table. Unlike clustered indexes, a table can have many nonclustered indexes.

### 8.2 Creating a Nonclustered Index

```sql
-- Create a nonclustered index on carrier_tracking_number
CREATE NONCLUSTERED INDEX idx_fact_reseller_carrier
ON fact_reseller_sales (carrier_tracking_number);
```

### 8.3 Query and Execution Plan

```sql
SELECT *
FROM fact_reseller_sales
WHERE carrier_tracking_number = 'YG8-2654';
```

**Before index — Heap:**
```
[Table Scan: 60,000 rows read] ──► [SELECT]
```

**After nonclustered index — Clustered Table:**
```
[Index Seek: 12 rows] ──► [Key Lookup: 12 rows] ──► [Nested Loop] ──► [SELECT]
```

> **Index Seek** is the best signal you can see in an execution plan. It tells you the optimizer found your index and used it to navigate directly to the right rows.

### 8.4 Verifying Your Index Was Used

In the Properties panel of the Index Seek node, look at the **Object** field:

```
Object: [AdventureWorksDW].[dbo].[fact_reseller_sales].[idx_fact_reseller_carrier]
```

If this matches the index you created, your indexing decision was correct. If it points to a different index, reconsider your index design.

## 9. Key Lookup and Covering Indexes

### 9.1 What Is a Key Lookup?

When a nonclustered index is used (Index Seek), it only contains the indexed column(s) plus the row locator. If your query requests additional columns not in the index, the engine must do a **Key Lookup** — going back to the clustered index (or heap) to fetch those missing columns.

```
[Index Seek]        ← finds the rows by carrier_tracking_number
[Key Lookup]        ← fetches all other columns (product_key, sales_amount, etc.)
[Nested Loop Join]  ← merges the two data streams
```

### 9.2 Why Key Lookup Can Be Expensive

For a small result set (12 rows), a Key Lookup is cheap. But for thousands of rows, each Key Lookup is an individual random I/O operation — this can be catastrophic for performance.

### 9.3 Eliminating Key Lookup with a Covering Index

A **covering index** includes all columns the query needs, so no lookup is necessary.

```sql
-- Covering index: includes carrier_tracking_number + all SELECT columns
CREATE NONCLUSTERED INDEX idx_fact_reseller_carrier_covering
ON fact_reseller_sales (carrier_tracking_number)
INCLUDE (product_key, sales_amount, order_date);
```

```sql
-- Now this query uses ONLY the index — no Key Lookup!
SELECT carrier_tracking_number, product_key, sales_amount
FROM fact_reseller_sales
WHERE carrier_tracking_number = 'YG8-2654';
```

**Execution Plan (with covering index):**
```
[Index Seek: 12 rows] ──► [SELECT]
    (No Key Lookup, No Nested Loop!)
```

## 10. Rowstore vs Columnstore — Analytical Workloads

### 10.1 What Is a Columnstore Index?

A **Columnstore Index** stores data column-by-column instead of row-by-row. This is dramatically more efficient for queries that aggregate large volumes of data across a few columns, because only the relevant columns are read from disk.

### 10.2 Creating a Clustered Columnstore Index

```sql
-- Convert the heap table to a Clustered Columnstore Index
CREATE CLUSTERED COLUMNSTORE INDEX idx_fact_reseller_heap_cs
ON fact_reseller_sales_heap;
-- No columns specified — the entire table becomes the columnstore
```

### 10.3 Execution Plan Comparison — Aggregation Query

```sql
SELECT
    p.english_name      AS product_name,
    SUM(f.sales_amount) AS total_sales
FROM fact_reseller_sales_heap f
JOIN dim_product p ON f.product_key = p.product_key
GROUP BY p.english_name;
```

**Execution Plan — Rowstore (Clustered Index):**
```
[Clustered Index Scan: fact table 71% cost]
    ──► [Hash Match Aggregate]
    ──► [Sort]
    ──► [Merge Join]
    ──► [Clustered Index Scan: dim table 29% cost]
    ──► [SELECT]
```

**Execution Plan — Columnstore:**
```
[Columnstore Index Scan: fact table 6% cost]  ← 71% → 6%!
    ──► [Hash Match Aggregate]
    ──► [Merge Join]
    ──► [Clustered Index Scan: dim table]
    ──► [SELECT]
```

| Metric | Rowstore | Columnstore |
|---|---|---|
| Fact table scan cost | ~71% | ~6% ✅ |
| CPU Cost | Higher | Much lower |
| I/O Cost | Higher | Much lower |
| Physical Operator | Clustered Index Scan | Columnstore Index Scan |
| Best For | OLTP (many small reads/writes) | OLAP (large aggregation queries) |

> **Performance Note:** For fact tables in a data warehouse, the Columnstore Index can reduce query cost by 10–20× on aggregation-heavy workloads. Always apply it to large fact tables in analytical systems.

### 10.4 Saving and Comparing Execution Plans

In SSMS:
1. Right-click on Execution Plan tab → **Save Execution Plan As** → save as `plan_rowstore.sqlplan`
2. Run the second query, right-click → **Compare Show Plan** → select the saved file
3. SSMS shows both plans side-by-side with cost comparison

## 11. Join Algorithms in the Execution Plan

The optimizer chooses between three join algorithms automatically. Understanding them helps you interpret execution plans and apply hints when needed.

| Algorithm | Symbol | Best For | Drawback |
|---|---|---|---|
| **Nested Loop** | ⟲ | Small outer + indexed inner table | Slow for large datasets |
| **Merge Join** | ⟳ | Both inputs pre-sorted, large data | Requires sorted inputs |
| **Hash Match** | # | Large unsorted data, aggregations | High memory usage |

```sql
-- Example showing all three in context:

-- Nested Loop: ideal when driving table is small
SELECT * FROM dim_product p
JOIN fact_reseller_sales f ON p.product_key = f.product_key
WHERE p.category = 'Bikes';          -- only 2 rows in dim → Nested Loop ✅

-- Hash Match: used when joining two large unsorted sets
SELECT * FROM fact_reseller_sales_heap f
JOIN dim_product p ON f.product_key = p.product_key;  -- 60k rows → Hash Match ✅

-- Merge Join: used when both sides are sorted (clustered indexes)
SELECT * FROM fact_reseller_sales f
JOIN dim_product p ON f.product_key = p.product_key
ORDER BY f.product_key;              -- Both sorted on product_key → Merge Join ✅
```

> **Warning:** Seeing a Nested Loop on a join of two large tables (tens of thousands of rows each) is a sign of trouble. It usually means the optimizer lacks statistics or proper indexes.

## 12. SQL Hints — Overriding the Query Optimizer

### 12.1 What Are SQL Hints?

**SQL Hints** are explicit directives embedded in your query that instruct the optimizer to use a specific execution strategy, bypassing its automatic decision-making. They are your manual override switch.

> Use hints **sparingly and temporarily**. They are not a permanent performance fix — they are emergency levers.

### 12.2 Forcing a Join Algorithm

```sql
-- Force the optimizer to use a Hash Join regardless of its own recommendation
SELECT o.order_id, c.customer_name
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
OPTION (HASH JOIN);                  -- Query hint at the end
```

Other join options:

```sql
OPTION (LOOP JOIN)    -- Force Nested Loop join
OPTION (MERGE JOIN)   -- Force Merge join
OPTION (HASH JOIN)    -- Force Hash join
```

### 12.3 Forcing Index Seek vs Scan (Table Hint)

```sql
-- Force SQL Server to use an Index Seek on the customers table
SELECT o.order_id, c.customer_name
FROM orders o
JOIN customers c WITH (FORCESEEK) ON o.customer_id = c.customer_id;
-- WITH (FORCESEEK) is a table hint — placed directly after the table name
```

```sql
-- Force an Index Scan instead of Seek
SELECT * FROM customers WITH (FORCESCAN);
```

### 12.4 Forcing a Specific Index

```sql
-- Force use of a specific index by name
SELECT o.order_id, c.customer_name
FROM orders o
JOIN customers c WITH (INDEX(idx_customers_pk)) ON o.customer_id = c.customer_id;
```

```sql
-- Check the execution plan's Object field to verify:
-- Object: [dbo].[customers].[idx_customers_pk]
```

### 12.5 Combining Multiple Hints

```sql
-- Force both a specific index and a join algorithm
SELECT o.order_id, c.customer_name
FROM orders o
JOIN customers c WITH (INDEX(idx_customers_pk))
    ON o.customer_id = c.customer_id
OPTION (LOOP JOIN);
```

> **Important:** Some combinations of table hints and query hints conflict. SQL Server will raise an error (as demonstrated in the source) if the hints are contradictory — always test each combination.

### 12.6 When to Use (and Not Use) Hints

| Scenario | Use Hints? | Reason |
|---|---|---|
| Emergency: slow query in production | ✅ Temporarily | Buy time while investigating root cause |
| Statistics are stale and not yet refreshed | ✅ Temporarily | Bypass bad plan until stats are updated |
| Development environment optimization | ⚠️ With caution | Dev data volume differs from production |
| Permanent fix for a recurring slow query | ❌ Never | Fix the root cause (index, statistics) instead |
| Testing index choices | ✅ Yes | Great for experimentation |

## 13. Cross-Database Comparison: MySQL, MSSQL, PostgreSQL

### 13.1 Viewing Execution Plans

| Feature | MySQL | MSSQL (SQL Server) | PostgreSQL |
|---|---|---|---|
| Estimated plan | `EXPLAIN SELECT ...` | Enable "Display Estimated Execution Plan" in SSMS | `EXPLAIN SELECT ...` |
| Actual plan | `EXPLAIN ANALYZE SELECT ...` | Enable "Include Actual Execution Plan" in SSMS | `EXPLAIN ANALYZE SELECT ...` |
| Format options | `FORMAT=JSON`, `FORMAT=TREE` | Graphical, XML, Text in SSMS | Text (default), JSON, XML, YAML |
| Live statistics | ❌ Not native | ✅ Live Query Statistics | ❌ Not native |

**MySQL:**
```sql
EXPLAIN SELECT * FROM fact_reseller_sales WHERE carrier_tracking_number = 'YG8-2654';
-- Output: type=ref (index scan), key=idx_carrier, rows=12

EXPLAIN ANALYZE SELECT * FROM fact_reseller_sales WHERE carrier_tracking_number = 'YG8-2654';
-- Output includes actual timing and loop counts
```

**MSSQL:**
```sql
-- Via SSMS GUI buttons or T-SQL:
SET STATISTICS IO ON;    -- Shows logical reads per table
SET STATISTICS TIME ON;  -- Shows CPU and elapsed time
SELECT * FROM fact_reseller_sales WHERE carrier_tracking_number = 'YG8-2654';
SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
```

**PostgreSQL:**
```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM fact_reseller_sales WHERE carrier_tracking_number = 'YG8-2654';
-- Output: Index Scan using idx_carrier, actual rows=12, buffers: shared hit=5
```

### 13.2 Scan Type Terminology

| Concept | MySQL Term | MSSQL Term | PostgreSQL Term |
|---|---|---|---|
| Full table read | `ALL` (type column) | Table Scan | Seq Scan |
| Index full read | `index` | Index Scan | Index Scan |
| Direct B-Tree seek | `ref`, `range`, `const` | Index Seek | Index Scan (with condition) |
| Heap fetch after index | N/A (clustered by default) | Key Lookup | Bitmap Heap Scan |

> **Note:** MySQL uses InnoDB, which stores all tables as clustered indexes (by primary key) by default. There is no "Heap" structure in MySQL. PostgreSQL stores data as heaps by default and uses separate B-Tree index structures.

### 13.3 Columnstore Index Support

| Feature | MySQL | MSSQL | PostgreSQL |
|---|---|---|---|
| Columnstore Index | ❌ Not supported | ✅ Clustered & Nonclustered | ❌ Not built-in (use extensions like `cstore_fdw` or Citus) |
| Equivalent solution | Use partitioning + covering indexes | Clustered Columnstore Index | `pg_partman` + BRIN indexes for time-series |

### 13.4 Forcing Join Algorithms (Hints)

**MySQL:**
```sql
SELECT * FROM orders o
STRAIGHT_JOIN customers c ON o.customer_id = c.customer_id;
-- STRAIGHT_JOIN forces left-to-right join order

SELECT * FROM orders o
JOIN customers c USE INDEX (PRIMARY) ON o.customer_id = c.customer_id;
-- USE INDEX hints a specific index
```

**MSSQL:**
```sql
SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
OPTION (HASH JOIN);

-- Force specific index:
SELECT * FROM customers WITH (INDEX(idx_customers_city));
```

**PostgreSQL:**
```sql
-- PostgreSQL uses GUC parameters to disable join types:
SET enable_hashjoin = OFF;
SET enable_nestloop = OFF;
SET enable_mergejoin = ON;

SELECT * FROM orders o JOIN customers c ON o.customer_id = c.customer_id;
-- Now only Merge Join is available

-- Reset after testing:
RESET enable_hashjoin;
RESET enable_nestloop;
```

### 13.5 Refreshing Statistics

| Database | Command |
|---|---|
| MySQL | `ANALYZE TABLE fact_reseller_sales;` |
| MSSQL | `UPDATE STATISTICS fact_reseller_sales;` or `EXEC sp_updatestats;` |
| PostgreSQL | `ANALYZE fact_reseller_sales;` or `VACUUM ANALYZE fact_reseller_sales;` |

### 13.6 Checking Missing Indexes

| Database | Method |
|---|---|
| MySQL | `EXPLAIN` output shows `possible_keys` vs `key` — gap = missing index |
| MSSQL | `sys.dm_db_missing_index_details` DMV |
| PostgreSQL | `pg_stat_user_tables` (seq_scan count high = candidate for index) |

## 14. Indexing Strategy — A Four-Phase Framework

> **The Golden Rule: Avoid Over-Indexing.** Every index added to a table slows down `INSERT`, `UPDATE`, and `DELETE` operations because the index must be updated, sorted, and rearranged with every data change. Too many indexes also confuse the optimizer, leading to bad execution plans. **Less is more.**

### 14.1 Phase 1 — Initial Indexing Strategy

**Step 1: Define the System Type**

Before creating any index, identify what type of system you are building.

| System Type | Full Name | Read/Write Pattern | Primary Goal |
|---|---|---|---|
| **OLAP** | Online Analytical Processing | Heavy reads, batch writes | Optimize read performance |
| **OLTP** | Online Transactional Processing | Constant reads AND writes | Optimize write performance |

**Step 2: Apply System-Appropriate Defaults**

For **OLAP (Data Warehouse):**
```sql
-- All large fact tables → Clustered Columnstore Index
CREATE CLUSTERED COLUMNSTORE INDEX idx_cs_fact_sales
ON fact_reseller_sales_heap;

-- All dimension tables → Clustered Index on Primary Key (default via PK)
-- Already handled by PRIMARY KEY constraint
```

For **OLTP (Transactional System):**
```sql
-- Every table → Clustered Index on Primary Key
-- Apply sparingly beyond this — each extra index slows writes
CREATE TABLE orders (
    order_id    INT PRIMARY KEY,  -- Clustered Index
    customer_id INT NOT NULL,
    order_date  DATE
);

-- Add nonclustered indexes ONLY for proven high-frequency lookups
-- Be conservative — OLTP writes must stay fast
```

### 14.2 Phase 2 — Usage Pattern Indexing

**Step 1: Identify Frequently Used Tables and Columns**

Analyze your query codebase to find patterns. Use AI assistance with this prompt template:

```
Analyze the following SQL queries and generate a report with:
- Total usage count per table
- Per-column usage breakdown (filtering, joining, grouping, aggregating)
- Columns appearing in WHERE, JOIN ON, GROUP BY, ORDER BY clauses

[Paste your SQL scripts here]
```

**Step 2: Classify Each High-Use Column**

| Column Usage | Recommended Index Type |
|---|---|
| Primary key | Clustered Index (default via PK) |
| JOIN condition (foreign key) | Nonclustered Index |
| WHERE filter (high selectivity) | Nonclustered Index |
| WHERE filter (low selectivity, e.g. boolean) | Skip — not selective enough |
| ORDER BY / GROUP BY | Nonclustered Index (consider INCLUDE columns) |
| Large fact table, many aggregations | Clustered Columnstore Index |
| Subset of rows always accessed (e.g. active records only) | Filtered Index |
| Unique constraint column | Unique Nonclustered Index |

**Step 3: Build and Test**

```sql
-- After identifying order_date as a frequent filter column:
CREATE NONCLUSTERED INDEX idx_fact_sales_order_date
ON fact_reseller_sales (order_date)
INCLUDE (product_key, sales_amount);  -- covering columns

-- Verify: run your query and check execution plan
-- Confirm: Object field shows idx_fact_sales_order_date
-- ✅ Index Seek appearing = good decision
-- ❌ Still showing Index Scan or Table Scan = wrong index, reconsider
```

### 14.3 Phase 3 — Scenario-Based Indexing

This phase targets specific slow queries identified from user reports or query log analysis.

**Process:**

```
1. Identify slow query
       │
       ▼
2. Run with Actual Execution Plan
       │
       ▼
3. Find the expensive operator
   (highest cost%, Table Scan, Nested Loop on large data)
       │
       ▼
4. Choose the right index type
       │
       ▼
5. Create the index
       │
       ▼
6. Re-run with Actual Execution Plan
       │
       ▼
7. Verify index is used (Object field)
   Compare cost% before and after
       │
       ├── ✅ Improved? → Done. Move to next slow query.
       └── ❌ No improvement? → Wrong index, investigate further.
```

**Example — Optimizing a Slow Aggregation Query:**

```sql
-- Slow query (identified from logs)
SELECT
    p.english_name,
    SUM(f.sales_amount) AS total_sales,
    COUNT(*)            AS order_count
FROM fact_reseller_sales f
JOIN dim_product p ON f.product_key = p.product_key
WHERE f.order_date BETWEEN '2023-01-01' AND '2023-12-31'
GROUP BY p.english_name;
```

```
Before index:
  Execution Plan: Table Scan (fact) → Hash Match → Sort → ...
  Fact table cost: 78% of total

After Columnstore Index on fact table:
  Execution Plan: Columnstore Index Scan → Hash Match → ...
  Fact table cost: 8% of total ✅
```

### 14.4 Phase 4 — Monitoring and Maintenance

**a) Monitor Index Usage**

```sql
-- MSSQL: Find unused indexes
SELECT
    OBJECT_NAME(i.object_id)    AS table_name,
    i.name                      AS index_name,
    i.type_desc                 AS index_type,
    u.user_seeks,
    u.user_scans,
    u.user_lookups,
    u.user_updates,
    u.last_user_seek,
    u.last_user_scan
FROM sys.indexes i
LEFT JOIN sys.dm_db_index_usage_stats u
    ON i.object_id = u.object_id
    AND i.index_id = u.index_id
    AND u.database_id = DB_ID()
WHERE i.type_desc <> 'HEAP'
  AND ISNULL(u.user_seeks, 0) + ISNULL(u.user_scans, 0) + ISNULL(u.user_lookups, 0) = 0
ORDER BY i.name;
-- Indexes with 0 reads but positive user_updates = dead weight, consider dropping
```

**b) Monitor Missing Indexes (MSSQL)**

```sql
-- Find indexes the optimizer wished existed
SELECT
    mid.statement                               AS table_name,
    migs.avg_total_user_cost * migs.avg_user_impact * (migs.user_seeks + migs.user_scans)
                                                AS improvement_measure,
    'CREATE INDEX [idx_missing_' + CONVERT(VARCHAR, mig.index_group_handle) + ']'
    + ' ON ' + mid.statement
    + ' (' + ISNULL(mid.equality_columns, '')
    + CASE WHEN mid.equality_columns IS NOT NULL AND mid.inequality_columns IS NOT NULL
           THEN ',' ELSE '' END
    + ISNULL(mid.inequality_columns, '') + ')'
    + ISNULL(' INCLUDE (' + mid.included_columns + ')', '') AS create_index_statement
FROM sys.dm_db_missing_index_groups mig
JOIN sys.dm_db_missing_index_group_stats migs ON migs.group_handle = mig.index_group_handle
JOIN sys.dm_db_missing_index_details mid ON mig.index_handle = mid.index_handle
ORDER BY improvement_measure DESC;
```

**c) Monitor Fragmentation**

```sql
-- MSSQL: Check index fragmentation
SELECT
    OBJECT_NAME(ips.object_id)  AS table_name,
    i.name                      AS index_name,
    ips.avg_fragmentation_in_percent,
    CASE
        WHEN ips.avg_fragmentation_in_percent < 10  THEN 'OK — No action needed'
        WHEN ips.avg_fragmentation_in_percent < 30  THEN 'REORGANIZE recommended'
        ELSE                                              'REBUILD required'
    END AS recommended_action
FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, 'LIMITED') ips
JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id
WHERE ips.avg_fragmentation_in_percent > 5
ORDER BY ips.avg_fragmentation_in_percent DESC;

-- Act on the results:
ALTER INDEX idx_fact_reseller_carrier ON fact_reseller_sales REORGANIZE;  -- 10-30%
ALTER INDEX idx_fact_reseller_carrier ON fact_reseller_sales REBUILD;     -- >30%
```

**d) Update Statistics Schedule**

```sql
-- MSSQL: Manual update (run weekly via SQL Agent job)
UPDATE STATISTICS fact_reseller_sales;
UPDATE STATISTICS dim_product;

-- Or update all tables at once:
EXEC sp_updatestats;

-- MySQL equivalent:
ANALYZE TABLE fact_reseller_sales;

-- PostgreSQL equivalent:
VACUUM ANALYZE fact_reseller_sales;
```

| Fragmentation | Action | MSSQL Command |
|---|---|---|
| 0 – 10% | None needed | — |
| 10 – 30% | Reorganize | `ALTER INDEX ... REORGANIZE` |
| > 30% | Full rebuild | `ALTER INDEX ... REBUILD` |

## 15. Common Mistakes

### 15.1 Over-Indexing

**Mistake:** Creating an index for every column or every query, thinking "more indexes = more speed."

**Reality:** Each index must be maintained on every write. 10 indexes on a heavily-written OLTP table means 10× the write overhead.

```sql
-- ❌ Bad: Index on a low-selectivity column (only 2 distinct values)
CREATE INDEX idx_is_active ON customers (is_active);
-- The optimizer will likely ignore this and do a full scan anyway

-- ✅ Better: Composite index with a selective lead column
CREATE INDEX idx_customer_active_date ON customers (signup_date, is_active);
```

### 15.2 Ignoring the Key Lookup Cost

**Mistake:** Creating an index without including the columns your SELECT needs, causing expensive Key Lookups.

```sql
-- ❌ Bad: Index only on the filter column, but SELECT needs more columns
CREATE INDEX idx_carrier ON fact_reseller_sales (carrier_tracking_number);
-- Causes Key Lookup for every other column in SELECT *

-- ✅ Better: Covering index
CREATE INDEX idx_carrier_covering ON fact_reseller_sales (carrier_tracking_number)
INCLUDE (product_key, sales_amount, order_date);
```

### 15.3 Using SQL Hints as Permanent Fixes

**Mistake:** Deploying `OPTION (HASH JOIN)` or `WITH (FORCESEEK)` hints in production and forgetting about them.

**Reality:** Hints bypass the optimizer. If your data distribution changes, the hint that once helped can now hurt — and you'll never know because the optimizer is not allowed to adapt.

### 15.4 Assuming Execution Plans Transfer Between Environments

**Mistake:** Tuning a query with hints in the development database (1,000 rows) and assuming it works the same in production (10 million rows).

**Rule:** Always test hints in production-scale environments before deploying.

### 15.5 Not Verifying Index Usage After Creation

**Mistake:** Creating an index and assuming the optimizer will use it.

**Rule:** Always run the query with the Actual Execution Plan enabled after creating an index. Check the **Object** field in the Properties pane of the scan/seek operator to verify your index name appears.

### 15.6 Ignoring Stale Statistics

**Mistake:** Trusting the optimizer's decisions without checking when statistics were last updated.

```sql
-- MSSQL: Check statistics last updated date
SELECT
    OBJECT_NAME(s.object_id)    AS table_name,
    s.name                      AS stat_name,
    STATS_DATE(s.object_id, s.stats_id) AS last_updated
FROM sys.stats s
WHERE OBJECT_NAME(s.object_id) = 'fact_reseller_sales'
ORDER BY last_updated;
```

## 16. Real-World Analytics Use Cases

### 16.1 OLAP: Data Warehouse Fact Table Aggregation

**Scenario:** A Power BI report aggregates 5 years of sales data across multiple dimensions. Report refresh takes 8 minutes.

```sql
-- Slow version: Rowstore clustered index on fact table
SELECT
    YEAR(f.order_date)  AS sales_year,
    p.category          AS product_category,
    SUM(f.sales_amount) AS total_revenue,
    COUNT(*)            AS transaction_count
FROM fact_reseller_sales f              -- 60M rows
JOIN dim_product p ON f.product_key = p.product_key
GROUP BY YEAR(f.order_date), p.category
ORDER BY sales_year, product_category;
-- Execution plan: Clustered Index Scan (60M rows) → Hash Match → ... → 8 min

-- Fix: Convert fact table to Columnstore
CREATE CLUSTERED COLUMNSTORE INDEX idx_cs_fact_reseller
ON fact_reseller_sales;
-- Re-run: Columnstore Index Scan → Hash Match → ... → 45 seconds ✅
```

### 16.2 OLTP: E-Commerce Order Lookup

**Scenario:** Customer service searches for all orders by a customer's email address. Response time is 4 seconds — unacceptable in a live chat context.

```sql
-- Find orders by customer email
SELECT o.order_id, o.order_date, o.total_amount, o.status
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE c.email = 'jane.doe@example.com'
ORDER BY o.order_date DESC;
-- Execution plan: Table Scan on customers (no index on email)

-- Fix: Add nonclustered index on email
CREATE NONCLUSTERED INDEX idx_customers_email
ON customers (email)
INCLUDE (customer_id);  -- covers the JOIN condition

-- Re-run: Index Seek on customers → Nested Loop → response < 50ms ✅
```

### 16.3 ETL Performance: Accelerating Nightly Load

**Scenario:** The nightly ETL process that loads the staging table into the fact table takes 3 hours.

```sql
-- Root cause: fact table has 8 nonclustered indexes
-- Every INSERT must update all 8 indexes

-- Fix: Disable non-essential indexes before bulk load, rebuild after
ALTER INDEX idx_fact_carrier ON fact_reseller_sales DISABLE;
ALTER INDEX idx_fact_date ON fact_reseller_sales DISABLE;

-- Run bulk INSERT
INSERT INTO fact_reseller_sales
SELECT * FROM staging_reseller_sales;

-- Rebuild indexes after load
ALTER INDEX idx_fact_carrier ON fact_reseller_sales REBUILD;
ALTER INDEX idx_fact_date ON fact_reseller_sales REBUILD;
-- ETL time: 3 hours → 55 minutes ✅
```

## 17. Best Practices

1. **Always check the Actual Execution Plan** (not Estimated) when diagnosing slow queries.

2. **Read plans right to left** — the rightmost operator is the first to execute.

3. **Index Seek is the goal** — if you see Table Scan or Index Scan on a filter query, investigate.

4. **Verify every new index** — run the target query after creation and confirm the index name appears in the execution plan's Object field.

5. **Use covering indexes** to eliminate Key Lookup operators when your queries return multiple columns.

6. **Apply Columnstore Indexes to large fact tables** in analytical (OLAP) systems.

7. **Avoid SQL Hints as permanent fixes** — they are workarounds, not solutions.

8. **Test hints in the target environment** — a hint optimal for 10K rows can be catastrophic for 10M rows.

9. **Schedule weekly statistics updates** — stale statistics cause the optimizer to make wrong decisions.

10. **Monitor and drop unused indexes** — they slow writes and confuse the optimizer for free.

11. **Commit the Golden Rule to your team guidelines:** *Avoid over-indexing. Less is more.*

12. **Rebuild fragmented indexes** — indexes above 30% fragmentation degrade both read and write performance.

## 18. Summary Table

| Topic | Key Concept | Best Practice |
|---|---|---|
| **Execution Plan** | Shows step-by-step how SQL processes a query | Always use Actual (not Estimated) for diagnosis |
| **Plan Cache** | Stores built plans for reuse | Benefit: faster repeated queries. Risk: stale plan on changed data |
| **Table Scan** | Reads every row in a Heap | Worst outcome — indicates missing index |
| **Index Scan** | Reads all or part of a B-Tree index | Acceptable for full-range reads; investigate for selective queries |
| **Index Seek** | Navigates directly to matching rows | Best outcome — confirms index is being used effectively |
| **Key Lookup** | Fetches missing columns after Index Seek | Eliminate with covering indexes (INCLUDE clause) |
| **Heap** | Table with no clustered index | Avoid for frequently queried tables; add at minimum a PK |
| **Clustered Index** | Data physically ordered by key | One per table; use primary key; eliminates sort operations |
| **Nonclustered Index** | Separate B-Tree pointing to base rows | Multiple allowed; use for JOIN/WHERE/ORDER columns |
| **Columnstore Index** | Column-oriented storage for aggregation | Apply to large fact tables in OLAP; massive aggregation speedup |
| **Filtered Index** | Index on a subset of rows | Use when queries always target a known subset (e.g. active records) |
| **Nested Loop Join** | Row-by-row join; great for small sets | Bad for large tables — look for this as a warning sign |
| **Merge Join** | Efficient join of two pre-sorted sets | Good when both sides have clustered indexes on the join key |
| **Hash Match** | Build hash table then probe; handles large data | Default for large unsorted joins and aggregations |
| **SQL Hints** | Manual override of optimizer decisions | Use only as temporary emergency fix; always test in target environment |
| **Stale Statistics** | Old row distribution data misleads optimizer | Refresh weekly; watch for Estimated vs Actual row count divergence |
| **Fragmentation** | Out-of-order pages reduce I/O efficiency | Reorganize at 10–30%; Rebuild above 30% |
| **Over-Indexing** | Too many indexes slows writes and confuses optimizer | Golden Rule: avoid it — less is more |
| **OLAP Strategy** | Analytics system; heavy reads, batch writes | Columnstore on facts; Clustered on dimension PKs |
| **OLTP Strategy** | Transactional system; constant reads and writes | Clustered PK + minimal selective nonclustered indexes |
| **Phase 1** | Define system objectives and index defaults | Know your system type (OLAP vs OLTP) before creating any index |
| **Phase 2** | Identify frequently used tables and columns | Use query analysis and AI tooling; choose right index type per usage |
| **Phase 3** | Optimize specific slow queries | Execution plan → identify bottleneck → add index → verify and compare |
| **Phase 4** | Ongoing monitoring and maintenance | Monitor usage, missing indexes, fragmentation, and statistics regularly |
