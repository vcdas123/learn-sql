# Window Functions (Analytical Functions) in SQL

---

# 1. Introduction

Window functions (also called analytical functions) are advanced SQL features that allow you to perform calculations across a subset of rows **without collapsing the result set**.

Unlike `GROUP BY`, window functions:

- Do NOT reduce the number of rows
- Preserve row-level detail
- Enable advanced analytics (ranking, running totals, comparisons)

They are essential for:

- Data analytics
- Reporting dashboards
- Time-series analysis
- Ranking & segmentation

This chapter provides a complete, structured, and practical understanding of SQL window functions.

---

# 2. Window Functions vs GROUP BY

## 2.1 GROUP BY Behavior

`GROUP BY` aggregates rows and reduces granularity.

### Example Dataset

```sql
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    ProductID INT,
    OrderDate DATE,
    OrderStatus VARCHAR(20),
    Sales INT
);
```

```sql
INSERT INTO Orders VALUES
(1, 101, '2024-01-01', 'Delivered', 10),
(2, 101, '2024-01-02', 'Shipped', 30),
(3, 102, '2024-01-03', 'Delivered', 5),
(4, 102, '2024-01-04', 'Shipped', 20);
```

### Using GROUP BY

```sql
SELECT ProductID, SUM(Sales) AS TotalSales
FROM Orders
GROUP BY ProductID;
```

### Output

| ProductID | TotalSales |
| --------- | ---------- |
| 101       | 40         |
| 102       | 25         |

Notice:

- Input rows: 4
- Output rows: 2
- Row-level detail is lost

---

## 2.2 Window Function Behavior

```sql
SELECT
    OrderID,
    ProductID,
    Sales,
    SUM(Sales) OVER (PARTITION BY ProductID) AS TotalSalesByProduct
FROM Orders;
```

### Output

| OrderID | ProductID | Sales | TotalSalesByProduct |
| ------- | --------- | ----- | ------------------- |
| 1       | 101       | 10    | 40                  |
| 2       | 101       | 30    | 40                  |
| 3       | 102       | 5     | 25                  |
| 4       | 102       | 20    | 25                  |

Notice:

- Input rows: 4
- Output rows: 4
- Aggregation + full detail preserved

---

# 3. Window Function Syntax

Window functions have two major components:

```sql
window_function(expression)
OVER (
    PARTITION BY column
    ORDER BY column
    frame_clause
)
```

## Components

| Component    | Purpose                   |
| ------------ | ------------------------- |
| Function     | SUM, AVG, RANK, etc.      |
| OVER         | Defines window context    |
| PARTITION BY | Divides data into windows |
| ORDER BY     | Sorts rows within window  |
| Frame Clause | Defines subset of rows    |

---

# 4. PARTITION BY Clause

Defines how data is divided into logical windows.

## 4.1 No Partition (Entire Table as One Window)

```sql
SELECT
    OrderID,
    Sales,
    SUM(Sales) OVER () AS TotalSales
FROM Orders;
```

### Output

| OrderID | Sales | TotalSales |
| ------- | ----- | ---------- |
| 1       | 10    | 65         |
| 2       | 30    | 65         |
| 3       | 5     | 65         |
| 4       | 20    | 65         |

Explanation:

- The window definition is `OVER ()`, which means no PARTITION BY and no ORDER BY.
- SQL treats the entire table as a single window.
- It calculates the total sales once: 10 + 30 + 5 + 20 = 65.
- Because window functions preserve row-level detail, the value 65 is repeated for every row instead of collapsing the result into one row.
- This differs from GROUP BY, which would return only one row.

---

## 4.2 Single Partition

```sql
SUM(Sales) OVER (PARTITION BY ProductID)
```

### Output

| OrderID | ProductID | Sales | TotalSalesByProduct |
| ------- | --------- | ----- | ------------------- |
| 1       | 101       | 10    | 40                  |
| 2       | 101       | 30    | 40                  |
| 3       | 102       | 5     | 25                  |
| 4       | 102       | 20    | 25                  |

Explanation: Sales are summed per ProductID partition and the product total is repeated for each row within that partition.

---

## 4.3 Multiple Partitions

```sql
SUM(Sales) OVER (PARTITION BY ProductID, OrderStatus)
```

### Output

| OrderID | ProductID | OrderStatus | Sales | TotalSalesByProductStatus |
| ------- | --------- | ----------- | ----- | ------------------------- |
| 1       | 101       | Delivered   | 10    | 10                        |
| 2       | 101       | Shipped     | 30    | 30                        |
| 3       | 102       | Delivered   | 5     | 5                         |
| 4       | 102       | Shipped     | 20    | 20                        |

Explanation (one line): The window is partitioned by both ProductID and OrderStatus, so each unique combination forms its own partition and the sum is calculated within that combination.

---

# 5. ORDER BY Clause in Windows

Required for:

- Ranking functions
- Value functions
- Frame definitions

Optional for aggregate window functions.

---

## Example: Ranking Orders by Sales

```sql
SELECT
    OrderID,
    Sales,
    RANK() OVER (ORDER BY Sales DESC) AS SalesRank
FROM Orders;
```

### Output

| OrderID | Sales | SalesRank |
| ------- | ----- | --------- |
| 2       | 30    | 1         |
| 4       | 20    | 2         |
| 1       | 10    | 3         |
| 3       | 5     | 4         |

Execution Flow:

1. Sort rows by Sales DESC
2. Assign rank sequentially

# 6. Frame Clause (Advanced)

The frame clause defines a dynamic subset of rows within each partition (window) that will participate in the calculation, and it works only when an ORDER BY is present; it specifies lower and upper boundaries (such as UNBOUNDED PRECEDING, CURRENT ROW, n PRECEDING, n FOLLOWING, UNBOUNDED FOLLOWING) that determine how the window frame slides row-by-row during execution.

Syntax:

```sql
ROWS BETWEEN lower_boundary AND upper_boundary
```

Common Boundaries:

| Boundary            | Meaning         |
| ------------------- | --------------- |
| UNBOUNDED PRECEDING | Start of window |
| CURRENT ROW         | Current row     |
| n PRECEDING         | n rows before   |
| n FOLLOWING         | n rows after    |
| UNBOUNDED FOLLOWING | End of window   |

---

## 6.1 Running Total Example

```sql
SELECT
    OrderDate,
    Sales,
    SUM(Sales) OVER (
        ORDER BY OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal
FROM Orders;
```

### Output

| OrderDate  | Sales | RunningTotal |
| ---------- | ----- | ------------ |
| 2024-01-01 | 10    | 10           |
| 2024-01-02 | 30    | 40           |
| 2024-01-03 | 5     | 45           |
| 2024-01-04 | 20    | 65           |

Explanation (one line): The frame `UNBOUNDED PRECEDING TO CURRENT ROW` accumulates sales from the first row up to the current row, producing a cumulative (running) total ordered by date.

---

## 6.2 Frame Shortcuts and Default Behavior

### Shortcut Syntax (Only for PRECEDING)

Instead of writing the full frame:

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

You can write the shortcut:

```sql
ROWS UNBOUNDED PRECEDING
```

Both produce the same result (running total).

Similarly:

```sql
ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
```

Shortcut:

```sql
ROWS 2 PRECEDING
```

⚠ Important Rules:

- Shortcut works **only with PRECEDING**.
- `ROWS UNBOUNDED FOLLOWING` ❌ Invalid as shortcut.
- `ROWS 1 FOLLOWING` ❌ Invalid as shortcut.
- FOLLOWING requires full BETWEEN syntax.

---

### Default Frame Behavior (Hidden Frame)

If you use ORDER BY but do NOT specify a frame clause:

```sql
SUM(Sales) OVER (ORDER BY OrderDate)
```

SQL automatically applies a hidden default frame:

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

That means:

- With ORDER BY → Default running total behavior
- Without ORDER BY → Entire partition is aggregated (no frame applied)

Example Without ORDER BY:

```sql
SUM(Sales) OVER ()
```

Result: Entire window aggregated for every row.

Example With ORDER BY but no frame:

```sql
SUM(Sales) OVER (ORDER BY OrderDate)
```

Result: Running total (because default frame is applied).

⚠ The default frame exists ONLY when ORDER BY is present.

---

If ORDER BY is present but frame is not specified:

Default =

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

# 7. Combining Multiple Windows

```sql
SELECT
    OrderID,
    ProductID,
    Sales,
    SUM(Sales) OVER () AS TotalSales,
    SUM(Sales) OVER (PARTITION BY ProductID) AS ProductTotal
FROM Orders;
```

### Output

| OrderID | ProductID | Sales | TotalSales | ProductTotal |
| ------- | --------- | ----- | ---------- | ------------ |
| 1       | 101       | 10    | 65         | 40           |
| 2       | 101       | 30    | 65         | 40           |
| 3       | 102       | 5     | 65         | 25           |
| 4       | 102       | 20    | 65         | 25           |

Explanation (one line): Multiple window functions operate independently in the same query, producing different aggregation levels (entire table vs per product) while preserving all original rows.

# 8. Rules & Limitations

## Rule 1: Allowed Only in SELECT and ORDER BY

Invalid:

```sql
WHERE SUM(Sales) OVER (...) > 100
```

## Rule 2: Cannot Nest Window Functions

Invalid:

```sql
SUM(SUM(Sales) OVER (...)) OVER (...)
```

## Rule 3: Executed After WHERE Filtering

Filtering happens first.

## Rule 4: Can Combine with GROUP BY (Same Columns Only)

Example:

```sql
SELECT
    CustomerID,
    SUM(Sales) AS TotalSales,
    RANK() OVER (ORDER BY SUM(Sales) DESC) AS RankCustomer
FROM Orders
GROUP BY CustomerID;
```

# 9. Performance Notes

- Requires sorting for ORDER BY
- May require memory for large partitions
- Index on partition/order columns improves performance
- Avoid unnecessary partitions

# 10. Common Mistakes

1. Forgetting ORDER BY in ranking functions
2. Misunderstanding default frame
3. Trying to filter with window function in WHERE
4. Mixing non-grouped columns with GROUP BY
5. Not understanding execution order

# 13. Additional Window Functions (Overview)

In addition to aggregate window functions (SUM, AVG, MIN, MAX, COUNT), SQL provides several other powerful analytical functions.

These will be explained in detail in later chapters.

## 13.1 Aggregate Window Functions

- COUNT(expr) → Input: any data type column | Returns: BIGINT/INT (row count) | Syntax: COUNT(expr) OVER (PARTITION BY ... ORDER BY ...)
- SUM(expr) → Input: numeric column | Returns: same numeric type | Syntax: SUM(expr) OVER (PARTITION BY ... ORDER BY ...)
- AVG(expr) → Input: numeric column | Returns: decimal/float | Syntax: AVG(expr) OVER (PARTITION BY ... ORDER BY ...)
- MIN(expr) → Input: comparable data type | Returns: same data type | Syntax: MIN(expr) OVER (PARTITION BY ... ORDER BY ...)
- MAX(expr) → Input: comparable data type | Returns: same data type | Syntax: MAX(expr) OVER (PARTITION BY ... ORDER BY ...)

## 13.2 Ranking Functions

- ROW_NUMBER() → Input: none (requires ORDER BY) | Returns: INT (sequential row number) | Syntax: ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)
- RANK() → Input: none (requires ORDER BY) | Returns: INT (rank with gaps on ties) | Syntax: RANK() OVER (PARTITION BY ... ORDER BY ...)
- DENSE_RANK() → Input: none (requires ORDER BY) | Returns: INT (rank without gaps) | Syntax: DENSE_RANK() OVER (PARTITION BY ... ORDER BY ...)
- CUME_DIST() → Input: none (requires ORDER BY) | Returns: DECIMAL (cumulative distribution 0–1) | Syntax: CUME_DIST() OVER (PARTITION BY ... ORDER BY ...)
- PERCENT_RANK() → Input: none (requires ORDER BY) | Returns: DECIMAL (percent rank 0–1) | Syntax: PERCENT_RANK() OVER (PARTITION BY ... ORDER BY ...)
- NTILE(n) → Input: integer n (number of buckets) | Returns: INT (bucket number) | Syntax: NTILE(n) OVER (PARTITION BY ... ORDER BY ...)

## 13.3 Value (Analytical) Functions

- LEAD(expr, offset, default) → Input: any data type column + optional offset/default | Returns: same data type as expr | Syntax: LEAD(expr, offset, default) OVER (PARTITION BY ... ORDER BY ...)
- LAG(expr, offset, default) → Input: any data type column + optional offset/default | Returns: same data type as expr | Syntax: LAG(expr, offset, default) OVER (PARTITION BY ... ORDER BY ...)
- FIRST_VALUE(expr) → Input: any data type column | Returns: same data type as expr | Syntax: FIRST_VALUE(expr) OVER (PARTITION BY ... ORDER BY ...)
- LAST_VALUE(expr) → Input: any data type column | Returns: same data type as expr | Syntax: LAST_VALUE(expr) OVER (PARTITION BY ... ORDER BY ...)

Note: Each of these functions serves a specific analytical purpose such as ranking rows, comparing previous/next values, or accessing boundary values within a partition. Detailed explanations and examples will be covered in subsequent sections.
