## 1. Introduction — Functional Indexes

Beyond the structural (clustered/non-clustered) and storage-based (row store/column store) categories, databases support a third category of indexes: **function-based indexes** — those that add **behavioral rules** or **scope constraints** on top of the standard B-Tree structure.

In this chapter we cover two:

| Index Type | What it does |
|---|---|
| **Unique Index** | Enforces that no two rows have the same value in the indexed column(s) |
| **Filtered Index** | Builds the index over only a **subset** of rows that satisfy a WHERE condition |

Both are compatible with clustered, non-clustered, and each other — with specific restrictions covered in detail below.

---

## 2. Unique Index — Definition & Purpose

A **Unique Index** is an index that enforces **uniqueness** — no two rows in the table may have the same value(s) in the indexed column(s).

> 💡 **Simple Definition:** A regular index speeds up lookups. A unique index speeds up lookups AND guarantees your data has no duplicates in that column.

### Two Key Reasons to Use a Unique Index

**Reason 1 — Data Integrity**

Columns like `email`, `product_code`, `username`, or `national_id` should never have duplicates. A unique index is the database-level enforcement of that business rule — not application-level, not a convention, but a hard constraint enforced by the engine itself.

```
Without unique index:
  Row 1: email = "alice@example.com"
  Row 2: email = "alice@example.com"  ← silently inserted ✅ (bad!)

With unique index:
  Row 1: email = "alice@example.com"
  Row 2: email = "alice@example.com"  ← rejected with error ❌ (correct!)
```

**Reason 2 — Query Performance**

When SQL knows a column is unique, it can **stop searching** as soon as it finds the first match — because there can be no second match. With a non-unique index, SQL must always check if more matching rows exist.

```
Non-unique index search for email = "alice@example.com":
  → Find first match → continue scanning (might be more) → eventually stop

Unique index search for email = "alice@example.com":
  → Find first match → STOP immediately (guaranteed no more) ✅ faster!
```

## 3. How the Unique Index Works Internally

The unique index uses the **same B-Tree structure** as a regular non-clustered index — but with an additional enforcement layer:

### Standard Non-Clustered Index (Non-Unique)

```
Root → Intermediate → Leaf Nodes (Row Identifiers, duplicates allowed)
```

### Unique Non-Clustered Index

```
Root → Intermediate → Leaf Nodes (Row Identifiers, uniqueness enforced)
                                          ↓
                              On every INSERT / UPDATE:
                              SQL checks: does this value already exist?
                              YES → reject with error
                              NO  → insert and update index
```

### Write Performance Impact

Because SQL must **verify uniqueness on every write**, building and maintaining a unique index is slightly **slower on writes** than a standard index:

```
Standard insert:     Write row → Update index B-Tree → Done
Unique insert:       Write row → Update index B-Tree
                               → Verify no duplicate exists → Done (extra step)
```

However, for read-heavy columns where uniqueness is expected (emails, IDs), the combined benefit of integrity + read speed far outweighs the minor write overhead.

### Read Performance Impact

```
Standard index scan for exact match:
  Navigate B-Tree → Find value → Continue scanning → Reach end of matches

Unique index scan for exact match:
  Navigate B-Tree → Find value → STOP (uniqueness guarantees only one) ← faster
```

## 4. Unique Index — Syntax & Rules

### Basic Syntax

```sql
CREATE UNIQUE [CLUSTERED | NONCLUSTERED] INDEX index_name
ON table_name (column1 [ASC|DESC], column2 [ASC|DESC], ...);
```

The `UNIQUE` keyword comes **before** `CLUSTERED` or `NONCLUSTERED`.

### Creating a Unique Non-Clustered Index

```sql
-- Enforce uniqueness on product name
CREATE UNIQUE NONCLUSTERED INDEX idx_products_product_name
ON sales.products (product_name);
```

### Creating a Unique Clustered Index

```sql
-- Primary keys automatically create a unique clustered index
-- But you can also create one explicitly
CREATE UNIQUE CLUSTERED INDEX idx_customers_customer_id
ON sales.customers (customer_id);
```

### Composite Unique Index (Uniqueness across multiple columns)

```sql
-- No two rows can have the SAME combination of first_name + last_name + birth_date
CREATE UNIQUE NONCLUSTERED INDEX idx_customers_name_dob
ON customers (last_name, first_name, birth_date);
```

> 💡 A composite unique index enforces uniqueness of the **combination**, not each column independently. Two customers can both have `last_name = 'Smith'`, as long as their full combinations differ.

### What Happens on Duplicate Insert

```sql
-- After creating unique index on product_name:
INSERT INTO sales.products (product_id, product_name)
VALUES (106, 'Caps');  -- 'Caps' already exists

-- Error:
-- Cannot insert duplicate key row in object 'sales.products'
-- with unique index 'idx_products_product_name'.
-- The duplicate key value is (Caps).
```

### Rules for Unique Index

| Rule | Detail |
|---|---|
| NULL values | In SQL Server, only **one NULL** is allowed (treated as a value). In PostgreSQL/MySQL, multiple NULLs are allowed (NULL ≠ NULL). |
| Existing duplicates | Cannot create a unique index on a column that already has duplicate values — must clean data first |
| Primary key | Creating a PRIMARY KEY automatically creates a unique clustered index (SQL Server) |
| Unique constraint | `UNIQUE CONSTRAINT` and `UNIQUE INDEX` are functionally equivalent in most databases |

## 5. Filtered Index — Definition & Purpose

A **Filtered Index** is a non-clustered index that covers only a **subset of rows** — those matching a specified WHERE condition. Instead of indexing every row in the table, it builds the B-Tree over only the rows relevant to your most frequent queries.

> 💡 **Simple Definition:** Instead of indexing the whole table, a filtered index says "only index the rows where `country = 'USA'`." The B-Tree becomes smaller, faster, and cheaper to store.

### Why Use a Filtered Index?

**Benefit 1 — Targeted Optimization**

If your application always queries `WHERE status = 'active'`, building an index that includes inactive customers is wasteful. A filtered index includes only the active rows — making it faster to navigate.

**Benefit 2 — Smaller Index Size**

A smaller B-Tree means:
- Fewer pages to read during navigation
- Faster query execution
- Less memory needed to cache the index

**Benefit 3 — Reduced Storage**

Fewer rows → fewer leaf-level index pages → less disk space consumed by the index.

```
Full non-clustered index on status:
  Leaf nodes: ALL rows → 1,000,000 entries

Filtered index (WHERE status = 'active'):
  Leaf nodes: ONLY active rows → 700,000 entries (if 70% are active)
```

---

## 6. How the Filtered Index Works Internally

A filtered index is a **standard non-clustered B-Tree** — but its leaf nodes contain Row Identifiers for **only the rows satisfying the filter condition**.

### Structure Comparison

**Full Non-Clustered Index on `country`:**

```
Leaf Nodes:
  country = 'Canada' → RID 1:100:20
  country = 'Germany' → RID 1:101:96
  country = 'India' → RID 1:100:40
  country = 'USA' → RID 1:101:12
  country = 'USA' → RID 1:102:56
  country = 'USA' → RID 1:103:80
  ... (ALL rows included)
```

**Filtered Non-Clustered Index on `country` WHERE `country = 'USA'`:**

```
Leaf Nodes:
  country = 'USA' → RID 1:101:12
  country = 'USA' → RID 1:102:56
  country = 'USA' → RID 1:103:80
  ... (ONLY USA rows — Germany, Canada, India excluded entirely)
```

The B-Tree is physically smaller — fewer nodes, fewer pages, faster traversal.

### When the Filtered Index IS Used

```sql
-- ✅ Query matches the filter condition → index used
SELECT * FROM customers WHERE country = 'USA';
SELECT customer_id, score FROM customers WHERE country = 'USA' AND score > 500;
```

### When the Filtered Index is NOT Used

```sql
-- ❌ Query asks for non-USA rows → filtered index ignored, full scan or other index used
SELECT * FROM customers WHERE country = 'Germany';

-- ❌ Query doesn't specify country → filtered index not applicable
SELECT * FROM customers WHERE score > 800;
```

> ⚠️ The query must include the filter column condition for the optimizer to consider using the filtered index. If you query for a value outside the filter, the optimizer falls back to a full scan or other available indexes.

## 7. Filtered Index — Syntax & Rules

### Basic Syntax

```sql
CREATE [UNIQUE] NONCLUSTERED INDEX index_name
ON table_name (column1, column2, ...)
WHERE filter_condition;
```

### Simple Filtered Index

```sql
-- Index only customers from USA
CREATE NONCLUSTERED INDEX idx_customers_country_usa
ON sales.customers (country)
WHERE country = 'USA';
```

### Filtered Index on Status Column

```sql
-- Index only active customers
CREATE NONCLUSTERED INDEX idx_customers_active
ON customers (customer_id, score)
WHERE status = 'active';
```

### Dropping a Filtered Index

```sql
DROP INDEX idx_customers_country_usa ON sales.customers;
```

### Restrictions (SQL Server)

| Restriction | Detail |
|---|---|
| Only non-clustered | Filtered indexes cannot be applied to clustered indexes (the entire table must be sorted — a partial sort makes no sense) |
| Only row store | Cannot be used with columnstore indexes |
| Filter column must be in the table | The WHERE condition must reference columns in the same table |
| Simple conditions only | Some complex expressions may not be supported in the filter |
| Statistics update | SQL Server must update filtered index statistics when data changes — this is automatic but adds slight write overhead |

## 8. Combining Unique + Filtered Index

You can apply **both** `UNIQUE` and `WHERE` to the same index. This enforces uniqueness within the filtered subset — rows outside the filter are not checked for uniqueness.

### Use Case — Unique Active Email

Business rule: "No two active users can share the same email. Inactive users are archived and may have duplicates (e.g., old deleted accounts)."

```sql
CREATE UNIQUE NONCLUSTERED INDEX idx_customers_active_email
ON customers (email)
WHERE status = 'active';
```

**Behaviour:**

```
email = "alice@example.com", status = "active"   → ✅ allowed
email = "alice@example.com", status = "active"   → ❌ duplicate! error
email = "alice@example.com", status = "inactive" → ✅ allowed (outside filter)
email = "alice@example.com", status = "inactive" → ✅ allowed (outside filter)
```

Two inactive accounts can share an email — only active accounts are unique-enforced.

## 9. Cross-Database Syntax — MSSQL vs MySQL vs PostgreSQL

### SQL Server (MSSQL)

```sql
-- Unique Non-Clustered Index
CREATE UNIQUE NONCLUSTERED INDEX idx_products_product_name
ON sales.products (product_name);

-- Unique Clustered Index
CREATE UNIQUE CLUSTERED INDEX idx_customers_customer_id
ON sales.customers (customer_id);

-- Filtered Index
CREATE NONCLUSTERED INDEX idx_customers_country_usa
ON sales.customers (country, score)
WHERE country = 'USA';

-- Unique + Filtered Index
CREATE UNIQUE NONCLUSTERED INDEX idx_customers_active_email
ON customers (email)
WHERE status = 'active';

-- Drop Index
DROP INDEX idx_products_product_name ON sales.products;

-- View indexes
SELECT name, is_unique, filter_definition, type_desc
FROM sys.indexes
WHERE object_id = OBJECT_ID('sales.customers');
```

**SQL Server Notes:**
- `filter_definition` column in `sys.indexes` shows the WHERE clause of a filtered index.
- Filtered indexes are only available in **SQL Server 2008 and later**.
- `UNIQUE CONSTRAINT` and `UNIQUE INDEX` are interchangeable in SQL Server.
- NULL handling: only **one NULL** is allowed in a unique index column.

---

### MySQL

```sql
-- Unique Index
CREATE UNIQUE INDEX idx_products_product_name
ON products (product_name);

-- Unique Index via ALTER TABLE
ALTER TABLE customers ADD UNIQUE INDEX idx_customers_email (email);

-- Unique constraint (equivalent to unique index in MySQL)
ALTER TABLE customers ADD CONSTRAINT uq_customers_email UNIQUE (email);

-- Composite Unique Index
CREATE UNIQUE INDEX idx_customers_name_dob
ON customers (last_name, first_name, birth_date);

-- ❌ Filtered Index: NOT supported natively in MySQL
-- MySQL does not support partial/filtered indexes on InnoDB tables.

-- Workaround 1: Create a generated (virtual) column and index it
ALTER TABLE customers
ADD COLUMN country_usa VARCHAR(20)
GENERATED ALWAYS AS (IF(country = 'USA', country, NULL)) VIRTUAL;

CREATE INDEX idx_customers_country_usa ON customers (country_usa);
-- Now querying WHERE country_usa IS NOT NULL effectively uses only USA rows

-- Workaround 2: Use a partial index via expression in MariaDB 10.5+
CREATE INDEX idx_customers_country_usa
ON customers (country)
WHERE country = 'USA';  -- MariaDB supports this; MySQL InnoDB does not

-- Drop Index
DROP INDEX idx_products_product_name ON products;

-- View indexes
SHOW INDEX FROM customers;

-- Check uniqueness
SELECT non_unique, index_name, column_name
FROM information_schema.statistics
WHERE table_name = 'customers';
```

**MySQL Notes:**
- MySQL **InnoDB does not support filtered/partial indexes** natively.
- **MariaDB 10.5+** (MySQL-compatible fork) added partial index support.
- Multiple NULL values **are allowed** in a unique index column in MySQL (NULL ≠ NULL semantics).
- `UNIQUE INDEX` and `UNIQUE KEY` are synonymous in MySQL.
- The generated column workaround is the most portable MySQL approach for simulating filtered indexes.

---

### PostgreSQL

PostgreSQL has **native support for partial indexes** (equivalent to filtered indexes) and handles unique indexes with more flexibility than SQL Server.

```sql
-- Unique Index
CREATE UNIQUE INDEX idx_products_product_name
ON products (product_name);

-- Unique Index with CONCURRENTLY (no table lock in production)
CREATE UNIQUE INDEX CONCURRENTLY idx_customers_email
ON customers (email);

-- Composite Unique Index
CREATE UNIQUE INDEX idx_customers_name_dob
ON customers (last_name, first_name, birth_date);

-- Partial Index (PostgreSQL's equivalent of filtered index)
CREATE INDEX idx_customers_country_usa
ON customers (country, score)
WHERE country = 'USA';

-- Unique Partial Index (unique + filtered combined)
CREATE UNIQUE INDEX idx_customers_active_email
ON customers (email)
WHERE status = 'active';

-- Unique Partial Index — another real-world example:
-- Only one 'pending' order per user at a time
CREATE UNIQUE INDEX idx_orders_pending_user
ON orders (user_id)
WHERE status = 'pending';

-- Drop Index
DROP INDEX idx_customers_country_usa;
-- Note: PostgreSQL DROP INDEX is not ON table_name like SQL Server

-- View partial indexes
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'customers'
  AND indexdef LIKE '%WHERE%';

-- View unique indexes
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'customers'
  AND indexdef LIKE '%UNIQUE%';
```

**PostgreSQL Notes:**
- PostgreSQL calls filtered indexes **partial indexes** — same concept, different name.
- **Multiple NULLs** are allowed in a unique index column (NULL ≠ NULL).
- `CONCURRENTLY` builds the index without locking the table — critical for production systems.
- Partial indexes can have **complex WHERE conditions**, including `IS NULL`, `IS NOT NULL`, and multi-column filters.
- PostgreSQL partial indexes are more **flexible** than SQL Server filtered indexes — they support a wider range of filter expressions.

---

### Cross-Database Feature Comparison

| Feature | SQL Server | MySQL (InnoDB) | MariaDB | PostgreSQL |
|---|---|---|---|---|
| Unique Index | ✅ | ✅ | ✅ | ✅ |
| Composite Unique Index | ✅ | ✅ | ✅ | ✅ |
| Filtered / Partial Index | ✅ (filtered) | ❌ (not native) | ✅ (partial, v10.5+) | ✅ (partial) |
| Unique + Filtered/Partial | ✅ | ❌ | ✅ | ✅ |
| NULL in unique index | 1 NULL only | Multiple NULLs | Multiple NULLs | Multiple NULLs |
| Filtered on clustered index | ❌ | ❌ | ❌ | ❌ |
| Filtered on columnstore | ❌ | ❌ | N/A | N/A |
| Non-locking index build | ✅ (online) | ✅ | ✅ | ✅ (`CONCURRENTLY`) |
| Syntax for filter | `WHERE condition` | N/A (generated col workaround) | `WHERE condition` | `WHERE condition` |

## 11. Query Execution Flow — Filtered Index in Action

### Query

```sql
SELECT customer_id, email, score
FROM customers
WHERE country = 'USA'
ORDER BY score DESC;
```

### Execution Without Filtered Index (Full Non-Clustered Index on `country`)

```
Navigate B-Tree → Leaf nodes contain ALL country values:
  Canada → RID
  Germany → RID
  Germany → RID
  UK → RID
  UK → RID
  USA → RID  ← needed
  USA → RID  ← needed
  USA → RID  ← needed
  USA → RID  ← needed
  USA → RID  ← needed

Pages read: All leaf pages (all countries)
Rows returned to filter: 10 (all countries)
Rows after filter: 5 (USA only)
```

### Execution With Filtered Index (`WHERE country = 'USA'`)

```
Navigate B-Tree → Leaf nodes contain ONLY USA rows:
  USA → RID  ← needed
  USA → RID  ← needed
  USA → RID  ← needed
  USA → RID  ← needed
  USA → RID  ← needed

Pages read: Fewer pages (only USA leaf pages)
Rows returned to filter: 5 (all already USA)
Rows after filter: 5
Unnecessary rows scanned: 0
```

**Result:** Smaller B-Tree → fewer I/O reads → faster query.

## 12. Real-World Use Cases

### Unique Index Use Cases

| Column | Why Unique Index |
|---|---|
| `email` | No two users should share a login email |
| `username` | Usernames must be globally unique |
| `product_sku` / `product_code` | Each SKU uniquely identifies a product in inventory |
| `national_id` / `passport_no` | Government identifiers must be unique per person |
| `order_reference` | Each order confirmation number must be unique |
| `slug` (blog/URL) | URL slugs must be unique for routing |

### Filtered Index Use Cases

| Scenario | Filtered Index Definition |
|---|---|
| App always queries active users | `WHERE status = 'active'` |
| Reports only cover current year orders | `WHERE YEAR(created_at) = 2024` |
| Searches always scoped to one country | `WHERE country = 'USA'` |
| Dashboard shows only unresolved tickets | `WHERE resolved = 0` |
| Email verification queue (unverified users only) | `WHERE email_verified = 0` |
| Premium user analytics | `WHERE plan = 'premium'` |

### Unique + Filtered Index Use Cases

| Business Rule | Index Definition |
|---|---|
| No two active users share an email | `UNIQUE ON (email) WHERE status = 'active'` |
| One pending order per user at a time | `UNIQUE ON (user_id) WHERE status = 'pending'` |
| One active session per device | `UNIQUE ON (device_id) WHERE session_active = 1` |
| Only one primary address per customer | `UNIQUE ON (customer_id) WHERE is_primary = 1` |

## 13. Common Mistakes

| Mistake | Problem | Fix |
|---|---|---|
| Creating a unique index on a column that already has duplicates | `Error: Cannot create unique index — duplicate key found` | Clean the data first (`DELETE` or `UPDATE` duplicates), then create the index |
| Creating a unique index on a column with low cardinality (e.g., `status` = 'active'/'inactive') | No benefit — values aren't unique; index creation fails or is wrong choice | Only apply unique index to columns that **should** be unique by business rule |
| Creating a filtered index on a clustered index | `Error: Filtered index is not allowed on clustered index` | Filtered indexes are non-clustered only — redesign accordingly |
| Creating a filtered index on a columnstore index | Not supported — different storage model | Use filtered non-clustered row store index alongside the columnstore |
| Querying with a condition outside the filter and expecting the filtered index to help | Filtered index is ignored — optimizer does full scan | Either widen the filter or create a second index for the other condition |
| Forgetting to include the filter condition in the query | Optimizer doesn't recognize the filtered index as applicable | Always include the filter column in your WHERE clause to trigger index use |
| Relying on application code alone to enforce uniqueness (no unique index) | Race conditions between simultaneous inserts can create duplicates even with app-level checks | Always enforce uniqueness at the **database level** with a unique index |
| Assuming NULL = NULL in a unique index | In SQL Server, only one NULL is allowed. In MySQL/PostgreSQL, multiple NULLs are permitted | Know your database's NULL uniqueness behaviour before designing nullable unique columns |

## 14. Best Practices

1. **Enforce uniqueness at the database level, not just the application level.** Application-level checks have race conditions — two simultaneous inserts can both pass the check and both succeed. A unique index is atomic and cannot be bypassed.

2. **Apply unique indexes to natural keys.** Columns that are logically unique by business rules (`email`, `username`, `sku`, `national_id`) should always have a unique index — regardless of whether they're the primary key.

3. **Use filtered indexes for dominant query patterns.** If 80%+ of your queries filter by a specific condition (`status = 'active'`, `country = 'USA'`), a filtered index dramatically reduces the B-Tree size and speeds up those queries at minimal cost.

4. **Filter on low-selectivity columns.** Filtered indexes are especially valuable for columns with low cardinality (few distinct values like `status`, `region`, `plan`) where a full index would contain many unwanted rows.

5. **Combine unique + filtered for conditional business rules.** "Only one pending order per user" or "active users must have unique emails" are perfect candidates for `UNIQUE + WHERE` filtered indexes.

6. **Don't create filtered indexes for infrequently queried conditions.** The optimizer will only use a filtered index when the query matches the filter. If your queries are diverse and don't consistently use that condition, a full index may serve better.

7. **Monitor with query execution plans.** After creating a filtered index, verify it's actually being used with `EXPLAIN` (MySQL/PostgreSQL) or `SET STATISTICS IO ON` + execution plan (SQL Server). A filtered index that isn't being used wastes storage and write overhead.

8. **In PostgreSQL, always use `CONCURRENTLY` in production** when creating unique or partial indexes on large tables — it prevents table locking during index build.

```sql
-- PostgreSQL: production-safe index creation
CREATE UNIQUE INDEX CONCURRENTLY idx_customers_email
ON customers (email);
```
