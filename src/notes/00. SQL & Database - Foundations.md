## 1. Introduction & Definitions

### 1.1 What is Data?
**Definition**: Data is raw information or facts that represent objects, events, or phenomena in the real world.

**Examples of Data**:
- Personal information (name, age, mobile number)
- Financial records (bank statements, transactions)
- Sales information (products, customers, transactions)
- Digital artifacts (files, documents, images)

### 1.2 The Data Problem
Organizations generate **massive amounts of data daily** through:
- Product operations
- Customer interactions
- Sales transactions
- Service logs
- Financial records

**Challenge**: Without proper storage and management, data becomes:
- Difficult to access
- Hard to organize
- Impossible to analyze at scale
- Prone to loss or corruption

## 2. What is SQL?

### 2.1 Definition
**SQL** stands for **Structured Query Language**. It is the standard programming language used to communicate with and manipulate relational databases.

**Pronunciation**: Both "SQL" (ess-que-el) and "Sequel" are accepted, though "SQL" is more common in formal contexts.

### 2.2 Purpose of SQL
SQL serves as a bridge between users/applications and databases:
- Query data (ask questions)
- Insert new data
- Update existing data
- Delete data
- Define database structure
- Manage database operations

### 2.3 Why SQL is Necessary

**Traditional Approach (Using Files)**:
```
File 1: customers.xlsx
File 2: sales.xlsx
File 3: products.txt
File 4: transactions.csv

Question: "What is the total spending by customer?"
Process: Open each file → Search → Combine data → Manual calculation → TIME CONSUMING
```

**SQL Approach (Using Database)**:
```sql
SELECT CustomerID, SUM(Amount) AS TotalSpending
FROM sales
GROUP BY CustomerID;

Result: INSTANT and ACCURATE
```

**Key Advantage**: One query returns results in milliseconds instead of hours.

## 3. What is a Database?

### 3.1 Definition
**Database**: A structured, organized container that stores data in a systematic way, designed for easy access, management, retrieval, and searching.

### 3.2 Why Use Databases Instead of Files?

| Aspect | Files/Spreadsheets | Database |
|--------|-------------------|----------|
| **Data Volume** | Limited (millions of rows crash spreadsheets) | Handles billions of rows efficiently |
| **Query Speed** | Slow (manual searching required) | Fast (millisecond responses) |
| **Data Integrity** | Prone to errors | Enforced constraints prevent errors |
| **Security** | Limited access control | Granular permission management |
| **Concurrent Access** | Problems with multiple users | Handles thousands of users simultaneously |
| **Data Consistency** | Manual verification | Automatic validation |
| **Backup & Recovery** | Manual process | Automated backup systems |

### 3.3 Real-World Example

**Scenario: Bank Data Management**

```
Using Spreadsheets:
- 1 million customer records
- 50 branches, 200 employees
- Each employee accessing data = CRASHES & CONFLICTS
- Finding customer history = HOURS

Using Database:
- All 1 million records stored efficiently
- 200 employees access simultaneously = NO ISSUES
- Finding customer history = MILLISECONDS
```

## 4. DBMS & SQL Servers

### 4.1 DBMS Definition
**DBMS** stands for **Database Management System**. It is specialized software that manages all interactions with the database.

### 4.2 Components of a Database Ecosystem

```
┌─────────────────────────────────────────────────────┐
│              USERS & APPLICATIONS                    │
│  (Employees, Web Apps, Mobile Apps, Tools)          │
└──────────────────┬──────────────────────────────────┘
                   │ (Send SQL queries)
                   ↓
┌─────────────────────────────────────────────────────┐
│   DATABASE MANAGEMENT SYSTEM (DBMS)                 │
│  (Manages requests, security, priority, execution)  │
└──────────────────┬──────────────────────────────────┘
                   │
                   ↓
┌─────────────────────────────────────────────────────┐
│         SERVER (Physical/Cloud Hardware)            │
│           (Runs 24/7, Always Available)             │
└──────────────────┬──────────────────────────────────┘
                   │
                   ↓
┌─────────────────────────────────────────────────────┐
│              DATABASE (Storage Layer)               │
│    (Organized tables, schemas, data)                │
└─────────────────────────────────────────────────────┘
```

### 4.3 DBMS Responsibilities

1. **Request Management**: Determines which queries execute first (priority queuing)
2. **Security Control**: Validates permissions before executing SQL
3. **Concurrency Control**: Manages simultaneous user access
4. **Query Optimization**: Finds the most efficient execution path
5. **Data Consistency**: Ensures data integrity across operations
6. **Backup & Recovery**: Maintains data safety

### 4.4 SQL Server

**Definition**: A SQL Server is a physical or virtual machine (computer) that runs a DBMS and stores database files.

**Deployment Options**:
- **On-Premises**: Server physically located in company data center
- **Cloud-Based**: Server hosted by cloud providers (AWS, Azure, GCP)

**Why Servers?**
- Local PC: Goes offline, limited storage, insufficient processing power
- Server: Available 24/7, unlimited storage (scalable), powerful processors

### 5.2 Relational Databases (SQL)

**Definition**: Databases that organize data into related tables with rows and columns, connected through relationships via primary and foreign keys.

**Structure**:
```
TABLE: Customers
┌────────────┬──────────┬─────────┐
│CustomerID  │  Name    │  Email  │
├────────────┼──────────┼─────────┤
│     1      │  Maria   │ maria@  │
│     2      │  John    │ john@   │
│     3      │  Peter   │ peter@  │
└────────────┴──────────┴─────────┘

TABLE: Orders
┌─────────┬────────────┬─────────┐
│ OrderID │ CustomerID │ Amount  │
├─────────┼────────────┼─────────┤
│    1    │     1      │  $100   │
│    2    │     2      │  $250   │
│    3    │     1      │  $150   │
└─────────┴────────────┴─────────┘

RELATIONSHIP: CustomerID links Customers to Orders
```

**Popular Relational Databases**:
- MySQL
- Microsoft SQL Server (MSSQL)
- PostgreSQL
- Oracle Database
- MariaDB

### 5.3 NoSQL Database Types

#### 5.3.1 Key-Value Databases
**Structure**: Stores data as key-value pairs (like a dictionary)
```
Key: "customer:1"       → Value: {name: "Maria", email: "maria@..."}
Key: "customer:2"       → Value: {name: "John", email: "john@..."}
```
**Examples**: Redis, Amazon DynamoDB, Memcached
**Use Case**: Caching, session management, real-time data

#### 5.3.2 Document Databases
**Structure**: Stores data as complete documents (JSON-like format)
```json
{
  "_id": 1,
  "name": "Maria",
  "email": "maria@...",
  "orders": [
    {"orderId": 1, "amount": 100},
    {"orderId": 2, "amount": 150}
  ]
}
```
**Examples**: MongoDB, Firestore, Couchbase
**Use Case**: Content management, user profiles, flexible schemas

#### 5.3.3 Column-Based Databases
**Structure**: Groups data by column instead of row
```
Traditional Row-Based:   Column-Based:
CustomerID | Name       CustomerID column: [1, 2, 3, 4...]
1          | Maria      Name column: [Maria, John, Peter...]
2          | John       Email column: [maria@, john@, peter@...]
3          | Peter
```
**Examples**: Apache Cassandra, Amazon Redshift, Google BigQuery
**Use Case**: Analytics, large-scale data warehousing, OLAP

#### 5.3.4 Graph Databases
**Structure**: Focuses on relationships between data points
```
     HAS_WORKED_AT
        ↙      ↘
    Maria    ─────  TechCorp
      │                │
      │ KNOWS      IS_CUSTOMER
      │                │
    John ─────────→ TechCorp
```
**Examples**: Neo4j, ArangoDB
**Use Case**: Social networks, recommendation engines, knowledge graphs

## 6. SQL Commands Overview

### 6.1 SQL Command Categories

SQL commands are organized into three main categories:

#### 6.1.1 DDL (Data Definition Language)
**Purpose**: Define and manage database structure
**Commands**:
- `CREATE`: Create new database objects (tables, schemas, indexes)
- `ALTER`: Modify existing database objects
- `DROP`: Delete database objects

**Example**:
```sql
-- Create a new table
CREATE TABLE Customers (
    CustomerID INT PRIMARY KEY,
    Name VARCHAR(100),
    Email VARCHAR(100)
);

-- Alter table to add a column
ALTER TABLE Customers ADD Phone VARCHAR(20);

-- Drop the table
DROP TABLE Customers;
```

#### 6.1.2 DML (Data Manipulation Language)
**Purpose**: Manage data within database objects
**Commands**:
- `INSERT`: Add new data rows
- `UPDATE`: Modify existing data
- `DELETE`: Remove data rows

**Example**:
```sql
-- Insert new customer
INSERT INTO Customers (CustomerID, Name, Email)
VALUES (1, 'Maria', 'maria@example.com');

-- Update customer email
UPDATE Customers
SET Email = 'maria.new@example.com'
WHERE CustomerID = 1;

-- Delete customer
DELETE FROM Customers
WHERE CustomerID = 1;
```

#### 6.1.3 DQL (Data Query Language)
**Purpose**: Retrieve and analyze data
**Command**:
- `SELECT`: Query data from database

**Example**:
```sql
-- Simple query
SELECT CustomerID, Name, Email
FROM Customers;

-- Query with conditions
SELECT * FROM Customers
WHERE Name = 'Maria';

-- Aggregate query
SELECT COUNT(*) AS TotalCustomers
FROM Customers;
```

### 6.2 SQL Command Execution Flow

```
┌─────────────────────────────────────┐
│  User writes SQL Query              │
└────────────────┬────────────────────┘
                 ↓
┌─────────────────────────────────────┐
│  SQL Parser validates syntax        │
└────────────────┬────────────────────┘
                 ↓
┌─────────────────────────────────────┐
│  Query Optimizer finds best plan    │
└────────────────┬────────────────────┘
                 ↓
┌─────────────────────────────────────┐
│  Query Compiler generates code      │
└────────────────┬────────────────────┘
                 ↓
┌─────────────────────────────────────┐
│  Query Engine executes              │
└────────────────┬────────────────────┘
                 ↓
┌─────────────────────────────────────┐
│  Database returns results           │
└─────────────────────────────────────┘
```

### 7.1 Complete Hierarchy

```
SERVER (Computer/Cloud Instance)
  └─ DATABASE 1 (sales_db)
      ├─ SCHEMA 1 (dbo)
      │   ├─ TABLE: Customers
      │   ├─ TABLE: Orders
      │   └─ TABLE: Products

          ├─ TABLE: Reports

      └─ SCHEMA 1 (dbo)
          ├─ TABLE: Employees
          ├─ TABLE: Departments
          └─ TABLE: Salaries
```

### 7.2 Schema Definition
**Definition**: A logical container/namespace that groups related database objects (tables, views, stored procedures, etc.)

**Purpose**:
- Organize tables by functionality
- Prevent naming conflicts
- Control access permissions
- Improve maintainability

**Example**:
```sql
-- Schema for order-related tables
CREATE SCHEMA dbo;
CREATE TABLE dbo.Customers (...);
CREATE TABLE dbo.Orders (...);

-- Schema for analytics
CREATE SCHEMA analytics;
CREATE VIEW analytics.MonthlySales AS ...;
CREATE TABLE analytics.Reports (...);

-- Access tables with schema prefix
SELECT * FROM dbo.Customers;
SELECT * FROM analytics.Reports;
```

## 8. Data Types in SQL

### 8.1 Integer Types

| Type | Range | Storage | Use Case |
|------|-------|---------|----------|
| TINYINT | 0 to 255 | 1 byte | Small counts, flags |
| SMALLINT | -32,768 to 32,767 | 2 bytes | Small numbers |
| INT | -2,147,483,648 to 2,147,483,647 | 4 bytes | General integers |
| BIGINT | Very large numbers | 8 bytes | Large identifiers |

**Example**:
```sql
CREATE TABLE Products (
    ProductID INT,              -- General use
    StockCount TINYINT,        -- 0-255 items
    WarehouseID SMALLINT       -- Small warehouse IDs
);
```

### 8.2 Decimal/Float Types

| Type | Precision | Use Case |
|------|-----------|----------|
| DECIMAL(p,s) | Exact decimal | Financial data, amounts |
| FLOAT | Approximate | Scientific calculations |
| REAL | Approximate | General floating point |

**Example**:
```sql
CREATE TABLE Orders (
    OrderID INT,
    Amount DECIMAL(10, 2),     -- $999,999.99 max, 2 decimal places
    Tax DECIMAL(5, 2),         -- $999.99
    Temperature FLOAT          -- Scientific data
);
```

### 8.3 Character/String Types

| Type | Size | Use Case |
|------|------|----------|
| CHAR(n) | Fixed n characters | Fixed-length (Phone, ZIP codes) |
| VARCHAR(n) | Up to n characters | Variable-length (Names, descriptions) |
| TEXT/LONGTEXT | Very large | Large documents, descriptions |

**CHAR vs VARCHAR**:
```sql
-- CHAR(5) - Always uses 5 characters
CHAR(5):  "AB   " (3 spaces padding)
          Size: 5 bytes

-- VARCHAR(5) - Uses only needed space
VARCHAR(5): "AB"
            Size: 2 bytes

-- Decision: Use VARCHAR for flexibility, CHAR for fixed formats
```

**Example**:
```sql
CREATE TABLE Customers (
    CustomerID INT,
    FirstName VARCHAR(50),     -- Variable length, up to 50 chars
    LastName VARCHAR(50),
    ZipCode CHAR(5),          -- Fixed 5-digit ZIP code
    Biography TEXT            -- Long text document
);
```

### 8.4 Date and Time Types

| Type | Format | Use Case |
|------|--------|----------|
| DATE | YYYY-MM-DD | Birth dates, order dates |
| TIME | HH:MM:SS | Times of day |
| DATETIME | YYYY-MM-DD HH:MM:SS | Complete timestamp |
| TIMESTAMP | Timezone-aware | System timestamps |

**Example**:
```sql
CREATE TABLE Events (
    EventID INT,
    EventDate DATE,            -- 2025-05-19
    EventTime TIME,            -- 14:30:45
    CreatedAt DATETIME,        -- 2025-05-19 14:30:45
    UpdatedAt TIMESTAMP        -- Automatic updates
);
```

### 8.5 Boolean Type

| Type | Values | Use Case |
|------|--------|----------|
| BOOLEAN/BIT | TRUE/FALSE or 1/0 | Flags, status |

**Example**:
```sql
CREATE TABLE Users (
    UserID INT,
    IsActive BOOLEAN,          -- TRUE or FALSE
    IsVerified BIT,            -- 1 or 0
    IsAdmin BOOLEAN DEFAULT FALSE
);
```

### 8.6 Special Types

| Type | Format | Use Case |
|------|--------|----------|
| UUID/GUID | Universally unique | Distributed systems |
| JSON | JSON document | Semi-structured data |
| XML | XML document | Complex hierarchical data |
| BLOB | Binary large object | Images, files |

**Example**:
```sql
CREATE TABLE UserProfiles (
    UserID INT,
    Metadata JSON,             -- {'age': 30, 'city': 'NYC'}
    Avatar BLOB,               -- Image file
    Preferences XML            -- Configuration data
);
```

### 9.1 Complete Structural Hierarchy

```
┌─────────────────────────────────────────────┐
│ SERVER (Physical Machine / Cloud Instance)  │
└────────────────┬────────────────────────────┘
                 │
     ┌───────────┴───────────┬────────────┐
     ↓                       ↓            ↓
┌─────────┐           ┌──────────┐  ┌─────────┐
│ DB: HR  │           │ DB: Sales│  │DB: Logs │
└────┬────┘           └────┬─────┘  └─────────┘
     │
     ├─ Schema: dbo
     │  ├─ Table: Employees
     │  │  ├─ Column: EmployeeID (INT)
     │  │  ├─ Column: FirstName (VARCHAR(50))
     │  │  ├─ Column: HireDate (DATE)
     │  │  ├─ Primary Key: EmployeeID
     │  │  └─ Row 1: [1, "Maria", 2020-01-15]
     │  │
     │  └─ Table: Departments
     │
     └─ Schema: analytics
        └─ View: EmployeeStats
```

### 9.2 Table Components Detailed

#### 9.2.1 Columns (Fields)

**Definition**: A column represents a specific attribute or property in a table.

**Characteristics**:
- Has a name (e.g., CustomerID, FirstName, Email)
- Has a data type (INT, VARCHAR, DATE, etc.)
- Contains values of that specific type

**Example**:
```sql
CREATE TABLE Customers (
    -- Column Name    Data Type      Constraints
    CustomerID     INT            PRIMARY KEY,
    FirstName      VARCHAR(50)    NOT NULL,
    Email          VARCHAR(100)   UNIQUE,
    JoinDate       DATE           DEFAULT GETDATE(),
    IsActive       BOOLEAN        DEFAULT TRUE
);
```

#### 9.2.2 Rows (Records)

**Definition**: A row represents a single instance or record of an entity.

**Example**:
```sql
-- Table: Customers
┌────────────┬───────────┬──────────────────┬──────────────┬──────────┐
│CustomerID  │FirstName  │Email             │JoinDate      │IsActive  │
├────────────┼───────────┼──────────────────┼──────────────┼──────────┤
│     1      │ Maria     │maria@example.com │ 2020-01-15   │   TRUE   │ ← Row 1
│     2      │ John      │john@example.com  │ 2021-03-20   │   TRUE   │ ← Row 2
│     3      │ Peter     │peter@example.com │ 2019-11-10   │   FALSE  │ ← Row 3
└────────────┴───────────┴──────────────────┴──────────────┴──────────┘
```

#### 9.2.3 Cells (Values)

**Definition**: A cell is the intersection of a column and row, containing a single data value.

```sql
SELECT FirstName FROM Customers WHERE CustomerID = 1;
-- Result: "Maria" (a single cell value)
```

#### 9.2.4 Primary Key

**Definition**: A column (or set of columns) that uniquely identifies each row in a table.

**Characteristics**:
- UNIQUE: No two rows have the same value
- NOT NULL: Must always have a value
- Acts as a "fingerprint" for each record
- Used for relationships between tables

**Example**:
```sql
-- Single column Primary Key
CREATE TABLE Customers (
    CustomerID INT PRIMARY KEY,  -- Unique identifier
    Name VARCHAR(100),
    Email VARCHAR(100)
);

-- Composite Primary Key (multiple columns)
CREATE TABLE OrderItems (
    OrderID INT,
    ItemID INT,
    Quantity INT,
    PRIMARY KEY (OrderID, ItemID)  -- Combination is unique
);
```

#### 9.2.5 Relationships Between Tables

**Definition**: Connections between tables using foreign keys to maintain data integrity.

**Example**:
```sql
-- Parent Table
CREATE TABLE Customers (
    CustomerID INT PRIMARY KEY,
    Name VARCHAR(100)
);

-- Child Table with Foreign Key
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT,             -- Foreign Key
    OrderDate DATE,
    Amount DECIMAL(10, 2),
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);

-- Now we can link data:
-- Order 1 belongs to Customer 1 (Maria)
-- Order 2 belongs to Customer 2 (John)
```

## 11. Common Mistakes

### 11.1 Structural Mistakes

#### Mistake 1: Not Understanding Data Types
```sql
-- ❌ WRONG: Using TEXT for fixed-length data
CREATE TABLE Users (
    PhoneNumber TEXT  -- Wastes space, no validation
);

-- ✓ CORRECT: Use fixed-length types
CREATE TABLE Users (
    PhoneNumber CHAR(10)  -- Space-efficient, validates format
);

-- ❌ WRONG: Using INT for money
CREATE TABLE Orders (
    Amount INT  -- Loses decimal precision
);

-- ✓ CORRECT: Use DECIMAL
CREATE TABLE Orders (
    Amount DECIMAL(10, 2)  -- Preserves cents
);
```

#### Mistake 2: No Primary Keys
```sql
-- ❌ WRONG: Table without primary key
CREATE TABLE Orders (
    OrderID INT,
    CustomerID INT,
    Amount DECIMAL(10, 2)
);

-- ✓ CORRECT: Always define primary key
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT NOT NULL,
    Amount DECIMAL(10, 2)
);
```

#### Mistake 3: Disorganized Schema
```sql
-- ❌ WRONG: Everything in one schema
CREATE TABLE dbo.Customers (...);
CREATE TABLE dbo.Products (...);
CREATE TABLE dbo.Analytics_Report_1 (...);
CREATE TABLE dbo.Analytics_Report_2 (...);

-- ✓ CORRECT: Organize by logical groups
CREATE SCHEMA sales;
CREATE SCHEMA analytics;
CREATE TABLE sales.Customers (...);
CREATE TABLE sales.Products (...);
CREATE TABLE analytics.Report_1 (...);
CREATE TABLE analytics.Report_2 (...);
```

### 11.2 Conceptual Mistakes

#### Mistake 4: Confusing Database and Server
```
❌ WRONG STATEMENT:
"Upload data to the server"
(Too vague - which database? which table?)

✓ CORRECT STATEMENT:
"Insert customer records into the Customers table
in the SalesDB database on the Production Server"
```

#### Mistake 5: Not Using Relationships
```sql
-- ❌ WRONG: Storing customer data in every order
CREATE TABLE Orders (
    OrderID INT,
    CustomerName VARCHAR(100),  -- Duplicates customer data
    CustomerEmail VARCHAR(100),
    Amount DECIMAL(10, 2)
);

-- ✓ CORRECT: Reference customer via foreign key
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT,  -- References Customers table
    Amount DECIMAL(10, 2),
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);
```

#### Mistake 6: Mixing Business Logic with Database
```sql
-- ❌ WRONG: Database only for storage
-- (Business logic implemented in application)

-- ✓ CORRECT: Use constraints and rules
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    Amount DECIMAL(10, 2) CHECK (Amount > 0),  -- Validate in database
    Status VARCHAR(20) CHECK (Status IN ('Pending', 'Shipped', 'Delivered')),
    OrderDate DATE DEFAULT GETDATE()  -- Default values in database
);
```

## 12. Real-World Use Cases

### 12.1 E-Commerce Platform

**Scenario**: Online retail company needs to manage customers, products, and orders.

**Database Design**:
```sql
CREATE SCHEMA ecommerce;

CREATE TABLE ecommerce.Customers (
    CustomerID INT PRIMARY KEY,
    FirstName VARCHAR(50),
    LastName VARCHAR(50),
    Email VARCHAR(100) UNIQUE,
    Phone CHAR(10),
    RegisterDate DATE,
    IsActive BOOLEAN DEFAULT TRUE
);

CREATE TABLE ecommerce.Products (
    ProductID INT PRIMARY KEY,
    ProductName VARCHAR(100),
    Description TEXT,
    Price DECIMAL(10, 2),
    StockQuantity INT,
    CategoryID INT
);

CREATE TABLE ecommerce.Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT NOT NULL,
    OrderDate DATE DEFAULT GETDATE(),
    TotalAmount DECIMAL(10, 2),
    Status VARCHAR(20) DEFAULT 'Pending',
    FOREIGN KEY (CustomerID) REFERENCES ecommerce.Customers(CustomerID)
);

CREATE TABLE ecommerce.OrderItems (
    OrderItemID INT PRIMARY KEY,
    OrderID INT NOT NULL,
    ProductID INT NOT NULL,
    Quantity INT,
    UnitPrice DECIMAL(10, 2),
    FOREIGN KEY (OrderID) REFERENCES ecommerce.Orders(OrderID),
    FOREIGN KEY (ProductID) REFERENCES ecommerce.Products(ProductID)
);
```

**Business Queries**:

    c.LastName,

-- Query: Find accounts with low balance (potential risk)
SELECT 
    a.AccountID,
    a.AccountType,
    a.Balance
FROM banking.Accounts a
WHERE a.Balance < 1000 AND a.IsActive = TRUE
ORDER BY a.Balance ASC;

-- Query: Total transactions in last 30 days
SELECT 
    a.AccountID,
    COUNT(t.TransactionID) AS TransactionCount,
    SUM(t.Amount) AS TotalAmount
FROM banking.Accounts a
LEFT JOIN banking.Transactions t ON a.AccountID = t.AccountID
    AND t.TransactionDate >= DATEADD(day, -30, GETDATE())
GROUP BY a.AccountID;

## 13. Best Practices

### 13.1 Database Design Best Practices

#### Practice 1: Always Define Primary Keys
```sql
-- ✓ GOOD
CREATE TABLE Customers (
    CustomerID INT PRIMARY KEY,
    Name VARCHAR(100) NOT NULL
);

-- Every table should have a unique identifier
```

#### Practice 2: Use Appropriate Data Types
```sql
-- ✓ GOOD
CREATE TABLE Products (
    ProductID INT,                    -- Integer for ID
    Name VARCHAR(100),                -- Variable string for text
    Price DECIMAL(10, 2),            -- Decimal for money
    CreatedDate DATE,                 -- DATE for dates only
    LastUpdated DATETIME,             -- DATETIME for timestamp
    IsActive BOOLEAN                  -- Boolean for flags
);
```

#### Practice 3: Implement NOT NULL and DEFAULT Constraints
```sql
-- ✓ GOOD
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT NOT NULL,          -- Cannot be empty
    OrderDate DATE DEFAULT GETDATE(), -- Automatic current date
    Status VARCHAR(20) DEFAULT 'Pending'
);
```

#### Practice 4: Use Foreign Keys for Relationships
```sql
-- ✓ GOOD
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT NOT NULL,
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);

-- Maintains referential integrity
-- Prevents orphaned records
```

#### Practice 5: Organize Tables in Schemas
```sql
-- ✓ GOOD
CREATE SCHEMA sales;
CREATE SCHEMA hr;
CREATE SCHEMA reporting;

CREATE TABLE sales.Orders (...);
CREATE TABLE sales.Customers (...);
CREATE TABLE hr.Employees (...);
CREATE TABLE reporting.Reports (...);
```

#### Practice 6: Use Meaningful Names
```sql
-- ✓ GOOD - Clear and descriptive
CustomerID, FirstName, OrderDate, IsActive

-- ❌ BAD - Unclear abbreviations
CID, FN, OD, IA
```

### 13.2 Query Best Practices

#### Practice 7: Write Readable Queries
```sql
-- ✓ GOOD - Well formatted
SELECT 
    c.CustomerID,
    c.FirstName,
    c.Email,
    COUNT(o.OrderID) AS TotalOrders
FROM Customers c
LEFT JOIN Orders o ON c.CustomerID = o.CustomerID
GROUP BY c.CustomerID, c.FirstName, c.Email
ORDER BY TotalOrders DESC;

-- ❌ BAD - Hard to read
SELECT c.CustomerID,c.FirstName,c.Email,COUNT(o.OrderID) AS TotalOrders FROM Customers c LEFT JOIN Orders o ON c.CustomerID=o.CustomerID GROUP BY c.CustomerID,c.FirstName,c.Email ORDER BY TotalOrders DESC;
```

#### Practice 8: Use Column Names Explicitly
```sql
-- ✓ GOOD
SELECT CustomerID, FirstName, Email FROM Customers;

-- ❌ BAD - SELECT * retrieves unnecessary columns
SELECT * FROM Customers;
```

#### Practice 9: Filter Early, Aggregate Late
```sql
-- ✓ GOOD - Filter before aggregating
SELECT 
    CategoryID,
    COUNT(*) AS ProductCount
FROM Products
WHERE IsActive = TRUE           -- Filter first
GROUP BY CategoryID;            -- Then aggregate

-- ❌ BAD - Aggregates then filters
SELECT 
    CategoryID,
    COUNT(*) AS ProductCount
FROM Products
GROUP BY CategoryID
HAVING COUNT(*) > 10;  -- Less efficient
```

#### Practice 10: Use Indexes for Large Tables
```sql
-- ✓ GOOD - Create indexes on frequently searched columns
CREATE INDEX idx_customer_email ON Customers(Email);
CREATE INDEX idx_order_date ON Orders(OrderDate);
CREATE INDEX idx_product_category ON Products(CategoryID);

-- Queries using these columns will be faster
SELECT * FROM Customers WHERE Email = 'maria@email.com';  -- Fast due to index
```

### 13.3 Security Best Practices

#### Practice 11: Use Parameterized Queries
```sql
-- ✓ GOOD - Prevents SQL injection
EXEC sp_executesql 
    N'SELECT * FROM Customers WHERE CustomerID = @ID',
    N'@ID INT',
    @ID = 1;

-- ❌ BAD - Vulnerable to SQL injection
EXEC ('SELECT * FROM Customers WHERE CustomerID = ' + @ID);
```

#### Practice 12: Implement Role-Based Access Control
```sql
-- ✓ GOOD - Different permissions for different roles
CREATE ROLE DataAnalyst;
GRANT SELECT ON Customers TO DataAnalyst;
GRANT SELECT ON Orders TO DataAnalyst;

CREATE ROLE DataAdmin;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES TO DataAdmin;
```

## 15. Summary

### 15.1 Key Takeaways

1. **SQL is Universal**
   - Standard language for all relational databases
   - Works on MySQL, MSSQL, PostgreSQL, Oracle, and more
   - Essential skill for any data professional

2. **Databases Solve Real Problems**
   - Handle massive data volumes (billions of rows)
   - Enable fast queries (milliseconds instead of hours)
   - Provide security and access control
   - Ensure data consistency and integrity

3. **DBMS and Servers Complete the Stack**
   - Database = Storage container
   - DBMS = Management software
   - Server = Physical hardware
   - SQL = Communication language

4. **Structure and Organization Matter**
   - Schemas organize related tables
   - Primary keys ensure data uniqueness
   - Foreign keys maintain relationships
   - Proper data types optimize storage and performance

5. **Relational Databases are Standard**
   - Tables with rows and columns
   - Relationships between tables
   - ACID properties for data reliability
   - Used by 90%+ of enterprises

|---------|-----------|---------|-----------|

### Common Errors and Solutions

| Error | Cause | Solution |
|-------|-------|----------|
| Table doesn't exist | Wrong table name | Check schema and table name: `SELECT * FROM schema.table` |
| Column doesn't exist | Wrong column name | Check column name spelling and case |
| Duplicate key value | Primary key violation | Ensure unique values or use different key |
| Foreign key constraint fails | Referenced record doesn't exist | Ensure parent record exists first |
| Data truncation | Value too long for column | Increase column size or use smaller value |
