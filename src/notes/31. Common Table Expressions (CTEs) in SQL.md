## 1. What Is a CTE?

A **Common Table Expression (CTE)** is a **temporary, named result set** — essentially a virtual table — that exists only for the duration of a single query. It is defined using the `WITH` clause and can be referenced one or more times within the same query, including in the main query and in other CTEs.

> **Key Definition:** A CTE is a named intermediate result set, scoped to the query in which it is defined. It does not persist after the query finishes.

### Anatomy of a CTE

```
WITH cte_name AS (
    -- CTE query (inner query)
    SELECT ...
    FROM   ...
    WHERE  ...
)
-- Main query (outer query)
SELECT *
FROM   cte_name;
```

The query inside the `WITH` block is called the **CTE query** (or inner query). The query that consumes its output is called the **main query** (or outer query).

---

## 2. Why Use CTEs?

CTEs solve several common problems that arise when writing complex SQL.

### 2.1 Breaking Down Complexity

Instead of writing one giant, hard-to-read query, CTEs let you decompose the problem into clearly named steps — each handling a single concern.

### 2.2 Readability

Each CTE is a self-contained, labelled block. A reader can understand what `cte_top_customers` does without deciphering a deeply nested subquery.

### 2.3 Modularity

Each CTE is responsible for one specific piece of logic. Adding or changing a step does not require restructuring the entire query.

### 2.4 Reusability (Key Advantage Over Subqueries)

The result of a CTE can be referenced **multiple times** in the main query. A subquery, by contrast, can only be used in the single position where it is written.

```sql
-- A subquery can only be used once:
SELECT *
FROM   (SELECT customer_id, SUM(sales) AS total_sales
        FROM   sales_orders
        GROUP  BY customer_id) AS sub
-- You cannot JOIN "sub" again elsewhere in this same query.

-- A CTE result can be used multiple times:
WITH total_sales AS (
    SELECT customer_id, SUM(sales) AS total_sales
    FROM   sales_orders
    GROUP  BY customer_id
)
SELECT c.customer_id,
       ts1.total_sales AS total_a,
       ts2.total_sales AS total_b   -- reused in two places
FROM   customers   c
JOIN   total_sales ts1 ON ts1.customer_id = c.customer_id
JOIN   total_sales ts2 ON ts2.customer_id = c.customer_id AND ts2.total_sales > 100;
```

### 2.5 Redundancy Reduction

Without CTEs, the same join or aggregation logic may have to be copy-pasted into multiple subqueries. A CTE lets you write that logic once and reuse it, reducing both code size and the risk of inconsistent changes.

## 3. How the Database Executes a CTE

Understanding execution order helps write correct, performant queries.

### Execution Steps

```
Step 1: Parse the full query.
Step 2: Identify the WITH clause → treat the CTE as high-priority.
Step 3: Execute the CTE query → store result in the cache (memory).
Step 4: Execute the main query.
         ├── Access physical tables from disk storage as needed.
         └── Access CTE results from the in-memory cache (fast).
Step 5: Return the final result to the client.
Step 6: Destroy the cached CTE result (cleanup).
```

### Why the Cache Matters

Retrieving data from the in-memory cache is significantly faster than reading from disk. When the main query references the same CTE multiple times, all subsequent reads come from the cache — not from disk — giving a real performance benefit for repeated access patterns.

> **Note:** In some database engines (e.g., PostgreSQL), the optimizer may choose to materialise the CTE result once (fence optimisation), or it may inline the CTE like a view depending on the version and query structure. MS SQL Server similarly decides whether to materialise or inline based on the execution plan.

## 5. Standalone CTE

### Definition

A **Standalone CTE** is a self-contained CTE that queries database tables directly. It does not depend on the result of any other CTE. It can be extracted and run independently.

### Sample Schema and Data

```sql
-- Table: customers
CREATE TABLE customers (
    customer_id   INT PRIMARY KEY,
    first_name    VARCHAR(50),
    last_name     VARCHAR(50)
);

-- Table: sales_orders
CREATE TABLE sales_orders (
    order_id    INT PRIMARY KEY,
    customer_id INT,
    sales       DECIMAL(10,2),
    order_date  DATE
);

-- Sample data
INSERT INTO customers VALUES (1,'Alice','Brown'),(2,'Bob','Smith'),
                             (3,'Carol','Jones'),(4,'Dan','White'),(5,'Eve','Black');

INSERT INTO sales_orders VALUES
  (101,1,120.00,'2024-01-05'),(102,2,45.00,'2024-01-10'),
  (103,3,200.00,'2024-02-01'),(104,1,80.00,'2024-02-14'),
  (105,3,60.00,'2024-03-01'),(106,2,30.00,'2024-03-15');
```

### Syntax

```sql
WITH cte_total_sales AS (
    -- CTE query: aggregates sales per customer
    SELECT   customer_id,
             SUM(sales) AS total_sales
    FROM     sales_orders
    GROUP BY customer_id
)
-- Main query: joins physical table with CTE result
SELECT   c.customer_id,
         c.first_name,
         c.last_name,
         cts.total_sales
FROM     customers         c
LEFT JOIN cte_total_sales  cts ON cts.customer_id = c.customer_id;
```

### Output

| customer_id | first_name | last_name | total_sales |
|-------------|------------|-----------|-------------|
| 1           | Alice      | Brown     | 200.00      |
| 2           | Bob        | Smith     | 75.00       |
| 3           | Carol      | Jones     | 260.00      |
| 4           | Dan        | White     | NULL        |
| 5           | Eve        | Black     | NULL        |

> Customer 4 and 5 return `NULL` because they have no orders in `sales_orders`. The `LEFT JOIN` preserves them.

### Restriction: No ORDER BY Inside a CTE

```sql
-- ❌ INVALID — ORDER BY is not allowed inside a CTE body
WITH cte_total_sales AS (
    SELECT   customer_id, SUM(sales) AS total_sales
    FROM     sales_orders
    GROUP BY customer_id
    ORDER BY customer_id  -- NOT ALLOWED
)
SELECT * FROM cte_total_sales;

-- ✅ VALID — ORDER BY belongs in the main query
WITH cte_total_sales AS (
    SELECT   customer_id, SUM(sales) AS total_sales
    FROM     sales_orders
    GROUP BY customer_id
)
SELECT * FROM cte_total_sales
ORDER BY customer_id;
```

## 6. Multiple Standalone CTEs

### Definition

Multiple CTEs are chained together under a single `WITH` clause. Each is separated by a comma. Only the **first** CTE uses the `WITH` keyword; all subsequent ones use only a comma.

### Syntax

```sql
WITH cte_one AS (
    SELECT ...
    FROM   table_a
),
cte_two AS (
    SELECT ...
    FROM   table_b
),
cte_three AS (
    SELECT ...
    FROM   table_c
)
SELECT ...
FROM   cte_one
JOIN   cte_two   ON ...
JOIN   cte_three ON ...;
```

### Extended Example: Customer Report (Step 1 + Step 2)

```sql
WITH cte_total_sales AS (
    -- Step 1: Total sales per customer
    SELECT   customer_id,
             SUM(sales) AS total_sales
    FROM     sales_orders
    GROUP BY customer_id
),
cte_last_order AS (
    -- Step 2: Most recent order date per customer
    SELECT   customer_id,
             MAX(order_date) AS last_order
    FROM     sales_orders
    GROUP BY customer_id
)
SELECT   c.customer_id,
         c.first_name,
         c.last_name,
         cts.total_sales,
         clo.last_order
FROM     customers        c
LEFT JOIN cte_total_sales cts ON cts.customer_id = c.customer_id
LEFT JOIN cte_last_order  clo ON clo.customer_id = c.customer_id
ORDER BY c.customer_id;
```

### Output

| customer_id | first_name | last_name | total_sales | last_order |
|-------------|------------|-----------|-------------|------------|
| 1           | Alice      | Brown     | 200.00      | 2024-02-14 |
| 2           | Bob        | Smith     | 75.00       | 2024-03-15 |
| 3           | Carol      | Jones     | 260.00      | 2024-03-01 |
| 4           | Dan        | White     | NULL        | NULL       |
| 5           | Eve        | Black     | NULL        | NULL       |

> **Reminder:** Never add a trailing comma after the last CTE definition and before the main query — that is a syntax error.

## 7. Nested CTE

### Definition

A **Nested CTE** is a CTE that references the result of a previously defined CTE instead of querying a physical table directly. It cannot be executed independently — it depends on the CTE(s) defined before it.

### Syntax

```sql
WITH cte_one AS (
    -- Standalone: reads from database
    SELECT ...
    FROM   physical_table
),
cte_two AS (
    -- Nested: reads from cte_one
    SELECT ...
    FROM   cte_one
    WHERE  ...
)
SELECT *
FROM   cte_two;
```

### Extended Example: Adding Customer Rank and Segment

```sql
WITH cte_total_sales AS (
    -- Step 1: Total sales per customer (Standalone)
    SELECT   customer_id,
             SUM(sales) AS total_sales
    FROM     sales_orders
    GROUP BY customer_id
),
cte_last_order AS (
    -- Step 2: Most recent order per customer (Standalone)
    SELECT   customer_id,
             MAX(order_date) AS last_order
    FROM     sales_orders
    GROUP BY customer_id
),
cte_customer_rank AS (
    -- Step 3: Rank customers by total sales (Nested — uses cte_total_sales)
    SELECT customer_id,
           total_sales,
           RANK() OVER (ORDER BY total_sales DESC) AS customer_rank
    FROM   cte_total_sales
),
cte_customer_segment AS (
    -- Step 4: Segment customers (Nested — uses cte_total_sales)
    SELECT customer_id,
           CASE
               WHEN total_sales > 100 THEN 'High'
               WHEN total_sales > 80  THEN 'Medium'
               ELSE                        'Low'
           END AS customer_segment
    FROM   cte_total_sales
)
-- Main query: assemble all steps
SELECT   c.customer_id,
         c.first_name,
         c.last_name,
         cts.total_sales,
         clo.last_order,
         ccr.customer_rank,
         ccs.customer_segment
FROM     customers            c
LEFT JOIN cte_total_sales     cts ON cts.customer_id = c.customer_id
LEFT JOIN cte_last_order      clo ON clo.customer_id = c.customer_id
LEFT JOIN cte_customer_rank   ccr ON ccr.customer_id = c.customer_id
LEFT JOIN cte_customer_segment ccs ON ccs.customer_id = c.customer_id
ORDER BY c.customer_id;
```

### Output

| customer_id | first_name | last_name | total_sales | last_order | customer_rank | customer_segment |
|-------------|------------|-----------|-------------|------------|---------------|-----------------|
| 1           | Alice      | Brown     | 200.00      | 2024-02-14 | 2             | High            |
| 2           | Bob        | Smith     | 75.00       | 2024-03-15 | 3             | Low             |
| 3           | Carol      | Jones     | 260.00      | 2024-03-01 | 1             | High            |
| 4           | Dan        | White     | NULL        | NULL       | NULL          | NULL            |
| 5           | Eve        | Black     | NULL        | NULL       | NULL          | NULL            |

> **Note on testing nested CTEs:** You cannot run a nested CTE in isolation — the CTE it depends on must also be included in the execution. When debugging, comment out the main query and include all upstream CTEs.

## 8. Recursive CTE

### Definition

A **Recursive CTE** is a CTE that **references itself**, executing repeatedly until a termination condition is met. It is primarily used for traversing hierarchical or graph-structured data (e.g., org charts, category trees, bill-of-materials).

> A non-recursive CTE executes exactly once. A recursive CTE loops — each iteration feeds its output into the next iteration — until no new rows are produced or a `WHERE` condition stops it.

### Structure

A recursive CTE always consists of exactly two parts connected by `UNION ALL`:

```sql
WITH RECURSIVE cte_name AS (
    -- Part 1: Anchor Query (runs once; provides the starting rows)
    SELECT ...
    FROM   physical_table
    WHERE  <initial condition>

    UNION ALL

    -- Part 2: Recursive Query (runs repeatedly; references cte_name itself)
    SELECT ...
    FROM   physical_table
    JOIN   cte_name ON <join condition>   -- self-reference
    WHERE  <termination condition>
)
SELECT * FROM cte_name;
```

| Part | Name | Runs | Purpose |
|------|------|------|---------|
| First SELECT | Anchor query | Once | Provides the seed/starting rows |
| Second SELECT | Recursive query | Repeatedly | Builds on previous iteration; stops when no rows match |

### Example 1: Generate a Number Sequence (1 to 20)

```sql
-- MS SQL Server / PostgreSQL
WITH num_sequence AS (
    -- Anchor: start at 1
    SELECT 1 AS my_number

    UNION ALL

    -- Recursive: add 1 each iteration; stop before 20
    SELECT my_number + 1
    FROM   num_sequence
    WHERE  my_number < 20
)
SELECT my_number FROM num_sequence;
```

#### Output

| my_number |
|-----------|
| 1         |
| 2         |
| ...       |
| 20        |

#### Controlling Maximum Recursion Depth (MS SQL Server)

```sql
-- Limit to 10 iterations
SELECT my_number FROM num_sequence
OPTION (MAXRECURSION 10);

-- Allow up to 5000 iterations
SELECT my_number FROM num_sequence
OPTION (MAXRECURSION 5000);

-- Unlimited (use with caution)
SELECT my_number FROM num_sequence
OPTION (MAXRECURSION 0);
```

> In PostgreSQL, use `SET max_recursion_depth` or rely on the query's `WHERE` clause. MySQL does not support `MAXRECURSION`; rely on the `WHERE` termination condition.

### Example 2: Employee Hierarchy Traversal

#### Schema and Data

```sql
CREATE TABLE employees (
    employee_id INT PRIMARY KEY,
    first_name  VARCHAR(50),
    department  VARCHAR(50),
    manager_id  INT NULL   -- NULL = top of the hierarchy (no manager)
);

INSERT INTO employees VALUES
  (1, 'Frank', 'Executive', NULL),
  (2, 'Kevin', 'Sales',     1),
  (3, 'Mary',  'Marketing', 1),
  (4, 'Michael','Sales',    2),
  (5, 'Carl',  'Marketing', 3);
```

> `manager_id` references `employee_id` within the same table — this self-referencing relationship signals a hierarchy.

#### Recursive CTE Query

```sql
-- MySQL uses WITH RECURSIVE; MS SQL Server / PostgreSQL accept plain WITH
WITH RECURSIVE employee_hierarchy AS (
    -- Anchor: the top manager (no manager above them)
    SELECT employee_id,
           first_name,
           manager_id,
           1 AS level
    FROM   employees
    WHERE  manager_id IS NULL

    UNION ALL

    -- Recursive: find direct reports of each row in the previous result
    SELECT e.employee_id,
           e.first_name,
           e.manager_id,
           eh.level + 1 AS level
    FROM   employees         e
    INNER JOIN employee_hierarchy eh ON e.manager_id = eh.employee_id
)
SELECT * FROM employee_hierarchy
ORDER BY level, employee_id;
```

#### Output

| employee_id | first_name | manager_id | level |
|-------------|------------|------------|-------|
| 1           | Frank      | NULL       | 1     |
| 2           | Kevin      | 1          | 2     |
| 3           | Mary       | 1          | 2     |
| 4           | Michael    | 2          | 3     |
| 5           | Carl       | 3          | 3     |

#### Organisation Chart (Visual)

```
Level 1:          Frank (CEO)
                  /          \
Level 2:       Kevin        Mary
               /                \
Level 3:    Michael             Carl
```

#### Execution Flow (Step-by-Step)

```
Iteration 0 — Anchor Query:
  SELECT WHERE manager_id IS NULL → returns Frank (level 1)
  CTE result so far: [Frank, level=1]

Iteration 1 — Recursive Query:
  JOIN employees ON manager_id = 1
  → returns Kevin (level=2), Mary (level=2)
  CTE result so far: [Frank, Kevin, Mary]

Iteration 2 — Recursive Query:
  JOIN employees ON manager_id IN (2, 3)
  → returns Michael (level=3), Carl (level=3)
  CTE result so far: [Frank, Kevin, Mary, Michael, Carl]

Iteration 3 — Recursive Query:
  JOIN employees ON manager_id IN (4, 5)
  → no rows found → loop terminates

Main query executes on the accumulated CTE result.
```

## 9. CTEs vs. Subqueries

| Feature | Subquery | CTE |
|---|---|---|
| Writing direction | Bottom-up (inner → outer) | Top-down (definition → usage) |
| Reusability | Used in **one place only** | Can be referenced **multiple times** |
| Readability | Decreases with nesting | Improves with named steps |
| Independence | Embedded inside the outer query | Defined separately before the main query |
| Recursive capability | No | Yes (recursive CTE) |
| Debugging | Must extract manually | Can run each CTE block in isolation (Standalone only) |
| Redundancy | May require duplication | Single definition, multiple uses |

## 10. Dialect Comparison: MySQL vs. MS SQL Server vs. PostgreSQL

### 10.1 Basic CTE Syntax

| Feature | MySQL 8.0+ | MS SQL Server | PostgreSQL |
|---|---|---|---|
| CTE support | ✅ (since 8.0) | ✅ | ✅ |
| Keyword for recursive | `WITH RECURSIVE` | `WITH` | `WITH RECURSIVE` |
| `ORDER BY` inside CTE | ❌ Not allowed | ❌ Not allowed | ❌ Not allowed |
| Multiple CTEs | Comma-separated | Comma-separated | Comma-separated |

### 10.2 Recursive CTE Syntax Differences

```sql
-- MySQL 8.0+ (requires RECURSIVE keyword)
WITH RECURSIVE cte AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1 FROM cte WHERE n < 10
)
SELECT n FROM cte;

-- MS SQL Server (no RECURSIVE keyword needed)
WITH cte AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1 FROM cte WHERE n < 10
)
SELECT n FROM cte
OPTION (MAXRECURSION 100);  -- MS SQL Server specific

-- PostgreSQL (requires RECURSIVE keyword)
WITH RECURSIVE cte AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1 FROM cte WHERE n < 10
)
SELECT n FROM cte;
```

### 10.3 Recursion Depth Control

| Database | Default Max Recursion | How to Change |
|---|---|---|
| MySQL | 1000 | `SET cte_max_recursion_depth = 5000;` |
| MS SQL Server | 100 | `OPTION (MAXRECURSION n)` at end of main query; `0` = unlimited |
| PostgreSQL | No hard limit | Relies on `WHERE` termination condition; `SET max_stack_depth` for memory |

```sql
-- MySQL: change recursion depth
SET SESSION cte_max_recursion_depth = 5000;

-- MS SQL Server: set in query
SELECT * FROM my_cte OPTION (MAXRECURSION 5000);

-- PostgreSQL: no special syntax needed; WHERE clause controls termination
```

### 10.4 CTE Materialisation Behaviour

| Database | Behaviour |
|---|---|
| MySQL 8.0+ | CTEs are always materialised (executed once, stored) |
| MS SQL Server | Optimizer decides to inline or materialise based on execution plan |
| PostgreSQL < 12 | CTEs always materialised (optimisation fence) |
| PostgreSQL ≥ 12 | Optimizer may inline simple CTEs; use `WITH ... AS MATERIALIZED` or `AS NOT MATERIALIZED` to control |

```sql
-- PostgreSQL 12+: force materialisation
WITH cte AS MATERIALIZED (
    SELECT * FROM large_table WHERE status = 'active'
)
SELECT * FROM cte;

-- PostgreSQL 12+: force inlining (let optimizer treat it as a view)
WITH cte AS NOT MATERIALIZED (
    SELECT * FROM large_table WHERE status = 'active'
)
SELECT * FROM cte;
```

## 11. Common Mistakes

### 11.1 Using `WITH` Instead of a Comma for Additional CTEs

```sql
-- ❌ WRONG
WITH cte_one AS (SELECT ...)
WITH cte_two AS (SELECT ...)   -- second WITH is a syntax error
SELECT * FROM cte_two;

-- ✅ CORRECT
WITH cte_one AS (SELECT ...),
     cte_two AS (SELECT ...)
SELECT * FROM cte_two;
```

### 11.2 Trailing Comma After the Last CTE

```sql
-- ❌ WRONG
WITH cte_one AS (SELECT ...),
     cte_two AS (SELECT ...),   -- trailing comma before main query
SELECT * FROM cte_two;

-- ✅ CORRECT
WITH cte_one AS (SELECT ...),
     cte_two AS (SELECT ...)
SELECT * FROM cte_two;
```

### 11.3 Using ORDER BY Inside a CTE

```sql
-- ❌ WRONG
WITH ranked AS (
    SELECT *, RANK() OVER (ORDER BY sales DESC) AS rnk
    FROM   sales_orders
    ORDER  BY rnk    -- not allowed inside CTE
)
SELECT * FROM ranked;

-- ✅ CORRECT
WITH ranked AS (
    SELECT *, RANK() OVER (ORDER BY sales DESC) AS rnk
    FROM   sales_orders
)
SELECT * FROM ranked ORDER BY rnk;
```

### 11.4 Referencing a CTE from Outside Its Query

```sql
-- ❌ WRONG — CTEs are not globally visible like physical tables
WITH cte_sales AS (SELECT customer_id, SUM(sales) AS total FROM sales_orders GROUP BY customer_id)
SELECT * FROM cte_sales;   -- first query

SELECT * FROM cte_sales;   -- ❌ second query — cte_sales no longer exists
```

### 11.5 Forgetting `UNION ALL` in a Recursive CTE

```sql
-- ❌ WRONG — UNION removes duplicates and may suppress recursive rows
WITH RECURSIVE cte AS (
    SELECT 1 AS n
    UNION        -- should be UNION ALL
    SELECT n + 1 FROM cte WHERE n < 5
)
SELECT n FROM cte;

-- ✅ CORRECT
WITH RECURSIVE cte AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1 FROM cte WHERE n < 5
)
SELECT n FROM cte;
```

### 11.6 No Termination Condition in Recursive CTE

```sql
-- ❌ DANGEROUS — infinite loop (database will hit max recursion limit and error)
WITH RECURSIVE cte AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1 FROM cte   -- no WHERE clause to stop the loop
)
SELECT n FROM cte;

-- ✅ CORRECT
WITH RECURSIVE cte AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1 FROM cte WHERE n < 100
)
SELECT n FROM cte;
```

### 11.7 Over-nesting CTEs (Readability Anti-Pattern)

Avoid creating a new CTE for every small calculation. More than 5 CTEs in a single query quickly becomes difficult to understand, even when each block is individually clear.

---

## 12. Best Practices

**Name CTEs descriptively.** `cte_top_customers_by_revenue` is far better than `cte1`.

**Limit CTE count to 3–5 per query.** Beyond that, consider splitting the work into multiple queries or a staging table.

**Consolidate where possible.** Before adding a new CTE, ask whether it can be merged into an existing one without sacrificing clarity.

**Use Standalone CTEs for reusable logic.** If the same aggregation is needed in two places, define it once as a CTE and reference it twice.

**Test CTEs incrementally.** For Standalone CTEs, run them in isolation to verify intermediate results before assembling the full query.

**Avoid ORDER BY inside CTE bodies.** Sorting belongs in the main query.

**Be explicit about JOIN types in the main query.** Use `LEFT JOIN` when you want to keep customers with no orders; use `INNER JOIN` when only matched rows are relevant.

**Use `UNION ALL` (not `UNION`) in recursive CTEs** unless you specifically need deduplication — `UNION` is slower and may suppress valid recursive rows.

**Always define a termination condition** in recursive CTEs to prevent infinite loops.

**In PostgreSQL 12+**, use `AS MATERIALIZED` explicitly if you need the CTE to act as an optimisation fence (preventing the optimizer from pushing predicates inside the CTE).

## 13. Real-World Analytics Use Cases

### 13.1 Customer Lifetime Value Report

```sql
WITH cte_spend AS (
    SELECT   customer_id,
             SUM(sales)   AS total_spend,
             COUNT(*)     AS order_count,
             MIN(order_date) AS first_order,
             MAX(order_date) AS last_order
    FROM     sales_orders
    GROUP BY customer_id
),
cte_segment AS (
    SELECT customer_id,
           total_spend,
           CASE
               WHEN total_spend >= 500 THEN 'Platinum'
               WHEN total_spend >= 200 THEN 'Gold'
               WHEN total_spend >= 100 THEN 'Silver'
               ELSE                         'Bronze'
           END AS tier
    FROM   cte_spend
)
SELECT   c.first_name,
         c.last_name,
         cs.total_spend,
         cs.tier
FROM     customers   c
JOIN     cte_segment cs ON cs.customer_id = c.customer_id
ORDER BY cs.total_spend DESC;
```

### 13.2 Rolling 7-Day Revenue Window

```sql
WITH daily_revenue AS (
    SELECT   order_date,
             SUM(sales) AS daily_sales
    FROM     sales_orders
    GROUP BY order_date
)
SELECT order_date,
       daily_sales,
       SUM(daily_sales) OVER (
           ORDER BY order_date
           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ) AS rolling_7day
FROM   daily_revenue
ORDER BY order_date;
```

### 13.3 Removing Duplicates Before Aggregation

```sql
WITH deduped AS (
    SELECT DISTINCT customer_id, order_id, sales
    FROM   sales_orders
    WHERE  sales > 0
)
SELECT   customer_id,
         SUM(sales) AS clean_total
FROM     deduped
GROUP BY customer_id;
```

### 13.4 Hierarchical Data — Full Org Tree with Path

```sql
WITH RECURSIVE org_tree AS (
    SELECT employee_id,
           first_name,
           manager_id,
           CAST(first_name AS VARCHAR(500)) AS path,
           1 AS level
    FROM   employees
    WHERE  manager_id IS NULL

    UNION ALL

    SELECT e.employee_id,
           e.first_name,
           e.manager_id,
           CONCAT(ot.path, ' → ', e.first_name) AS path,
           ot.level + 1
    FROM   employees  e
    JOIN   org_tree   ot ON e.manager_id = ot.employee_id
)
SELECT employee_id, first_name, level, path
FROM   org_tree
ORDER BY level, employee_id;
```

#### Output

| employee_id | first_name | level | path |
|---|---|---|---|
| 1 | Frank | 1 | Frank |
| 2 | Kevin | 2 | Frank → Kevin |
| 3 | Mary | 2 | Frank → Mary |
| 4 | Michael | 3 | Frank → Kevin → Michael |
| 5 | Carl | 3 | Frank → Mary → Carl |

### 13.5 Year-over-Year Sales Comparison

```sql
WITH yearly_sales AS (
    SELECT   YEAR(order_date)  AS yr,
             SUM(sales)        AS total_sales
    FROM     sales_orders
    GROUP BY YEAR(order_date)
)
SELECT cur.yr,
       cur.total_sales                                           AS current_year,
       prev.total_sales                                          AS prior_year,
       ROUND(100.0 * (cur.total_sales - prev.total_sales)
             / NULLIF(prev.total_sales, 0), 2)                  AS yoy_pct_change
FROM   yearly_sales  cur
LEFT JOIN yearly_sales prev ON prev.yr = cur.yr - 1
ORDER BY cur.yr;
```

## 14. Summary Table

| Dimension | Standalone CTE | Nested CTE | Recursive CTE |
|---|---|---|---|
| **Reads from** | Physical database tables | Another CTE's result | Itself (self-reference) |
| **Can run independently?** | ✅ Yes | ❌ No (depends on upstream CTEs) | ❌ No (requires anchor + recursive parts together) |
| **Execution count** | Once | Once | Multiple times (loops until condition met) |
| **Main use case** | Intermediate aggregation, filtering | Chained multi-step logic, reuse of upstream CTEs | Hierarchies, trees, graphs, sequences |
| **Reusable in main query?** | ✅ Yes (multiple JOINs) | ✅ Yes | ✅ Yes |
| **Termination condition required?** | No | No | ✅ Yes (mandatory) |
| **MySQL syntax note** | `WITH` | `WITH` | `WITH RECURSIVE` |
| **MS SQL Server syntax note** | `WITH` | `WITH` | `WITH` + `OPTION (MAXRECURSION n)` |
| **PostgreSQL syntax note** | `WITH` | `WITH` | `WITH RECURSIVE`; `AS MATERIALIZED` option ≥ v12 |
