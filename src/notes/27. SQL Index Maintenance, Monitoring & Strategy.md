## 1. Why Index Maintenance Matters

Creating indexes is only the beginning. Over time, indexes **degrade** through:

- **Fragmentation** — inserts, updates, and deletes cause gaps and out-of-order pages in the B-Tree
- **Stale statistics** — the query optimizer makes wrong decisions because it believes row counts / distributions that no longer reflect reality
- **Unused indexes** — created for a feature or query that no longer exists, silently slowing down every write
- **Duplicate indexes** — multiple developers unknowingly indexing the same column, wasting storage and write performance
- **Missing indexes** — queries that perform full table scans because the right index was never created

> 💡 **Analogy:** An index is like a car. You can build a perfect car, but without regular oil changes, tire checks, and tune-ups — it degrades. The same applies to database indexes. Creating them is the easy part; maintaining them is what separates good DBAs and developers from great ones.

The five maintenance tasks covered in this chapter:

| # | Task | Purpose |
|---|---|---|
| 1 | Monitor Index Usage | Find unused indexes to drop |
| 2 | Identify Missing Indexes | Find query patterns that need new indexes |
| 3 | Find Duplicate Indexes | Eliminate redundant indexes on same columns |
| 4 | Update Statistics | Keep the query optimizer's data accurate |
| 5 | Monitor Fragmentation | Reorganize or rebuild degraded indexes |

## 2. Task 1 — Monitor Index Usage

### Why This Matters

If an index is never used by any query, it is:
- **Wasting storage** — the B-Tree occupies disk space for no benefit
- **Slowing writes** — every INSERT/UPDATE/DELETE still updates the index, paying the cost without gaining the read benefit

> 💡 **Pro tip from the field:** In 15+ years of SQL work, approximately **90% of indexes created in real projects are totally unused**. Auditing and dropping them on day one of a new project is one of the fastest wins you can deliver.

---

### Step 1A — Quick Index Overview with `sp_helpindex`

```sql
-- List all indexes on a specific table
EXEC sp_helpindex 'sales.db_customers';
```

**Sample Output:**

| index_name | index_description | index_keys |
|---|---|---|
| idx_db_customers_customer_id | clustered, unique | customer_id |
| idx_db_customers_country | nonclustered | country |
| idx_db_customers_first_name | nonclustered | first_name |
| idx_db_customers_cs | nonclustered columnstore | (no keys) |

---

### Step 1B — Full Index Inventory Query

```sql
SELECT
    tbl.name                    AS table_name,
    idx.name                    AS index_name,
    idx.type_desc               AS index_type,
    idx.is_primary_key          AS is_primary_key,
    idx.is_unique               AS is_unique,
    idx.is_disabled             AS is_disabled
FROM sys.indexes idx
INNER JOIN sys.tables tbl
    ON idx.object_id = tbl.object_id
WHERE idx.type > 0   -- exclude heaps (type = 0)
ORDER BY tbl.name, idx.name;
```

---

### Step 1C — Usage Statistics with Dynamic Management Views (DMV)

SQL Server tracks usage statistics for every index in memory via `sys.dm_db_index_usage_stats`.

```sql
SELECT
    tbl.name                        AS table_name,
    idx.name                        AS index_name,
    idx.type_desc                   AS index_type,
    idx.is_unique                   AS is_unique,
    s.user_seeks                    AS user_seeks,
    s.user_scans                    AS user_scans,
    s.user_lookups                  AS user_lookups,
    s.user_updates                  AS user_updates,
    COALESCE(
        s.last_user_seek,
        s.last_user_scan
    )                               AS last_used
FROM sys.indexes idx
INNER JOIN sys.tables tbl
    ON idx.object_id = tbl.object_id
LEFT JOIN sys.dm_db_index_usage_stats s
    ON idx.object_id = s.object_id
    AND idx.index_id = s.index_id
    AND s.database_id = DB_ID()   -- current database only
WHERE idx.type > 0
ORDER BY tbl.name, idx.name;
```

**Reading the Output:**

| Column | Meaning |
|---|---|
| `user_seeks` | How many times a B-Tree seek was used (fastest — ideal) |
| `user_scans` | How many times the index was scanned fully or partially |
| `user_lookups` | How many key lookups happened via this index |
| `user_updates` | How many times the index was updated (INSERT/UPDATE/DELETE) |
| `last_used` | Timestamp of last read usage |

**Interpretation:**

```
user_seeks = 0, user_scans = 0, user_lookups = 0, user_updates = 50
→ Index is never READ but costs writes on every data change → CANDIDATE FOR DROPPING

user_seeks = 1500, user_updates = 50
→ Index is heavily read, low write cost → KEEP

All nulls (LEFT JOIN found no row in DMV)
→ Index has never been used since last server restart → INVESTIGATE
```

> ⚠️ **Important:** These statistics are **in-memory only** — they reset when SQL Server restarts. In production environments, statistics accumulate for days or weeks and reflect realistic usage. In local development, they reset frequently.

---

### Step 1D — Find Completely Unused Indexes

```sql
-- Indexes that have ZERO reads since last restart
SELECT
    tbl.name    AS table_name,
    idx.name    AS index_name,
    idx.type_desc AS index_type,
    s.user_updates AS write_cost_with_no_reads
FROM sys.indexes idx
INNER JOIN sys.tables tbl
    ON idx.object_id = tbl.object_id
LEFT JOIN sys.dm_db_index_usage_stats s
    ON idx.object_id = s.object_id
    AND idx.index_id = s.index_id
    AND s.database_id = DB_ID()
WHERE idx.type > 0
  AND idx.is_primary_key = 0      -- don't drop primary keys
  AND COALESCE(s.user_seeks, 0) = 0
  AND COALESCE(s.user_scans, 0) = 0
  AND COALESCE(s.user_lookups, 0) = 0
ORDER BY COALESCE(s.user_updates, 0) DESC; -- worst offenders first
```

## 3. Task 2 — Identify Missing Indexes

### Why This Matters

SQL Server's query optimizer, while executing queries, detects when a missing index would have significantly improved performance. It logs these **recommendations** in memory via a Dynamic Management View. This gives you database-driven guidance on which indexes to create next.

---

### Missing Index Recommendations Query

```sql
SELECT
    mid.statement                           AS table_name,
    mid.equality_columns                    AS equality_filter_cols,
    mid.inequality_columns                  AS range_filter_cols,
    mid.included_columns                    AS included_cols,
    migs.avg_total_user_cost * migs.avg_user_impact * (migs.user_seeks + migs.user_scans)
                                            AS estimated_improvement_score,
    migs.user_seeks                         AS times_seek_would_have_used_it,
    migs.avg_user_impact                    AS pct_improvement_estimated,
    migs.last_user_seek                     AS last_time_needed
FROM sys.dm_db_missing_index_details mid
INNER JOIN sys.dm_db_missing_index_groups mig
    ON mid.index_handle = mig.index_handle
INNER JOIN sys.dm_db_missing_index_group_stats migs
    ON mig.index_group_handle = migs.group_handle
ORDER BY estimated_improvement_score DESC;
```

**Sample Output:**

| table_name | equality_filter_cols | range_filter_cols | estimated_improvement_score |
|---|---|---|---|
| `[dbo].[DimProduct]` | `[Color]` | NULL | 8540 |
| `[dbo].[FactInternetSales]` | `[ProductKey]` | NULL | 6210 |
| `[dbo].[FactInternetSales]` | NULL | `[OrderDateKey]` | 4890 |

**Understanding the columns:**

| Column | Meaning |
|---|---|
| `equality_columns` | Columns used in `WHERE col = value` conditions |
| `inequality_columns` | Columns used in `WHERE col > value` or `BETWEEN` range conditions |
| `included_columns` | Columns that appear in `SELECT` but not `WHERE` — can be added as non-key includes |
| `estimated_improvement_score` | Higher = more benefit from creating this index |
| `avg_user_impact` | Estimated % query cost reduction if index existed |

---

### Using the Recommendations

```sql
-- The DMV suggested this for DimProduct.Color:
CREATE NONCLUSTERED INDEX idx_dim_product_color
ON DimProduct (Color)
INCLUDE (ProductName, ListPrice);
-- INCLUDE adds columns to the leaf level without being part of the key

-- The DMV suggested this for FactInternetSales.ProductKey (join column):
CREATE NONCLUSTERED INDEX idx_fact_internet_sales_product_key
ON FactInternetSales (ProductKey);
```

> ⚠️ **Warning:** Do NOT blindly create an index for every recommendation. Ask:
> 1. Is this query run frequently enough to justify the write overhead?
> 2. Does a similar index already exist that could be extended?
> 3. Is the `estimated_improvement_score` significant enough?
> 4. Will this new index create a duplicate of an existing one?

## 4. Task 3 — Find Duplicate Indexes

### Why This Matters

In team environments, different developers often create indexes on the same columns independently. Duplicate indexes mean:
- Double the write overhead on that column (every data change updates both indexes)
- Double the storage for the B-Trees
- Confusing maintenance — which one do you keep?

---

### Duplicate Index Detection Query

```sql
SELECT
    tbl.name            AS table_name,
    idx.name            AS index_name,
    idx.type_desc       AS index_type,
    col.name            AS column_name,
    ic.key_ordinal      AS key_position,
    COUNT(*) OVER (
        PARTITION BY tbl.name, col.name
    )                   AS indexes_on_this_column  -- > 1 means duplicate
FROM sys.indexes idx
INNER JOIN sys.tables tbl
    ON idx.object_id = tbl.object_id
INNER JOIN sys.index_columns ic
    ON idx.object_id = ic.object_id
    AND idx.index_id = ic.index_id
INNER JOIN sys.columns col
    ON ic.object_id = col.object_id
    AND ic.column_id = col.column_id
WHERE idx.type > 0
ORDER BY tbl.name, col.name, indexes_on_this_column DESC;
```

**Sample Output:**

| table_name | index_name | index_type | column_name | indexes_on_this_column |
|---|---|---|---|---|
| customers | idx_customers_country | nonclustered | country | **2** ← duplicate! |
| customers | idx_customers_cs | nonclustered columnstore | country | **2** ← duplicate! |
| customers | idx_customers_first_name | nonclustered | first_name | **2** ← duplicate! |
| customers | idx_customers_name_cs | nonclustered columnstore | first_name | **2** ← duplicate! |
| products | idx_products_category | nonclustered | category | 1 |

**Filtering only duplicates:**

```sql
-- Add HAVING or filter on the window function result
SELECT *
FROM (
    SELECT
        tbl.name        AS table_name,
        idx.name        AS index_name,
        idx.type_desc   AS index_type,
        col.name        AS column_name,
        COUNT(*) OVER (PARTITION BY tbl.name, col.name) AS duplicate_count
    FROM sys.indexes idx
    INNER JOIN sys.tables tbl ON idx.object_id = tbl.object_id
    INNER JOIN sys.index_columns ic ON idx.object_id = ic.object_id AND idx.index_id = ic.index_id
    INNER JOIN sys.columns col ON ic.object_id = col.object_id AND ic.column_id = col.column_id
    WHERE idx.type > 0
) base
WHERE duplicate_count > 1
ORDER BY table_name, column_name;
```

## 5. Task 4 — Update Statistics

### What Are Statistics?

When SQL Server executes a query, it needs to build an **execution plan** — a roadmap for how to retrieve data most efficiently. To do this, it evaluates multiple options:
- Table scan
- Index scan
- Index seek

To choose between them, the optimizer reads **statistics** — metadata about the table that describes:
- Total row count
- Value distribution (histogram) in each column
- Number of distinct values

```
Query: SELECT * FROM customers WHERE status = 'active'

Without up-to-date statistics:
  Optimizer reads stats: "this table has 50 rows"
  Decision: "small table, just do a full scan, no need for index"
  Reality: table now has 1,000,000 rows → full scan is catastrophic

With up-to-date statistics:
  Optimizer reads stats: "this table has 1,000,000 rows, 70% are active"
  Decision: "large table, use the index on status for 700,000 rows"
  Reality: correct decision, fast query
```

**Statistics go stale when:**
- Large bulk data loads happen (migration, ETL pipelines)
- Frequent INSERTs/UPDATEs/DELETEs accumulate
- The server was restarted (stats persist, but new data changes them)

---

### Monitor Statistics Health

```sql
SELECT
    tbl.name                AS table_name,
    st.name                 AS statistics_name,
    sp.last_updated         AS last_updated,
    sp.rows                 AS total_rows,
    sp.rows_sampled         AS rows_sampled,
    sp.modification_counter AS modifications_since_last_update
FROM sys.stats st
INNER JOIN sys.tables tbl
    ON st.object_id = tbl.object_id
CROSS APPLY sys.dm_db_stats_properties(st.object_id, st.stats_id) sp
ORDER BY sp.modification_counter DESC; -- worst offenders first
```

**Sample Output:**

| table_name | statistics_name | last_updated | total_rows | modifications_since_last_update |
|---|---|---|---|---|
| db_customers | _WA_Sys_status | 2024-10-19 | 5 | 15 |
| customers | idx_customers_email | 2024-10-23 | 1000000 | 0 |

**Interpretation:**

| `modifications_since_last_update` | Action |
|---|---|
| 0 | ✅ Up to date — no action needed |
| Low (< 5% of rows) | ✅ Usually fine |
| Moderate (5–20% of rows) | ⚠️ Consider updating |
| High (> 20% of rows) | ❌ Update immediately |

---

### Update Statistics — Three Levels

**Level 1 — Update one specific statistic:**

```sql
UPDATE STATISTICS sales.db_customers
    [_WA_Sys_00000003_6383C8BA];   -- specific statistics name
```

**Level 2 — Update all statistics on one table:**

```sql
UPDATE STATISTICS sales.db_customers;
-- Automatically updates all statistics (table-level + all indexes) for this table
```

**Level 3 — Update all statistics in the entire database:**

```sql
EXEC sp_updatestats;
-- Updates all out-of-date statistics across every table in the current database
-- SQL Server skips tables where modification_counter is zero (smart, no waste)
```

> 💡 **Production strategy:** Schedule `sp_updatestats` as a nightly or weekly SQL Server Agent job. Run it immediately after large data migrations or ETL loads.

## 6. Task 5 — Monitor & Fix Index Fragmentation

### What is Fragmentation?

As data is inserted, updated, and deleted over time, the **physical order of data in B-Tree pages** diverges from the **logical order**. Pages develop:
- **Internal fragmentation** — gaps within pages (pages are partially empty after deletes)
- **External fragmentation** — pages are out of logical order (page 5 comes before page 3 in the B-Tree but physically sits elsewhere on disk)

Result: SQL Server must read more pages and do more disk seeks to retrieve data, slowing down queries.

---

### Check Fragmentation Levels

```sql
SELECT
    tbl.name                            AS table_name,
    idx.name                            AS index_name,
    idx.type_desc                       AS index_type,
    ips.avg_fragmentation_in_percent    AS fragmentation_pct,
    ips.page_count                      AS page_count
FROM sys.dm_db_index_physical_stats(
    DB_ID(),    -- current database
    NULL,       -- all tables
    NULL,       -- all indexes
    NULL,       -- all partitions
    'LIMITED'   -- scan mode: LIMITED is fast; DETAILED is slow but thorough
) ips
INNER JOIN sys.indexes idx
    ON ips.object_id = idx.object_id
    AND ips.index_id = idx.index_id
INNER JOIN sys.tables tbl
    ON idx.object_id = tbl.object_id
WHERE ips.avg_fragmentation_in_percent > 0
ORDER BY ips.avg_fragmentation_in_percent DESC;
```

---

### Interpreting Fragmentation Percentage

| Fragmentation % | Health | Recommended Action |
|---|---|---|
| 0% – 10% | ✅ Healthy | No action needed |
| 10% – 30% | ⚠️ Moderate | **REORGANIZE** the index |
| > 30% | ❌ Severely fragmented | **REBUILD** the index |

---

### Fix Option 1 — REORGANIZE (Light Maintenance)

`REORGANIZE` defragments the **leaf level** of the B-Tree in-place, compacting pages and restoring logical order. It is:
- **Online** — the table remains accessible to users during the operation
- **Interruptible** — can be stopped and progress is kept
- **Light** — suitable for 10–30% fragmentation

```sql
-- Reorganize a single index
ALTER INDEX idx_customers_country
ON sales.customers
REORGANIZE;

-- Reorganize all indexes on a table
ALTER INDEX ALL
ON sales.customers
REORGANIZE;
```

---

### Fix Option 2 — REBUILD (Heavy Maintenance)

`REBUILD` **drops the entire index and recreates it from scratch**. This:
- Eliminates all fragmentation completely (internal + external)
- Updates statistics automatically as part of the rebuild
- Is **offline** by default (table locked) — use `ONLINE = ON` to allow concurrent access (Enterprise Edition)
- Is a **heavyweight** operation — takes longer, uses more I/O

```sql
-- Rebuild a single index (offline)
ALTER INDEX idx_customers_country
ON sales.customers
REBUILD;

-- Rebuild with online access (Enterprise Edition only)
ALTER INDEX idx_customers_country
ON sales.customers
REBUILD WITH (ONLINE = ON);

-- Rebuild all indexes on a table
ALTER INDEX ALL
ON sales.customers
REBUILD;
```

### REORGANIZE vs REBUILD — Decision Table

| Feature | REORGANIZE | REBUILD |
|---|---|---|
| Fragmentation fixed | Leaf level only | Full index (all levels) |
| Table access during operation | ✅ Online (always) | ⚠️ Offline by default |
| Updates statistics | ❌ No | ✅ Yes |
| Speed | ✅ Fast | ⚠️ Slower |
| Storage used during operation | ✅ Minimal | ⚠️ Temporary extra space |
| Recommended for | 10%–30% fragmentation | > 30% fragmentation |

---

### Automated Fragmentation Maintenance Script

```sql
-- Dynamically reorganize or rebuild based on fragmentation level
DECLARE @table_name   NVARCHAR(200);
DECLARE @index_name   NVARCHAR(200);
DECLARE @frag_pct     FLOAT;
DECLARE @sql          NVARCHAR(500);

DECLARE frag_cursor CURSOR FOR
    SELECT
        tbl.name,
        idx.name,
        ips.avg_fragmentation_in_percent
    FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, 'LIMITED') ips
    INNER JOIN sys.indexes idx ON ips.object_id = idx.object_id AND ips.index_id = idx.index_id
    INNER JOIN sys.tables tbl ON idx.object_id = tbl.object_id
    WHERE ips.avg_fragmentation_in_percent > 10
      AND ips.page_count > 100;   -- only act on indexes with substantial pages

OPEN frag_cursor;
FETCH NEXT FROM frag_cursor INTO @table_name, @index_name, @frag_pct;

WHILE @@FETCH_STATUS = 0
BEGIN
    IF @frag_pct BETWEEN 10 AND 30
        SET @sql = 'ALTER INDEX [' + @index_name + '] ON [' + @table_name + '] REORGANIZE';
    ELSE IF @frag_pct > 30
        SET @sql = 'ALTER INDEX [' + @index_name + '] ON [' + @table_name + '] REBUILD';

    EXEC sp_executesql @sql;
    PRINT 'Processed: ' + @table_name + '.' + @index_name + ' (' + CAST(@frag_pct AS VARCHAR) + '%)';

    FETCH NEXT FROM frag_cursor INTO @table_name, @index_name, @frag_pct;
END

CLOSE frag_cursor;
DEALLOCATE frag_cursor;
```

## 7. Choosing the Right Index Type — Decision Guide

### Index Type Selection Framework

```
Does the table need any index at all?
│
├── NO: Temporary/staging tables, write-only log tables, small tables
│   → Use HEAP (no index) — fastest writes, no overhead
│
└── YES ↓

Is this column the PRIMARY KEY?
│
├── YES → Use CLUSTERED (ROW STORE) INDEX
│         Primary keys: unique, never updated → perfect clustered index candidate
│
└── NO ↓

Is this table used primarily for ANALYTICS / aggregations on large data?
│
├── YES → Use CLUSTERED COLUMNSTORE INDEX
│         Data warehouse, BI reports, Power BI, large aggregations
│
└── NO ↓

Is this a non-primary-key column used in WHERE / JOIN / ORDER BY?
│
├── YES → Use NON-CLUSTERED (ROW STORE) INDEX
│         Foreign keys, join columns, search columns
│
└── Is your query always filtering on a specific subset?
    │
    ├── YES → Use FILTERED INDEX
    │         WHERE status = 'active', WHERE country = 'USA'
    │
    └── Does the column need to enforce no-duplicates?
        │
        └── YES → Use UNIQUE INDEX
                  email, username, product_sku, national_id
```

---

### Index Type Quick Reference

| Index Type | When to Use | When NOT to Use |
|---|---|---|
| **Heap (no index)** | Staging tables, temp tables, write-heavy with no reads, small tables | Anything that is queried by users or reports |
| **Clustered Row Store** | Primary keys, date columns for range queries, the main sort order of a table | Non-unique columns, frequently updated columns |
| **Non-Clustered Row Store** | Foreign keys, JOIN columns, WHERE/ORDER BY columns (non-PK) | Primary keys (already covered by clustered) |
| **Clustered Columnstore** | OLAP, data warehouse, fact tables, BI reports, tables with complex aggregations | OLTP tables with frequent single-row reads/writes |
| **Non-Clustered Columnstore** | Mixed OLTP + analytics — companion index for analytical queries | Pure OLTP with no analytics |
| **Filtered Index** | Queries always scoped to a subset (active records, current year, single country) | Highly variable query patterns with no dominant subset |
| **Unique Index** | Enforcing business uniqueness rules (email, SKU, username) | Low-cardinality columns or columns that intentionally allow duplicates |

## 8. Cross-Database Maintenance Syntax — MSSQL vs MySQL vs PostgreSQL

### Monitor Index Usage

**SQL Server:**
```sql
SELECT idx.name, s.user_seeks, s.user_scans, s.user_updates
FROM sys.indexes idx
LEFT JOIN sys.dm_db_index_usage_stats s
    ON idx.object_id = s.object_id AND idx.index_id = s.index_id
WHERE idx.object_id = OBJECT_ID('customers');
```

**MySQL:**
```sql
-- MySQL 8.0+ performance_schema
SELECT object_name, index_name, count_read, count_write, count_fetch
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE object_schema = 'mydb'
  AND object_name = 'customers'
ORDER BY count_read ASC;  -- low read count = potentially unused
```

**PostgreSQL:**
```sql
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan    AS times_used,
    idx_tup_read AS tuples_read,
    idx_tup_fetch AS tuples_fetched
FROM pg_stat_user_indexes
WHERE tablename = 'customers'
ORDER BY idx_scan ASC;  -- 0 = never used since last stats reset
```

---

### Update Statistics

**SQL Server:**
```sql
UPDATE STATISTICS sales.customers;        -- one table
EXEC sp_updatestats;                      -- whole database
```

**MySQL:**
```sql
ANALYZE TABLE customers;                  -- one table
-- Full database: run ANALYZE TABLE for each table (no single command)
```

**PostgreSQL:**
```sql
ANALYZE customers;                        -- one table
ANALYZE;                                  -- entire database
VACUUM ANALYZE customers;                 -- vacuum + analyze combined (recommended)
```

---

### Monitor & Fix Fragmentation

**SQL Server:**
```sql
-- Check fragmentation
SELECT idx.name, ips.avg_fragmentation_in_percent
FROM sys.dm_db_index_physical_stats(DB_ID(), OBJECT_ID('customers'), NULL, NULL, 'LIMITED') ips
INNER JOIN sys.indexes idx ON ips.object_id = idx.object_id AND ips.index_id = idx.index_id;

-- Reorganize
ALTER INDEX idx_customers_email ON customers REORGANIZE;

-- Rebuild
ALTER INDEX idx_customers_email ON customers REBUILD;
ALTER INDEX idx_customers_email ON customers REBUILD WITH (ONLINE = ON); -- Enterprise only
```

**MySQL:**
```sql
-- Check fragmentation (data_free = wasted space in bytes)
SELECT table_name, data_free, data_length, index_length
FROM information_schema.tables
WHERE table_schema = 'mydb' AND table_name = 'customers';

-- Reorganize / optimize (combines both reorganize and rebuild)
OPTIMIZE TABLE customers;

-- Rebuild single index
ALTER TABLE customers DROP INDEX idx_customers_email,
                      ADD INDEX idx_customers_email (email);
```

**PostgreSQL:**
```sql
-- Check bloat (fragmentation equivalent)
-- Using pgstattuple extension:
CREATE EXTENSION IF NOT EXISTS pgstattuple;
SELECT * FROM pgstattuple('customers');
-- leaf_fragmentation shows % fragmentation

-- Reorganize equivalent (removes dead tuples, compacts)
VACUUM customers;

-- Rebuild equivalent (exclusive lock)
REINDEX INDEX idx_customers_email;

-- Rebuild without locking (PostgreSQL 12+)
REINDEX INDEX CONCURRENTLY idx_customers_email;

-- Rebuild all indexes on a table
REINDEX TABLE customers;
REINDEX TABLE CONCURRENTLY customers;  -- PostgreSQL 12+
```

### Missing Index Recommendations

**SQL Server:**
```sql
SELECT mid.statement, mid.equality_columns, mid.inequality_columns, migs.avg_user_impact
FROM sys.dm_db_missing_index_details mid
INNER JOIN sys.dm_db_missing_index_groups mig ON mid.index_handle = mig.index_handle
INNER JOIN sys.dm_db_missing_index_group_stats migs ON mig.index_group_handle = migs.group_handle
ORDER BY migs.avg_user_impact DESC;
```

**MySQL:**
```sql
-- MySQL doesn't have a built-in missing index advisor.
-- Use EXPLAIN to identify full table scans manually:
EXPLAIN SELECT * FROM customers WHERE country = 'USA';
-- If "type" = "ALL" → full table scan → consider adding an index

-- Or use MySQL Workbench's Visual EXPLAIN and Performance Schema reports
```

**PostgreSQL:**
```sql
-- PostgreSQL doesn't have a built-in missing index advisor.
-- Use EXPLAIN ANALYZE to find sequential scans on large tables:
EXPLAIN ANALYZE SELECT * FROM customers WHERE country = 'USA';
-- "Seq Scan" on large tables → candidate for an index

-- pg_stat_user_tables can help identify heavily sequentially-scanned tables:
SELECT relname, seq_scan, idx_scan
FROM pg_stat_user_tables
WHERE seq_scan > idx_scan
ORDER BY seq_scan DESC;
-- High seq_scan relative to idx_scan → table may be missing indexes
```

### Cross-Database Maintenance Feature Comparison

| Maintenance Task | SQL Server | MySQL | PostgreSQL |
|---|---|---|---|
| Index usage tracking | `sys.dm_db_index_usage_stats` | `performance_schema` (v8.0+) | `pg_stat_user_indexes` |
| Missing index advisor | ✅ Built-in DMV | ❌ Manual EXPLAIN | ❌ Manual EXPLAIN / `pg_stat_user_tables` |
| Update table statistics | `UPDATE STATISTICS` | `ANALYZE TABLE` | `ANALYZE` |
| Update all DB statistics | `sp_updatestats` | Per-table only | `ANALYZE` (whole DB) |
| Check fragmentation | `dm_db_index_physical_stats` | `information_schema.tables` (data_free) | `pgstattuple` extension |
| Light defragmentation | `REORGANIZE` | `OPTIMIZE TABLE` | `VACUUM` |
| Full rebuild | `REBUILD` | Rebuild via `ALTER TABLE` | `REINDEX` |
| Non-locking rebuild | `REBUILD WITH (ONLINE = ON)` | ✅ (InnoDB online DDL) | `REINDEX CONCURRENTLY` (v12+) |
| Duplicate index detection | `sys.indexes` + `sys.index_columns` | `information_schema.statistics` | `pg_indexes` + `pg_stat_user_indexes` |

## 11. Common Mistakes

| Mistake | Problem | Fix |
|---|---|---|
| Never checking index usage | Accumulate unused indexes → write overhead grows silently | Schedule monthly usage audits |
| Blindly creating every missing index recommendation | Over-indexing → too many indexes slow writes dramatically | Evaluate frequency and importance before creating |
| Never updating statistics after bulk data loads | Optimizer makes wrong index choices → slow queries despite having indexes | Always run `UPDATE STATISTICS` or `ANALYZE` after large data loads |
| Using `REBUILD` when `REORGANIZE` would suffice | Unnecessary table lock and I/O overhead for low fragmentation | Use `REORGANIZE` for 10–30%, `REBUILD` only for > 30% |
| Rebuilding on production without `ONLINE = ON` | Table is locked — users get timeouts | Use `REBUILD WITH (ONLINE = ON)` or schedule during maintenance window |
| Ignoring DMV statistics because "we're in development" | Missing the habit of monitoring, which must exist in production | Practice the monitoring queries even locally |
| Treating duplicate indexes as harmless | Each duplicate slows every INSERT/UPDATE/DELETE on that column | Audit regularly and drop confirmed duplicates |

## 12. Best Practices

1. **Audit unused indexes on day one of any new project.** Run the usage stats query before making any changes. You will almost always find unused indexes to drop — freeing storage and improving write speed instantly.

2. **Schedule statistics updates as a recurring job.** Run `sp_updatestats` (SQL Server) or `ANALYZE` (PostgreSQL) nightly or weekly. Always run immediately after large data migrations.

3. **Automate fragmentation checks and maintenance.** The cursor-based script in Section 6 can be deployed as a SQL Server Agent job to automatically REORGANIZE and REBUILD based on thresholds.

4. **Never drop an index without discussing with the team first.** An index that appears unused in a DMV might be used by a monthly batch job or rarely-triggered report. Check with stakeholders before dropping.

5. **For PostgreSQL, prefer `VACUUM ANALYZE` combined.** It handles dead tuple cleanup (fragmentation) and statistics in one operation — more efficient than running them separately.

6. **For MySQL, use `OPTIMIZE TABLE` during low-traffic windows.** It rebuilds the table and indexes, but may lock the table on older MySQL versions (InnoDB online DDL in MySQL 5.6+ reduces this risk).

7. **Use `REINDEX CONCURRENTLY` in PostgreSQL for production.** It rebuilds without locking the table — available from PostgreSQL 12 onward.

8. **Be conservative with missing index recommendations.** Start with the top recommendations by `estimated_improvement_score`. Create them one at a time, measure query performance before and after, and confirm the improvement.

## 13. Final Summary Table

| Task | What to Check | Key Tool / Query | Action |
|---|---|---|---|
| **Monitor Usage** | Which indexes are read? Which have zero reads? | `sys.dm_db_index_usage_stats` / `pg_stat_user_indexes` | Drop unused indexes |
| **Missing Indexes** | Which queries do full scans that an index would fix? | `sys.dm_db_missing_index_details` / `EXPLAIN ANALYZE` | Create high-priority missing indexes |
| **Duplicate Indexes** | Which columns are covered by multiple indexes? | `sys.index_columns` + window function / `pg_indexes` | Drop redundant indexes after team review |
| **Update Statistics** | Are row counts / distributions stale in metadata? | `sys.dm_db_stats_properties` / `pg_stat_user_tables` | `UPDATE STATISTICS` / `ANALYZE` |
| **Fragmentation** | What % of index pages are out of order or wasted? | `sys.dm_db_index_physical_stats` / `pgstattuple` | REORGANIZE (10–30%) or REBUILD (>30%) |

| Index Type | Best For | Avoid When |
|---|---|---|
| **Heap** | Staging, temp, write-only tables | Queried by users or reports |
| **Clustered Row Store** | Primary keys, date ranges | Frequently updated columns |
| **Non-Clustered Row Store** | WHERE / JOIN / ORDER BY non-PK columns | Columns already covered by clustered |
| **Clustered Columnstore** | OLAP, data warehouse, BI aggregations | OLTP single-row transactions |
| **Filtered** | Dominant subset queries (active, USA) | Variable query patterns |
| **Unique** | Business uniqueness rules (email, SKU) | Low-cardinality or intentionally duplicate cols |
