## 1. What Is a Database Table?

A **database table** is a structured collection of data organised into columns and rows — conceptually similar to a spreadsheet, but stored as optimised binary files on disk managed entirely by the database engine.

```
Table: employees

┌─────────────┬────────────┬─────────┬─────────┐
│ employee_id │ first_name │ country │ salary  │
├─────────────┼────────────┼─────────┼─────────┤
│ 1           │ Alice      │ Germany │ 72000   │  ← row (record)
│ 2           │ Bob        │ USA     │ 85000   │
│ 3           │ Carol      │ France  │ 68000   │
└─────────────┴────────────┴─────────┴─────────┘
      ↑              ↑
   column          column
(field/attribute) (field/attribute)

Intersection of a row and column = cell (single piece of data)
```

### Where Tables Sit in the Database Hierarchy

```
SQL Server
└── Database (e.g., SalesDB)
    └── Schema (e.g., sales)
        ├── Tables   ← physical data lives here
        └── Views    ← virtual; no data stored
```

Tables live at the **Logical Level** of the three-tier database architecture — the middle layer. You do not need to be a database administrator to work with tables, but you do need to understand DDL commands (`CREATE`, `ALTER`, `DROP`) and basic data modelling.

### Key Characteristics

- Data is **stored permanently** in the database as physical files on disk.
- Each column has a defined **data type** (INT, VARCHAR, DATE, DECIMAL…).
- Rows represent individual **records**; columns represent **attributes**.
- Data persists until explicitly deleted or the table is dropped.
- End users query the table; the database engine reads the underlying files.

## 2. Table Types Overview

```
Tables
├── Permanent Tables     (persist indefinitely until explicitly dropped)
│   ├── Create + INSERT  (define structure first, then insert data)
│   └── CTAS             (Create Table As Select — structure + data from a query)
│
└── Temporary Tables     (dropped automatically when the session ends)
    ├── Local temp tables  (#table_name in MS SQL Server)
    └── Global temp tables (##table_name in MS SQL Server)
```

This chapter focuses primarily on **permanent tables** and the **CTAS** method. Temporary tables are covered in [Section 10](#10-temporary-tables).

## 3. Two Ways to Create a Permanent Table

### Method 1: CREATE + INSERT (Classical)

Two explicit steps: first define the structure (empty table), then populate it.

```
Step 1: CREATE TABLE → empty table with defined structure
Step 2: INSERT INTO  → data loaded into the table
```

```sql
-- Step 1: Define the table structure
CREATE TABLE sales.monthly_orders (
    order_month   VARCHAR(20),
    total_orders  INT
);

-- Step 2: Insert data manually (or from a file, application, migration, etc.)
INSERT INTO sales.monthly_orders (order_month, total_orders)
VALUES ('January', 42),
       ('February', 38),
       ('March', 55);
```

**When to use:** Building tables from scratch, migrations from external systems, application-driven inserts, or when you need fine-grained control over data types, constraints, and indexes before any data exists.

### Method 2: CTAS (Create Table As Select)

One step: the query's result set defines both the structure and the initial data.

```
Step 1: Write a SELECT query
         ↓
The query result defines the table structure (column names, data types)
         ↓
The query result is immediately inserted as the table's data
         ↓
New permanent table exists in the database
```

```sql
-- CTAS: create a table directly from a query result
CREATE TABLE sales.monthly_orders AS
(
    SELECT   DATENAME(month, order_date) AS order_month,
             COUNT(order_id)             AS total_orders
    FROM     sales.orders
    GROUP BY DATENAME(month, order_date)
);
```

**When to use:** Materialising the results of a complex query for performance, creating snapshots, building physical data mart tables, or whenever the source data already exists in the database.

### Side-by-Side Comparison

| Dimension | CREATE + INSERT | CTAS |
|---|---|---|
| Steps | 2 (define, then populate) | 1 (query defines and populates) |
| Structure defined by | Developer explicitly | Query result automatically |
| Data source | External / manual / application | Existing database tables |
| Data type control | Full manual control | Inferred from query result |
| Constraint support | Full (PKs, FKs, indexes, defaults) | Limited — constraints added after |
| Use case | New tables, migrations, app tables | Materialised queries, snapshots, marts |

## 4. CTAS vs. Views

Both CTAS and views produce a named object derived from a query. The critical difference is whether data is stored.

### Creation Time Behaviour

```
CREATE VIEW:
  → Stores the SQL query in the catalog
  → Executes nothing
  → No data written anywhere
  → Instant

CREATE TABLE AS SELECT (CTAS):
  → Executes the SELECT query immediately
  → Stores the result as physical rows in a new table
  → Takes as long as the query takes to run
  → Data is now frozen in the table
```

### Query Time Behaviour

```
Querying a VIEW:
  User query → Database fetches stored SQL from catalog
             → Executes view's SQL against source tables (FRESH data)
             → Applies user's filters
             → Returns result
  (Two queries execute; always reflects current state of source tables)

Querying a CTAS table:
  User query → Database reads directly from the table's stored rows
             → Returns result
  (One query; fast; but data is a SNAPSHOT from when the table was created)
```

### The Pizza Analogy

> **View = Restaurant pizza.** Every time you order, the chef makes it fresh from the current ingredients. You always get the latest.
>
> **CTAS table = Frozen pizza.** It was prepared earlier and stored. Fast to serve, but it is the state of the pizza from when it was frozen — not today's ingredients.

### Feature Comparison

| Dimension | View | CTAS Table |
|---|---|---|
| Stores data | ❌ No | ✅ Yes |
| Data freshness | Always current (re-queries source) | Snapshot at creation time |
| Query performance | Slower (view SQL + user SQL) | Faster (direct table read) |
| Auto-updates when source changes | ✅ Yes | ❌ No — must recreate |
| Maintenance effort | Low (update the query) | Higher (drop + recreate to refresh) |
| Dropping deletes data? | No | Yes — table rows are lost |
| Best for | Dynamic, always-fresh data products | Pre-computed, performance-optimised results |

## 5. CTAS Syntax Across Dialects

### MySQL & PostgreSQL: Standard CTAS

```sql
-- MySQL 8.0+ and PostgreSQL
CREATE TABLE sales.monthly_orders AS
(
    SELECT   DATE_FORMAT(order_date, '%Y-%m-01') AS order_month,   -- MySQL
             COUNT(order_id)                      AS total_orders
    FROM     sales.orders
    GROUP BY DATE_FORMAT(order_date, '%Y-%m-01')
);

-- PostgreSQL variant
CREATE TABLE sales.monthly_orders AS
(
    SELECT   DATE_TRUNC('month', order_date) AS order_month,
             COUNT(order_id)                 AS total_orders
    FROM     sales.orders
    GROUP BY DATE_TRUNC('month', order_date)
);
```

### MS SQL Server: SELECT INTO

MS SQL Server uses a different syntax — the `INTO` keyword is embedded between `SELECT` and `FROM`:

```sql
-- MS SQL Server: SELECT ... INTO new_table FROM ...
SELECT   DATENAME(month, order_date) AS order_month,
         COUNT(order_id)             AS total_orders
INTO     sales.monthly_orders        -- ← DDL lives here, inside the query
FROM     sales.orders
GROUP BY DATENAME(month, order_date);
```

> **Note:** The `SELECT INTO` syntax places the DDL in the middle of the query. For long, complex queries this can be easy to miss. Prefer the standard `CREATE TABLE AS` form when your database supports it.

### Data Types Are Inferred Automatically

The database determines column data types from the query result:

| Query Expression | Inferred Type |
|---|---|
| `COUNT(order_id)` | INT |
| `SUM(sales)` | DECIMAL / FLOAT |
| `DATENAME(month, order_date)` | VARCHAR |
| `DATE_TRUNC('month', order_date)` | DATE / TIMESTAMP |
| `CAST(value AS INT)` | INT |

You can always override inferred types by wrapping expressions in explicit `CAST` or `CONVERT` calls before the table is created.

## 6. How to Refresh a CTAS Table

Because a CTAS table stores a snapshot, it does not update automatically. To get fresh data, you must drop and recreate it.

### Naive Approach (Two Steps)

```sql
-- Step 1: Drop the old table
DROP TABLE sales.monthly_orders;

-- Step 2: Recreate with fresh data
SELECT   DATENAME(month, order_date) AS order_month,
         COUNT(order_id)             AS total_orders
INTO     sales.monthly_orders
FROM     sales.orders
GROUP BY DATENAME(month, order_date);
```

### MS SQL Server: T-SQL Conditional Refresh (Preferred)

Handles both cases — whether the table already exists or not — in a single execution:

```sql
-- Drop if exists, then recreate
IF OBJECT_ID('sales.monthly_orders', 'U') IS NOT NULL
    DROP TABLE sales.monthly_orders;
GO

SELECT   DATENAME(month, order_date) AS order_month,
         COUNT(order_id)             AS total_orders
INTO     sales.monthly_orders
FROM     sales.orders
GROUP BY DATENAME(month, order_date);
GO
```

### MySQL & PostgreSQL: DROP IF EXISTS + CREATE

```sql
-- MySQL / PostgreSQL
DROP TABLE IF EXISTS sales.monthly_orders;

CREATE TABLE sales.monthly_orders AS
(
    SELECT   DATE_TRUNC('month', order_date) AS order_month,
             COUNT(order_id)                 AS total_orders
    FROM     sales.orders
    GROUP BY DATE_TRUNC('month', order_date)
);
```

### PostgreSQL: TRUNCATE + INSERT (Preserve Table Structure)

When the table structure is stable and only the data needs refreshing, `TRUNCATE` is faster than `DROP + CREATE` because it preserves indexes and constraints:

```sql
-- Truncate existing data (keep structure, indexes, constraints)
TRUNCATE TABLE sales.monthly_orders;

-- Re-insert fresh data
INSERT INTO sales.monthly_orders (order_month, total_orders)
SELECT   DATE_TRUNC('month', order_date),
         COUNT(order_id)
FROM     sales.orders
GROUP BY DATE_TRUNC('month', order_date);
```

### Scheduling Refreshes

In production, CTAS refresh scripts are typically scheduled as automated jobs:

| Platform | Scheduling Tool |
|---|---|
| MS SQL Server | SQL Server Agent (scheduled jobs) |
| MySQL | MySQL Event Scheduler |
| PostgreSQL | `pg_cron` extension or external cron job |
| Cloud (general) | Airflow, dbt, Azure Data Factory, AWS Glue |

## 7. Use Case 1 — Optimise Query Performance

### The Problem

Some views contain complex multi-table joins, heavy aggregations, or window functions. When many users query such a view simultaneously, each user triggers a full re-execution of the view's SQL — which may take 10, 20, or 30 minutes per query. This is unacceptable in a production analytics environment.

```
View approach (slow):
  User 1 queries v_complex_report → 30 min execution ← unhappy
  User 2 queries v_complex_report → 30 min execution ← unhappy
  User 3 queries v_complex_report → 30 min execution ← unhappy
  (Each user pays the full cost every time)

CTAS table approach (fast):
  Night job runs CTAS → 30 min to build the table (once)
  User 1 queries the table → < 1 second ← happy
  User 2 queries the table → < 1 second ← happy
  User 3 queries the table → < 1 second ← happy
```

### The Solution: Pre-compute Overnight with CTAS

Move the expensive computation to an off-hours scheduled job. Users query the pre-built table, not the live view.

```sql
-- Sample Schema
CREATE TABLE sales.orders (
    order_id    INT,
    customer_id INT,
    product_id  INT,
    order_date  DATE,
    sales       DECIMAL(10,2),
    quantity    INT
);

CREATE TABLE sales.customers (
    customer_id  INT,
    first_name   VARCHAR(50),
    last_name    VARCHAR(50),
    country      VARCHAR(50)
);

CREATE TABLE sales.products (
    product_id  INT,
    product     VARCHAR(100),
    category    VARCHAR(50)
);
```

```sql
-- Step 1: Nightly refresh script (runs at 02:00 via SQL Agent / cron)

-- MS SQL Server
IF OBJECT_ID('sales.monthly_category_summary', 'U') IS NOT NULL
    DROP TABLE sales.monthly_category_summary;
GO

SELECT   DATENAME(month, o.order_date)        AS order_month,
         YEAR(o.order_date)                   AS order_year,
         p.category,
         COUNT(DISTINCT o.order_id)           AS total_orders,
         SUM(o.sales)                         AS total_revenue,
         SUM(o.quantity)                      AS total_units,
         AVG(o.sales)                         AS avg_order_value,
         COUNT(DISTINCT o.customer_id)        AS unique_customers
INTO     sales.monthly_category_summary
FROM     sales.orders   o
LEFT JOIN sales.products p ON p.product_id = o.product_id
GROUP BY DATENAME(month, o.order_date),
         YEAR(o.order_date),
         p.category;
GO
```

```sql
-- Step 2: Analysts query the pre-built table (instant)
SELECT *
FROM   sales.monthly_category_summary
WHERE  order_year = 2024
ORDER  BY order_year, order_month, total_revenue DESC;
```

### Sample Output

| order_month | order_year | category | total_orders | total_revenue | total_units | avg_order_value | unique_customers |
|---|---|---|---|---|---|---|---|
| January | 2024 | Electronics | 15 | 14985.00 | 15 | 999.00 | 12 |
| January | 2024 | Furniture | 8 | 2000.00 | 16 | 250.00 | 7 |
| February | 2024 | Electronics | 22 | 21978.00 | 22 | 999.00 | 18 |

## 8. Use Case 2 — Persistent Data Snapshots for Quality Analysis

### The Problem

When a data quality issue is discovered (e.g., duplicate records, wrong values, unexpected NULLs), the data is often actively being updated by upstream systems. By the time you investigate, the problematic records may have changed, making it impossible to reproduce or understand the original issue.

```
Timeline:
  08:00  Data quality issue discovered in sales.orders
  08:30  Analyst starts investigating
  09:00  Upstream ETL job runs → records updated / new records inserted
  09:30  Analyst can no longer reproduce the original issue
         → Root cause analysis fails
```

### The Solution: Freeze the State with CTAS

At the moment the issue is discovered, immediately capture a snapshot of the affected data into a dedicated table. Analysis then proceeds on the frozen state — unaffected by any subsequent updates.

```sql
-- Capture the snapshot immediately when the issue is discovered
-- MS SQL Server
SELECT   *
INTO     sales.orders_snapshot_20240315     -- include date in name for clarity
FROM     sales.orders
WHERE    order_date >= '2024-03-01';        -- scope to the affected period

-- MySQL / PostgreSQL
CREATE TABLE sales.orders_snapshot_20240315 AS
(
    SELECT *
    FROM   sales.orders
    WHERE  order_date >= '2024-03-01'
);
```

Now analysis can proceed on the snapshot safely:

```sql
-- Investigate duplicate order_ids in the snapshot
SELECT   order_id,
         COUNT(*) AS occurrence_count
FROM     sales.orders_snapshot_20240315
GROUP BY order_id
HAVING   COUNT(*) > 1
ORDER BY occurrence_count DESC;

-- Compare snapshot vs. current state to see what changed
SELECT s.order_id, s.sales AS snapshot_sales, o.sales AS current_sales
FROM   sales.orders_snapshot_20240315 s
JOIN   sales.orders                   o ON o.order_id = s.order_id
WHERE  s.sales <> o.sales;
```

### When to Use Snapshots

- Investigating data quality or pipeline bugs
- Auditing a batch load before and after
- Preserving a baseline for A/B or before/after analysis
- Regulatory or compliance requirements (point-in-time state)
- Debugging ETL transformations where source data is mutable

## 9. Use Case 3 — Physical Data Marts in a Data Warehouse

### Architecture Background

As covered in the Views chapter, virtual data marts (views on top of a warehouse) are the recommended starting point. But as complexity and user volume grow, views can become a performance bottleneck.

```
Virtual Data Mart (Views):          Physical Data Mart (CTAS Tables):

Warehouse Tables                    Warehouse Tables
      │                                    │
      │  (query every time)                │  (nightly CTAS job)
      ▼                                    ▼
   View Layer          ─────────►    Table Layer (Mart)
      │                                    │
      │  (slow for complex models)         │  (fast direct reads)
      ▼                                    ▼
  Power BI / Reports                  Power BI / Reports
```

### When to Switch from Virtual to Physical Marts

| Signal | Action |
|---|---|
| Dashboard takes minutes to load | Switch affected mart views to CTAS tables |
| Many concurrent users hit the same mart | Pre-compute with CTAS |
| Mart logic involves heavy aggregations or many joins | CTAS for performance |
| Data freshness of T-1 (yesterday's data) is acceptable | CTAS nightly job |
| Data must be real-time | Keep as view |

### Recommended Evolution Strategy

```
Phase 1: Start with virtual marts (views)
  → Fast to implement
  → Always fresh
  → Low maintenance
  → Easy to change business logic

Phase 2: Profile performance
  → Identify slow marts (> acceptable load time for users)

Phase 3: Convert slow marts to physical (CTAS)
  → Build a nightly refresh job
  → Mart becomes a pre-computed table
  → Remaining fast/simple marts stay as views
```

### Example: Physical Mart Nightly Build

```sql
-- Nightly job: build the Sales Data Mart table
-- PostgreSQL
DROP TABLE IF EXISTS datamart.sales_mart;

CREATE TABLE datamart.sales_mart AS
(
    SELECT
        o.order_date,
        DATE_TRUNC('month', o.order_date)                AS order_month,
        c.country                                         AS customer_country,
        p.category,
        COUNT(DISTINCT o.order_id)                        AS total_orders,
        COUNT(DISTINCT o.customer_id)                     AS unique_customers,
        SUM(o.sales)                                      AS total_revenue,
        SUM(o.quantity)                                   AS total_units,
        ROUND(SUM(o.sales) / NULLIF(SUM(o.quantity),0), 2) AS avg_unit_price
    FROM       warehouse.fact_orders     o
    LEFT JOIN  warehouse.dim_customers   c ON c.customer_key = o.customer_key
    LEFT JOIN  warehouse.dim_products    p ON p.product_key  = o.product_key
    WHERE      o.is_deleted = 0
    GROUP BY   o.order_date,
               DATE_TRUNC('month', o.order_date),
               c.country,
               p.category
);

-- Add indexes for BI query patterns
CREATE INDEX idx_sales_mart_month    ON datamart.sales_mart (order_month);
CREATE INDEX idx_sales_mart_country  ON datamart.sales_mart (customer_country);
CREATE INDEX idx_sales_mart_category ON datamart.sales_mart (category);
```

> **Tip:** After creating the table, add indexes on the columns your BI tool filters and groups by most. CTAS does not carry over indexes from source tables — they must be added explicitly.

## 10. Temporary Tables

### Definition

A **temporary table** is a table that exists only for the duration of a session (or, for global temp tables, until all sessions referencing it disconnect). The database engine drops it automatically when the session ends.

Temporary tables are useful for:
- Breaking a complex multi-step query into intermediate steps (similar to CTEs, but the intermediate result can be indexed)
- Passing data between stored procedures or scripts within a session
- Holding staging data before a final INSERT

### Syntax

```sql
-- MS SQL Server: local temp table (# prefix; session-scoped)
SELECT   customer_id, SUM(sales) AS total_sales
INTO     #customer_totals           -- # prefix = temp table
FROM     sales.orders
GROUP BY customer_id;

-- Use it later in the same session
SELECT c.first_name, t.total_sales
FROM   sales.customers  c
JOIN   #customer_totals t ON t.customer_id = c.customer_id;

-- MS SQL Server: global temp table (## prefix; all sessions)
SELECT * INTO ##shared_staging FROM sales.orders WHERE order_date = '2024-01-01';

-- MySQL: temp table
CREATE TEMPORARY TABLE customer_totals AS
(
    SELECT customer_id, SUM(sales) AS total_sales
    FROM   sales.orders
    GROUP BY customer_id
);

-- PostgreSQL: temp table
CREATE TEMP TABLE customer_totals AS
(
    SELECT customer_id, SUM(sales) AS total_sales
    FROM   sales.orders
    GROUP BY customer_id
);
```

### Temporary Tables vs. CTEs vs. Permanent Tables

| Dimension | CTE | Temp Table | Permanent CTAS Table |
|---|---|---|---|
| Lifespan | One query only | One session | Until explicitly dropped |
| Stored on disk | No (in-memory) | Yes (tempdb / temp schema) | Yes |
| Indexable | No | Yes | Yes |
| Reusable within session | No | Yes | Yes |
| Visible to other sessions | No | No (local) / Yes (global) | Yes |
| Auto-cleanup | Yes (query ends) | Yes (session ends) | No — must `DROP TABLE` |
| Best for | Readability within one query | Multi-step session work | Persisted materialised results |

## 11. Dialect Comparison: MySQL vs. MS SQL Server vs. PostgreSQL

### 11.1 CTAS Syntax

| Feature | MySQL 8.0+ | MS SQL Server | PostgreSQL |
|---|---|---|---|
| Standard CTAS | `CREATE TABLE t AS (SELECT ...)` | ❌ Not supported | `CREATE TABLE t AS (SELECT ...)` |
| SELECT INTO | ❌ Not supported | `SELECT ... INTO t FROM ...` | `SELECT ... INTO t FROM ...` (creates temp table in PG) |
| Preferred idiom | `CREATE TABLE ... AS` | `SELECT ... INTO` | `CREATE TABLE ... AS` |
| Data types inferred | ✅ Yes | ✅ Yes | ✅ Yes |

```sql
-- MySQL
CREATE TABLE sales.monthly_orders AS
SELECT DATE_FORMAT(order_date, '%Y-%m-01') AS order_month,
       COUNT(order_id) AS total_orders
FROM   sales.orders
GROUP BY DATE_FORMAT(order_date, '%Y-%m-01');

-- MS SQL Server
SELECT DATENAME(month, order_date) AS order_month,
       COUNT(order_id) AS total_orders
INTO   sales.monthly_orders
FROM   sales.orders
GROUP BY DATENAME(month, order_date);

-- PostgreSQL
CREATE TABLE sales.monthly_orders AS
SELECT DATE_TRUNC('month', order_date) AS order_month,
       COUNT(order_id) AS total_orders
FROM   sales.orders
GROUP BY DATE_TRUNC('month', order_date);
```

### 11.2 Drop Table Syntax

```sql
-- MySQL, PostgreSQL — DROP IF EXISTS (no error if missing)
DROP TABLE IF EXISTS sales.monthly_orders;

-- MS SQL Server 2016+
DROP TABLE IF EXISTS sales.monthly_orders;

-- MS SQL Server (older, pre-2016) — check OBJECT_ID
IF OBJECT_ID('sales.monthly_orders', 'U') IS NOT NULL
    DROP TABLE sales.monthly_orders;
```

### 11.3 Temporary Tables

| Feature | MySQL | MS SQL Server | PostgreSQL |
|---|---|---|---|
| Local temp table syntax | `CREATE TEMPORARY TABLE t AS ...` | `SELECT ... INTO #t ...` | `CREATE TEMP TABLE t AS ...` |
| Global temp table | ❌ | `##t` | ❌ (use unlogged tables or schemas) |
| Auto-drop | On session end | On session end (local) / all sessions disconnect (global) | On session end |
| Temp table in default db | Separate `TEMPORARY` schema | `tempdb` system database | `pg_temp` schema |

### 11.4 Refresh Patterns

```sql
-- MySQL
DROP TABLE IF EXISTS sales.monthly_orders;
CREATE TABLE sales.monthly_orders AS (SELECT ...);

-- MS SQL Server
IF OBJECT_ID('sales.monthly_orders', 'U') IS NOT NULL
    DROP TABLE sales.monthly_orders;
GO
SELECT ... INTO sales.monthly_orders FROM ...;
GO

-- PostgreSQL: drop + recreate
DROP TABLE IF EXISTS sales.monthly_orders;
CREATE TABLE sales.monthly_orders AS (SELECT ...);

-- PostgreSQL: truncate + insert (preserves indexes/constraints)
TRUNCATE TABLE sales.monthly_orders;
INSERT INTO sales.monthly_orders SELECT ...;
```

### 11.5 Adding Indexes After CTAS

CTAS does not transfer indexes from source tables. Always add indexes manually after creation:

```sql
-- MySQL
CREATE INDEX idx_month ON sales.monthly_orders (order_month);

-- MS SQL Server
CREATE INDEX idx_month ON sales.monthly_orders (order_month);

-- PostgreSQL
CREATE INDEX idx_month ON sales.monthly_orders (order_month);
-- Concurrent index (no table lock in PostgreSQL)
CREATE INDEX CONCURRENTLY idx_month ON sales.monthly_orders (order_month);
```

## 12. Common Mistakes

### 12.1 Querying a CTAS Table and Expecting Fresh Data

```sql
-- ❌ This returns stale data if the source changed after table creation
SELECT * FROM sales.monthly_orders;

-- ✅ Understand that this table is a snapshot; refresh it explicitly if needed
-- Run the drop + recreate script before querying
```

### 12.2 Running CTAS Twice Without Dropping First

```sql
-- ❌ Error: table already exists
SELECT DATENAME(month, order_date) AS order_month, COUNT(*) AS total_orders
INTO   sales.monthly_orders
FROM   sales.orders
GROUP BY DATENAME(month, order_date);
-- "There is already an object named 'monthly_orders' in the database."

-- ✅ Always drop first (or use conditional T-SQL / DROP IF EXISTS)
DROP TABLE IF EXISTS sales.monthly_orders;
SELECT ...
INTO   sales.monthly_orders
FROM   sales.orders ...;
```

### 12.3 Forgetting That Dropping a CTAS Table Destroys the Data

```sql
-- ⚠️ Unlike dropping a VIEW, dropping a TABLE removes all rows permanently
DROP TABLE sales.monthly_orders;
-- The rows are gone. The source table (sales.orders) is unaffected,
-- but you will need to re-run the CTAS script to get the data back.
```

### 12.4 Not Adding Indexes After CTAS

```sql
-- ❌ CTAS table with no indexes — full table scans on every BI query
CREATE TABLE datamart.sales_mart AS (SELECT ... FROM ...);

-- ✅ Add indexes on commonly filtered / grouped columns
CREATE TABLE datamart.sales_mart AS (SELECT ... FROM ...);
CREATE INDEX idx_month    ON datamart.sales_mart (order_month);
CREATE INDEX idx_country  ON datamart.sales_mart (customer_country);
```

### 12.5 Using SELECT INTO in PostgreSQL (Wrong Semantics)

```sql
-- ❌ In PostgreSQL, SELECT INTO creates a temp table, not a permanent one
SELECT order_id, sales
INTO   sales.monthly_orders   -- this behaves differently in PostgreSQL!
FROM   sales.orders;

-- ✅ Use CREATE TABLE AS in PostgreSQL
CREATE TABLE sales.monthly_orders AS
SELECT order_id, sales FROM sales.orders;
```

### 12.6 Immediately Switching Views to Tables Without Profiling First

Don't preemptively replace all views with CTAS tables. Start with views (dynamic, zero maintenance), profile actual query times in production, then convert only the views that are demonstrably too slow.

## 13. Best Practices

**Start with views, switch to CTAS when needed.** Views are easier to build and maintain. Only materialise into a table when query performance is genuinely a problem.

**Always name CTAS tables clearly** and consider including a schema prefix (`sales.monthly_orders`, not just `monthly_orders`) and, for snapshots, a date suffix (`orders_snapshot_20240315`).

**Schedule CTAS refreshes during off-peak hours.** The refresh takes the same time as the original query. Run it overnight so users always find fresh data ready in the morning.

**Add indexes after every CTAS.** CTAS infers column types but creates no indexes. Always add indexes on the columns your queries filter, join, or group by.

**Use `TRUNCATE + INSERT` instead of `DROP + CREATE`** when the table structure is stable and you need to preserve indexes and constraints between refreshes (PostgreSQL / MS SQL Server).

**Use snapshots proactively.** When you detect a data quality issue, capture a snapshot immediately — before the next ETL run overwrites the evidence.

**Document what each CTAS table represents.** Add a comment or a metadata entry recording: the source query, the refresh schedule, the date it was last refreshed, and who owns it.

**Do not use CTAS for real-time data needs.** If users need data that is current to the minute or hour, use a view or a streaming solution. CTAS is for T-1 (yesterday's data) use cases.

**In a data warehouse, keep virtual marts until performance demands otherwise.** Virtual data marts (views) give you a single source of truth and zero ETL maintenance. Physical marts (CTAS) add speed but require a refresh pipeline.
