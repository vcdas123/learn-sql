## 1. Introduction

Window aggregate functions in SQL allow you to perform calculations across a set of rows **while still preserving individual row-level details**.

Unlike traditional aggregate functions (e.g., GROUP BY), window functions:

* Do NOT collapse rows
* Return a value for each row
* Enable powerful analytical queries

---

## 2. What are Window Aggregate Functions?

Window aggregate functions perform aggregation over a "window" (subset) of rows.

### Key Idea:

* SQL scans rows
* Applies aggregation within a window
* Returns result for EACH row

### Common Functions:

* COUNT()
* SUM()
* AVG()
* MIN()
* MAX()

---

## 3. Syntax Overview

```sql
FUNCTION_NAME(expression)
OVER (
    PARTITION BY column
    ORDER BY column
    ROWS BETWEEN ...
)
```

### Notes:

* `expression` is required
* `PARTITION BY` is optional
* `ORDER BY` is optional
* Frame clause is optional

### Data Type Rules:

* COUNT → accepts all data types
* Others → require numeric values

## 5. Execution Flow of Window Functions

SQL executes window functions in the following order:

1. FROM / WHERE
2. GROUP BY (if present)
3. SELECT (window functions executed here)
4. ORDER BY

### Internal Steps:

1. Partition data (if PARTITION BY used)
2. Order rows (if ORDER BY used)
3. Define window frame
4. Apply aggregation per row

## 6. Window Function: COUNT()

### Definition

Returns number of rows within a window.

### Example: Count rows per product

```sql
SELECT
    product,
    COUNT(*) OVER (PARTITION BY product) AS total_orders
FROM orders;
```

### Output

| product | total_orders |
| ------- | ------------ |
| Caps    | 3            |
| Gloves  | 3            |

---

### COUNT(*) vs COUNT(column)

```sql
-- Counts all rows (includes NULLs)
COUNT(*)

-- Counts non-null values
COUNT(sales)
```

### Example

```sql
SELECT
    product,
    COUNT(sales) OVER (PARTITION BY product) AS sales_count
FROM orders;
```

* NULL values are ignored

---

### Special Case

```sql
COUNT(1) == COUNT(*)
```

No significant performance difference.

---

### Use Cases

* Total records
* Per-category counts
* NULL detection
* Duplicate detection

## 7. Window Function: SUM()

### Definition

Returns total sum within a window.

### Example

```sql
SELECT
    product,
    sales,
    SUM(sales) OVER (PARTITION BY product) AS total_sales
FROM orders;
```

### Behavior

* Ignores NULL values (by default)
* Works only on numeric data

#### Including NULL Values in Calculations

By default, SQL aggregate functions like SUM ignore NULL values. However, in many business scenarios, NULL may represent a meaningful value (e.g., 0 sales).

To include NULLs in calculations, you must explicitly handle them using functions like `COALESCE()`.

```sql
SELECT
    product,
    SUM(COALESCE(sales, 0)) OVER (PARTITION BY product) AS total_sales_including_nulls
FROM orders;
```

### Explanation:

* `COALESCE(sales, 0)` replaces NULL with 0
* Ensures NULL values are included in aggregation

### When to Use:

* When NULL means "no value" (e.g., 0 sales)
* When business logic requires counting missing values

⚠️ Important:
Always confirm the business meaning of NULL before replacing it, otherwise you may produce incorrect analytics results.

---

### Overall Total

```sql
SELECT
    order_id,
    SUM(sales) OVER () AS total_sales
FROM orders;
```

---

### Use Case: Percentage Contribution

```sql
SELECT
    order_id,
    sales,
    ROUND(
        CAST(sales AS FLOAT) /
        SUM(sales) OVER () * 100,
        2
    ) AS percentage
FROM orders;
```

## 8. Window Function: AVG()

### Definition

Returns average value within a window.

### Example

```sql
SELECT
    product,
    AVG(sales) OVER (PARTITION BY product) AS avg_sales
FROM orders;
```

---

### Handling NULLs

By default:

* NULLs are ignored

To treat NULL as 0:

```sql
SELECT
    AVG(COALESCE(sales, 0)) OVER () AS avg_sales
FROM orders;
```

---

### Use Case: Above Average

```sql
SELECT *
FROM (
    SELECT
        order_id,
        sales,
        AVG(sales) OVER () AS avg_sales
    FROM orders
) t
WHERE sales > avg_sales;
```

---

## 9. Window Functions: MIN() & MAX()

### Definition

* MIN → lowest value
* MAX → highest value

---

### Example

```sql
SELECT
    product,
    MIN(sales) OVER (PARTITION BY product) AS min_sales,
    MAX(sales) OVER (PARTITION BY product) AS max_sales
FROM orders;
```

---

### Use Case: Find Top Performers

```sql
SELECT *
FROM (
    SELECT
        order_id,
        sales,
        MAX(sales) OVER () AS max_sales
    FROM orders
) t
WHERE sales = max_sales;
```

---

### Deviation Analysis

```sql
SELECT
    order_id,
    sales,
    sales - MIN(sales) OVER () AS deviation_from_min,
    MAX(sales) OVER () - sales AS deviation_from_max
FROM orders;
```

## 10. Running Total vs Rolling Total

### Running Total

```sql
SELECT
    order_date,
    SUM(sales) OVER (ORDER BY order_date) AS running_total
FROM orders;
```

### Behavior

* Aggregates from beginning → current row
* Uses default frame:

```sql
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

---

### Rolling Total (Fixed Window)

```sql
SELECT
    order_date,
    SUM(sales) OVER (
        ORDER BY order_date
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS rolling_total
FROM orders;
```

### Behavior

* Fixed window size
* Old rows drop out

---

## 11. Moving Average

### Definition

Running average over time.

```sql
SELECT
    order_date,
    AVG(sales) OVER (
        PARTITION BY product
        ORDER BY order_date
    ) AS moving_avg
FROM orders;
```

---

### Rolling Average (Next Row Only)

```sql
SELECT
    order_date,
    AVG(sales) OVER (
        PARTITION BY product
        ORDER BY order_date
        ROWS BETWEEN CURRENT ROW AND 1 FOLLOWING
    ) AS rolling_avg
FROM orders;
```

## 12. Common Mistakes

1. Using window functions in WHERE clause
   ❌ Invalid

   ```sql
   WHERE sales > AVG(sales) OVER ()
   ```

   ✅ Use subquery

2. Ignoring NULL handling

3. Using wrong COUNT variant

4. Forgetting ORDER BY for running calculations

5. Confusing GROUP BY with window functions

## 13. Performance Notes

* Window functions are computed after joins/filtering
* PARTITION BY increases computation cost
* ORDER BY is expensive (sorting)

### Optimization Tips

* Index partition/order columns
* Avoid unnecessary partitions
* Use smaller datasets when possible
