## 🔹 JOIN

- Combines **columns** (side-by-side)
- Requires a **key column**
- Expands table width

```
Table A columns + Table B columns → Wider table
```

## 🔹 SET OPERATORS

- Combine **rows** (one under another)
- No key required
- Requires same structure

```
Query 1 rows
Query 2 rows
----------------
One long table
```

📌 Decision Rule:

If combining columns → use JOIN
If combining rows → use SET operators

---

# 3️⃣ Basic Syntax

```sql
SELECT col1, col2 FROM table1
SET_OPERATOR
SELECT col1, col2 FROM table2;
```

Where SET_OPERATOR can be:

- UNION
- UNION ALL
- EXCEPT
- INTERSECT

---

# 4️⃣ The 6 Rules of Set Operators

## ✅ Rule 1: ORDER BY

- Can be used only once
- Must be at the end of the entire query

❌ Not allowed inside individual SELECTs

---

## ✅ Rule 2: Same Number of Columns

Both queries must return the same number of columns.

✔ Valid

```sql
SELECT first_name, last_name FROM customers
UNION
SELECT first_name, last_name FROM employees;
```

❌ Invalid

```sql
SELECT id, first_name, last_name FROM customers
UNION
SELECT first_name, last_name FROM employees;
```

---

## ✅ Rule 3: Matching Data Types

Corresponding columns must have compatible data types.

If first query has INT and second has VARCHAR → Error.

---

## ✅ Rule 4: Same Column Order

SQL matches columns by position, NOT by name.

First column maps to first column.

❌ Wrong mapping:

```sql
SELECT last_name, id FROM customers
UNION
SELECT employee_id, last_name FROM employees;
```

---

## ✅ Rule 5: First Query Controls Output

The first SELECT determines:

- Column names
- Aliases
- Data types

```sql
SELECT customer_id AS id FROM customers
UNION
SELECT employee_id FROM employees;
```

Output column name = id

---

## ✅ Rule 6: Logical Mapping Is Your Responsibility

Even if SQL runs without errors, data can be logically wrong.

Always ensure:

- first_name maps to first_name
- last_name maps to last_name

SQL checks structure — not meaning.

---

# 5️⃣ UNION

## Definition

Returns all distinct rows from both queries.

- Removes duplicates
- Ensures unique rows

```sql
SELECT first_name FROM customers
UNION
SELECT first_name FROM employees;
```

### Behavior

1. Takes column names from first query
2. Combines rows
3. Removes duplicates

### Performance

Slower than UNION ALL (due to duplicate removal)

---

# 6️⃣ UNION ALL

## Definition

Returns all rows from both queries.

- Does NOT remove duplicates
- Fastest set operator

```sql
SELECT first_name FROM customers
UNION ALL
SELECT first_name FROM employees;
```

### When to Use

- When duplicates are acceptable
- When performance matters
- When checking data quality

---

# 7️⃣ EXCEPT

## Definition

Returns rows from first query that are NOT in second query.

Order matters.

```sql
SELECT first_name FROM employees
EXCEPT
SELECT first_name FROM customers;
```

Returns employees who are NOT customers.

### Key Points

- Order of queries matters
- Removes duplicates
- Only returns data from first query

---

# 8️⃣ INTERSECT

## Definition

Returns only common rows between both queries.

```sql
SELECT first_name FROM employees
INTERSECT
SELECT first_name FROM customers;
```

Returns employees who are also customers.

### Key Points

- Order does NOT affect result
- Removes duplicates
- Similar to INNER JOIN (but works on rows)

---

# 9️⃣ Execution Behavior (Behind the Scenes)

## UNION

- Append rows
- Remove duplicates

## UNION ALL

- Append rows
- No checks

## EXCEPT

- Compare first set against second
- Keep only non-matching rows

## INTERSECT

- Keep only matching rows

# 🔟 Real-World Use Cases

---

## ✅ Use Case 1: Combine Similar Tables

Example:

- employees
- customers
- suppliers
- students

Instead of analyzing separately:

```sql
SELECT ... FROM employees
UNION ALL
SELECT ... FROM customers
UNION ALL
SELECT ... FROM suppliers
```

Create one unified dataset (persons) and analyze once.

Benefits:

- Cleaner logic
- Easier maintenance
- Consistent reporting

---

## ✅ Use Case 2: Combine Partitioned Tables

Example:

- orders_2022
- orders_2023
- orders_2024

```sql
SELECT * FROM orders_2022
UNION ALL
SELECT * FROM orders_2023
UNION ALL
SELECT * FROM orders_2024;
```

Create unified reporting dataset.

---

## ✅ Use Case 3: Delta Detection (Data Engineering)

Find new records:

```sql
SELECT * FROM day2
EXCEPT
SELECT * FROM day1;
```

Returns only new rows.

Used in ETL pipelines.

---

## ✅ Use Case 4: Data Completeness Check

Verify migration:

```sql
SELECT * FROM dbA.table
EXCEPT
SELECT * FROM dbB.table;
```

Then reverse:

```sql
SELECT * FROM dbB.table
EXCEPT
SELECT * FROM dbA.table;
```

If both return empty → tables identical.

---

# 🔟+1 Best Practices

✔ Avoid SELECT \* in production
✔ Always list columns explicitly
✔ Ensure correct column order
✔ Add source column when combining tables
✔ Prefer UNION ALL when duplicates impossible
✔ Use EXCEPT for validation checks

---

# 🎯 Summary

Set Operators combine rows from multiple queries.

| Operator  | Removes Duplicates | Order Matters | Purpose               |
| --------- | ------------------ | ------------- | --------------------- |
| UNION     | Yes                | No            | Combine distinct rows |
| UNION ALL | No                 | No            | Combine all rows      |
| EXCEPT    | Yes                | Yes           | First minus second    |
| INTERSECT | Yes                | No            | Common rows           |

---

# Core Philosophy

JOIN → Combine columns
SET Operators → Combine rows

Use them to reshape data before analysis, validation, and reporting.
